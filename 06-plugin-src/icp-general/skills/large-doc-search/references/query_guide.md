## 1. 定位知識庫

- 用 Glob 搜尋 `**/index.json`（maxdepth 3）
- 如果有多個 KB，列出所有可用的 KB：`kb_name` + `total_pages` + `total_chapters`
- 使用者提到的品牌、產品名稱、型號 → 用 `kb_name` / `source_file` 匹配
- 使用者說「那個西門子的手冊」→ 找 kb_name 含 "Siemens" 或 "SIP" 的 KB

## 2. 查詢類型分流

收到問題後，先判斷類型再選策略。不要每次都走相同的流程。

| 問題特徵 | 類型 | 策略 |
|---------|------|------|
| 含數值、單位、「多少」「範圍」 | 規格查詢 | 表格優先（§3） |
| 含「比較」「差異」「vs」或兩個品牌 | 跨 KB 比較 | 比較協議（§4） |
| 含「有沒有」「哪裡提到」 | 存在性查詢 | Grep 快篩（§5） |
| 問原理、流程、「怎麼運作」 | 概念查詢 | 傳統流程（讀 index → 載入章節） |
| 問「在第幾章」「目錄」 | 定位查詢 | 直接從 index 回答 |

## 3. 表格優先策略

當問題涉及數值規格時（溫度、電壓、精度、設定範圍等），表格通常比章節正文更精準。

**流程**：
1. 用 Glob 列出 `tables/*.md` 所有檔案
2. 從檔名判斷哪些表格可能相關（如 `tbl05_technical_specifications.md`）
3. 同時檢查 index 中各章節的 `related_tables` 欄位
4. 直接載入相關表格（每個通常 <2K chars，非常便宜）
5. 如果表格能完整回答 → 直接回答，不用載入整章
6. 如果表格不夠 → 再載入對應章節正文補充

**為什麼這樣做更好**：載入一個表格 (~1K chars) 比載入整章 (~20K chars) 省 95% context，而且表格的數值通常比散文中的更準確。

## 4. 跨 KB 比較協議

當使用者要求比較兩個產品/品牌/方案時，需要結構化的比較流程。

**流程**：

### Step 1：讀取雙方 index
同時讀取兩個 KB 的 index.json，了解各自的章節結構。

### Step 2：建立功能維度對照表
將雙方章節按功能域配對。例如：

| 功能維度 | KB-A 章節 | KB-B 章節 |
|---------|----------|----------|
| 產品架構 | ch01 Overview | ch01 Introduction |
| 技術規格 | ch14 Technical Data | ch05 Specifications |
| 補償機制 | ch05 Compensation | ch04 Compensation |

### Step 3：逐維度載入與比較
- 每次只處理一個維度，載入雙方對應的章節（或表格）
- 如果某個維度在某一方 KB 中沒有對應章節，明確標記「未涵蓋」
- 提取具體參數值做並列比較，不是只寫「有支援」

### Step 4：產出結構化比較
- 每個維度一張比較表
- 每格標註來源：`{KB名} {章節名} (p.XXX)`
- 結尾附上「關鍵差異總結」，列出各方的獨特優勢

### 比較表格式範例

```markdown
| 參數 | 產品 A | 產品 B |
|------|-------|-------|
| 命令精度 | 50 μs (ch12, p.147) | ±10 μs (ch05, p.39) |
| 溫度範圍 | -5~+55°C (ch11, p.141) | -50~+70°C (ch05, p.25) |
```

## 5. Grep 快篩策略

當使用者問「有沒有提到 X」這類存在性問題時，用 Grep 先確認再載入。

**流程**：
1. 用 Grep 搜尋 `chapters/` 和 `tables/` 目錄，output_mode="count"
2. 如果匹配次數 = 0 → 直接回答「未提及」
3. 如果匹配次數 > 0 → 按匹配次數排序，載入最相關的 1-2 個章節
4. 回答時標註：「在 {章節名} 中提及 X 次」

**為什麼比傳統方式好**：避免靠「語意猜測」載入不相關的大章節。Grep 的結果是確定性的——有就是有，沒有就是沒有。

## 6. 載入控制

### 額度管理
- **單次總載入上限**：120,000 字元（約 30K tokens）
- **表格不計入主要額度**：表格通常 <2K chars，可以多載
- **優先小章節**：同等相關度時，先載 char_count 較小的

### 超過上限時
1. 先回答已載入部分
2. 列出未載入但可能相關的章節
3. 問使用者：「要我繼續查看 {章節名} 嗎？」

## 7. 回答格式

### 來源標註（必須）
每段回答都要標註來源，格式：

> 根據 {KB名} 第 X 章 *{章節名}*（p.XXX-YYY），...

### 數值規格
- 用表格呈現，方便比對
- 標註原始頁碼和表格編號

### 找不到答案
- 明確說：「在 {KB名} 知識庫中未找到關於 XXX 的資訊」
- 不猜測、不編造
- 如果可能在其他 KB 中，建議使用者

### 跨 KB 回答
標註每段資訊的出處 KB：

> 根據 **SIP5_PoW_Manual** ch12 (p.147)，命令精度為 50 μs；
> 而 **SynchroTeq_Datasheet** ch05 (p.39) 標示為 ±10 μs。

## 8. 多知識庫場景

1. **自動選擇**：問題提到品牌/產品 → 匹配 `kb_name`
2. **列出選項**：不確定時列出所有 KB 讓使用者選
3. **跨庫查詢**：問題涉及多個 KB 時，走比較協議（§4）
4. **同品牌多 KB**：如 Siemens 有 SIP5_PoW_Manual、SIPROTEC5_Catalog 等多個 KB，根據問題性質選最相關的那個（POW 功能問 PoW_Manual，通訊問 7SJ82-85_Manual）
