## 第一部分　部門整體治理定位

### 一、部門存在必要性

系統設計部門之設立，並非基於技術產出的效率考量，而是基於三類財務風險之前置控制需求。

第一類為「標準缺位風險」。當組織缺乏公司級的設計標準擁有單位，各專案團隊將依據各自理解執行設計，其結果是品質基線無法統一，設計偏差只能在交付階段或客戶端才被發現。此類風險直接轉化為返工成本與客戶索賠，且事後補救所需資源通常為設計階段前置控制成本的三至五倍。

第二類為「責任歸屬模糊風險」。在缺乏治理框架的組織中，設計責任與執行責任之間沒有明確的分界點。當問題發生時，責任追溯無從著手。對財務而言，這意味著風險無法量化、保險無法精算、損失無法歸責。

第三類為「合規暴露風險」。系統設計部門所擁有之 IEC 62443 合規方法與安全工程標準，是組織通過外部稽核與驗證的基礎。若此能力不存在於一個有標準擁有權的單位中，合規將成為每個專案各自應對的負擔，整體合規成本將顯著上升，且稽核一致性無法保障。

### 二、治理機制核心原則

本部門之治理架構建立在三項原則之上。

其一，標準擁有權制度。組織內每一項設計標準皆指定明確的角色擁有者，該角色對標準的完整性、可執行性與時效性負最終責任。標準非建議性文件，而是強制遵循事項。受治理單位須遵循標準，或透過正式例外程序申請偏離並留存紀錄。

其二，Gate 控制邏輯。設計流程透過四道 Gate（Gate 0 至 Gate 3）實施階段性品質攔截。Gate 0 控制需求受理，Gate 1 確認設計基線，Gate 2 管控重大設計變更，Gate 3 驗證交付完整性。每道 Gate 皆有指定的核准者與必要輸入文件，且審查紀錄須完整保存供稽核。

其三，責任封裝模型。設計責任與執行責任以 Gate 3 為分界，設計責任在 Gate 3 交接後正式移轉至執行單位。部門內部各角色之間亦有明確的責任邊界，不存在責任重疊或責任真空。此封裝確保每一項失敗皆可追溯至特定角色與特定階段。

### 三、對 EBITDA 的間接影響

本部門並非直接產生營收的單位，但其對 EBITDA 的間接影響體現在兩個維度。首先是成本波動控制：透過設計階段的標準化與 Gate 攔截，將設計缺陷發現的時點從執行階段前移至設計階段，從而降低返工、變更與索賠所導致的成本波動。其次是決策前置化：透過 Pre-Gate 0 的需求技術釐清與風險預揭露，使商務承諾建立在經過技術評估的基礎上，而非建立在未經驗證的假設上。此二者共同作用，使設計相關成本從不可預測的變動支出，轉化為可規劃的固定治理投入。

## 第二部分　部門角色 JD 核心摘要

### Head of System Design

本角色為部門最高權責者，擁有標準體系的整體擁有權。其核心定位為標準方向決策者與例外裁決的最終權責。本角色負責 Gate 0（需求受理）與 Gate 2（重大變更）之核准決策，以及所有 Level 3 爭議之最終裁決。其責任邊界在於標準治理的制度層面，不涉入標準的技術內容細節。本角色不可替代性在於：組織內必須存在一個對標準體系整體一致性負最終責任的角色，否則標準之間的衝突將無人仲裁，例外偏離將無人裁決。若此角色失效，標準體系將喪失更新動力與一致性保障，治理將退化為無約束狀態。

### System Design Governance Lead

本角色為治理框架標準之擁有者，負責確保治理原則、Gate 機制與合規要求成為全公司設計的強制遵循標準。其核心職責為 Gate 1 與 Gate 3 之核准決策、標準合規驗證、Level 2 爭議仲裁，以及設計交接之正式執行。其責任邊界在於治理程序的正確執行，不涉及設計內容的技術判斷。此角色的不可替代性在於：Gate 機制的有效性依賴一個獨立於設計執行者的審查權責。若此角色失效，Gate 審查將流於形式，合規驗證將失去一致性，設計交接將缺乏正式紀錄。

### System Architect

本角色為架構設計標準與技術基線標準之擁有者，負責全公司系統設計之技術一致性基礎。其核心當責涵蓋設計基線文件集之完整性、技術可行性評估、設計變更影響分析，以及 Gate 3 最終設計文件包之編製。其責任邊界在於設計階段的技術正確性，Gate 3 之後的執行結果不在其責任範圍。此角色的不可替代性在於：設計基線若無明確的當責擁有者，設計版本將失去控制，設計變更將無人評估影響。若此角色失效，組織將無法產出可稽核的設計基線，Gate 1 至 Gate 3 的品質攔截將喪失技術基礎。

### Security Engineering Role

本角色為設計階段安全工程標準與風險評估標準之擁有者，負責 IEC 62443 合規方法的定義與執行、威脅分析的完整性，以及安全等級（SL）判定的全生命週期追蹤。其責任邊界嚴格限於設計階段，不涵蓋實作階段的安全測試與滲透測試。此角色的不可替代性在於：IEC 62443 合規是許多專案的合約義務或市場准入條件。若此角色失效，殘餘風險將無法正式追溯至分析來源，安全合規將成為不可稽核的宣稱，組織將面臨合規暴露。

### Design QA Role

本角色為設計品質標準與追溯性標準之擁有者，負責設計文件完整性驗證、需求追溯矩陣建立與維護、Gate 3 交付檢查表之執行，以及殘餘風險的追溯性抽查。其責任邊界在於設計產出的品質驗證，不涉及生產階段的品質檢驗。此角色的不可替代性在於：設計文件若無獨立的品質驗證機制，文件版本不一致、追溯斷鏈等問題將只能在交付後才被發現。若此角色失效，Gate 3 交付檢查將失去獨立性，設計文件的可信度將無法保障。

### Design Governance Coordinator

本角色為治理流程執行與文件管理之協調者，確保 Gate 流程順利執行、會議紀錄完整、文件版本控制有序。本角色不擁有標準，但負責治理機制的日常運作支撐。其核心當責為 Gate 會議之安排與紀錄、待辦事項追蹤，以及治理文件之版本控制維護。其責任邊界在於流程協調與紀錄完整性，不涉及任何核准或技術決策。此角色的不可替代性在於：Gate 機制若無專職的流程協調者，會議紀錄將不完整，版本控制將失序，待辦事項將遺失追蹤，治理機制的可稽核性將受到損害。

### Pre-Gate Design Support（Concept System Architect / Feasibility Owner）

本角色為 Gate 0 階段之正式技術執行角色，非商務角色。其核心當責為產出 Gate 0 所需之全部輸入文件，包含需求釐清紀錄、風險預揭露清單、技術可行性初評、Commercial BOM（CBOM）及概念系統架構圖（Concept Architecture）。CBOM 標註為「商務可用、設計不拘束」，可供報價使用但不構成設計基線。Gate 0 核准後，設計責任正式移轉至 System Architect，本角色之概念產出僅供參考。其責任邊界在於 Gate 0 輸入品質，不承擔 Gate 0 核准決策責任，亦不承擔 Gate 1 之後的設計責任。此角色的不可替代性在於：若 Gate 0 輸入文件品質無專責角色把關，商務承諾將建立在未經技術評估的基礎上，導致後續設計返工與成本波動。

### 角色於設計流程中的定位

設計流程自需求受理至設計交付，依序經過 Pre-Gate 0、Gate 0、Gate 1、Gate 2 及 Gate 3 五個階段。各角色於流程中的定位如下表所示。

| Gate 階段 | 核准者 | 當責執行角色 | 關鍵產出 |
|----------|--------|-------------|---------|
| Pre-Gate 0 | 無正式核准 | Pre-Gate Design Support | 需求釐清紀錄、風險預揭露清單、CBOM、概念架構圖 |
| Gate 0（需求受理） | Head of System Design | Pre-Gate Design Support（輸入文件）、System Architect（可行性審核）、Security Engineering Role（SL 提案） | Gate 0 核准紀錄、目標 SL 提案、責任移轉紀錄 |
| Gate 1（設計基線確認） | System Design Governance Lead | System Architect（設計基線與 EBOM）、Security Engineering Role（風險評估）、Design QA Role（追溯矩陣） | 設計基線文件集、EBOM、整合式風險評估報告、SR 檢查表、需求追溯矩陣 |
| Gate 2（設計變更管控） | Head of System Design（重大）/ System Architect（次要） | System Architect（變更紀錄）、Security Engineering Role（影響分析） | 設計變更紀錄、影響分析文件、更新之 SL Decision Record |
| Gate 3（交付驗證） | System Design Governance Lead | System Architect（最終文件包）、Security Engineering Role（殘餘風險清單）、Design QA Role（交付檢查表） | 最終設計文件包、殘餘風險清單、設計交付檢查表、交接會議紀錄 |

Design Governance Coordinator 於所有 Gate 階段擔任流程協調與會議紀錄之職責，確保每道 Gate 之會議安排、紀錄保存與待辦事項追蹤皆完整執行。

責任移轉的關鍵節點在於 Gate 0 與 Gate 3。Gate 0 核准後，設計責任自 Pre-Gate Design Support 正式移轉至 System Architect。Gate 3 核准後，設計責任自系統設計部門正式移轉至執行單位。兩次移轉皆須書面紀錄並經正式簽核。

## 第三部分　KPI 架構與可稽核模型

### 一、KPI 設計原則

本體系之 KPI 設計遵循一項根本原則：KPI 僅驗證治理是否存在，不評估努力與情緒性指標。

具體而言，KPI 從角色的「當責」推導而非從「活動」推導。評估的是標準是否存在、紀錄是否完整、審查是否執行、追溯是否可行。評估的不是投入多少時間、參加多少會議、產出多少文件數量。這意味著 KPI 不設「Gate 通過率」（避免鼓勵放水）、不設「文件數量」（避免鼓勵膨脹）、不設「時效 KPI」（避免鼓勵草率），也明確禁止以商務結果（如成交率）作為技術角色的績效指標。

每一項 KPI 皆強制對應至少一項可審查、可稽核、可重現的證據來源，證據須為角色可控或可取得之文件與紀錄。

### 二、責任可觀測化機制

以財務語言描述，KPI 體系的核心價值在於可追溯性（traceability）。當組織面臨客戶索賠、外部稽核或內部爭議時，可追溯性決定了組織能否將損失歸屬至正確的責任階段與責任角色。

本體系確保：每一項標準皆有擁有角色、每一道 Gate 皆有核准紀錄、每一項設計變更皆有影響分析、每一項殘餘風險皆可追溯至分析來源。此結構使「責任」從抽象的組織宣稱，轉化為可在稽核場景中被驗證的具體紀錄鏈。

### 三、年度治理循環

治理體系透過曆年制的年度週期運作。每年 12 月完成角色指派確認、標準年度審查與下年度目標設定。年度內以季為單位執行治理活動，Q2 末進行半年審查，Q4 末完成證據蒐集與數據凍結。次年 1 月完成 KPI 計算、聚合與結算，次年 2 月完成獎酬分配。

此週期的治理意義在於：標準體系每年至少經歷一次完整審查，任何標準的時效性、可執行性問題將在年度審查中被識別。制度的穩定性來自於週期的可預測性，而制度的進化能力來自於年度審查的閉環設計。

|-----|------|---------|
| Gate 0 輸入文件完整率 | 25 | 100% → 25 分；任何文件缺失 → 0 分 |
| 需求釐清採用率 | 20 | 100% → 20 分；80-99% → 16 分；60-79% → 10 分；< 60% → 0 分 |
| 返工降低達成率 | 18 | 100% → 18 分；90-99% → 14 分；80-89% → 8 分；< 80% → 0 分 |
| 風險預揭露有效率 | 17 | 100% → 17 分；80-99% → 14 分；60-79% → 8 分；< 60% → 0 分 |
| 概念產出標註合規率 | 12 | 100% → 12 分；任何標註缺失 → 0 分 |
| Gate 0 責任移轉完成率 | 8 | 100% → 8 分；任何延遲 → 0 分 |

本角色之 KPI 分數具備後向驗證機制。返工降低達成率須待 Gate 1 後 30 日方可確認，風險預揭露有效率須待 Gate 3 完成後方可確認。未驗證前，返工降低達成率暫計滿分，風險預揭露有效率暫計 80% 基準。

#### 分數詮釋基準

| 分數區間 | 詮釋 |
|---------|------|
| 95 至 100 分 | 完全達成當責要求 |
| 85 至 94 分 | 基本達成，有改善空間 |
| 75 至 84 分 | 部分達成，需檢討 |
| 低於 75 分 | 未達最低要求，需改善計畫 |
| 100 分以上（含例外加分） | 超越基本當責，展現治理領導力 |

基本分低於 75 分者，例外加分不予採計。例外加分不得使未達最低門檻之 KPI 視為達成。加分紀錄獨立呈現，不掩蓋基本分缺失。

## 第四部分　多角色公平與風險保護

本組織允許一人擔任多個功能性角色。此配置在小型專業團隊中具有現實必要性，但同時產生兩項公平性問題：多角色擔任者是否在評估結構上處於劣勢？以及，多角色擔任者是否可透過角色膨脹獲取不當利益？

針對第一項問題，體系設置了 Role Load Stabilizer（RLS）。其目的為移除多角色評估面向增加所帶來的結構性波動劣勢。RLS 並非獎勵，而是穩定器，其調整幅度受到嚴格上限控制，不會因角色數量無限增加而無限膨脹。

針對第二項問題，體系設置了 Multi-Role Excellence Multiplier（MREM）。其設計邏輯為：多角色皆維持高品質表現，代表更高的實質貢獻，應被合法量化。但 MREM 採用全有或全無機制，啟動條件極為嚴格，要求每個角色皆達到高品質門檻，且未觸發任何治理失敗。角色數量本身不構成 MREM 的啟動條件，品質一致性才是。

此外，體系設置了 Critical Role Failure Protection。此機制確保任何角色的重大失敗不會被其他角色的高表現掩蓋。當任一角色出現關鍵失敗時，個人最終分數將被強制壓低，且治理失敗標記將獨立於分數存在，不可被任何乘數覆蓋。此設計的治理意義在於：組織寧可犧牲整體分數的膨脹空間，也要確保關鍵角色的失敗被正式揭露。

## 第五部分　分數使用邊界與管理裁量

### 合法使用範圍

績效分數可用於以下場景：獎金池分配上限之設定、晉升資格之必要條件判定、發展資源之優先順序參考，以及績效面談之事實依據。所有使用皆須遵守貢獻對獎酬映射表之規則式對應，分數對應層級為公開資訊。

管理層在獎酬分配上享有的裁量空間，限於映射表所設定的上限之內。管理層可決定實際給付金額（不超過上限）、可決定晉升人選（於合格者中），但不得調整分數計算本身，不得突破上限，不得跳過必要程序。

### 不可逾越的治理邊界

績效分數不得作為基本薪資調整的依據、不得作為解僱決策的依據、不得作為懲處決策的依據、不得用於人格評價、不得用於公開排名、不得用於跨部門比較、不得跨評估週期累積或抵銷。

分數計算規則、治理保護機制之觸發、CRM 層級對應、上限設定，皆屬強制性規則，管理層對此不具裁量權。裁量權的行使須遵循一致性、透明性、可預測性與合理性原則，且重要裁量須留存紀錄。

分數不可成為政治工具。當分數被用於制度設計以外的目的時，其客觀性即遭到破壞，所有依賴此客觀性的治理機制將連帶失效。制度對此設有稽核追蹤與申訴機制，裁量權濫用屬可申訴事項。

## 結論

本制度的本質，是將技術責任轉化為可控制財務風險的治理架構。標準擁有權確保責任可歸屬，Gate 機制確保品質可攔截，KPI 確保當責可觀測，公平機制確保制度可持續，使用規則確保分數不被異化。此架構的價值不在於其評估結果，而在於其存在本身所提供的制度性風險控制能力。
