## 公司級技術權責單位之定位

### 使命聲明

系統設計部門之存在目的，在於作為組織之**公司級技術權責單位（Company-Level Technical Authority）**，負責：

1. **定義並擁有**全公司系統設計之標準框架、標準流程及標準工具
2. **建立並維護**系統設計之技術基線，作為所有專案與業務單位之共同遵循標準
3. **守護**所有對外交付服務之技術一致性與設計品質

本部門**非**交付單位，**非**執行單位，**非**諮詢單位。本部門之產出為**強制性標準與設計時控制機制**，非建議性指引。

### 組織定位特性

| 定位維度 | 說明 |
|---------|------|
| **組織層級** | 公司級（Company-Level），非專案級或業務單位級 |
| **運作模式** | 橫向跨專案、跨業務單位運作 |
| **權限範圍** | 設計時（Design-Time）權限，非執行時（Execution-Time）權限 |
| **產出性質** | 標準、框架、範本、決策機制；非實作產出 |
| **對外關係** | 賦能（Enablement）而非代勞（Execution-on-behalf） |

### 標準權威性聲明

本部門所定義之標準具備以下權威性：

1. **強制性**：本部門產出之標準為強制遵循事項，非參考性建議
2. **合規義務**：所有受治理之專案與單位須遵循本部門標準，或透過正式例外程序申請偏離
3. **設計時控制**：本部門透過標準化與設計時控制保護對外服務品質，而非透過運行時介入
4. **一致性基線**：本部門標準構成全公司系統設計之技術一致性基線

## 公司級定位與績效治理設計之關聯

### 為何績效指標須從公司級定位推導

本部門作為公司級技術權責單位，其績效指標之設計須體現以下邏輯關係：

```
公司級使命 → 標準擁有權當責 → 角色當責分配 → 可觀測 KPI → 客觀評估
```

| 層級 | 內涵 | 績效治理對應 |
|-----|------|-------------|
| 公司級使命 | 定義標準、維護基線、守護品質 | 績效評估須反映使命達成程度 |
| 標準擁有權當責 | 各標準皆有擁有角色，負最終責任 | KPI 須從當責推導，非從活動推導 |
| 角色當責分配 | 每角色對特定標準負當責 | 每角色之 KPI 反映其當責範圍 |
| 可觀測 KPI | 客觀、可稽核、可重現 | 避免主觀評量、避免努力導向 |

### 績效指標設計之治理意圖

本部門績效指標之設計，非以激勵最大化為目標，而是以**治理可觀測性（Governance Observability）**為核心：

| 設計意圖 | 說明 |
|---------|------|
| 驗證當責履行 | 確認角色確實履行其標準擁有權責 |
| 偵測治理失效 | 及早發現標準失效、流程崩壞、責任真空 |
| 支持決策回溯 | 提供可追溯之決策紀錄供稽核與改善 |
| 維護公平性 | 避免主觀偏見，確保評估之一致性 |

### KPI 與公司級使命之追溯矩陣

| 公司級使命 | 對應標準類別 | 擁有角色 | 衍生 KPI 類別 |
|-----------|-------------|---------|--------------|
| 定義標準框架 | 治理框架標準 | System Design Governance Lead | 框架時效性、審查完整性 |
| 維護技術基線 | 技術基線標準 | System Architect | 基線時效性、可行性評估覆蓋度 |
| **確保 Gate 0 輸入品質** | **Pre-Gate 0 需求釐清** | **Pre-Gate Design Support** | **需求釐清採用率、返工降低率、風險預揭露品質** |
| 守護設計品質 | 設計品質標準 | Design QA Role | 文件完整性、追溯性、版本一致性 |
| 確保安全合規 | 安全工程標準 | Security Engineering Role | SR 可追溯率、威脅分析完整性 |
| 標準體系治理 | 例外管理標準 | Head of System Design | 例外紀錄完整性、決策可追溯性 |

## 部門核心職能

本部門整合以下職能於單一當責邊界內：

| 職能 | 部門內角色 | 標準擁有權 |
|-----|----------|-----------|
| 設計治理 (Design Governance) | System Design Governance Lead | 治理框架、Gate 機制、合規驗證標準 |
| 系統架構 (System Architecture) | System Architect | 架構設計標準、技術基線、設計文件標準 |
| **概念設計與可行性 (Concept Design & Feasibility)** | **Pre-Gate Design Support** | **無正式標準擁有權（Gate 0 執行角色）** |
| 安全工程 (Security Engineering) | Security Engineering Role | 設計階段安全標準、IEC 62443 對應方法、風險評估標準 |
| 設計品質保證 (Design QA) | Design QA Role | 文件品質標準、追溯性標準、符合性驗證標準 |
| 設計工程管理 (Design Engineering Management) | Head of System Design | 標準擁有權裁決、例外管理、技術方向決策 |

## 標準遵循與偏離機制

所有受本部門治理之專案與單位，對於本部門標準須採取以下立場之一：

| 立場 | 說明 | 程序 |
|-----|------|-----|
| **完全遵循** | 依標準執行，無偏離 | 無須額外程序 |
| **核准偏離** | 因正當理由無法完全遵循，經核准偏離 | 須提出例外申請，經 Head of System Design 核准，紀錄於例外清冊 |
| **未經核准偏離** | 未遵循標準且未取得例外核准 | Gate 審查不予通過；已通過之 Gate 須重新評估 |

**違反標準之處置**：未經核准偏離本部門標準之設計，不得通過 Gate 審查。已通過 Gate 審查後發現未遵循標準之情事，須啟動設計變更流程或正式例外申請。

## 最終當責

### 標準品質當責

本部門對以下事項負最終責任：

- **標準完整性**：所有標準涵蓋其所宣稱之範圍，無遺漏
- **標準可執行性**：所有標準可被受治理單位具體執行
- **標準可稽核性**：所有標準之合規狀態可被客觀驗證

### 設計品質當責

- **設計內容正確性**：所有於本部門治理框架下產出之設計文件的準確性與完整性
- **設計架構合理性**：系統架構決策之技術適當性與可實作性
- **設計階段安全性**：設計階段所有安全要求之識別、分析與對應
- **設計文件品質**：設計文件之完整性、追溯性及版本一致性

### 治理當責

- **風險揭露完整性**：確保所有已識別風險皆經正式記錄、評估，且可追溯至其分析來源
- **合規對齊**：確保設計符合所宣告安全等級（SL）之 IEC 62443 要求
- **責任邊界執行**：維護設計責任與執行責任之間的明確分界
- **Gate 機制有效性**：確保 Gate 機制有效攔截不合格設計

### 可否證條件

若觀察到以下任一情況，即視為本部門失職：

- 本部門標準存在無法執行或無法驗證之條款
- 於設計階段依合理專業判斷標準應可識別之設計缺陷，未揭露於殘餘風險清單中
- Gate 3 交接在未完成 Gate 3 符合性檢查表所定義之完整文件的情況下執行
- 最終風險清單中之殘餘風險無法追溯至至少一個有效風險來源 ID
- IEC 62443 SR 檢查表中標註為「Implemented」之項目缺乏對應設計證據
- 系統架構決策缺乏可追溯之技術可行性評估紀錄
- 設計階段安全分析未涵蓋適用之 IEC 62443-3-2 威脅情境
- 設計文件未通過內部 Design QA 審查即進入 Gate 審核

## 核心部門能力

本部門須維持以下基本能力以履行標準擁有者之職責：

| 能力 | 說明 |
|-----|------|
| 標準制定能力 | 能夠識別標準化需求、起草標準文件、驗證標準可執行性、發布並維護標準體系 |
| **概念設計與可行性評估能力** | **能夠於 Gate 0 前執行技術需求釐清、產出概念架構與初階 BOM、識別並預揭露潛在技術風險** |
| 系統架構能力 | 能夠產出並評估系統架構設計，包含架構圖、介面定義、資料流圖，以及符合 IEC 62443-3-2 之 Zone & Conduit 圖 |
| 整合式風險評估能力 | 能夠執行並審查 IEC 62443-3-2 威脅情境分析、FMEA、HAZOP |
| IEC 62443 合規能力 | 能夠詮釋並應用 IEC 62443-2-4、IEC 62443-3-2 及 IEC 62443-3-3 於設計決策 |
| 治理流程執行能力 | 能夠管理多階段 Gate 審查流程、維護文件清冊、執行版本控制標準 |
| 跨功能協調能力 | 能夠與各相關單位介接，同時維護清晰之責任邊界 |
| 技術文件能力 | 能夠產出、審查並維護符合版本要求之設計文件集 |
| 設計品質管理能力 | 能夠執行設計文件之完整性稽核、追溯性驗證及符合性審查 |

## 假設條件

本文件之適用基於以下假設：

1. **組織授權**：組織已授權系統設計部門作為公司級技術權責單位，具備定義、發布及強制執行設計標準之權限
2. **組織架構**：「系統設計部門」係以專責功能單位運作
3. **標準強制性**：本部門所定義之標準為強制遵循事項；受治理單位須遵循標準或透過正式例外程序申請偏離
4. **文件層級**：本文件從屬於《System Design Governance》主文件（v1.2）及《跨部門專案交付 SOP》（v1.0）
