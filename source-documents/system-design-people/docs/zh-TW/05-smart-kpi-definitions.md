# 角色 SMART KPI 定義

## Role-Based SMART KPI Definitions

**文件編號：** GOV-03-001
**文件版本：** 1.0
**生效日期：** 2026-02-06
**文件層級：** 績效評估文件
**依據文件：** GOV-02-002 角色 KPI 與證據模型

---

## 文件目的

本文件將角色當責推導之 KPI 轉化為符合 SMART 原則之量化績效指標，供管理層進行客觀、一致且可重複之績效評估。

---

## SMART 原則詮釋

| 原則 | 定義 | 本文件應用 |
|-----|------|-----------|
| **S**pecific（具體） | 明確描述可觀測之條件 | 避免抽象品質描述 |
| **M**easurable（可量測） | 使用計數、比率、百分比或二元判定 | 每個 KPI 皆有量測公式 |
| **A**chievable（可達成） | 正常運作條件下可達成 | 不假設超常努力 |
| **R**elevant（相關） | 直接對應角色當責 | 每個 KPI 映射至少一項當責 |
| **T**ime-bound（有時限） | 明確量測頻率與評估週期 | 指定評估週期 |

## Head of System Design KPI

### KPI 概覽

本 KPI 組合衡量 Head of System Design 對標準體系完整性、例外治理、Gate 決策及爭議裁決之當責履行。所有 KPI 聚焦於決策可追溯性與治理完整性，而非決策內容之對錯。

### KPI 清單

#### KPI 1: 標準體系完成率

| 項目 | 內容 |
|-----|------|
| **定義** | 所有宣稱擁有之標準中，已正式定義、發布並納入版本控制之比率 |
| **公式** | `(具備完整文件之標準數 / Section 2 標準總數) × 100%` |
| **最低可接受** | ≥ 95% |
| **期望目標** | 100% |
| **評估週期** | 每季評估，年度彙總 |
| **證據來源** | 標準清冊、各標準文件元資料頁、版本控制系統紀錄 |

#### KPI 2: 標準衝突回報數

| 項目 | 內容 |
|-----|------|
| **定義** | 年度內因標準間衝突導致之正式回報或例外申請次數 |
| **公式** | `年度內標準衝突類別之回報次數 + 因標準衝突之例外申請次數` |
| **最低可接受** | ≤ 2 件/年 |
| **期望目標** | 0 件/年 |
| **評估週期** | 年度評估 |
| **證據來源** | 標準回饋紀錄、例外申請紀錄 |

#### KPI 3: 例外裁決紀錄完整率

| 項目 | 內容 |
|-----|------|
| **定義** | 所有例外裁決紀錄具備必要欄位之比率 |
| **公式** | `(必要欄位皆完整之例外紀錄數 / 例外紀錄總數) × 100%` |
| **最低可接受** | 100% |
| **期望目標** | 100% |
| **評估週期** | 每季評估 |
| **證據來源** | 例外裁決紀錄、例外清冊 |

#### KPI 4: Gate 0/Gate 2 決策紀錄覆蓋率

| 項目 | 內容 |
|-----|------|
| **定義** | 所有 Gate 0 及 Gate 2 決策具備完整核准紀錄之比率 |
| **公式** | `(具備完整紀錄之 Gate 0/2 決策數 / Gate 0/2 決策總數) × 100%` |
| **最低可接受** | 100% |
| **期望目標** | 100% |
| **評估週期** | 每季評估 |
| **證據來源** | Gate 0/Gate 2 核准紀錄、Gate 審查紀錄 |

#### KPI 5: Level 3 爭議結案率

| 項目 | 內容 |
|-----|------|
| **定義** | 所有 Level 3 爭議經正式裁決並結案之比率 |
| **公式** | `(已結案之 Level 3 爭議數 / Level 3 爭議總數) × 100%` |
| **最低可接受** | 100%（於評估週期內） |
| **期望目標** | 100% |
| **評估週期** | 每季評估 |
| **證據來源** | Level 3 爭議裁決紀錄、爭議清單 |

#### KPI 6: SL Decision Record 核准完整率

| 項目 | 內容 |
|-----|------|
| **定義** | 所有 SL Decision Record 經正式核准之比率 |
| **公式** | `(具備核准簽核之 SL Decision Record 數 / SL Decision Record 總數) × 100%` |
| **最低可接受** | 100% |
| **期望目標** | 100% |
| **評估週期** | 每 Gate 檢查，每季彙總 |
| **證據來源** | SL Decision Record、Gate 審查紀錄 |

## Design Governance Coordinator KPI

### KPI 概覽

本 KPI 組合衡量 Design Governance Coordinator 對 Gate 流程協調、會議紀錄完整性及版本控制維護之當責履行。

### KPI 清單

#### KPI 1: Gate 會議紀錄完整率

| 項目 | 內容 |
|-----|------|
| **定義** | 所有 Gate 會議具備完整紀錄之比率 |
| **公式** | `(紀錄完整之 Gate 會議數 / Gate 會議總數) × 100%` |
| **最低可接受** | 100% |
| **期望目標** | 100% |
| **評估週期** | 每 Gate 即時檢查，每季彙總 |
| **證據來源** | Gate 會議紀錄、Gate 審查紀錄 |

#### KPI 2: 待辦事項結案率

| 項目 | 內容 |
|-----|------|
| **定義** | Gate 審查待辦事項於下一 Gate 前結案或展延之比率 |
| **公式** | `(已結案或展延之待辦事項數 / 待辦事項總數) × 100%` |
| **最低可接受** | ≥ 95% |
| **期望目標** | 100% |
| **評估週期** | 每 Gate 間隔檢查，每季彙總 |
| **證據來源** | 流程追蹤表、Gate 會議紀錄 |

#### KPI 3: 治理文件版本控制覆蓋率

| 項目 | 內容 |
|-----|------|
| **定義** | 所有治理文件納入版本控制之比率 |
| **公式** | `(納入版本控制之治理文件數 / 治理文件總數) × 100%` |
| **最低可接受** | 100% |
| **期望目標** | 100% |
| **評估週期** | 每季評估 |
| **證據來源** | 版本控制紀錄、治理文件清單 |

#### KPI 4: 流程提醒準時發送率

| 項目 | 內容 |
|-----|------|
| **定義** | Gate 流程提醒於規定時點發送之比率 |
| **公式** | `(準時發送之提醒數 / 應發送提醒總數) × 100%` |
| **最低可接受** | ≥ 95% |
| **期望目標** | 100% |
| **評估週期** | 每 Gate 週期檢查，每季彙總 |
| **證據來源** | 流程提醒發送紀錄、流程追蹤表 |

## Pre-Gate Design Support KPI

**角色定位**：Concept System Architect / Feasibility Owner - Gate 0 執行角色

### KPI 概覽

本 KPI 組合衡量 Pre-Gate Design Support 對 Gate 0 輸入文件品質、需求釐清採用度、返工降低效果及概念產出完整性之當責履行。所有 KPI 須經後續 Gate 驗證方可生效。

### KPI 清單

#### KPI 1: Gate 0 輸入文件完整率

| 項目 | 內容 |
|-----|------|
| **定義** | Gate 0 所有必要輸入文件皆已產出且具備必要欄位之比率 |
| **公式** | `(完整產出之必要文件數 / 5) × 100%` |
| **最低可接受** | 100% |
| **期望目標** | 100% |
| **評估週期** | 每 Gate 0 即時檢查 |
| **證據來源** | Gate 0 文件包檢查表、5 項必要文件 |

#### KPI 2: 需求釐清採用率

| 項目 | 內容 |
|-----|------|
| **定義** | 需求釐清紀錄於 Gate 0 核准文件中被引用之比率 |
| **公式** | `(被引用之釐清項目數 / 釐清項目總數) × 100%` |
| **最低可接受** | ≥ 80% |
| **期望目標** | 100% |
| **評估週期** | Gate 0 核准後評估 |
| **證據來源** | Gate 0 核准紀錄、需求釐清紀錄、引用追溯表 |

#### KPI 3: 返工降低達成率

| 項目 | 內容 |
|-----|------|
| **定義** | Gate 1 後因「需求澄清不足」導致之設計變更比例 |
| **公式** | `1 - (需求澄清不足類變更數 / 設計變更總數)` |
| **最低可接受** | ≥ 90%（即需求澄清不足類變更 ≤ 10%） |
| **期望目標** | 100%（無需求澄清不足類變更） |
| **評估週期** | Gate 1 後 30 日評估 |
| **證據來源** | 設計變更紀錄、變更類別分析報告 |

#### KPI 4: 風險預揭露有效率

| 項目 | 內容 |
|-----|------|
| **定義** | 預揭露風險於後續 Gate 未重複出現為新識別風險之比率 |
| **公式** | `1 - (重複識別風險數 / 預揭露風險總數)` |
| **最低可接受** | ≥ 80% |
| **期望目標** | 100% |
| **評估週期** | Gate 3 完成後評估 |
| **證據來源** | 風險預揭露清單、殘餘風險清單、交叉比對紀錄 |

#### KPI 5: CBOM 完整性

| 項目 | 內容 |
|-----|------|
| **定義** | Feasibility BOM 具備正確 Non-binding 標註、BOM Type 宣告完整之比率 |
| **公式** | `(標註正確之 CBOM 數 / CBOM 總數) × 100%` |
| **最低可接受** | 100% |
| **期望目標** | 100% |
| **評估週期** | Gate 0 提交時即時檢查 |
| **證據來源** | Feasibility BOM、標註檢查紀錄 |

#### KPI 6: Concept Architecture 完整性

| 項目 | 內容 |
|-----|------|
| **定義** | Concept Architecture 具備 Conceptual/Non-binding 標註之比率 |
| **公式** | `(標註正確之 Concept Architecture 數 / Concept Architecture 總數) × 100%` |
| **最低可接受** | 100% |
| **期望目標** | 100% |
| **評估週期** | Gate 0 提交時即時檢查 |
| **證據來源** | Concept Architecture、標註檢查紀錄 |

#### KPI 7: Gate 0 責任移轉完成率

| 項目 | 內容 |
|-----|------|
| **定義** | Gate 0 核准後責任移轉紀錄於期限內完成之比率 |
| **公式** | `(期限內完成移轉之專案數 / Gate 0 核准專案總數) × 100%` |
| **最低可接受** | 100% |
| **期望目標** | 100% |
| **評估週期** | Gate 0 核准後 15 日評估 |
| **證據來源** | Gate 0 責任移轉紀錄、簽核紀錄 |

### KPI 生效條件

**重要說明**：本角色之 KPI 分數須待後續 Gate 驗證方可生效：

| KPI | 驗證時點 | 生效條件 |
|-----|---------|---------|
| Gate 0 輸入文件完整率 | Gate 0 提交時 | 即時生效 |
| 需求釐清採用率 | Gate 0 核准後 | Gate 0 核准紀錄確認引用 |
| 返工降低達成率 | Gate 1 後 30 日 | 設計變更分析完成 |
| 風險預揭露有效率 | Gate 3 完成後 | 風險清單交叉比對完成 |
| CBOM 完整性 | Gate 0 提交時 | 即時生效 |
| Concept Architecture 完整性 | Gate 0 提交時 | 即時生效 |
| Gate 0 責任移轉完成率 | Gate 0 後 15 日 | 移轉紀錄確認 |

## 跨角色一致性驗證

### KPI 定義一致性

| 驗證項目 | 結果 |
|---------|-----|
| 標準可執行性 KPI 定義一致（Head/Architect/Security/QA） | ✓ |
| 文件完整性 KPI 定義一致 | ✓ |
| 百分比計算公式格式一致 | ✓ |
| 證據來源與角色職責對應 | ✓ |

### KPI 重複性驗證

| 潛在重複 | 區分說明 |
|---------|---------|
| 設計文件完整性（Architect vs QA） | Architect 負責產出，QA 負責驗證；KPI 分別衡量 |
| Gate 審查紀錄（Governance Lead vs Coordinator） | Lead 負責決策紀錄，Coordinator 負責會議紀錄；職責互補 |

### 跨角色激勵衝突檢查

| 潛在衝突 | 解決方式 |
|---------|---------|
| QA 發現 vs Architect 設計品質 | QA KPI 不計發現數量；Architect KPI 不計被發現數量 |
| Gate 通過率 vs 品質把關 | 不設 Gate 通過率 KPI；僅評估紀錄完整性 |

## 假設條件

1. **基線數據**：部分門檻數值（標註為「需基線」者）需經實際運作後調整
2. **紀錄系統**：組織具備支援 KPI 量測之文件管理與紀錄系統
3. **評估週期**：每季評估適用於穩定運作之組織；初期可縮短為每月
4. **抽查機制**：抽查式 KPI 需建立標準化抽查程序
