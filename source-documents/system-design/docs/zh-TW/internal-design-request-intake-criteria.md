# Internal Design Request Intake Criteria

## Document Classification {#doc-classification}

**文件分類**：內部控制文件（Internal Control Document）

**重要聲明**：

- 本文件**非**《System Design Governance》主文件或其規範性附錄之一部分
- 本文件為系統設計部門**內部作業指引**，用於協助團隊評估 Pre-Gate0 需求之就緒程度
- 本文件所列之準則**不構成外部單位之提交義務**
- Design Requesting Function 不受本文件約束，其義務以主文件為準

---

## Purpose {#purpose}

本文件提供系統設計部門內部使用之設計需求受理評估準則，協助團隊：

1. 評估 Pre-Gate0 需求是否達到可進入 Gate 0 審查之品質門檻
2. 識別需補充之資訊，以協助 Design Requesting Function 準備
3. 提供一致之評估基準，減少主觀判斷差異

---

## Intake Evaluation Rubric {#evaluation-rubric}

### 評估維度與指引

以下評估維度對應主文件所定義之 Gate 0 品質門檻（可理解性、可評估性、需求擁有者明確性、範圍穩定性）。

#### 維度 1：需求目標清晰度（對應：可理解性）

| 評分 | 描述 |
|------|------|
| ✅ 充分 | 需求目標明確，預期成果可描述，團隊可理解設計需要達成什麼 |
| ⚠️ 部分 | 需求目標大致可理解，但部分細節需釐清 |
| ❌ 不足 | 需求目標模糊，團隊無法理解需要設計什麼 |

**補充建議**（若評分為「部分」或「不足」）：
- 請 DRF 說明：「這個設計完成後，業務 / 用戶可以做什麼以前做不到的事？」
- 請 DRF 說明：「如果不做這個設計，會有什麼影響？」

#### 維度 2：範圍邊界識別度（對應：可評估性 + 範圍穩定性）

| 評分 | 描述 |
|------|------|
| ✅ 充分 | 設計範圍有明確邊界，知道什麼在範圍內、什麼不在範圍內 |
| ⚠️ 部分 | 範圍大致可識別，但邊界模糊或仍在變動 |
| ❌ 不足 | 範圍完全不確定，或仍處於探索階段 |

**補充建議**（若評分為「部分」或「不足」）：
- 請 DRF 說明：「這個需求涵蓋哪些系統 / 模組 / 功能？」
- 請 DRF 說明：「哪些是明確不在這次設計範圍內的？」

#### 維度 3：利害關係人與需求擁有者（對應：需求擁有者明確性）

| 評分 | 描述 |
|------|------|
| ✅ 充分 | 有明確之 Design Requesting Function，可為需求之商業合理性負責 |
| ⚠️ 部分 | 有聯絡人但職責不明，或尚未確認其授權 |
| ❌ 不足 | 沒有明確之需求擁有者，或多頭馬車無法決策 |

**補充建議**（若評分為「部分」或「不足」）：
- 確認：「誰有權決定這個需求的範圍與優先順序？」
- 確認：「若需求發生變更，誰是最終確認者？」

#### 維度 4：技術與風險資訊（對應：可評估性）

| 評分 | 描述 |
|------|------|
| ✅ 充分 | 提供足夠資訊進行初步技術可行性評估與風險識別 |
| ⚠️ 部分 | 有部分資訊，但不足以識別主要風險假設 |
| ❌ 不足 | 資訊極度不足，無法進行任何技術評估 |

**補充建議**（若評分為「部分」或「不足」）：
- 詢問：「有哪些已知的技術限制或相依性？」
- 詢問：「是否有安全、合規或效能方面的特殊要求？」
- 詢問：「是否有既有系統或介面需要整合？」

#### 維度 5：急迫性與驅動因素（輔助評估）

| 評分 | 描述 |
|------|------|
| ✅ 充分 | 有明確之時程驅動因素（如合約、法規、商務承諾） |
| ⚠️ 部分 | 有時程期望但驅動因素不明 |
| ❌ 不足 | 無時程資訊或「越快越好」但無具體依據 |

**補充建議**（若評分為「部分」或「不足」）：
- 詢問：「是否有外部時程限制（合約、法規、商務）？」
- 詢問：「延遲的後果是什麼？」

## Intake Decision Matrix {#decision-matrix}

### 評估結果判定

| 維度 1-4 評分組合 | 建議決策 |
|-----------------|---------|
| 全部 ✅ | **可進入 Gate 0 審查** |
| 含 1-2 個 ⚠️，無 ❌ | **有條件進入**：列出待補充事項，DRF 補充後可進入 Gate 0 |
| 含 ❌ | **維持 Pre-Gate0 釐清**：協助 DRF 釐清，待改善後重新評估 |

### 拒絕受理之正當理由（供內部參考）

以下情境可作為拒絕進入 Gate 0 審查之正當理由：

- 無法識別 Design Requesting Function
- 需求目標無法被團隊理解
- 範圍仍在探索階段，尚未穩定
- 資訊不足以識別任何基本風險假設
- 需求方無法回答「如果不做會怎樣」

## Clarification vs. Rejection {#clarification-vs-rejection}

### 內部行為指引

| 情境 | 建議行為 |
|------|---------|
| DRF 積極配合，資訊逐步補足 | 持續協助釐清，待就緒後進入 Gate 0 |
| DRF 無法提供關鍵資訊，但非不願意 | 協助識別資訊來源，建議分階段釐清 |
| DRF 拒絕提供基本資訊 | 記錄溝通過程，正式告知無法進入 Gate 0 |
| 需求持續變動，無法穩定 | 建議待需求穩定後再提出 |

### 溝通建議用語

**協助釐清時**：
> 「為了讓我們能更準確評估這個需求，能否請您補充以下資訊：...」

**拒絕進入 Gate 0 時**：
> 「目前提供的資訊尚不足以進行正式的技術可行性評估。建議補充以下內容後，再提出 Gate 0 審查申請：...」
