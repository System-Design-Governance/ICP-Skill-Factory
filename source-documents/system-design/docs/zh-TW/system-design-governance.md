# System Design Governance

## Purpose and Authority

本文件定義公司於**系統設計階段之治理原則、責任邊界與決策流程**，其目的在於：

- 確保系統設計決策於安全、品質與合規層級具備一致性
- 將設計責任、風險揭露與後續執行責任清楚區隔
- 作為專案啟動、設計交付與稽核判定之正式依據

### 治理地位與強制性聲明

**本文件為系統設計相關治理文件之上層依據，具備以下強制性要求：**

1. **強制適用範圍**：凡涉及系統架構、安全關鍵設計或風險假設之專案，其設計決策與交付流程，**必須遵循**本文件及其附錄所定義之治理要求。

2. **授權來源**：本文件由 System Design Governance Function 發布，並經 Engineering Management 核准執行。

3. **生效日期**：2026-01-08 起，所有新啟動專案必須遵守本文件規範。既有進行中專案若涉及重大設計變更，亦須適用本文件 Gate 2 與 Gate 3 之要求。

### 不遵守之制度性後果

未遵循本文件治理要求之專案，將面臨以下制度性後果：

- **Gate 0 未通過**：專案不得啟動，不得對外承諾交期或資源
- **Gate 1 未完成**：不得進入詳細設計階段，不得開始實作
- **Gate 2 未執行**：設計變更不得實施，專案暫停直至完成偏離紀錄
- **Gate 3 未交付**：設計責任無法移轉，系統設計部門不承認設計交付完成
- **稽核不符合**：專案無法通過 IEC 62443-2-4 合規稽核，影響產品認證與對外交付

**治理宗旨聲明**：本文件之目的並非阻擋專案執行，而是確保設計決策能被負責任地做出與交接。治理的價值，不在於避免所有風險，而在於確保每一個風險之承擔者皆已明確認定並經正式確認。

## Pre-Gate0 Clarification Boundary {#pre-gate0-boundary}

### 治理責任起始點聲明 {#governance-start-point}

**本治理框架之責任適用，以 Gate 0 核准為明確起始點。**

在 Gate 0 核准之前，任何形式之討論、釐清、諮詢、概念探索或協助，皆不構成以下效力：

1. **不構成受理**：Pre-Gate0 階段之交流不代表系統設計部門已接受設計需求。設計需求之正式受理，以 Gate 0 核准為唯一判定依據。

2. **不構成責任承擔**：Pre-Gate0 階段之任何回饋、建議或初步分析，不構成系統設計部門對設計結果、時程或品質之承諾或責任。

3. **不構成資源承諾**：Pre-Gate0 階段之討論不代表正式開發資源已被分配。資源分配須待 Gate 0 核准後，依專案管理程序另行安排。

4. **不構成交付承諾**：在 Gate 0 核准之前，不得以系統設計部門之名義對外承諾交期、功能範圍或設計方向。

### Pre-Gate0 釐清活動之定位 {#pre-gate0-clarification}

系統設計部門**得**於 Pre-Gate0 階段提供以下協助，但此類協助**不構成治理責任之開始**：

- 初步技術諮詢與概念討論
- 協助 Design Requesting Function 釐清需求範圍
- 評估資訊補充建議（協助需求方準備 Gate 0 所需資訊）
- 高層級可行性初判（非正式評估）

**重要聲明**：上述活動僅為善意協助，不改變「Gate 0 核准為治理責任起始點」之原則。若需求方將 Pre-Gate0 討論內容引用為設計承諾或責任依據，系統設計部門得拒絕並要求回歸正式治理流程。

### 釐清與受理之區分 {#clarification-vs-acceptance}

| 階段 | 性質 | 治理責任 | 資源承諾 | 交付義務 |
|------|------|---------|---------|---------|
| **Pre-Gate0 釐清** | 需求探索與資訊補充 | 無 | 無 | 無 |
| **Gate 0 審查中** | 正式評估技術可行性與風險可承接性 | 無（審查進行中） | 無 | 無 |
| **Gate 0 核准後** | 治理責任正式開始 | **是** | **依核准內容** | **依核准範圍** |

## Applicability and Scope

### 適用範圍判定準則

本治理文件適用於符合以下**任一條件**之專案，且**不得例外跳過**：

#### 強制適用類型

| 專案類型 | 判定準則 | 最低 Gate 要求 |
|---------|---------|--------------|
| **新系統開發** | 從零開始設計之系統或平台 | Gate 0 至 Gate 3 全部執行 |
| **重大架構調整** | 影響系統核心架構、資料流或外部介面 | Gate 1 至 Gate 3 |
| **安全關鍵功能** | 涉及認證、授權、加密、關鍵資料保護 | Gate 1 至 Gate 3 |
| **合規風險變更** | 影響 IEC 62443 SL 等級或法規合規性 | Gate 2 至 Gate 3 |

#### 適用性判定流程

若專案發起人或專案經理對「是否適用本文件」有疑問，應遵循以下流程：

1. **初步自評**：專案經理依據上述判定準則進行初步評估
2. **諮詢確認**：若仍有疑問，向 System Design Governance Function 提出書面諮詢
3. **正式判定**：由 System Design Governance Function 於 3 個工作日內給予書面判定
4. **紀錄留存**：判定結果納入專案文件，作為稽核依據

**判定原則**：當有疑義時，採取「從嚴認定」原則，即：**有疑問時視為適用**。

### 例外處理機制

若因時程、商務或技術限制需偏離本治理流程，必須遵循以下例外處理機制：

#### 例外申請條件

例外申請僅限於以下情境：

- **緊急安全修補**：已發生之安全事件需立即修補，且無法等待完整 Gate 流程
- **法規強制要求**：外部法規或主管機關要求之強制性變更，且有明確時限
- **商務關鍵承諾**：已簽約之商務承諾，且違約將造成重大法律或財務損失

**不得以下列理由申請例外**：「專案時程緊迫」、「資源不足」、「客戶要求加速」、「已經開始做了」

#### 例外申請流程

1. **申請人**：專案經理或部門主管提出書面例外申請
2. **申請內容**必須包含：偏離哪些 Gate 或治理要求、為何無法遵循正常流程（必須符合上述例外申請條件）、風險評估與緩解措施、預計何時補足治理要求
3. **核准權限**：
   - Gate 0 / Gate 1 例外：需 Engineering Management 核准
   - Gate 2 例外：需 System Design Governance Function 核准
   - Gate 3 例外：需 Engineering Management + 風險接受單位共同核准
4. **紀錄留存**：所有例外申請與核准紀錄納入專案文件，並於年度稽核時檢討

#### 例外後果與補救

即使獲得例外核准，仍須遵守以下要求：

- **風險承擔**：例外期間產生之風險，由例外核准者與專案執行單位共同承擔
- **補救要求**：例外核准文件必須載明需補回的 Gate 與補件日期，逾期則升級至 Engineering Management 裁決並可暫停專案或交付
- **稽核追蹤**：例外案件納入重點稽核清單，直至補救措施完成

## Governance Model and Design Gates

系統設計治理以 **Gate 機制** 運作，確保設計決策於適當時點被檢視、確認與交接。

### 設計治理核心原則

本治理框架建立於以下核心原則，所有 Gate 審查必須驗證這些原則是否被落實：

- **Security by Design**：安全需求必須於設計初期即被納入考量，而非事後補強。Gate 0 必須識別安全需求，Gate 1 必須定義安全控制措施。

- **Traceability**：所有關鍵設計決策須可回溯至需求與風險假設。Gate 1 至 Gate 3 所有設計文件必須包含需求追溯矩陣。

- **Compliance Alignment**：設計須符合適用之法規與標準（如 IEC 62443）。Gate 1 必須完成 IEC 62443 對應檢查（詳見 Appendix A）。

- **Risk-based Decision Making**：設計取捨須以風險為基礎，而非僅以技術可行性為考量。Gate 0 至 Gate 3 所有重大決策必須包含風險評估紀錄。

### 治理構件定義

本治理框架區分以下三種治理構件，各具獨立之存在性要求：

| 治理構件 | 定義 | 治理目的 |
|---------|------|---------|
| **設計文件集（Design Artifacts）** | 記載設計內容、決策依據與技術規格之文件 | 作為設計內容之正式紀錄與實作依據 |
| **文件清冊（Document Register）** | 列舉所有受治理管控之設計文件及其版本狀態 | 作為設計基線認定與交付範圍之唯一依據 |
| **設計標的清冊（Design Object Inventory）** | 列舉設計所涵蓋之系統、子系統、元件或資產 | 作為風險評估範圍與責任移轉標的之界定依據 |

**治理構件之存在性要求**：

- **Gate 1**：文件清冊必須存在並經核准，作為設計基線承認之正式依據。未列入清冊之文件，不得作為 Gate 1 審查之輸入。
- **Gate 3**：設計責任移轉僅限於文件清冊所列且經核准版本之設計文件，以及設計標的清冊所列之系統與元件。未列入清冊之項目，不在責任移轉範圍內。

**清冊管理原則**：本文件不規定清冊之格式或管理工具。清冊之建立與維護方式，由專案依組織標準自行決定，惟必須滿足上述存在性與可稽核性要求。

### 整合式設計風險評估

本治理框架要求系統設計階段之風險評估**必須採用整合式方法論**，結合以下三種分析方法：

1. **IEC 62443-3-2**：識別資安威脅、評估安全風險、定義 Zone/Conduit、選擇安全控制措施
2. **FMEA**：識別系統/子系統/元件之失效模式、評估失效後果、定義偵測與緩解措施
3. **HAZOP**：識別操作流程偏差、評估偏差後果、定義保護措施與操作程序

**整合原則**：三種方法不可互相取代，必須全部執行。Gate 3 殘餘風險必須可追溯至三種方法之至少一種來源，否則視為風險紀錄不成立。

#### 風險評估之治理時程

| Gate | 風險評估要求 | 最低可接受形式 | 責任角色 |
|------|-------------|---------------|---------|
| Gate 0 | 提出風險評估策略與範圍 | 書面說明三種方法之使用範圍與深度 | System Architect |
| Gate 1 | 完成初步風險評估 | Lite 形式（見下方定義） | System Architect |
| Gate 2 | 設計變更導致風險重評 | 更新受影響之分析項目 | System Architect |
| Gate 3 | 殘餘風險追溯與接受 | 完整版（見下方定義） | System Architect + Risk Acceptance Authority |

#### Gate 1 Lite 風險評估定義

若專案於 Gate 1 採用 Lite 形式，各方法之最低可接受內容如下：

- **IEC 62443-3-2 Lite**：Zone & Conduit Diagram 初版、威脅情境清單、目標 SL 等級初步對應
- **FMEA Lite**：Top-level Failure Modes、S/O/D 評分、高風險項目控制措施
- **HAZOP Lite**：關鍵流程 Guide Words 分析、高後果偏差保護措施

**Lite 共同要求**：必須有書面紀錄、版本號、日期、分析參與人員簽名、審查簽核紀錄

#### 完整版風險評估要求

完整版風險評估必須於 Gate 3 前完成，各方法須包含：

- **IEC 62443-3-2 完整版**：最終版 Zone & Conduit Diagram、完整 Threat Modeling、Security Requirements Traceability Matrix、Risk Treatment Plan
- **FMEA 完整版**：完整 RPN 計算（S×O×D）、所有高風險項目 Action Plan、改善後 RPN 驗證
- **HAZOP 完整版**：完整 Guide Words 分析、Risk Ranking、操作程序建議

**完整版共同要求**：必須使用正式模板（見 Appendix D）、完整審查簽核鏈、版本管理與變更歷程

**Appendix D 未發布時之過渡規則**：若 Appendix D 尚未正式發布，專案得使用符合本節內容要求之等效格式，但不得於任何文件中宣稱「符合 Appendix D」，亦不得以此作為 Gate 1 或 Gate 3 完備之依據。

#### Lite 升級規則與限制

若採用 Lite 形式通過 Gate 1，須於 Gate 3 前升級為完整版本。若未升級，Gate 3 不得放行。

**Lite 適用範圍限制**：對上述定義之「安全關鍵功能」與「重大架構調整」專案，Gate 1 不得以 Lite 作為最終交付依據。此類專案必須在進入任何實作或採購前，至少完成 IEC 62443-3-2 完整版。

#### 設計變更風險重評觸發條件

當 Gate 2 設計變更符合以下任一條件，必須更新對應之風險評估：

- **觸發 IEC 62443-3-2 更新**：Zone/Conduit 邊界變更、外部連接介面變更、安全控制措施變更、SL 等級調整
- **觸發 FMEA 更新**：系統架構變更、失效偵測機制變更、冗餘設計變更、關鍵元件變更
- **觸發 HAZOP 更新**：操作流程變更、控制邏輯變更、人機介面變更、安全聯鎖變更

更新後之文件版本號必須遞增，並重新審查簽核。

#### 殘餘風險追溯要求

Gate 3 殘餘風險清單中之每個風險項目，必須可追溯至以下至少一種來源：

- IEC 62443-3-2 Threat Scenario ID
- FMEA Failure Mode ID
- HAZOP Deviation ID

若某風險無法提供來源 ID，視為風險紀錄不成立，Gate 3 不得放行。QA Team 於 Gate 3 審查時，必須抽查至少 20% 高風險項目之追溯有效性。

### 設計治理 Gate 定義

#### 治理角色定義

本文件採用以下角色模型：

- **Owner**：本文件之 Owner 為 System Design Governance Function
- **Accountable**：對 Gate 輸出內容負責之角色，通常為 System Architect
- **Approver**：有權決定 Gate 通過與否之單一角色
- **Required Reviewers**：必須完成審查並簽核，但不單獨擁有放行權
- **Co-signers**：針對風險接受等事項需共同簽核承擔責任，但不代表 Gate 放行權
- **Design Requesting Function**：設計需求之合法提出來源（詳見下方定義）

**重要原則**：每個 Gate 僅有一個 Approver，避免權責分散。

#### Design Requesting Function（需求提出單位）

Design Requesting Function 為設計需求之唯一合法提出來源，負責確保所提出之設計需求具備商業、營運或策略上之合理性。

**治理責任**：

1. **需求提出責任**：所有設計需求須由 Design Requesting Function 正式提出，並對其商業目的、營運必要性或策略合理性負責。未經 Design Requesting Function 提出之需求，不得作為設計工作之依據。

2. **Gate 0 成立基礎**：Gate 0 專案啟動審查之前提，為存在經 Design Requesting Function 確認之設計需求。無正式設計需求之專案，不得進入 Gate 0 審查。

3. **需求變更確認**：設計需求之實質變更（範圍擴大、目標調整、優先順序變更），須由原 Design Requesting Function 發起或書面確認。未經確認之需求變更，不得作為 Gate 2 設計變更之依據。

**責任邊界聲明**：

- Design Requesting Function **不具** Gate 1 至 Gate 3 之 Approver 身分
- Design Requesting Function **不對**設計解法、安全架構或風險控制負技術決策責任
- Design Requesting Function **不參與**設計內容審查，僅就需求範圍與商業合理性提供確認

**角色映射說明**：Design Requesting Function 依組織實際情況，可由 Business Owner、Product Owner、Requirement Owner、或經授權之專案發起人擔任。無論由何種職稱擔任，其治理責任不變。

#### Design Requesting Function 與 Risk Acceptance Authority 之關係 {#drf-raa-relationship}

Design Requesting Function 與 Risk Acceptance Authority 為不同之治理角色，職責邊界如下：

| 角色 | 職責範圍 | 出現時點 |
|------|---------|---------|
| **Design Requesting Function** | 需求提出、商業合理性確認、需求變更確認 | Gate 0（主要）、Gate 2（變更時） |
| **Risk Acceptance Authority** | 殘餘風險接受簽核 | Gate 3 |

**角色重疊情境**：當 Business Owner 同時擔任 Design Requesting Function 與 Risk Acceptance Authority（High Risk 情境）時，其職責不相混淆：

- 作為 DRF：對需求之商業合理性負責
- 作為 RAA：對殘餘風險之接受決策負責

此二責任之行使時點不同，且簽核紀錄須分別留存。

## Governance Interface Compatibility {#governance-interface-compatibility}

### 治理介面相容性原則 {#interface-compatibility-principle}

本治理框架設計為可與其他組織單位之治理流程銜接之**管線介面（Pipeline Interface）**。當外部單位（無論作為需求提出方、設計執行方或後續承接方）需與本治理框架銜接時，須滿足以下介面相容性要求。

**重要聲明**：本框架**不要求**外部單位採用相同之流程名稱、Gate 編號或管理工具。相容性判定以**功能等效性**為準，而非形式一致性。

### 介面相容性要件 {#interface-requirements}

任何與本治理框架銜接之外部單位，須提供以下等效治理介面：

#### 1. 角色映射（Role Mapping）

外部單位須能識別並映射以下治理角色（名稱可不同，但職責須等效）：

| 本框架角色 | 必要職責等效性 |
|-----------|---------------|
| Design Requesting Function | 具備需求提出與商業合理性確認之權責單位 |
| Risk Acceptance Authority | 具備殘餘風險接受決策權限之簽核者 |
| Accountable Role | 對交付內容負責之明確角色 |

#### 2. 決策點映射（Decision Point Mapping）

外部單位須具備與本框架 Gate 等效之決策點機制（名稱可不同）：

| 本框架 Gate | 等效決策點之必要特徵 |
|------------|-------------------|
| Gate 0 | 正式受理 / 啟動決策點 |
| Gate 3 | 交付 / 責任移轉決策點 |

**最低要求**：至少須具備「受理」與「責任移轉」兩個可稽核之決策點。

#### 3. 可稽核性（Auditability）

外部單位須能提供以下可稽核之治理證據：

- **清冊存在性**：類似文件清冊或設計標的清冊之交付範圍界定機制
- **決策紀錄**：決策點之核准 / 拒絕紀錄
- **責任歸屬紀錄**：各階段責任歸屬之書面或系統紀錄

### 介面相容性驗證 {#interface-verification}

當外部單位首次與本治理框架銜接時，由 System Design Governance Function 執行介面相容性驗證：

1. **角色映射確認**：確認外部單位可指認等效角色
2. **決策點對應**：確認外部單位具備等效決策機制
3. **證據格式協議**：協議責任移轉時之證據交換格式

**驗證結果**：
- **相容**：可進行正式銜接，責任移轉依本框架 Gate 3 規範執行
- **部分相容**：須補充特定介面要件，或於銜接時增加額外驗證步驟
- **不相容**：不得進行正式銜接，責任移轉不生效力

### 不相容情境之處理 {#incompatibility-handling}

若外部單位無法提供等效治理介面，則：

- 本框架不承認該單位作為有效之需求提出方或責任承接方
- Gate 0 不得核准受理（若為需求提出方）
- Gate 3 責任移轉不生效力（若為責任承接方）
- 系統設計部門之責任不因外部單位之治理缺失而免除或移轉

**補救措施**：外部單位得透過補充內部規範或簽署責任協議之方式，滿足介面相容性要件。具體補救措施由 System Design Governance Function 與外部單位協商決定。

#### Gate 架構總覽

| Gate | 目的 | 觸發時機 | 最低輸出要求 | 阻擋條件 |
|------|------|---------|-------------|---------|
| Gate 0 | 技術可行性與風險可承接性判定 | 專案啟動前 | 技術可行性評估、初步風險清單、風險評估策略 | 關鍵風險無法接受 |
| Gate 1 | 設計基線建立與風險初步揭露 | 初步設計完成後 | 設計基線文件、整合式風險評估、IEC 62443 對應檢查表、需求追溯矩陣 | 安全要求未定義 |
| Gate 2 | 設計變更與偏離管理 | 設計變更發生時 | 變更影響分析、風險重新評估 | 重大變更未經核准 |
| Gate 3 | 設計交付與殘餘風險責任移轉 | 設計完成交付前 | 最終設計文件、殘餘風險清單、交接會議紀錄 | 殘餘風險未獲接受 |

#### Gate 0：專案啟動審查（受理決策）

**目的**：Gate 0 為**受理決策（Acceptance Decision）**，判定系統設計部門是否正式接受此設計需求並開始承擔治理責任。Gate 0 **不是文件清單驗證**，而是評估需求是否達到可被評估與可被承接的品質門檻。

**觸發時機**：專案概念提出後、正式啟動前，在資源分配與對外承諾前執行。

**受理品質門檻（Acceptance Quality Thresholds）**：

Gate 0 審查須確認設計需求符合以下抽象品質門檻，方可核准受理：

1. **可理解性（Comprehensibility）**：設計需求之目標、範圍與預期成果可被系統設計部門理解，足以進行技術可行性評估。

2. **可評估性（Evaluability）**：所提供之資訊足以評估技術可行性與初步風險假設。若資訊不足以識別基本風險，則不滿足可評估性要求。

3. **需求擁有者明確性（Accountable Owner）**：存在明確之 Design Requesting Function，對需求之商業目的、營運必要性與策略合理性負責。

4. **範圍穩定性（Scope Stability）**：需求之目標與範圍已足夠穩定，可作為設計基線之基礎。若需求仍處於探索或頻繁變動狀態，應維持於 Pre-Gate0 釐清階段。

**拒絕受理之正當性**：

若提出之資訊不足以評估技術可行性或識別基本風險假設，系統設計部門**有權拒絕進入 Gate 0 審查**，需求將維持於未受理狀態，直至 Design Requesting Function 補足所需資訊。此拒絕權限為系統設計部門之專業判斷，不構成對需求方之不當阻擋。

**Gate 0 核准之輸出**：

Gate 0 核准後，須產出以下治理紀錄（形式不限，但必須可稽核）：

- 技術可行性初步判定
- 初步風險識別與風險評估策略（明確後續將採用之風險分析方法及其範圍深度）
- 目標 IEC 62443 SL 等級初步提案
- 資源與時程初步估算

**阻擋條件**：關鍵技術風險無單位願意承接、明顯違反法規或公司政策、資源需求遠超公司負荷能力、需求未達上述品質門檻

**未通過後果**：專案不得啟動、不得對外承諾交期或功能、不得分配正式開發資源

**責任角色**：

- Accountable：System Architect
- Approver：Engineering Management

#### Gate 1：設計基線建立

**目的**：建立設計基線，確保設計方向正確且風險已充分識別。

**觸發時機**：初步設計完成後、開始詳細設計前。

**最低輸出要求**：

1. 設計基線文件（包含系統架構圖、介面定義、資料流圖）
2. 整合式風險評估報告（依上述要求，Lite 或完整版）
3. IEC 62443 對應檢查表（詳見 Appendix A）
4. 需求追溯矩陣
5. 文件清冊（Document Register）：列舉所有構成設計基線之文件及其版本
6. 設計標的清冊（Design Object Inventory）：列舉設計所涵蓋之系統、子系統與元件

**阻擋條件**：安全要求未定義或控制措施不足、IEC 62443 對應檢查未完成、關鍵設計決策無法追溯至需求、重大風險未識別、文件清冊不存在或未經核准

**未通過後果**：不得進入詳細設計階段、不得開始實作或採購

**責任角色**：

- Accountable：System Architect
- Approver：System Design Governance Function
- Required Reviewers：Security Team、QA Team

#### Gate 2：設計變更管理

**目的**：管理設計變更，確保變更不引入未評估之風險。

**觸發時機**：設計基線建立後，發生任何設計變更時。

**最低輸出要求**：

1. 變更描述與理由
2. 影響分析
3. 風險重新評估（依上述觸發條件更新）
4. 更新後的設計文件（版本號遞增）

**阻擋條件**：重大變更未經 Engineering Management 核准、變更導致 SL 等級下降且未獲接受、變更引入新高風險且無控制措施

**未通過後果**：變更不得實施、專案暫停直至完成偏離紀錄

**責任角色**：

- Accountable：System Architect
- Approver：Engineering Management（可委任 Governance Function 執行審查與彙整，但放行權與責任仍屬 Engineering Management，文件上 Approver 不得出現第二人）

**重大變更定義**：影響系統核心架構或外部介面、改變 SL 等級、增加或移除安全控制措施、影響已簽約承諾

#### Gate 3：設計交付與責任移轉

**目的**：完成設計交付，將設計責任移轉至專案執行單位，並確保殘餘風險已被接受。

**觸發時機**：詳細設計完成後、交付實作前。

**最低輸出要求**：

1. 最終設計文件包（所有文件標註版本號、發布日期、文件擁有者，引用版本號必須一致）
2. 殘餘風險清單（使用 Appendix C 模板，必須符合上述追溯要求）
3. 設計交付檢查表
4. 交接會議紀錄（必須記載「專案執行單位已完成交接確認」，雙方簽核）

**阻擋條件**：殘餘風險未獲簽核接受、風險清單存在無法追溯項目、文件不完整或版本不一致、Gate 1 Lite 未升級為完整版、交接會議紀錄未記載交接確認

**未通過後果**：設計責任無法移轉、專案執行單位不得宣稱「設計已交付」、未完成 Gate 3 仍進入實作視為偏離治理流程，必須啟動例外機制並由 Engineering Management 裁決責任承擔

**責任角色**：

- Accountable：System Architect
- Approver：System Design Governance Function（放行前提：殘餘風險已由下述權限者簽核、交接會議紀錄已記載交接確認）
- Required Reviewers：QA Team
- Risk Acceptance Authority：由下方殘餘風險管理定義，與 Approver 分離

**設計交付後責任分界**：

- 系統設計部門：對設計內容正確性與風險揭露完整性負責，但不對實作結果負責
- 專案執行單位：對實作結果、殘餘風險承擔與後續營運風險負責
- 爭議處理：由 Engineering Management 裁決

**責任移轉之法律邊界聲明**：

Gate 3 通過所確立之設計責任移轉，其效力範圍受以下條件限制：

1. **移轉範圍限定**：設計責任移轉僅限於文件清冊所列且經 Gate 3 核准版本之設計文件，以及設計標的清冊所列之系統與元件。未列入清冊之項目，其設計責任不因 Gate 3 通過而移轉。

2. **風險揭露前提**：責任移轉以殘餘風險清單所揭露之風險為限。系統設計部門對於已揭露並經風險接受權限者簽核之殘餘風險，於 Gate 3 通過後不再承擔責任。

3. **未揭露風險之責任保留**：對於設計階段應識別而未識別、應揭露而未揭露之設計缺陷或風險，系統設計部門之責任不因 Gate 3 通過而免除。此類責任之認定，以設計時點之合理專業判斷標準為依據。

4. **惡意或重大過失不適用**：因惡意隱匿或重大過失導致之設計缺陷，不適用上述責任移轉規則。

5. **爭議舉證責任**：若專案執行單位主張設計缺陷屬「應揭露而未揭露」，應負舉證責任證明該缺陷於設計時點依合理專業判斷應可識別。

## Roles, Handover, and Accountability

### 角色分離原則

本治理框架明確區分三個責任主體：

| 責任主體 | 責任範圍 | 責任終點 |
|---------|---------|---------|
| 系統設計部門 | 設計內容正確性、風險揭露完整性、IEC 62443 合規性 | Gate 3 交付完成 |
| 專案執行部門 | 實作結果、殘餘風險承擔決策、執行期間風險管理 | 專案結案 |
| 營運維護部門 | 營運期間風險、系統維護、事件回應 | 系統除役 |

**重要原則**：

- 系統設計部門不對實作結果負責
- 專案執行部門不對設計內容正確性負責
- 營運維護部門不對設計或實作缺陷負責

### 交付與責任移轉時機

責任移轉發生於以下時間點：

1. **Gate 3 完成時**：設計責任從系統設計部門移轉至專案執行部門
2. **專案結案時**：實作責任從專案執行部門移轉至營運維護部門
3. **系統除役時**：所有責任終止

**責任移轉必要條件**：責任移轉須同時滿足以下要件，缺一不可：

- 對應 Gate 審查已通過
- 書面交接文件已建立並經雙方確認
- 交接會議已召開並留存正式紀錄
- 接收單位已完成書面簽核確認

**責任移轉失敗之效果**：若上述要件未全數滿足，責任移轉不生效力，責任仍由原單位承擔。接收單位有權拒絕承接後續風險，直至要件補足。

### 爭議處理與升級機制

當跨部門對責任歸屬有爭議時，遵循以下升級路徑：

- **Level 1**（3 個工作日內）：System Architect + Project Manager + QA Team 三方協商
- **Level 2**（5 個工作日內）：提交至 System Design Governance Function 仲裁
- **Level 3**（10 個工作日內）：提交至 Engineering Management 最終決策

**重要原則**：爭議期間專案不得進入下一階段。例外情境：若涉及上述定義之「緊急安全修補」，則不中斷緊急處置，但須同步啟動例外機制。

詳細角色與責任矩陣，請參閱 **Appendix B: RACI Matrix**。

## Risk and Compliance Alignment

### IEC 62443 標準對齊

本治理模式與 IEC 62443-2-4 及 IEC 62443-3-3 之對應關係，請參閱 **Appendix A: IEC 62443 Alignment**。

**IEC 62443 SL 判定時程**：

- Gate 0：提出目標 SL 等級（初步假設）
- Gate 1：完成 SL 等級判定與對應檢查
- Gate 2：設計變更若影響 SL 等級，必須重新評估並經核准
- Gate 3：確認殘餘風險不影響目標 SL 等級之宣告

**不符合處理**：若設計無法滿足目標 SL 等級，必須降低目標 SL 等級（需 Stakeholders 簽核接受）或重新設計。不得以「技術限制」為由跳過 IEC 62443 要求。

### 殘餘風險管理

設計階段無法完全消除之風險，稱為**殘餘風險**，必須於 Gate 3 交付時明確揭露並完成責任移轉。

**紀錄要求**：使用 Appendix C 模板，每個殘餘風險必須包含風險描述、固有風險等級、已實施控制措施、殘餘風險等級、風險接受決策與簽核。

**風險等級判定依據**：

- 風險等級（Low / Medium / High / Critical）必須依據 Appendix C 所定義之風險矩陣
- 不得僅憑主觀判斷標註風險等級
- 必須保留 Likelihood、Impact 評分依據與計算過程
- 稽核時必須能重新計算出相同結果，否則視為風險評估不成立

**FMEA RPN 特殊要求**：必須記錄 Severity、Occurrence、Detection 三項評分依據與 RPN 閾值，改善後 RPN 必須重新計算。

**風險等級一致性要求**：同一風險在不同文件或章節出現時，風險等級必須一致，且以殘餘風險清單為準，若有差異必須在交接會議紀錄中說明。

**殘餘風險接受權限**：

- Low Risk：由 Project Manager 接受
- Medium Risk：由 Project Manager + Security Lead 共同接受
- High Risk：由 Project Manager + Security Lead + Business Owner 共同接受
- Critical Risk：必須升級至 Engineering Management，並制定額外緩解計畫

**殘餘風險未被接受之後果**：Gate 3 無法完成、設計責任無法移轉、專案不得進入實作階段

**殘餘風險接受之治理原則**：殘餘風險之接受，係專案執行單位之責任與權利。系統設計部門不得代替專案執行單位接受殘餘風險，亦不得以任何形式承諾風險接受。風險接受決策一經簽核，其效力及於該風險之所有後果。

本文件之修訂流程：變更提案 → 影響評估 → 利害關係人審查 → Engineering Management 核准發布
