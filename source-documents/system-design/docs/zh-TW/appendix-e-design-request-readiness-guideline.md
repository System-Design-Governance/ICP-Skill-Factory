# Appendix E: Design Request Readiness Guideline

## Document Classification {#appE-classification}

**⚠️ Informative Appendix — Non-normative Guidance**

本附錄為**參考性指引（Informative Guidance）**，不構成《System Design Governance》主文件之強制性要求。

**與規範性附錄之區別**：

| 附錄類型 | 遵循義務 | 本附錄定位 |
|---------|---------|-----------|
| 規範性附錄（Appendix A-D） | 必須遵循 | ❌ 不適用 |
| 參考性附錄（本文件） | 建議參考 | ✅ 適用 |

**使用目的**：本附錄旨在協助 Design Requesting Function 了解如何準備設計需求，以提高 Gate 0 審查效率並減少後續爭議。

---

## Purpose {#appE-purpose}

本指南說明：

1. 什麼樣的設計需求容易被理解與評估
2. 常見的需求準備不足情境及其影響
3. 如何透過良好的需求準備，減少責任爭議與返工

**重要聲明**：本指南**不創設新的強制性要求**。Design Requesting Function 之義務，以主文件所定義之治理責任為準。

---

## Why Request Readiness Matters {#appE-why-readiness}

### 對 Design Requesting Function 的好處

| 好處 | 說明 |
|------|------|
| **加速審查流程** | 資訊充分的需求可快速進入 Gate 0 審查，減少來回釐清 |
| **減少責任爭議** | 清晰的需求範圍可避免後續「這不是我要的」爭議 |
| **提升設計品質** | 設計團隊能更準確理解需求，產出更符合期望的設計 |
| **風險提前識別** | 充分的資訊有助於早期識別風險，避免後期發現重大問題 |

### 準備不足的常見後果

| 情境 | 可能後果 |
|------|---------|
| 需求目標模糊 | 設計方向偏差，需多次返工 |
| 範圍不明確 | Gate 3 時發生「這也要包含」的範圍爭議 |
| 無明確需求擁有者 | 決策延遲，無人可為變更負責 |
| 時程驅動因素不明 | 優先順序無法判定，資源分配困難 |

## Good vs. Poor Request Examples {#appE-examples}

### 範例 1：需求目標描述

**❌ 較差的描述**：
> 「我們需要一個新的報表系統。」

**問題**：
- 不知道報表要呈現什麼內容
- 不知道要解決什麼問題
- 無法評估技術可行性

**✅ 較好的描述**：
> 「我們需要一個每日營運報表系統，讓營運主管能在每天早上 9 點前看到前一天的銷售數據（含通路別、產品別銷售額）、庫存水位、以及異常訂單清單，以便在晨會做出當日營運決策。」

**優點**：
- 明確的使用者（營運主管）
- 明確的使用情境（晨會決策）
- 可識別的資料需求（銷售、庫存、異常訂單）
- 可評估的時程限制（每天早上 9 點前）

---

### 範例 2：範圍邊界定義

**❌ 較差的描述**：
> 「範圍包含所有相關功能。」

**問題**：
- 「相關」無法定義
- 無法識別設計邊界
- 後續易發生範圍蔓延

**✅ 較好的描述**：
> **在範圍內**：
> - 報表資料擷取 API
> - 報表呈現 Web 介面
> - 報表匯出（PDF、Excel）
>
> **明確不在範圍內**：
> - 資料來源系統的修改
> - 報表資料的手動編輯功能
> - 行動裝置版本

**優點**：
- 明確列出包含項目
- 明確列出排除項目
- 可作為 Gate 3 驗收依據

---

### 範例 3：急迫性與驅動因素

**❌ 較差的描述**：
> 「越快越好。」

**問題**：
- 無法判定真實優先順序
- 無法評估延遲的影響
- 易造成資源錯置

**✅ 較好的描述**：
> 「此系統需於 2026 年 Q2 上線，因為：
> 1. 與 A 客戶的合約要求 2026-06-30 前提供此功能
> 2. 若無法如期交付，將面臨違約金 XXX 萬元
> 3. 此外，目前手動作業每月耗費 20 人日，上線後可節省此成本」

**優點**：
- 明確的時程限制來源（合約）
- 可量化的延遲後果（違約金）
- 額外的價值說明（成本節省）

## Readiness Self-Assessment Checklist {#appE-checklist}

Design Requesting Function 可使用以下清單自我評估需求準備程度（非強制，僅供參考）：

### 基本就緒

- [ ] 我能用 2-3 句話說明這個需求要解決什麼問題
- [ ] 我能說明「如果不做這個設計，會有什麼影響」
- [ ] 我能指出誰是這個需求的最終負責人（Design Requesting Function）

### 範圍就緒

- [ ] 我能列出這個設計應該包含哪些主要功能/模組
- [ ] 我能列出這個設計明確不包含什麼
- [ ] 我能說明這個設計需要與哪些既有系統整合

### 風險資訊就緒

- [ ] 我能說明是否有特殊的安全/合規要求
- [ ] 我能說明已知的技術限制或風險
- [ ] 我能提供相關的參考文件或既有設計

### 時程就緒

- [ ] 我能說明期望的完成時間
- [ ] 我能說明時程限制的來源（合約、法規、商務）
- [ ] 我能說明延遲的具體後果

**評估結果參考**：

- **全部勾選**：需求準備充分，可順利進入 Gate 0 審查
- **大部分勾選**：需求基本就緒，可能需要少量補充
- **少部分勾選**：建議先與系統設計部門進行 Pre-Gate0 釐清

## Common Pitfalls to Avoid {#appE-pitfalls}

### 陷阱 1：「先開始做，邊做邊定義」

**風險**：
- 設計方向可能偏離實際需求
- Gate 3 時可能發生重大驗收爭議
- 責任歸屬不清，難以追溯

**建議**：
- 至少在 Gate 0 前確定核心需求目標
- 可接受部分細節待後續釐清，但核心範圍應穩定

### 陷阱 2：「技術細節由設計團隊決定就好」

**風險**：
- 設計團隊可能做出不符業務期望的技術選擇
- 後續發現「這不是我要的」時，責任歸屬不清

**建議**：
- 說明業務期望與限制，讓設計團隊在正確的框架內做技術決策
- 重大技術選擇仍應與 Design Requesting Function 確認

### 陷阱 3：「時程很趕，先跳過 Gate 0」

**風險**：
- 依主文件規定，跳過 Gate 0 需啟動例外機制
- 例外機制有額外審批與風險承擔要求
- 即使例外核准，仍須補回治理要求

**建議**：
- 評估是否真的無法等待 Gate 0
- 若確實緊急，依主文件例外申請流程辦理
