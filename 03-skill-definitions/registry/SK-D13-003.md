## Description

Develop and deploy AI-assisted tools for automated design review, including rule-based checking of naming conventions and document completeness, pattern matching to detect common design errors and inconsistencies, and consistency verification across multi-document deliverables. AI-assisted review augments but does not replace human design review conducted by qualified personnel (SK-D02-012: Design Review — Detailed; SK-D11-003: Design Change and Interface Control). The tool provides objective, repeatable checking that increases review efficiency and catches systematic errors early, allowing human reviewers to focus on higher-level architectural and safety-critical decisions. This skill bridges design engineering and automation, implementing the quality assurance objectives of SK-D11-006 (Quality Plan Development).

## Inputs

- Design artifacts in standardized formats (requirements specifications, architecture diagrams, datasheets, drawings, interface control documents)
- Design standards and templates (from SK-D11-010: Internal Design Standards Maintenance)
- Rule sets and pattern definitions (naming conventions, completeness criteria, error patterns)
- Design review checklist (from SK-D11-006: Quality Plan Development)
- Historical design review findings and lessons learned
- Tool platform specifications (cloud vs. on-premise, API interfaces, compliance requirements)

## Outputs

- AI-assisted review tool (implemented as script, plugin, or service)
- Rule configuration library:
  - Naming convention rules (zone names, conduit IDs, device tags, document sections)
  - Completeness checks (required fields, cross-references, traceability matrix entries)
  - Consistency checks (nomenclature consistency across documents, matching definitions, aligned data flows)
  - Error pattern detectors (common design mistakes, missing security controls, untraceable requirements)
- Review automation workflow specification (input documents, tool invocation, output report generation)
- User interface/command-line specification for design engineers and reviewers
- Calibration report demonstrating false-positive/false-negative rates on representative design sets
- Tool integration documentation (connection to document management system SK-D09-005, CI/CD pipeline if applicable)
- Training and change management plan for adoption

## Tools

- NLP/ML frameworks (spaCy, transformers, Claude API for semantic analysis)
- Rules engine (Drools, OPA, or custom DSL for design rule execution)
- Document parsing (PDF, Word, Markdown parsers for requirement/diagram extraction)
- Version control and CI/CD platforms (Git, GitHub Actions, GitLab CI for automated review triggering)
- Document management systems (SharePoint, Confluence, or internal DMS integration)
- Visualization tools for presenting review findings (HTML reports, dashboards)

## Roles

- SYS (System Architect) — primary: owns tool requirements, rule set development, integration strategy
- DES (Design Engineer) — execution: applies tool to design artifacts, resolves findings, resubmits for review
- QA (Quality Assurance) — supports: tool validation, calibration, metrics tracking
- SAC (Security Architect) — reviews: security-related rule sets and error detection for completeness

## Dependencies

### Hard Dependencies (Required Before Execution)

- SK-D11-006: Quality Plan Development — must define review checklist and acceptance criteria that the AI tool enforces
- SK-D11-010: Internal Design Standards Maintenance — rule sets must be based on standards and templates library
- Design standards and templates finalized and versioned

### Soft Dependencies (Enhances Effectiveness)

- SK-D02-012: Design Review — Detailed — understanding human review process ensures AI tool complements rather than duplicates effort
- SK-D11-003: Design Change and Interface Control — AI tool can assist in change impact analysis and traceability maintenance
- SK-D09-005: Document Management System — integration with DMS enables seamless review workflow

## Acceptance Criteria

1. Rule set covers all items in SK-D11-006 Quality Plan design review checklist (100% coverage minimum)
2. Tool correctly detects naming convention violations, missing required fields, and broken cross-references on test design sets
3. Pattern matching identifies ≥95% of injected error scenarios (common design mistakes) with false-positive rate <5%
4. Consistency verification confirms traceability matrix completeness and validates data flow diagram alignment with detailed design
5. Tool produces actionable review report: each finding includes rule violated, artifact location, and guidance for remediation
6. Performance meets usability threshold: review execution on typical design package < 5 minutes
7. Integration with document management and/or CI/CD pipeline tested and operational
8. Tool and rules approved by SYS, SAC, and QA; training materials prepared for design engineer adoption

## Estimated Effort

| Level | Effort | Notes |
|-------|--------|-------|
| Junior (< 2 yr) | 25–35 person-days | Includes tool development, rule configuration, validation on design sets; may require learning NLP/ML frameworks |
| Senior (5+ yr) | 15–20 person-days | Same scope; leverages prior automation experience and internal ML infrastructure; can iterate quickly on rule refinement |

Notes: Tool sophistication and rule set complexity drive effort. Initial deployment targets 20–30 design rules; expansion to 50+ rules may require additional 10–15 person-days. False-positive/false-negative calibration may require 3–5 iterations on real design data.
