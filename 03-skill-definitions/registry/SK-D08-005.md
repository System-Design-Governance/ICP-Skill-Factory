## Description

Execute security-specific acceptance tests that verify the implementation status of every applicable Security Requirement (SR) defined in the FR/SR mapping matrix, confirming that the as-built system meets its Security Level Targets (SL-T) per zone. This skill is the definitive SR verification activity: each SR is tested against its expected implementation and marked as Implemented / Planned / N/A with formal sign-off. Under ICP governance (GOV-SD), SR verification is a Gate 3 blocker — the system cannot be handed over unless all applicable SRs are verified. Per BOUNDARY-007, security acceptance testing is a specialized subset of SAT (SK-D08-004) focused exclusively on cybersecurity SR compliance, distinct from general functional SAT and integration SIT. This skill covers security test execution and SR status determination; the security FAT test case design falls under SK-D08-002 ⏳, and the overall SAT coordination falls under SK-D08-004.

## Inputs

- FR/SR mapping matrix (from SK-D01-007) — defines which SRs apply per zone/component
- Security Functional Description Specification (from SK-D09-002 ⏳) — detailed SR implementation descriptions per IEC 62443-2-4 SP mapping
- FAT security test results — baseline verification from factory environment
- Zone/Conduit architecture (from SK-D01-001) — SL-T per zone determines SR applicability
- Endpoint hardening configurations (from SK-D01-019) — expected hardening state to verify
- Account and access control policy (from SK-D01-020) — RBAC implementation to verify
- Malware protection deployment (from SK-D01-023) — antivirus and endpoint protection to verify
- Security test procedure (from SK-D08-001 / SK-D08-003 ⏳) — test cases for security categories
- ID14 exemplar: SR-specific test categories (SR 1.01 through SR 2.05+ cybersecurity test items)

## Outputs

- SR Verification Report:
  - Per-SR status matrix: SR ID, SR description, zone applicability, verification method (test/inspection/review), test result, status (Implemented / Planned / N/A), evidence reference, verifier signature
  - SR coverage summary: total applicable SRs, count per status (Implemented / Planned / N/A), coverage percentage
  - Failed SR register: SRs that did not pass with root cause, compensating control proposal, and risk acceptance path
  - SL-T achievement summary per zone: achieved SL vs. target SL, delta analysis
- Security Test Evidence Package:
  - Per test category: test logs, screenshots, scan results, configuration exports
  - Network scan results (Nmap, Nessus) confirming firewall rules and open ports
  - Access control verification: authentication test results, privilege escalation attempts, RBAC validation
  - Encryption verification: TLS certificate validation, protocol analysis captures
- SR Status Sign-Off Sheet: formal sign-off per SR by Security Engineering Role, confirming verification completeness
- Gate 3 Security Evidence Package: compiled security test results formatted for the 12-item Gate 3 delivery checklist

## Automation Potential

**Level:** Partial

Vulnerability scanning and port verification can be automated (Nessus, Nmap scripted scans). Configuration compliance checking can be partially automated (CIS-CAT benchmarks, custom audit scripts). SR status tracking and report generation can be automated. However, certain SR verification requires manual testing (physical security controls, social engineering resistance), compensating control evaluation requires security expert judgment, and SL-T gap analysis requires contextual risk interpretation.

## Acceptance Criteria

1. 100% of applicable SRs in the FR/SR mapping matrix have a verified status (Implemented / Planned / N/A) with evidence reference — no SR left unverified
2. Each SR verification includes: documented test method, actual result, and evidence (scan report, screenshot, log excerpt, or inspection record)
3. SL-T achievement per zone documented: achieved SL matches or exceeds target SL; any shortfall has documented compensating controls and risk acceptance per GOV-SD hierarchy
4. Failed SRs have root cause analysis, compensating control proposal, and risk acceptance sign-off at the appropriate authority level (PM → PM+Security → PM+Security+Business → Engineering Management)
5. 20% sampling verification completed per GOV-SD Gate 3 requirement: random sample of SR evidence independently re-verified by Design QA Role
6. Gate 3 security evidence package compiled: all SR verification results formatted for inclusion in the 12-item Gate 3 delivery checklist

## Estimated Effort

| Level | Effort | Notes |
|-------|--------|-------|
| Junior (< 2 yr) | 15–20 person-days | Assumes ~60 applicable SRs, single-site; includes scanning, manual testing, evidence capture, and SR status documentation |
| Senior (5+ yr) | 8–12 person-days | Same scope; senior leverages automated scanning pipelines and efficient SR-to-evidence mapping |

Notes: Effort scales with SR count (proportional to zone count × SL level). High-SL zones (SL-3, SL-4) have significantly more applicable SRs. ID14 exemplar shows security test categories covering SR 1.01 through SR 2.05+ for a medium-complexity project. Failed SRs requiring compensating control design add 1–3 person-days per failure. The 20% sampling verification adds ~2 person-days for Design QA Role.
