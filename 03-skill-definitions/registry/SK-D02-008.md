## Description

Design edge computing deployment architecture for operational technology (OT) environments, placing computational resources and data processing logic at or near the data sources (e.g., substations, generation facilities, transmission line sensors, manufacturing floors) to enable real-time processing with reduced latency and dependency on centralized cloud or data center infrastructure. Edge computing design encompasses edge node placement strategy, local processing and data filtering/aggregation logic, edge-to-cloud communication protocols and security, and integration with existing Supervisory Control and Data Acquisition (SCADA) systems, Energy Management Systems (EMS), or other industrial control systems. The architecture must address data sovereignty, bandwidth constraints of network links to central facilities, failure modes when edge-to-cloud communication is unavailable (offline resilience), and security implications of distributed processing logic across multiple sites. This skill outputs specifications that feed SK-D02-006 (High-Availability Architecture Design) with edge node redundancy requirements, SK-D02-003 (Interface Control Document Development) with edge-to-cloud interface specifications, and ensure integration with SK-D01-001 (Zone/Conduit Architecture Design) to define security zones for edge nodes and guard against unauthorized edge-to-cloud data exfiltration.

## Inputs

- System Architecture Design (from SK-D02-001: System Partitioning and Functional Architecture)
- Zone/Conduit Architecture Design (from SK-D01-001: Zone/Conduit Architecture Design)
- Data Flow Diagram (from SK-D02-004 ⏳: Data Flow Diagram Development)
- Real-Time Processing Requirements (latency targets, throughput requirements, processing complexity per data stream)
- Network Topology and Available Bandwidth (site locations, network link capacities, latency between sites and central facility)
- Existing SCADA/EMS System Architecture (legacy system constraints, protocol compatibility requirements)
- Data Sovereignty and Privacy Requirements (data retention policies, where data can reside, aggregation rules)
- Threat and Risk Assessment (from SK-D01-006 ⏳: Threat and Risk Assessment — Preliminary; data exfiltration risks, edge node compromise scenarios)

## Outputs

- Edge Computing Architecture Diagram (D2 / Visio format)
  - Physical site layout showing edge node locations
  - Logical architecture showing edge-to-cloud communication paths
  - Data processing pipeline showing filtering/aggregation at edge and central facility
- Edge Node Specification Document
  - Edge node placement strategy: which data sources have co-located edge nodes vs. regional aggregation points
  - Edge node hardware and software requirements (compute, storage, OS, container runtime)
  - Local processing and filtering logic per edge node (pseudo-code or detailed specifications)
  - Data buffering and offline resilience procedures (how much local storage, retention policy, synchronization when connectivity restored)
- Edge-to-Cloud Communication Specification
  - Communication protocols (MQTT, OPC UA, REST, custom protocols)
  - Data aggregation and sampling rates (raw sensor frequency vs. aggregated frequency sent to cloud)
  - Message format and size estimates
  - Bandwidth requirements and impact on existing network utilization
- Edge Node Security Architecture
  - Authentication and authorization for edge node management
  - Encryption requirements for edge-to-cloud communication
  - Data classification per edge-to-cloud interface
  - Access controls and credential management for edge node operators
- Integration Checklist (compatibility with SCADA/EMS APIs, data model mappings, legacy protocol bridges)
- Bandwidth and Cost Analysis (network costs of edge data transmission, comparison with centralized processing)
- Offline Resilience Procedures (procedures when edge-to-cloud communication is unavailable, synchronization logic when connectivity restored)

- STC (Security Technical Consultant) — reviews edge node security implications, edge-to-cloud communication security

## Acceptance Criteria

1. Edge node placement strategy is justified: for each edge node location, document why edge processing is beneficial (latency reduction, bandwidth savings, or offline resilience requirement)
2. Local processing logic is specified to sufficient detail that it can be implemented: pseudo-code, data transformation rules, filtering thresholds, aggregation windows
3. Edge-to-cloud communication is fully specified: protocol, message format, frequency, bandwidth requirements, and impact on existing network
4. All edge-to-cloud interfaces have been added to the Zone/Conduit Architecture or explicitly approved as exceptions
5. Edge node security architecture is complete: authentication/authorization mechanisms, encryption requirements, data classification per interface, credential management procedures
6. Offline resilience procedures are documented: local data buffering capacity, retention policies, synchronization logic when connectivity restored
7. Integration with existing SCADA/EMS systems is validated: compatibility with legacy APIs, data model mappings, protocol bridges specified
8. Edge computing architecture has been reviewed and approved by SYS, STC, INF, and PE/O
