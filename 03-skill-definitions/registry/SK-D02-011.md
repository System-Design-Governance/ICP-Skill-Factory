## Description

Develop the Simple Network Diagram (SND) for the SuC, providing an overview-level visualization of the network topology using the Purdue Enterprise Reference Architecture (PERA) level model as the organizing framework. The SND maps all major system components to their Purdue levels (Level 0 through Level 5 / DMZ), shows physical and logical connectivity between network segments, and delineates the SuC boundary. Under ICP governance (GOV-SD), the SND is a Gate 1 input that establishes the network architecture baseline for subsequent security design. This skill covers topology-level network diagramming; the detailed data flow analysis that maps communication paths over this topology falls under SK-D02-004, and the detailed OT network topology design with redundancy and routing falls under SK-D02-001 ⏳.

## Inputs

- Asset inventory with device types and network connectivity (from SK-D01-005)
- Customer/employer network architecture documentation (existing site drawings for brownfield)
- System requirements specification — system components and their functional roles
- Purdue Enterprise Reference Architecture (PERA) model reference
- Zone/Conduit architecture concept (draft, from SK-D01-001 or SK-D14-013 ⏳) — zone boundaries overlay the network topology
- SuC scope definition (from project requirements / Gate 0 decision package)

## Outputs

- Simple Network Diagram (per ID10 exemplar format):
  - Purdue level layout: Level 0 (Process), Level 1 (Basic Control), Level 2 (Area Supervisory), Level 3 (Site Operations), Level 3.5/DMZ, Level 4 (Enterprise), Level 5 (External)
  - All major network components placed at appropriate Purdue levels (PLCs/RTUs at L1, HMI/SCADA servers at L2/L3, engineering workstations, historians, firewalls, switches, routers)
  - Network segment identification with IP subnet assignments where known
  - SuC boundary clearly delineated (what is in-scope for security design vs. external/customer responsibility)
  - Key interconnection points between Purdue levels (firewalls, DMZ connections, air gaps)
  - Legend with component symbols and network link types
- Purdue Level Assignment Table: per-device mapping to Purdue level with justification
- SuC Boundary Definition Document: list of in-scope vs. out-of-scope systems with responsibility assignment

## Automation Potential

**Level:** Partial

Network topology discovery in brownfield environments can be partially automated (SNMP walk, Nmap scanning, switch MAC/ARP table extraction). Diagram generation from structured device inventories can be automated (D2/Mermaid scripting with Purdue level layout templates). However, Purdue level assignment requires engineering judgment (device function determines level, not just network location), SuC boundary definition is a project scoping decision requiring stakeholder input, and diagram layout optimization for readability requires human design sense.

## Acceptance Criteria

1. SND covers 100% of devices in the approved asset inventory with each device placed at the correct Purdue level
2. Purdue level assignments documented with justification per device; no devices remain unassigned
3. SuC boundary clearly delineated on diagram with in-scope/out-of-scope systems explicitly identified
4. All inter-level connectivity shown with network link types (Ethernet, serial, wireless) and key infrastructure (firewalls, switches, DMZ) identified
5. Network segments labeled with IP subnet assignments where known; segment isolation points (firewalls, air gaps) clearly marked
6. SND reviewed by SYS and SAC; accepted as Gate 1 input (per GOV-SD)

## Estimated Effort

| Level | Effort | Notes |
|-------|--------|-------|
| Junior (< 2 yr) | 4–6 person-days | Assumes single-site, ~50 assets, greenfield with clear requirements; includes Purdue level analysis, diagram production, and boundary documentation |
| Senior (5+ yr) | 2–3 person-days | Same scope; senior leverages Purdue model familiarity and template-based diagramming |

Notes: Brownfield projects with undocumented networks may require 1.5–2× effort for discovery. Multi-site projects require one SND per site plus an inter-site connectivity overlay. ID10 exemplar shows a medium-complexity SCADA project SND with 5 Purdue levels — this serves as the baseline reference. Complex projects with multiple control system vendors may require 1.5× effort for integration point documentation.
