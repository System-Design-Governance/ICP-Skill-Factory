## Description

Design and implement automated workflows that chain multiple engineering skills into repeatable pipelines, enabling complex multi-step processes to execute with minimal manual intervention. Examples include: automated asset inventory ingestion → threat and risk assessment → zone/conduit generation; or point list → database configuration → HMI screen generation. This skill encompasses workflow orchestration (defining skill dependencies, sequencing, branching), comprehensive input/output validation to ensure interface contract compliance between chained skills, and robust error handling with logging and rollback procedures. Skill automation reduces manual effort, improves consistency, and accelerates delivery on repetitive design and implementation tasks. Implementation platforms may range from simple Python scripts with validation logic to enterprise workflow engines (Apache Airflow, cloud Functions-as-a-Service).

## Inputs

- Skill definitions and interface contracts (inputs/outputs for each skill to be chained)
- Example or baseline execution data (input test sets, expected outputs)
- Requirements for the specific pipeline (e.g., "asset inventory to zone/conduit" or "point list to HMI")
- Validation rules (data schemas, acceptable value ranges, required fields)
- Error handling requirements (retry logic, notification triggers, manual escalation criteria)
- Orchestration platform specifications (on-premise, cloud, latency/throughput constraints)
- Change management and rollback procedures for pipeline failures

## Outputs

- Orchestration workflow definition (DAG/pipeline configuration, skill invocation order, dependencies)
- Input/output validation schema and implementation (JSON Schema, Pydantic models, or equivalent)
- Workflow execution engine/service (script, containerized function, or cloud orchestration job)
- Error handling and logging implementation:
  - Exception catching and classification
  - Detailed event logging for audit and diagnosis
  - Rollback procedures (data cleanup, state reversal if pipeline fails mid-execution)
  - Alerting/notification on pipeline failures
- Pipeline execution testing report (success/failure scenarios, latency benchmarks, resource utilization)
- Documentation (pipeline architecture, usage guide, troubleshooting procedures)
- Integration specifications for human review/approval checkpoints (if pipeline outputs require sign-off before downstream execution)

## Dependencies

### Hard Dependencies (Required Before Execution)

- Skill definitions and interface contracts finalized for all skills to be chained — cannot orchestrate undefined or volatile interfaces
- Input data format and validation schema defined — orchestration requires data contract compliance
- Execution platform selected and available — choice of platform (cloud, on-premise, serverless) drives implementation approach

### Soft Dependencies (Enhances Effectiveness)

- SK-D09-005: Document Management System — DMS integration enables seamless access to input documents and storage of outputs
- SK-D13-003: AI-Assisted Design Review — review automation can be integrated into post-execution validation step
- SK-D11-006: Quality Plan Development — quality gates and acceptance criteria drive pipeline error handling and approval checkpoints

## Acceptance Criteria

1. Workflow DAG/pipeline configuration documents all skill dependencies and execution order; no circular dependencies or unvalidated state transitions
2. Input validation detects 100% of schema violations (missing required fields, invalid data types, out-of-range values) before skill execution
3. Output validation confirms each skill's outputs match expected schema before passing to next skill
4. Pipeline execution latency meets specification for target use case (e.g., < 30 minutes for asset inventory → zone/conduit generation)
5. Error handling tested: exceptions caught, logged, and escalated correctly; rollback procedures validated on multi-step failures
6. Pipeline successfully executes end-to-end on representative input sets (minimum 3 distinct test cases per scenario)
7. Monitoring and alerting functional: pipeline failures trigger notifications within acceptable time window
8. Code review completed and approved by SYS, DES, and QA; pipeline operations runbook prepared
