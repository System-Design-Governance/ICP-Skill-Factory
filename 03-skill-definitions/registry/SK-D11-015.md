## Description

Integrate cybersecurity requirements into procurement processes per organizational procurement security procedure (ID22 Tier 3 standard), ensuring that vendor selection, purchase orders (POs), product specifications, and supply chain management include appropriate security specifications, evaluation criteria, and contractual clauses that align with technical security risk assessments and security level requirements. This skill operationalizes the connection between security architecture (D01 domain skills, which define what technical controls are needed) and procurement execution (which acquires products and services that meet those requirements). The scope includes: translating security control requirements into vendor specifications and procurement language, defining vendor security evaluation criteria (certifications, audit results, security claims), specifying security requirements in RFQ/RFP documents, incorporating security acceptance criteria into purchase orders, establishing vendor onboarding security assessments, and maintaining compliance with procurement security procedure throughout the vendor lifecycle. This skill works in partnership with SK-D01-024 (Vendor Security Risk Assessment), which provides the technical security assessment of a specific vendor or product; this skill ensures that those assessments are systematically incorporated into procurement governance. The boundary with D01 technical skills is clear: D01 determines what security controls are required; D11-015 ensures those requirements are reflected in vendor selection and contractual terms.

## Inputs

- Security architecture and security level requirements from D01 design (specifically: Zone/Conduit Architecture SK-D01-001, Network Security Device Requirements SK-D01-007 ⏳, etc.)
- Procurement Security Procedure ID22 (organizational standard for security in vendor selection, RFQ/RFP requirements, vendor agreements, supply chain risk management)
- Preliminary vendor list and product options (identified during architectural design phase)
- Risk assessment findings relevant to supply chain and third-party components (from SK-D01-006 ⏳)
- Security control requirements and specifications (from detailed design, D01 domain skills)
- Vendor certifications and security claims documentation (ISO 27001, IEC 62443, NIST compliance, etc.)
- Legacy vendor agreements or frameworks (for consistency and renegotiation planning)
- Procurement process templates and organizational contracting standards

## Outputs

- **Security Requirements Specification for Procurement (SRSP)**: document translating security control requirements into procurement language, including:
  - Required vendor certifications/attestations (ISO 27001, IEC 62443 Curve certification, NIST compliance, etc.)
  - Technical security specifications for products (e.g., encryption algorithms, key management, logging capabilities, FIPS compliance)
  - Vendor security evaluation criteria and scoring matrix (security features, audit results, industry reputation, response time to vulnerabilities)
  - Supply chain integrity requirements (secure delivery, tamper detection, software bill of materials (SBOM) requirements)
  - Support and vulnerability response SLA expectations
- **RFQ/RFP Security Appendix**: questionnaire and requirements document provided to vendors, clearly specifying:
  - Security features required for the solution
  - Evaluation scoring criteria and pass/fail gates
  - Mandatory certifications or compliance evidence
  - Data handling and confidentiality terms
  - Post-sale security obligations (vulnerability notification, patch support, incident response)
- **Security Requirements in Purchase Order (PO) Terms**: contractual clauses specifying:
  - Product security features and specifications as acceptance criteria
  - Vendor security obligations (certifications, SLAs for security updates, incident notification)
  - Acceptance testing protocols (including security verification tests)
  - Indemnification and liability for security failures
  - Compliance with ID22 and organizational security policy
- **Vendor Security Assessment Checklist**: evaluation form for assessing vendor security maturity, to be completed prior to award (input to SK-D01-024 technical assessment)
- **Vendor Security Onboarding Package**: standardized security requirements for all new vendors, including:
  - Access control and authentication requirements for vendors to organizational systems
  - Data handling and confidentiality agreements
  - Incident reporting and vulnerability disclosure procedures
  - Audit and compliance verification schedule
- **Procurement Security Compliance Monitoring Log**: tracking of vendor compliance with security requirements throughout contract lifecycle, including audit findings, corrective actions, and renewal decisions

## Standards

- IEC 62443-2-1 §5.2.1 (Supply Chain Security) and §7.2 (Third-Party Risk Management) — framework for managing security throughout supply chain
- IEC 62443-4-1 §11.2: Vendor Requirements — specific controls for managing vendor security risks
- ID22 (Tier 3 Procurement Security Procedure) — organizational standard for security in vendor selection, contracts, and supply chain
- ISO/IEC 27001:2022 Annex A.6.5: Supplier Relations — requirements for managing security in vendor relationships
- NIST SP 800-53 SA-3 (System Development Life Cycle) and SA-12 (Supply Chain Risk Management) — supplementary guidance on integrating security into procurement

## IEC 62443 Lifecycle Stages

- R0 (Concept) — identify high-level vendor categories and security expectations
- R1 (Security Requirements) — develop SRSP and RFQ/RFP security appendix based on security requirements
- R2 (Detailed Design) — refine technical security specifications in POs and vendor contracts as design details are finalized
- R3 (Implementation) — monitor vendor compliance with security requirements during development and delivery; incorporate security testing into acceptance
- R5 (Operational Maintenance) — maintain vendor security compliance, manage vulnerability notifications, plan renewal negotiations with updated security requirements

## Roles

- **Procurement Manager / Procurement Officer** — primary: leads RFQ/RFP process, negotiates vendor contracts, manages vendor relationships
- **Security Architect (SAC) / Security Requirements Owner** — defines security specifications and evaluation criteria, reviews vendor security claims
- **Legal / Contracts** — incorporates security clauses into vendor agreements, ensures compliance with organizational policies
- **Vendor Management / Third-Party Risk** — assesses vendor security maturity, monitors ongoing compliance, coordinates remediation
- **Technical Evaluator (STC / Design Engineer)** — participates in vendor security evaluation, defines technical acceptance criteria
- **Project Manager / SuC Owner** — ensures vendor security requirements align with project schedule and scope

## Dependencies

### Hard Dependencies (Required Before Execution)

- D01 Security Architecture and Requirements (from SK-D01-001 Zone/Conduit Architecture, SK-D01-007 ⏳ Network Security Device Requirements, etc.) — must know what security controls are required before specifying vendor requirements
- ID22: Procurement Security Procedure (Tier 3) — organizational standard that this skill operationalizes
- SK-D11-001: Organizational Governance Structure — authorities and roles for procurement decisions

### Soft Dependencies (Enhances Effectiveness)

- SK-D01-024: Vendor Security Risk Assessment — provides technical security evaluation of specific vendors; this skill ensures those assessments feed into procurement governance
- SK-D01-006 ⏳: Threat and Risk Assessment — risk findings inform criticality of vendor security requirements and evaluation weighting
- SK-D14-001: Documentation and Change Management — vendor documentation (SBOM, security claims, compliance certificates) requires version control and change tracking
- SK-D11-014: Contracts and Liability — legal framework for vendor agreements complements security procurement terms

## Acceptance Criteria

1. SRSP document maps 100% of identified security control requirements from D01 architecture to procurement language (no orphaned security controls)
2. RFQ/RFP Security Appendix includes explicit evaluation scoring criteria for each security requirement, with pass/fail gates clearly stated (no subjective evaluation such as "adequate security")
3. All vendor evaluation criteria are observable and measurable (e.g., "ISO 27001 certification," "FIPS 140-2 algorithm list," "< 30-day CVSS 9+ patch response SLA") rather than subjective
4. 100% of executed purchase orders for security-relevant products/services reference security specifications from SRSP or contain equivalent security requirements
5. Vendor Security Assessment Checklist has been completed for all vendors in scope prior to contract signature
6. All vendor security agreements include explicit confidentiality, data handling, incident notification, and compliance verification terms aligned with ID22
7. Vendor Security Compliance Monitoring Log is current within 30 days, with documented evidence of at least one verification activity per vendor per year

## Estimated Effort

| Level | Effort | Notes |
|-------|--------|-------|
| Junior (< 2 yr) | 6–10 person-days | Developing SRSP and RFQ/RFP appendix; assumes 3–5 primary vendor categories; requires SAC/legal review cycles |
| Senior (5+ yr) | 3–5 person-days | Leverages procurement security templates and vendor scorecard patterns |

Notes: Large vendor base (>10 vendors) or highly complex security specifications (e.g., cryptographic hardware or secure supply chain) may require 1.5–2× effort and external procurement specialist involvement. Ongoing vendor monitoring and compliance tracking averages 1–2 person-days per month; annual contract renewals typically require 2–4 person-days per vendor.
