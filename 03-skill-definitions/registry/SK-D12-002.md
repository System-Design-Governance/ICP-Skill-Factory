## Description

Develop protocol parsers and drivers for collecting data from operational technology (OT) devices using industrial protocols (Modbus, IEC 61850 MMS, DNP3, OPC UA). This skill encompasses protocol-specific message parsing, data extraction from device registers/points, comprehensive error handling, and security validation (including rejection of malformed packets and protocol anomalies). The developed parsers feed directly into SK-D12-001 (Data Acquisition Architecture), providing the protocol translation layer between heterogeneous OT devices and the centralized data acquisition framework. This skill is execution-focused, implementing the architecture-level data collection strategy defined in SK-D12-001.

## Inputs

- Data Acquisition Architecture specification (from SK-D12-001)
- Device inventory and communication specifications (protocol type, register maps, point lists)
- Protocol standards documentation (Modbus TCP/RTU, IEC 61850 MMS, DNP3, OPC UA specifications)
- Security requirements from design review (data validation rules, allowed message types, anomaly thresholds)
- System environment specifications (hardware, operating system, runtime constraints)
- Test data sets and device communication logs for validation

## Outputs

- Protocol parser/driver source code (C, Python, or language per architecture specification)
- Parser configuration files (register maps, message templates, protocol-specific parameters)
- Error handling and logging specifications (malformed packet detection, security event logging)
- Protocol-specific security validation rules and implementation code
- Unit test suite with protocol-level edge cases (oversized packets, invalid message types, sequence violations)
- Integration test results demonstrating parser functioning with live/simulated devices
- Parser documentation (API reference, configuration guide, troubleshooting)

## Tools

- Programming language environment (C, Python, Java per system requirements)
- Protocol analysis tools (Wireshark, protocol sniffers for validation and debugging)
- Unit testing frameworks (pytest, googletest, or equivalent per language)
- Protocol reference implementations and simulators (for testing without live hardware)
- Code review and version control (Git)

- STC (Security Technical Consultant) — supports: security validation rules, malformed packet detection

- SYS (System Architect) — reviews: parser interface compliance with SK-D12-001 architecture

## Dependencies

### Hard Dependencies (Required Before Execution)

- SK-D12-001: Data Acquisition Architecture — parser design must conform to the specified data acquisition framework and interface contracts
- Device inventory and communication specifications — cannot develop protocol-specific parsers without knowing target devices and protocols

### Soft Dependencies (Enhances Effectiveness)

- SK-D01-001: Zone/Conduit Architecture Design — understanding network segmentation informs parser security boundaries
- SK-D02-001: OT Network Topology Design — network topology may affect protocol selection and parser robustness requirements
- SK-D05-008: OT Device Configuration Design — understanding device capabilities and constraints improves parser completeness

## Acceptance Criteria

1. Parser correctly interprets all specified industrial protocol message formats (Modbus, IEC 61850, DNP3, OPC UA as applicable)
2. Data extraction from device registers/points matches documented specifications with 100% accuracy on test data sets
3. All malformed packet types (oversized, invalid message structure, out-of-sequence, CRC/checksum failures) are detected and logged without crashing
4. Security validation rules reject unauthorized message types and anomalous communication patterns as defined in design review
5. Error handling and logging meet completeness criteria: every error condition is logged with sufficient context for diagnosis
6. Unit test coverage ≥ 90% for parser code; integration testing confirms successful data collection with 3+ different device instances per protocol type
7. Parser performance meets architecture specification: message latency < specification, no memory leaks under sustained load
8. Code review completed and approved by both DES and STC
