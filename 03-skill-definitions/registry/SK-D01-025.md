## Description

Analyze and manage Software Bill of Materials (SBOM) for OT/ICS components, including SBOM generation and collection from vendors, vulnerability matching against CVE databases, component lifecycle tracking, and license compliance verification. This skill ensures supply chain security transparency and identifies known vulnerabilities in third-party software components early in the system lifecycle. It covers SBOM acquisition from hardware and software vendors (in CycloneDX, SPDX, or proprietary formats), normalization and consolidation of SBOMs across the system, systematic matching of identified components against NIST NVD and other CVE databases, tracking of component versions through the system lifecycle (greenfield and brownfield), verification of open-source license compatibility with organizational policy, and continuous monitoring for disclosed vulnerabilities in deployed components. The output feeds into SK-D01-026 (Third-Party Component Verification) for risk assessment and mitigation planning. The boundary of this skill is SBOM analysis and tracking; vulnerability risk assessment falls under SK-D01-026, and patch management procedures fall under D08.2.

## Inputs

- Component list from system design (hardware BOM, firmware versions, operating system versions, application software package list from SK-D02-001 ⏳ and detailed design documents)
- SBOMs from hardware/software vendors (CycloneDX, SPDX, or proprietary formats; sourced through vendor datasheets, supply agreements, or vendor portals)
- NIST National Vulnerability Database (NVD) access and/or commercial CVE database subscriptions (NVD, CVE Details, Qualys, Rapid7, etc.)
- Existing component license database (if available from procurement/legal)
- System architecture and component deployment mapping (which components run in which zones, criticality levels)
- Change management records for component updates or substitutions during project lifecycle

## Outputs

- Consolidated SBOM in standardized format (CycloneDX or SPDX XML): component name, version, vendor, component type, hash/checksum for verification
- SBOM Mapping Table: system component → SBOM entry → version → deployment location (zone/system) → criticality level
- CVE Matching Report: component name, version, identified CVEs (CVE ID, base score, affected versions), publication date, criticality classification per component function
- Component Lifecycle Tracking Register: component ID, introduction date, version history, replacement dates, end-of-life status
- License Compliance Inventory: component name, version, license type (proprietary, Apache 2.0, GPL v3, etc.), organizational license policy compatibility (approved/flagged/requires legal review)
- Third-Party Component Risk Summary (input to SK-D01-026): total components analyzed, vulnerable component count, critical/high/medium/low vulnerability distribution
- Continuous Monitoring Plan: schedule for CVE database checks, alert thresholds for new vulnerabilities, escalation procedure

## Tools

- SBOM generation tools (Cyclone DX Maven Plugin, SPDX License List tools, CycloneDx.Net)
- SBOM analysis and management platforms (SPDX tools, Dependency-Track, Black Duck, Synopsys, JFrog Xray)
- CVE databases (NIST NVD API, CVE Details, Qualys, Rapid7, commercial feeds)
- License scanning tools (FOSSA, Black Duck, Synopsys, FOSSID)
- Spreadsheet / structured markdown for SBOM consolidation and lifecycle tracking
- Git / version control for tracking SBOM versions and change history

## Standards

- IEC 62443-4-2: Supply Chain & Lifecycle Requirements — SBOM and component security management
- NTIA "Minimum Elements for Software Transparency": SBOM format and content requirements
- NIST SP 800-53 Rev. 5, SA-3 System Development Life Cycle: supply chain risk management, component tracking
- NIST Cybersecurity Framework (CSF): Govern and Manage supply chain risks
- SPDX (Software Package Data Exchange): SBOM interchange format standard
- CycloneDX: lightweight, XML-based SBOM format optimized for supply chain components

## IEC 62443 Lifecycle Stages

- R3 (Implementation — SBOM collection and analysis during system build)
- R4 (Maintenance and Review — continuous monitoring for new vulnerabilities)

## Roles

- STC (Security Technical Consultant) — primary: acquires SBOMs, performs CVE matching, tracks lifecycle
- Procurement / Supply Chain — sources SBOMs from vendors, manages component version control
- Legal / Compliance — verifies license compliance, manages license risk
- SAC (Security Architect) — reviews component risk summary, feeds into risk assessment
- Development / DevOps — maintains SBOM updates during design changes and patches

## Dependencies

### Hard Dependencies (Required Before Execution)

- SK-D02-001 ⏳: OT Network Topology Design (provides component list) or detailed system design documentation with full bill of materials

### Soft Dependencies (Enhances Effectiveness)

- SK-D01-026: Third-Party Component Verification — component risk assessment uses SBOM analysis as input
- SK-D01-007 ⏳: Threat and Risk Assessment (Detailed) — SBOM supports supply chain risk dimension of overall threat landscape
- D08.2 ⏳: Patch Management Procedures — SBOM provides foundation for patch applicability assessment

## Acceptance Criteria

1. SBOMs collected for 100% of third-party software components and 80%+ of hardware components (some legacy hardware may not have SBOMs)
2. SBOM consolidation is complete: all component versions deployed in the system are mapped to SBOM entries and CVE databases
3. CVE matching is performed against current NVD snapshot; all critical and high-severity CVEs are identified and documented
4. Component lifecycle register accounts for all component introductions, version updates, and end-of-life transitions during project timeline
5. License compliance inventory is complete and reviewed by legal; no unapproved license types are present in the final system
6. Continuous monitoring plan is established with defined check cadence (at least quarterly for deployed systems) and alert procedures
7. SBOM and associated analyses are version-controlled and audit-traceable (change history retained)

## Estimated Effort

| Level | Effort | Notes |
|-------|--------|-------|
| Junior (< 2 yr) | 10–15 person-days | Assumes ~80–120 components, partial SBOM availability from vendors, manual CVE matching; includes license review |
| Senior (5+ yr) | 4–7 person-days | Same scope; senior can leverage automated SBOM tools, batch CVE matching, and established vendor relationships |

Notes: Greenfield projects with good vendor SBOM support may require 5–8 days less effort. Brownfield projects with legacy components and no available SBOMs may require 1.5–2× effort due to reverse engineering and manual component discovery. Highly regulated environments (nuclear, critical infrastructure) may require additional legal and compliance review.
