## Description

Develop and maintain the Data Flow Diagram (DFD) for the SuC, mapping all logical data flows between zones, subsystems, and external interfaces — including protocol identification, data direction, data classification, and communication endpoints. The DFD serves as the foundational input for Zone/Conduit architecture design (SK-D01-001) by making inter-zone communication visible and auditable. Under ICP governance (GOV-SD), the DFD is a Gate 1 mandatory deliverable as it directly feeds the Zone/Conduit diagram. This skill covers data flow identification, numbering, classification, and diagram production; the physical network topology that carries these data flows falls under SK-D02-011, and the conduit security specifications derived from data flows fall under SK-D01-001.

## Inputs

- Asset inventory with system roles and connectivity (from SK-D01-005)
- Customer requirements specification / SOW — expected data exchanges and integration points
- Zone/Conduit architecture (draft or approved, from SK-D01-001) — zone boundaries determine how data flows are grouped into conduits
- Network architecture documentation (existing site diagrams for brownfield)
- Protocol specifications for SCADA/EMS/DERMS communication (IEC 61850, DNP3, Modbus, OPC UA)
- Security Level Targets per zone (from SK-D01-010 ⏳) — SL determines data flow security requirements

## Outputs

- Data Flow Diagram (per ID08 exemplar format):
  - Numbered data flows (DF-001 through DF-nnn) with unique identifiers
  - Per data flow: source system → destination system, protocol, port, direction (unidirectional/bidirectional), data classification (per BOUNDARY-008), purpose/function description
  - Inter-zone flows highlighted with zone boundary crossings explicitly marked
  - External interface flows identified (connections to customer/employer networks, internet, remote access)
- Data Flow Register: tabular inventory of all data flows with attributes (flow ID, source zone, destination zone, protocol, port, direction, classification, conduit ID mapping)
- Data Flow Security Assessment: per-flow evaluation of authentication, encryption, and integrity requirements based on zone SL-T
- Updated Zone/Conduit input: data flows grouped into logical conduits for SK-D01-001 consumption

## Tools

- Microsoft Visio / draw.io (DFD production)
- D2 / Mermaid (technical diagram generation, per ICP arch-diagram skill)

- Wireshark / network traffic analysis tools (brownfield: discovery of undocumented data flows)

## Standards

- IEC 62443-3-2: Security Risk Assessment for System Design — data flow analysis as input to zone/conduit methodology
- IEC 62443-1-1: Terminology, concepts and models — conduit definition includes data flow characterization
- ISO 27001 Annex A.13: Communications Security — network security and information transfer
- GOV-SD: Gate 1 mandatory — DFD must be available before Zone/Conduit architecture design can proceed

## IEC 62443 Lifecycle Stages

- R1 (Security Requirements — initial DFD based on requirements and preliminary architecture)
- R2 (Detailed Design — DFD refined with protocol-level detail after architecture decisions)

## Roles

- SAC (Security Architect) — primary: identifies inter-zone data flows, classifies security requirements per flow
- SYS (System Architect) — provides system communication requirements, validates data flow completeness
- DES (Design Engineer) — documents protocol details, produces DFD diagrams
- STC (Security Technical Consultant) — reviews data flow security assessment against IEC 62443 requirements

## Automation Potential

**Level:** Partial

Data flow discovery in brownfield environments can be partially automated through network traffic capture and analysis (Wireshark, Nmap). Diagram generation from structured data flow registers can be automated (D2/Mermaid scripting). However, data flow identification in greenfield projects requires design interpretation, data classification requires domain judgment, and security requirement assessment per flow requires expert evaluation of threat context.

## Acceptance Criteria

1. DFD covers 100% of identified system-to-system communication paths in the asset inventory; no undocumented data flows remain
2. Every data flow has a unique identifier (DF-nnn) with all required attributes populated: source, destination, protocol, port, direction, classification, purpose
3. All inter-zone data flows explicitly identified with zone boundary crossings marked; each maps to a conduit ID in the Zone/Conduit architecture
4. External interface data flows documented with security classification and authentication/encryption requirements specified
5. Data flow register is traceable: every flow can be linked to an operational requirement or system function
6. DFD reviewed and approved by SAC and SYS prior to Gate 1 submission (per GOV-SD)

Multi-protocol environments (IEC 61850 + DNP3 + Modbus + OPC UA) increase complexity. ID08 exemplar shows 16 numbered inter-zone flows for a medium-complexity SCADA project — this serves as the baseline reference.
