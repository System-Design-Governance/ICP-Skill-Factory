## Description

Design network redundancy architecture for OT networks to meet operational availability requirements while integrating with Zone/Conduit security architecture. This skill encompasses the selection and specification of redundancy mechanisms (Rapid Spanning Tree Protocol [RSTP], ring topology, dual-homing strategies) to eliminate single points of failure in critical network segments, along with the failover timing and recovery procedures. The design must account for security zone boundaries and ensure that redundancy mechanisms do not introduce unintended communication paths that bypass firewall rules or security controls. The boundary of this skill is network redundancy topology and protocol selection; device-level redundancy configuration (e.g., RSTP priority, ring supervisor, port cost tuning) falls under device-level configuration skills (D05.5), and operational failover testing and timing validation fall under commissioning and verification skills (D06.x, D09.x).

## Inputs

- Approved Zone/Conduit Architecture Design with network segmentation policy (from SK-D01-001)
- OT Network Topology Design (physical and logical) with identified critical segments (from SK-D02-001 ⏳)
- Asset Inventory Register with network connectivity and criticality (from SK-D01-005 ⏳)
- Operational Availability Requirements specification: acceptable downtime (RTO), recovery time objective (RPO) per network segment or subsystem
- Firewall architecture and device placement (from SK-D01-003 ⏳: Firewall Rule Planning)
- Existing redundancy infrastructure (if brownfield: current switches, links, protocol configuration)
- Vendor specifications for redundancy protocols and failover timing (switch/router datasheets)

## Outputs

- Network Redundancy Architecture Specification: chosen redundancy topology (RSTP mesh, ring, dual-homing), technology rationale (why RSTP vs. ring topology), failover time requirements (milliseconds/seconds) per critical segment, and integration with security zones
- Redundancy Topology Diagram: network topology with redundant links highlighted, link costs/priorities indicated, root bridge/ring supervisor placement, and clear demarcation of redundancy boundaries (i.e., where RSTP or ring rules apply)
- Failover Timing and Recovery Plan: calculated failover time for each critical redundancy path, convergence validation criteria, and timing test procedure (per commissioning phase)
- VLAN/Zone Compatibility Analysis: verification that redundant links do not create unintended communication paths between security zones, and confirmation that firewall rules remain effective across failover scenarios
- Redundancy Configuration Template: switch/router configuration snippets for RSTP priority, ring supervisor setup, dual-homing interface configuration, and convergence tuning parameters
- Operational Handover Procedure: runbook for network operators describing redundancy monitoring, failover expectations, and manual intervention triggers (if any)

## Roles

- SYS (System Architect) — primary: selects redundancy strategy, defines failover timing, integrates with system architecture
- STC (Security Technical Consultant) — validates VLAN/zone compatibility and ensures redundancy does not circumvent security controls
- DES (Design Engineer) — incorporates redundancy topology into network design drawings
- Network Operations Team — provides operational availability requirements (RTO/RPO) and validates failover procedures
- Vendor / Network Integrator — supplies redundancy protocol expertise and device configuration guidance

## Automation Potential

**Level:** Partial

Failover timing calculations can be partially automated based on device specifications and link parameters. Redundancy topology templates can be generated for standard topologies (RSTP mesh, ring). However, topology selection (RSTP vs. ring vs. dual-homing) and integration with security zones requires manual analysis and trade-off assessment.

## Acceptance Criteria

1. Chosen redundancy mechanism (RSTP, ring, dual-homing) is justified with explicit trade-off analysis (fault tolerance, convergence time, cost, operational complexity)
2. Redundancy topology diagram shows all redundant links with root bridge/ring supervisor placement and link costs/priorities clearly labeled
3. Failover timing requirement is specified for each critical segment (in milliseconds or seconds) with rationale tied to operational availability targets (RTO/RPO)
4. VLAN/zone compatibility analysis confirms that redundant links do not bypass firewall rules or create unintended inter-zone communication
5. Failover recovery procedure is documented with convergence validation criteria and timing test plan for commissioning
6. Configuration template covers all devices involved in redundancy (switches, routers, managed equipment) with vendor-specific syntax
7. Operational handover procedure is prepared for network operations team with runbook and monitoring expectations

## Estimated Effort

| Level | Effort | Notes |
|-------|--------|-------|
| Junior (< 2 yr) | 6–10 person-days | Assumes standard RSTP topology, ~10 redundant links; includes topology design, timing analysis, and configuration template |
| Senior (5+ yr) | 3–6 person-days | Same scope; senior leverages RSTP knowledge and vendor tool familiarity |

Notes: Complex multi-protocol redundancy (RSTP + ring hybrid, customer-specific availability targets) may add 2–4 days. Brownfield environments with legacy redundancy requiring modernization may require 1.5–2× effort. Multi-site redundancy designs (WAN-level) may scale effort linearly per site.
