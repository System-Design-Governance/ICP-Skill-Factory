> **Design Review Checklist Development**: A foundational governance skill that standardizes the verification and validation of engineering deliverables across all design phases. Checklists ensure that completeness, correctness, consistency, and compliance criteria are uniformly applied, reducing design defects and supporting the critical review gates required by IEC 62443.

## Description

Develop standardized design review checklists covering completeness, correctness, consistency, and compliance criteria for engineering deliverables across all design phases (D01, D02, D03, D04, D05, D09–D12). These checklists serve as control gates that feed downstream skills including SK-D02-012 (Architecture Review Facilitation) and SK-D11-003 (Critical Security Design Review). Per ID02 Annex C, specific checklists exist for verification categories C.1 through C.8, covering requirements traceability, design correctness, design-implementation consistency, and compliance with applicable standards. This skill covers the definition, structuring, and maintenance of checklists; the execution of reviews using checklists falls under SK-D02-012 and SK-D11-003.

## Inputs

- IEC 62443 series security requirements (ID01, ID02, ID03, ID04)
- Engineering domain standards and design practices (IEEE Std 1012 for verification/validation; domain-specific codes per D01–D12)
- Project-specific design standards and customer requirements (SOW, design guidelines)
- Identified design defects from prior projects (lessons learned, quality metrics database)
- Regulatory and compliance requirements (NERC CIP, DO-178C, IEC 61508, jurisdiction-specific codes)
- Role definitions and competency requirements (who executes and reviews what)

## Outputs

- Design Review Checklist Template (structured markdown or Excel) with governance-mandated sections
- Domain-specific checklists for each engineering domain (D01, D02, D03, D04, D05, D09, D10, D11, D12)
- Verification category checklists (C.1–C.8 from ID02 Annex C: Requirements Traceability, Completeness, Design Correctness, Design-Implementation Consistency, Security Requirements Implementation, Standards Compliance, and Cross-Domain Consistency)
- Checklist usage guidance document (when to execute, roles, escalation procedures)
- Periodic checklist effectiveness review report (defect escape analysis, process improvement recommendations)

## Standards

- IEC 62443-3-2: Security Risk Assessment for System Design — design review and verification requirements
- IEC 62443-3-3: System Security Requirements and Security Levels — verification gates aligned to design phases
- IEEE Std 1012: Software Verification and Validation — structured V&V framework (applicable to automation system design)
- ISO/IEC/IEEE 42010: System and software engineering — architecture description framework, design review scope
- ID02 Annex C: Design Verification Checklist — primary normative reference for C.1–C.8 categories

## IEC 62443 Lifecycle Stages

- R1 (Security Requirements — initial design review against security requirements)
- R2 (Detailed Design — verification that detailed design implements all security requirements)
- R3 (Implementation — verification of implementation correctness against detailed design)

## Roles

- Head of System Design (GOV-SDP) — primary: owns checklist development and maintenance, approves updates
- SAC (Security Architect) — provides security-specific verification criteria for C.6 (Security Requirements Implementation)
- DES (Design Engineer) — executes checklists, identifies needed updates based on field experience
- Quality Assurance Lead (QA) — analyzes checklist effectiveness, recommends improvements
- Domain Leads (D01–D12) — provide domain-specific verification criteria

## Acceptance Criteria

1. Checklists exist for all engineering domains (D01, D02, D03, D04, D05, D09, D10, D11, D12) and major deliverable types
2. Each checklist explicitly maps to verification categories C.1–C.8 per ID02 Annex C or provides documented exception
3. Checklists include objective, measurable criteria (not vague or qualitative items)
4. Completeness criteria: checklist items cover 100% of identified design verification requirements from IEC 62443 standards
5. Security-specific checklists (C.6) include all security requirements from the approved Security Functional Description Specification (SFDS)
6. Checklist effectiveness has been validated: documented review of defect escape rates; recommended improvements implemented or documented
7. All design and engineering team members have received training on checklist execution and closure procedures
