## Description

Design security alarm rules and correlation logic for detecting cybersecurity threats in OT/ICS environments. This skill defines the abstract alarm categories, trigger conditions, severity classifications, and escalation pathways that guide the operational detection and response to security incidents. It covers identification of key security events (authentication failure, unauthorized access, anomalous traffic patterns, policy violation), specification of alarm hierarchy and severity levels, definition of correlation logic (multi-event patterns, temporal sequences, threshold-based anomalies), and integration with SK-D05-006 (Alarm Hierarchy) for unified alarm management across the system. The output of this skill feeds into SK-D01-014 (SIEM Configuration and Tuning) for implementation in security monitoring platforms. The boundary of this skill is rule design; the operational configuration of the SIEM tool falls under SK-D01-014, and incident response execution falls under D07.4.

## Inputs

- Approved Zone/Conduit Architecture (from SK-D01-001: Zone/Conduit Architecture Design)
- Security Level Targets (SL-T) per zone (from SK-D01-010 ⏳: Security Level Target Assessment)
- Preliminary and Detailed Threat and Risk Assessment reports (from SK-D01-006 ⏳ and SK-D01-007 ⏳)
- Approved network and data flow diagrams (from SK-D02-004 ⏳)
- Asset inventory with system criticality and function (from SK-D01-005 ⏳)
- Security monitoring requirements and KPIs (from SK-D01-009 ⏳: Security Monitoring Plan Development)
- Incident history and threat intelligence from OT domain (if available from customer or industry databases)
- System operational constraints and alert fatigue thresholds

## Outputs

- Alarm Category Specification: list of security events to monitor (authentication events, access control violations, anomalous data flows, policy breaches, hardware faults, configuration changes)
- Alarm Hierarchy and Severity Classification Table: alarm ID, alarm name, trigger condition (English description), severity level (Critical/High/Medium/Low), affected asset/zone, escalation pathway
- Correlation Rule Specification: rule ID, rule name, component events (alarm triggers that feed the rule), time window, aggregation logic, resulting alarm severity, business impact
- Anomaly Detection Baseline Specification: metric (e.g., event rate, data volume, user behavior), baseline value (derived from historical data or operational norms), anomaly threshold, detection method
- Alarm-to-Incident Mapping: which alarms trigger incident classification per D07.3 (Incident Severity Classification)
- Integration mapping with SK-D05-006 (Alarm Hierarchy) documenting how functional alarms and security alarms coexist in the unified alarm system
- Security Monitoring Plan section (narrative) documenting alarm coverage against top threats from risk assessment

## Acceptance Criteria

1. Every material threat from the detailed risk assessment has at least one corresponding alarm rule
2. Alarm hierarchy covers at least 5 security event categories: authentication, authorization, anomalous traffic, policy violation, and system fault
3. Severity levels are consistently applied: Critical (immediate escalation, business impact), High (investigation required within SLA), Medium/Low (logged but routine review)
4. Correlation rules are technically feasible (e.g., multi-event correlation windows are realistic given system latency)
5. Alarm categories are validated by SAC and OT operations stakeholders; feedback is documented
6. Mapping to SK-D05-006 is complete; no naming conflicts or ambiguities in unified alarm system
7. Design includes documented rationale for threshold values and anomaly detection baselines (tied to historical data or operational norms)
