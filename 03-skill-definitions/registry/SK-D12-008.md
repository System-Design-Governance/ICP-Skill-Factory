## Description

This skill encompasses the design of data access policies that define who can read, write, or modify operational data across OT/IT systems, implementing the principle of least privilege through role-based access control (RBAC). Data access policies translate abstract data classification (SK-D01-033: Data Classification Policy) into concrete access rules: by role (operator, administrator, analyst, auditor), by data type/element, and by access mode (read-only, read-write, execute). Policies coordinate with account and access control design (SK-D01-020: Account and Access Control) to ensure that authentication and authorization mechanisms support data access restrictions. The scope includes policy development, RBAC matrix definition, enforcement mechanisms (database-level, application-level, or both), audit and logging requirements to track data access, and support for secure data sharing between OT and IT systems while maintaining security boundaries.

## Inputs

- Data Classification Policy (from SK-D01-033: Data Classification Policy) — data sensitivity levels drive access control decisions
- Account and Access Control Policy (from SK-D01-020: Account and Access Control) — role definitions and account management procedures
- Information Asset Classification (from SK-D11-013: Information Asset Classification) — asset sensitivity and ownership
- Data Acquisition Architecture (from SK-D12-001: Data Acquisition Architecture) — identifies data types and systems storing data
- Data Dictionary (from SK-D12-007: Data Dictionary Development) — detailed data element definitions
- Time-Series Database Configuration (from SK-D12-003: Time-Series Database Selection) — database RBAC capabilities
- Regulatory and Compliance Requirements: data protection regulations (GDPR, HIPAA) and industry-specific mandates
- Business Requirements for Data Sharing: cross-functional use cases requiring controlled data access

## Outputs

- Data Access Policy Document: policy statement specifying data access principles, role definitions, access control requirements, and audit requirements
- Role and Permission Matrix: table defining which roles have read/write/execute access to which data categories/elements; matrix includes justification for each access grant
- Access Control Specifications: technical specifications for implementing RBAC in database, application, and data platform layers
- Enforcement Mechanism Design: architectural design showing where access control is enforced (database-level triggers, application-level authorization, API gateway)
- Audit and Logging Plan: requirements for logging all data access attempts, what information must be logged, log retention, and access monitoring procedures
- Data Sharing Agreement Templates: templates for documenting controlled data sharing between OT and IT systems or external entities
- Access Request and Approval Process: procedure for requesting and approving access changes; workflow and approval authority levels

## IEC 62443 Lifecycle Stages

- R2 (Detailed Design) — data access policy is designed and RBAC matrix is defined
- R3 (Implementation) — access controls are configured in databases, applications, and data platforms
- R4 (Commissioning/Startup) — access controls are tested and verified during commissioning; access provisioning procedures are validated
- R5 (Operations) — access control enforcement is maintained; access reviews are conducted periodically; new user access is provisioned per approved process

## Acceptance Criteria

1. Data Access Policy document clearly states access control principles (least privilege, segregation of duties) and defines role-based access model; at least 5 distinct roles (operator, administrator, analyst, auditor, manager) are defined with responsibilities and access needs
2. Role and Permission Matrix comprehensively maps roles to data categories/elements with access modes (read, write, execute, delete); every row is justified (e.g., "operators need read access to real-time sensor data for operational monitoring")
3. Access control specifications detail whether enforcement is at database layer, application layer, or both; for database-level controls, specific GRANT/DENY statements or equivalent are documented; for application controls, authorization framework (OAuth, SAML) is specified
4. Audit and Logging Plan requires logging of all data access attempts with: user identity, data accessed, access mode (read/write), timestamp, access grant/denial status; log retention is at least 12 months
5. Data Sharing Agreements (if applicable) document controlled access for external parties or cross-functional teams; data minimization principles are evident (only minimum necessary data is shared)
6. Access policy has been reviewed and approved by Security Architect, Compliance Officer, and Data Governance Officer; technical feasibility in selected database and application platforms has been confirmed
