## Description

Define Recovery Time Objective (RTO) and Recovery Point Objective (RPO) targets for each system tier and critical service, and design the backup and recovery architecture to meet those targets. RTO specifies the maximum tolerable time interval between a failure event and the restoration of service to an acceptable operational state; RPO specifies the maximum tolerable amount of data loss, expressed in time (e.g., "no more than 1 hour of data loss"). RTO/RPO targets drive decisions across SK-D02-006 (High-Availability Architecture Design), SK-D01-022 (Backup and Restore Planning), and incident response capabilities. This skill requires understanding of business criticality, cost-benefit trade-offs between backup strategies (full backup, incremental, differential, snapshot-based), backup frequency, storage requirements, and recovery validation procedures. RTO/RPO planning integrates security considerations: backup repositories must be isolated from production systems and protected at a security level commensurate with the data being backed up. This skill outputs specifications and target matrices that directly feed SK-D02-006 (HA Architecture) architectural feasibility assessments and SK-D01-022 (Backup and Restore) implementation planning.

## Inputs

- System Architecture Design (from SK-D02-001: System Partitioning and Functional Architecture)
- Business Continuity Requirements (customer SOW, SLAs, regulatory mandates)
- Service Criticality and Priority Rankings (business impact assessment per service tier)
- Threat and Risk Assessment (from SK-D01-006 ⏳: Threat and Risk Assessment — Preliminary; potential failure scenarios and probabilities)
- Infrastructure Constraints (storage capacity, network bandwidth for backup operations, backup site availability)
- Zone/Conduit Architecture (from SK-D01-001: Zone/Conduit Architecture Design; backup communication paths and storage security zones)
- Cost and Schedule Constraints (budget for backup infrastructure, time available for implementation)

## Outputs

- RTO/RPO Target Matrix (table format)
  - Rows: each system tier / critical service
  - Columns: Service Name, Business Criticality (H/M/L), RTO Target (hours/minutes), RPO Target (hours/minutes), Rationale
- Recovery Procedures Specification (high-level steps per service tier)
  - Full system recovery procedure
  - Service-level partial recovery procedures
  - Data corruption recovery procedures
- Backup Strategy Selection Document
  - Backup frequency (daily, hourly, real-time replication)
  - Backup type per system tier (full, incremental, differential, snapshot-based, log-based)
  - Backup retention policies (how long backups are retained before deletion)
  - Off-site backup requirements and synchronization frequency
- Backup Storage Requirements Estimate (capacity, bandwidth, redundancy level)
- Recovery Validation and Testing Plan (restore test schedule, success criteria, documentation requirements)
- Integration with Security Architecture (backup repository security classification, encryption requirements, access control requirements)
- Cost-Benefit Analysis (cost of achieving each RTO/RPO target vs. business impact of service loss)

## IEC 62443 Lifecycle Stages

- R1 (Security Requirements) — initial RTO/RPO targets based on preliminary business continuity requirements and threat landscape
- R2 (Detailed Design) — complete RTO/RPO architecture with detailed backup strategies, recovery procedures, and storage specifications

## Roles

- SYS (System Architect) — primary: owns RTO/RPO targets, integrates recovery objectives with system design
- BUS (Business Analyst) — defines business criticality rankings and tolerance for service loss / data loss
- DES (Design Engineer) — translates RTO/RPO targets into backup and recovery technology selections
- INF (Infrastructure/Storage Architect) — evaluates backup infrastructure requirements and feasibility
- QA (Quality Assurance) — develops and executes recovery validation and testing procedures
- PE/O (Project Engineer/Owner) — reviews and approves RTO/RPO targets and backup strategies for feasibility and cost alignment

## Acceptance Criteria

1. All system tiers and critical services identified in the system architecture have assigned RTO and RPO targets
2. Each RTO/RPO target has documented business rationale: why this service requires this recovery capability
3. RTO/RPO targets are feasible given the infrastructure constraints; infeasible targets are flagged with cost/schedule implications
4. For each RTO/RPO target, a corresponding backup/recovery strategy has been selected (full backup frequency, incremental frequency, off-site synchronization frequency, recovery automation level)
5. Backup repository security classification and access controls are specified and aligned with the security level of the data being backed up
6. Recovery procedures are documented to sufficient detail that they can be validated: step-by-step instructions, command syntax, validation criteria
7. Recovery validation test schedule has been defined; at least one recovery test per major backup strategy is planned before production deployment
8. RTO/RPO targets and backup strategies have been reviewed and approved by SYS, BUS, INF, and PE/O; cost-benefit trade-offs have been accepted
