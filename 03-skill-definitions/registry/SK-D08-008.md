## Description

Execute application-level security testing for OT/ICS software components including SCADA servers, HMI (Human-Machine Interface) applications, web-based monitoring interfaces, engineering workstations, and custom automation code. Application security testing adapts OWASP (Open Web Application Security Project) testing methodology and other industry application security standards to OT environments, where safety criticality and operational continuity create unique constraints. Testing covers input validation (injection attacks, buffer overflows), authentication and access control verification, session management and token handling, API security (if REST/web services are used), secure configuration review, secure coding practices validation, and cryptographic implementations. Testing is executed during FAT (Factory Acceptance Test) and SAT (Site Acceptance Test) phases, with results documented in security test reports that feed formal compliance verification and risk acceptance decisions.

## Inputs

- Security test plan and test case designs (from SK-D08-002: Security FAT Test Case Design)
- Application design documentation: architecture, component diagrams, data flows (from SK-D05-001 ⏳ and domain D05)
- Security requirements specification: authentication, authorization, input validation, cryptography requirements (from SK-D01-005, SK-D01-006)
- OWASP Top 10 and OT-adapted application security requirements (from organizational security standards)
- Application source code (for code review and white-box testing, if available)
- Compiled applications, installers, and deployment configurations
- Test environment setup: isolated test SCADA/HMI systems with test data
- User account provisioning and authentication credentials for testing
- Known vulnerability databases and security advisories for dependent libraries

## Outputs

- Application Security Test Execution Report: executive summary, scope, test methodology, test environment description, findings summary (by severity and category)
- Test Case Execution Log: per-test-case results (pass/fail), evidence (screenshots, logs), observed behavior, any exceptions or deviations
- Vulnerability Findings Report: discovered vulnerabilities organized by severity (Critical, High, Medium, Low), with detailed description, affected component, reproduction steps, impact assessment, and remediation recommendation
- Input Validation Test Report: injection attack attempts (SQL injection, command injection, LDAP injection, XXE), buffer overflow attempts, format string attacks; results and protective mechanisms validated
- Authentication and Access Control Test Report: weak credential policies, authentication bypass attempts, privilege escalation paths, unauthorized access attempts, multi-factor authentication verification
- Session Management Test Report: session token generation, token lifetime and invalidation, concurrent session handling, session fixation and hijacking attempts, secure cookie attributes
- API Security Test Report (if applicable): authentication and authorization for API endpoints, input validation on API parameters, rate limiting, API versioning and deprecation
- Secure Configuration Review Report: secure defaults validation, unnecessary services disabled, security headers present (if web-based), TLS/SSL configuration assessment, logging and monitoring enabled
- Code Review Findings (if white-box testing): secure coding practices validation, use of safe APIs, input validation and output encoding patterns, cryptographic key management, error handling and exception safety
- Remediation Recommendations: prioritized list of required fixes, suggested fixes, and implementation guidance for addressing identified vulnerabilities

## Tools

- OWASP ZAP (Zed Attack Proxy): automated web application security scanning and penetration testing
- Burp Suite: comprehensive web application security testing platform with proxy, scanner, and manual testing tools
- Qualys WASP or similar: SaaS-based application vulnerability scanning
- HMI/SCADA-specific testing tools: vendor-supplied test utilities, protocol simulators, HMI emulators
- Static Code Analysis Tools: SonarQube, Checkmarx, Fortify for source code vulnerability detection
- Dynamic Analysis Tools: Valgrind, AddressSanitizer, fuzzing frameworks for runtime vulnerability detection
- Authentication Testing Tools: custom test scripts, LDAP/AD testing utilities, OAuth/SAML validators
- API Testing Tools: Postman, REST Assured, API Fortress for API security testing
- Manual Testing: browser developer tools, Wireshark for network inspection, custom test scripts

## Roles

- STC (Security Technical Consultant) — primary: leads application security test planning, executes advanced security testing, interprets findings
- QA (Quality Assurance Engineer) — coordinates test execution, manages test environment, documents results
- Developers — provide application context, assist with debugging findings, implement remediation

## Acceptance Criteria

1. Test plan is approved by STC and development team: scope, test environment, test cases, success criteria defined
2. All test cases from SK-D08-002 are executed; any skipped test cases have documented justification and risk acceptance
3. Test environment is isolated and does not impact production systems
4. Input validation testing covers: SQL injection, command injection, LDAP injection, XXE, XXS, buffer overflows, format string attacks; at least 5 attack variants per category
5. Authentication testing includes: weak credential policies, authentication bypass, privilege escalation, unauthorized access; MFA validated if required
6. Session management testing covers: token generation randomness, token lifetime, session fixation, concurrent session handling, secure cookie attributes
7. API testing (if applicable) verifies: endpoint authentication, input validation, rate limiting, error handling
8. Secure configuration testing validates: unnecessary services disabled, security headers present, TLS/SSL configured per standard, logging enabled
9. Code review (if white-box testing) checks: use of safe APIs, input/output encoding, cryptographic key management, secure error handling
10. All findings documented with severity, affected component, reproduction steps, impact, and remediation recommendation
11. Critical and High findings have remediation plan approved before system acceptance
12. Test execution report is reviewed and approved by STC and project management

## Estimated Effort

| Level | Effort | Notes |
|-------|--------|-------|
| Junior (< 2 yr) | 10–16 person-days | Assumes moderate application complexity, OWASP Top 10 coverage, manual testing primarily; includes test case execution, documentation, and initial analysis |
| Senior (5+ yr) | 5–10 person-days | Same scope; senior can leverage automated tools, prior application testing experience, and faster remediation assessment |

Notes: Complex applications with many custom components, APIs, or cryptographic functions may require 1.5–2× effort due to advanced testing requirements. Applications with weak development documentation may add 2–5 days for discovery and context gathering. Enterprise applications with many integrations (AD, SSO, external APIs) may add 3–7 days for integration-specific testing.
