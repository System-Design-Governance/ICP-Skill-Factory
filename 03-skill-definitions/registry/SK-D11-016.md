## Description

Develop the comprehensive Security Management Plan (SMP) for System Integrator (SI) and Security Maintenance (SM) projects per IEC 62443-2-4, using ID06 as the Tier 2 format exemplar (45+ page template). The plan establishes the security governance framework for the entire project lifecycle, covering security organization, roles, policies, procedures, and compliance requirements. This skill translates the approved security requirements (from SK-D14-001) into an operational security governance structure that governs all subsequent security activities throughout all IEC 62443 lifecycle stages (R0–R5). The plan is an R0 deliverable (foundational to project initiation) and serves as the authoritative reference for security management throughout the project. The boundary of this skill is security management plan development; the actual execution of plan elements falls under their respective domain skills (SK-D01-024 for vendor security management, D08 for security testing, etc.).

## Inputs

- Approved Security Requirements Specification (from SK-D14-001: Security Requirements Development)
- Project scope and contract: customer requirements, project timeline, resource allocation, budget
- Organization security policies and standards (from SK-D11-019: Standards Ownership & Exception Arbitration)
- ID06 (Tier 2, 45+ pages): SI/SM Security Management Plan Template — provides the procedural baseline and section structure for SI/SM projects
- Preliminary risk assessment or threat landscape analysis (from SK-D01-006 ⏳: Threat and Risk Assessment — Preliminary, if available at project initiation)
- Organizational competency framework and personnel assignments (from SK-D11-011: Engineering Competency Framework Development)
- Lessons learned from prior projects (if available)
- Customer-imposed security governance requirements (if applicable, from contract SOW)

## Outputs

- Security Management Plan document (per ID06 structure, 45+ pages):
  - Executive summary: project security governance overview, key stakeholders, lifecycle applicability
  - Security organization and roles: organizational structure, role definitions, responsibilities matrix (RACI: Responsible, Accountable, Consulted, Informed)
  - Security policies and procedures: policy statements for design, implementation, testing, operation, and maintenance security activities
  - Security governance workflow: decision-making processes, escalation paths, approval workflows for security-related decisions
  - Compliance and audit requirements: traceability to IEC 62443 requirements, regulatory obligations, customer contractual requirements
  - Risk management approach: risk assessment process, risk acceptance criteria, risk treatment strategy per IEC 62443-2-4
  - Security change management process: change request evaluation, security impact analysis, approval authority
  - Security training and competency requirements: personnel training plan linked to organizational competency framework
  - Incident response and corrective action procedure: incident classification, investigation, escalation, closure
  - Lessons learned process: continuous improvement mechanism for capturing and incorporating project security insights
  - Appendices: templates, checklists, forms referenced throughout the plan
- Security Management Plan Approval Record: sign-off by PM, customer representative, and security oversight role
- Traceability Matrix: SMP sections linked to IEC 62443-2-4 requirements and approved Security Requirements Specification items
- Governance Implementation Schedule: timeline for establishing security governance activities, personnel assignments, and resource allocation

## IEC 62443 Lifecycle Stages

- R0 (Foundational: Project Initiation) — SMP is an R0 deliverable that must be approved before project design begins (R1)
- R1–R5 (Continuous) — SMP governs security management activities throughout all subsequent lifecycle stages

## Roles

- PM (Project Manager) — primary owner: develops SMP, coordinates with security roles and customer, obtains approvals
- Security Engineering Role (from SK-D11-011) — contributes security-specific policies, procedures, governance workflows, and competency requirements
- SAC (Security Architect) — reviews SMP for alignment with security architecture and design requirements
- Customer Project Representative — reviews SMP for compliance with contractual security requirements; approves plan
- Organizational Security Officer / Head of System Design (GOV-SDP) — reviews and approves SMP for organizational security policy compliance

## Dependencies

### Hard Dependencies (Required Before Execution)

- SK-D14-001: Security Requirements Development — approved security requirements specification defines the security objectives that the SMP must operationalize; cannot develop SMP without knowing what security must be achieved

### Soft Dependencies (Enhances Effectiveness)

- SK-D01-024: Vendor Security Management and Oversight — if third-party components are part of the project, vendor security management procedure should be integrated into SMP; can be added as an addendum if vendor scope becomes clear during project execution
- SK-D11-011: Engineering Competency Framework Development — competency framework informs personnel training and qualification requirements within the SMP
- SK-D11-019: Standards Ownership & Exception Arbitration — organizational standards ownership and exception process should be referenced in SMP change management section

## Acceptance Criteria

1. SMP document is complete per ID06 section checklist: all 45+ page template sections populated with project-specific content (no template placeholders remaining)
2. RACI matrix explicitly assigns responsibility for every security management activity listed in the plan (no unassigned activities)
3. Every item in the approved Security Requirements Specification is traceable to at least one section of the SMP (traceability matrix at 100%)
4. SMP procedures specify decision authority (who approves) for every security-related decision point (design approval, change approval, risk acceptance, exception approval)
5. Security governance implementation schedule defines: start date, resource allocation, and completion target for each governance activity (e.g., security committee establishment, training program launch)
6. SMP is signed off by PM, customer representative, and organizational security oversight role (three-party approval record completed)

## Estimated Effort

| Level | Effort | Notes |
|-------|--------|-------|
| Junior (< 2 yr) | 12–18 person-days | Assumes single-site project, ~50–100 assets, with ID06 template; includes customization, stakeholder review, and revisions |
| Senior (5+ yr) | 6–10 person-days | Same scope; senior leverages prior project SMP and ID06 template; faster customization and approval coordination |

Notes: Multi-customer or multi-site projects may require 1.5–2× effort due to multiple approval cycles and governance customization per customer. Brownfield projects with existing organizational governance may reduce effort by 20–30% (leverage existing policies). Initial SMP creation (first project in organization) may require 25–35 person-days to establish baseline governance policies. ID06 provides the procedural template; customization effort depends on variance from template requirements. Approval cycle typically requires 2–4 weeks for stakeholder review and sign-off.
