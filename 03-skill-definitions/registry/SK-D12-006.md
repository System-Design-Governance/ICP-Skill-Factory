## Description

This skill encompasses the design of monitoring and analytics dashboards that provide operators and managers with real-time visibility into operational performance, security events, and system health. Monitoring dashboards integrate data from multiple sources (SK-D12-001: Data Acquisition, SK-D01-014: SIEM, operational historians) and present information through tailored views that enable quick decision-making and root cause analysis. Dashboards may serve different audiences: operational dashboards for plant operators focus on system status and alarms; security dashboards for security teams focus on anomalies and events; executive dashboards show high-level KPIs and trends. The skill covers dashboard architecture, widget selection and layout design, data refresh rates and performance optimization, role-based access control and data filtering, drill-down navigation between summary and detailed views, and integration with SK-D05-005 (HMI Screen Design) for consistency.

## Inputs

- Data Acquisition Architecture (from SK-D12-001: Data Acquisition Architecture) — available data sources and schema
- SIEM Design (from SK-D01-014: SIEM Design and Implementation) — security event data structure and queries
- KPI and Metrics Requirements: business, operational, and security KPIs to be displayed (from project requirements or operations team)
- HMI Screen Design (from SK-D05-005: HMI Screen Design) — design consistency and screen layout standards
- Access Control Policies (from SK-D01-020: Account and Access Control) — role-based dashboard access and data filtering
- System Performance and Data Refresh Requirements: latency tolerance, acceptable query load on backend systems
- Dashboard Tool Capabilities: selected visualization platform (Grafana, Tableau, PowerBI, Kibana, custom web application)
- User Experience Requirements: audience profiles, usability goals, accessibility standards

## Outputs

- Dashboard Design Specification: dashboard layout, widget definitions, data sources for each widget, refresh rates, drill-down navigation paths
- Dashboard Prototype or Wireframes: visual mockups showing dashboard layout and widget placement for each dashboard type (operational, security, executive)
- Data Integration Specification: data sources for each dashboard, queries or API calls to retrieve data, data transformation logic (if needed)
- Widget and Layout Templates: reusable dashboard components and layout patterns for consistency across dashboards
- Role-Based Access Control (RBAC) Matrix: role definitions, dashboards accessible per role, data filtering rules per role
- Performance and Scalability Plan: query optimization techniques, caching strategies, acceptable query response times, system load testing results
- Drill-Down Navigation Map: how users navigate from summary dashboards to detailed analysis views; breadcrumb or navigation structure
- Security Event Summary Dashboard Design: layout and widgets for presenting security events and alerts to security team

## Tools

- Dashboard and visualization tools: Grafana, Kibana, Tableau, Microsoft Power BI, Apache Superset, or custom web frameworks (React, Vue.js)
- Wireframing and mockup tools: Balsamiq, Figma, Adobe XD (for dashboard prototype design)
- Query languages: InfluxQL, TimescaleDB SQL, ESQL (Elasticsearch Query Language) (for data retrieval)
- Performance testing tools: JMeter, Locust (for load testing and optimization)
- Data modeling and transformation tools: Python pandas, dbt (data build tool) (for complex transformations)
- Design system tools: Storybook or similar for documenting dashboard design patterns

## IEC 62443 Lifecycle Stages

- R2 (Detailed Design) — dashboard requirements and design are finalized
- R3 (Implementation) — dashboards are developed and integrated with data sources
- R4 (Commissioning/Startup) — dashboards are tested with production data and optimized for performance
- R5 (Operations) — dashboards are operationalized; new dashboards may be added based on operational feedback

## Acceptance Criteria

1. Dashboard Design Specification documents all dashboard types (operational, security, executive) with purpose, audience, KPIs, and data sources; at least 3 distinct dashboards are designed
2. For each dashboard widget: data source, query/API call, refresh rate, and visualization type are specified; performance impact (estimated query time) is documented
3. Drill-down navigation is clearly mapped: users can navigate from summary metrics to detailed data views with breadcrumb navigation or back-button; at least 2 drill-down levels are designed
4. Role-based access control matrix specifies which roles can view which dashboards; data filtering rules per role are documented (e.g., operators see only their area's data)
5. Dashboard prototype or wireframes have been reviewed by operational stakeholders (Operations Lead, plant operators, security team); feedback has been incorporated
6. Performance testing confirms dashboard queries complete within acceptable response times (e.g., < 2 seconds for typical queries) under normal operational load; query optimization techniques are documented
