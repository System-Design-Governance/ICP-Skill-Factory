# Skill Definition — SK-D11-020

> **Design Change Impact Analysis & SL Recertification**: A critical governance skill for managing security implications of design changes post-approval. This skill determines whether design changes require Security Level reassessment, Zone/Conduit redesign, or full Security Level recertification, ensuring that change management does not inadvertently degrade security posture.

## Description

Analyze the security impact of design changes per GOV-SD Gate 2 trigger criteria (Tier 1 governance decision point), determining whether changes require Security Level-Tier (SL-T) reassessment, Zone/Conduit redesign, impact on data flow diagrams and communication conduits, or full Security Level recertification before implementation. This skill produces SL Decision Records documenting impact analysis results and any SL-T changes. It is triggered by SK-D10-002 (Change Request Evaluation) when a change request has been identified as having potential security impact. The skill ensures that the change control process maintains security requirements traceability and prevents degradation of the approved Security Functional Description Specification (SFDS).

## Inputs

- Change Request (CR) from SK-D10-002 evaluation, including:
  - Description of proposed change (equipment replacement, network topology modification, protocol change, etc.)
  - Change rationale (e.g., obsolescence, performance improvement, regulatory requirement)
  - Preliminary impact assessment (systems affected, estimated effort)
- Approved design baseline:
  - Current Zone/Conduit Architecture Diagram and specification (from SK-D01-001 ⏳)
  - Approved Security Functional Description Specification (SFDS) with SL-T assignments per zone (from SK-D01-001 ⏳)
  - Approved Threat and Risk Assessment (Detailed) (from SK-D01-007 ⏳)
  - Current system SL certification record and SL-T decision record
  - Network topology and data flow diagrams (from SK-D02-001 ⏳, SK-D02-004 ⏳)
- Design change impact assessment criteria (from GOV-SD governance framework — Gate 2 trigger matrix)
- SL-T recertification requirements per IEC 62443-3-2

## Outputs

- SL Decision Record documenting:
  - Change ID, description, and affected system/zone(s)
  - Impact analysis results: Does the change trigger SL-T reassessment? (Yes/No/Conditional)
  - SL-T change recommendation: If SL-T is affected, what are new SL-T assignments per zone?
  - Zone/Conduit redesign requirement: Does the change require updates to Zone/Conduit Architecture? (Yes/No)
  - Data flow impact: Are communication conduits or data flows affected? (Yes/No)
  - Threat profile impact: Are the assumed threats or vulnerabilities in the TRA still valid? (Yes/No)
  - Recertification scope: Is full SL recertification required, or only partial re-verification? (Full / Partial / None)
  - Mitigations required before implementation (if any)
  - SL reassessment decision: Proceed with change / Require design changes / Reject change (with rationale)
- Updated SL Decision Record filed in change management system (Jira / Azure DevOps)
- Traceability link to SK-D10-002 Change Request Evaluation

- R2 (Detailed Design — change impact analysis applied to design documentation)
- R3 (Implementation — impact analysis applied to implementation changes)
- R4 (Operations & Maintenance — change control for operational modifications)

## Roles

- SAC (Security Architect) — primary: leads impact analysis, makes SL-T reassessment decision
- SK-D10-002 Change Request Evaluator (typically SYS or Technical Lead) — provides change description and preliminary impact from Change Request
- DES (Design Engineer) — provides technical details on feasibility and impact on design artifacts
- Compliance Lead — advises on regulatory implications of SL-T changes
- PE/O (Project Executive / Owner) — approves SL-T changes and any associated recertification scope

## Dependencies

### Hard Dependencies (Required Before Execution)

- SK-D10-002 ⏳: Change Request Evaluation — must identify and route change to this skill when security impact is suspected
- SK-D01-001 ⏳: Zone/Conduit Architecture Design — baseline architecture required for impact comparison
- SK-D01-001 ⏳: SFDS Development — approved SFDS with SL-T assignments is the baseline for impact analysis

### Soft Dependencies (Enhances Effectiveness)

- SK-D01-007 ⏳: Threat and Risk Assessment (Detailed) — threat profile informs impact assessment on risk posture
- SK-D02-001 ⏳: OT Network Topology Design — network topology changes may trigger zone redesign analysis
- SK-D02-004 ⏳: Data Flow Diagram Development — data flow changes may affect Zone/Conduit design

## Acceptance Criteria

1. SL Decision Record is complete and documents all required elements: change ID, impact analysis results, SL-T recommendation, redesign requirements
2. Impact analysis is objective and traceable to baseline design documents (Zone/Conduit Diagram, SFDS, TRA)
3. If SL-T change is recommended, new SL-T assignments are justified with reference to threat profile and risk appetite
4. If Zone/Conduit redesign is required, the scope of redesign is clearly documented and tied to SK-D01-001 execution
5. If data flow changes are identified, traceability to SK-D02-004 execution is documented
6. Recertification scope decision is clearly stated: Full / Partial / None, with supporting rationale
7. If mitigations are required before implementation, they are specific, measurable, and assigned to responsible parties with target completion dates
8. SL Decision Record has been reviewed and approved by SAC, with sign-off by PE/O if SL-T changes are approved

## Estimated Effort

| Level | Effort | Notes |
|-------|--------|-------|
| SAC / Senior Security Architect | 2–4 person-days | Impact analysis for single zone or moderate scope change; no Zone/Conduit redesign required |
| SAC with Design Engineer support | 5–8 person-days | Complex multi-zone or multi-system impact; may trigger Zone/Conduit redesign (SK-D01-001 execution included) |

Notes: Changes triggering full SL recertification may require 15–20 additional days for re-verification. Changes affecting cryptography or key management may require STC consultation (add 1–2 days).
