## Description

Develop the Factory Acceptance Test (FAT) procedure document that defines the test protocol for verifying security and functional requirements at the vendor/integrator premises before system shipment to the project site. The FAT procedure specifies test categories, individual test cases, pass/fail criteria, test environment requirements, and documentation standards. Under ICP governance (GOV-SD), FAT completion evidence is part of the Gate 3 delivery checklist, and FAT results feed into the security acceptance verification. This skill covers FAT procedure authoring and test case design; the actual FAT execution falls under the broader testing execution skills, the SAT procedure falls under SK-D08-003 ⏳, and the SIT protocol falls under SK-D08-013 ⏳. Per BOUNDARY-007, FAT is factory testing at vendor premises (pre-shipment), distinct from SAT (site testing post-installation) and SIT (end-to-end integration with employer systems).

## Inputs

- Security Functional Description Specification (from SK-D09-002 ⏳) — SR requirements that must be verified
- FR/SR mapping matrix (from SK-D01-007) — specifies which SRs apply per zone/device
- Zone/Conduit architecture (from SK-D01-001) — defines the security topology to be tested
- Endpoint hardening specifications (from SK-D01-019) — hardening configurations to verify during FAT
- Account and access control policy (from SK-D01-020) — RBAC implementation to verify
- Malware protection deployment plan (from SK-D01-023) — antivirus configuration to verify
- Customer/employer FAT requirements and acceptance criteria
- IEC 62443-2-4 verification requirements per SP category

## Outputs

- FAT Procedure Document (per ID14 exemplar format):
  - Test scope and objectives: what is being verified, what is out of scope
  - Test environment requirements: lab setup, network configuration, simulated loads
  - Test categories (per ID14 14-category structure):
    - Physical security inspection
    - Network architecture verification
    - Firewall/ACL rule verification
    - Account and access control verification
    - Endpoint hardening verification
    - Malware protection verification
    - Logging and monitoring verification
    - Backup and restore verification
    - Patch management verification
    - Remote access security verification
    - Communication encryption verification
    - Incident response procedure verification
    - Documentation completeness verification
    - Functional integration verification
  - Per test case: test ID, test description, pre-conditions, test steps, expected result, pass/fail criteria, actual result (blank for procedure), tester signature
  - Test execution schedule and resource requirements
  - Defect reporting and severity classification procedures
  - Re-test criteria: conditions under which failed tests must be repeated
- FAT Test Case Register: tabular list of all test cases with SR traceability (which SR each test verifies)
- FAT Environment Specification: lab configuration requirements, test data, simulation tools

## Automation Potential

**Level:** Partial

Test case generation from SR matrices can be partially automated (template-driven test case derivation). Test execution reporting and pass/fail tracking can be automated through test management platforms. However, test procedure design requires expert judgment (determining appropriate verification methods per SR), test environment specification requires understanding of system architecture, and pass/fail criteria definition requires domain expertise to ensure meaningful verification without false positives.

## Acceptance Criteria

1. FAT procedure covers all 14 test categories (per ID14 exemplar) or documents explicit rationale for excluded categories
2. Every applicable SR in the FR/SR mapping matrix has at least one corresponding FAT test case; SR traceability documented in test case register
3. Each test case includes: unique ID, clear steps, expected result, and unambiguous pass/fail criteria (no subjective language like "adequate" or "satisfactory")
4. Test environment specification is sufficient to reproduce the test: lab configuration, network setup, test data, and simulation requirements documented
5. Re-test criteria defined: conditions for re-execution of failed tests, maximum re-test iterations, escalation path for persistent failures
6. FAT procedure reviewed and approved by Security Engineering Role and Design QA Role before test execution (per GOV-SD Gate 3 evidence requirements)

## Estimated Effort

| Level | Effort | Notes |
|-------|--------|-------|
| Junior (< 2 yr) | 10–15 person-days | Assumes ~30 applicable SRs, 14 test categories, ~100 individual test cases; includes procedure authoring, test case design, and review cycle |
| Senior (5+ yr) | 5–8 person-days | Same scope; senior leverages reusable test case templates and SR-to-test-case mapping patterns |

Notes: Effort scales with SR count and system complexity. Multi-vendor systems requiring separate FAT sessions per vendor multiply effort. ID14 exemplar shows 14 categories with 33 pages of test protocol for a medium-complexity SCADA project — this serves as the baseline reference. Test case design effort is front-loaded; subsequent FAT procedures for similar projects can reuse 60–70% of test cases.
