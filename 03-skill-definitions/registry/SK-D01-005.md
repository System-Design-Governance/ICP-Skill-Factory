## Description

Develop a comprehensive inventory of all physical and logical assets within the System under Consideration (SuC), including their criticality ratings, zone assignments, and network connectivity relationships. This skill translates the initial project scope into a structured asset register that feeds downstream risk assessment, zone/conduit architecture design, and security requirements allocation. The inventory covers OT devices (HMI, IED, PLC/RTU, switches, wireless AP), IT infrastructure, software components, and communication pathways. The boundary of this skill is asset identification and classification; the actual risk analysis of these assets falls under SK-D01-006 ⏳ (Preliminary TRA), and physical/logical network topology design falls under SK-D02-001 ⏳.

## Inputs

- System under Consideration (SuC) scope definition from SOW / contract
- Preliminary network architecture diagram (from SK-D02-001 ⏳: OT Network Topology Design, if available)
- Customer-provided equipment lists, single-line diagrams, or existing CMDB exports
- Site survey findings (from SK-D14-011 ⏳: Site Survey and Constraint Documentation, for brownfield projects)
- Existing infrastructure inventory (from SK-D14-012 ⏳: Existing Infrastructure Inventory, for brownfield)
- IEC 62443-3-2 zone/conduit framework guidance

## Outputs

- Asset Inventory Register (structured spreadsheet or database): asset ID, asset name, asset type, manufacturer/model, firmware/software version, IP/MAC address, physical location, zone assignment, conduit connectivity, criticality rating (High/Medium/Low), data classification level
- Zone Assignment Summary Table: zone ID → list of contained assets with justification
- Asset-to-Conduit Mapping: which assets communicate through which conduits, with protocol and direction
- Asset Criticality Rationale Document: per-asset or per-group justification for assigned criticality level
- Assumptions and External Factors Log: documented assumptions about assets not directly observable (e.g., vendor-managed components)

## Tools

- Microsoft Excel / structured spreadsheet (primary asset register format)
- CMDB tools (ServiceNow, Lansweeper) for automated discovery where available
- Nmap / passive network scanning tools (for brownfield asset discovery — with PtW approval)
- Microsoft Visio / draw.io (for visual zone-to-asset mapping)
- ICP standard asset inventory template (based on ID09 format: zone-based columnar layout)

## Standards

- IEC 62443-3-2: Security Risk Assessment for System Design — asset identification as input to zone/conduit model
- IEC 62443-2-1: Security Management System for IACS — asset management requirements
- IEC 62443-1-1: Terminology, concepts and models — asset and zone definitions
- ISO 27001 Annex A.8: Asset Management — information asset identification and classification

## IEC 62443 Lifecycle Stages

- R0 (Project initiation — preliminary asset identification from SOW/site survey)
- R1 (Security requirements — complete asset inventory as input to preliminary TRA)

## Roles

- SAC (Security Architect) — primary: defines asset classification criteria, assigns zone memberships, validates completeness
- SYS (System Architect) — provides system topology and subsystem boundaries for asset scoping
- DES (Design Engineer) — contributes device-level specifications and network connectivity details
- Pre-Gate Design Support — provides existing infrastructure inventory from feasibility phase (if available)

## Dependencies

### Hard Dependencies (Required Before Execution)

- Project SOW / scope definition — cannot inventory assets without knowing the SuC boundary
- SK-D14-011 ⏳: Site Survey and Constraint Documentation — brownfield projects require physical site data to identify existing assets

### Soft Dependencies (Enhances Effectiveness)

- SK-D02-001 ⏳: OT Network Topology Design — network topology informs asset connectivity mapping; can be developed iteratively
- SK-D14-012 ⏳: Existing Infrastructure Inventory — pre-gate infrastructure data accelerates brownfield inventory
- SK-D01-001: Zone/Conduit Architecture Design — zone definitions guide asset grouping; often developed iteratively with inventory

## Automation Potential

**Level:** Partial

Automated network scanning tools can discover IP-addressable assets and their basic properties. However, criticality assessment, zone assignment rationale, and non-networked asset identification (physical security devices, manual interfaces) require human judgment and domain expertise. Template generation and data formatting are automatable.

## Acceptance Criteria

1. Inventory covers 100% of assets identified in the SuC scope definition, with no known omissions undocumented
2. Every asset has a unique ID, assigned zone, and criticality rating with documented rationale
3. Zone assignment column is populated for all assets; no asset is "unassigned"
4. Asset-to-conduit mapping identifies all inter-zone communication paths with protocol and direction
5. Assumptions log documents all inferred or estimated asset data with flagged confidence level
6. Inventory format follows ICP standard template (zone-based columnar layout per ID09 exemplar)

## Estimated Effort

| Level | Effort | Notes |
|-------|--------|-------|
| Junior (< 2 yr) | 8–12 person-days | Assumes single-site, ~50 assets, greenfield with equipment lists provided |
| Senior (5+ yr) | 4–6 person-days | Same scope; senior leverages prior project templates and systematic discovery methods |

Notes: Brownfield projects with undocumented legacy systems may require 2–3× effort due to physical discovery overhead. Multi-site deployments scale roughly linearly per site. Active network scanning requires PtW approval and may add 1–2 days for coordination.
