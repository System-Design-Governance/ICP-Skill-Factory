## Description

Configure protocol gateway devices that translate between different industrial protocols, enabling interoperability across heterogeneous OT/ICS systems. This skill covers the selection, deployment, and hardening of gateway hardware that bridges incompatible protocol environments (e.g., Modbus to DNP3, IEC 61850 to OPC UA). Configuration encompasses protocol mapping tables, data point mapping, failover and redundancy settings, and comprehensive security hardening including firmware updates, disabling unused services, and enforcing access control. The gateway serves as a critical protocol translation layer within the network architecture; configuration must comply with zone/conduit boundaries (SK-D01-001) and integrate with network segmentation controls (SK-D02-003 ⏳).

## Inputs

- Protocol Gateway hardware specifications and datasheet (from vendor or procurement)
- Network architecture and zone/conduit diagram (from SK-D01-001)
- Data flow diagram identifying required protocol translations (from SK-D02-004 ⏳)
- Source and destination protocol specifications (Modbus, DNP3, IEC 61850, OPC UA technical standards)
- Security requirements and access control policies (from SK-D01-002, SK-D01-003)
- Failover and redundancy strategy (from SK-D02-006 ⏳)
- Vendor configuration guides and security bulletins

## Outputs

- Protocol Gateway Configuration Document: hardware model, firmware version, enabled protocols, protocol mapping table
- Protocol Mapping Table: source protocol → destination protocol, translation rules, data type conversions, timing and latency specifications
- Data Point Mapping Table: source device → source data point → gateway mapping → destination device → destination data point, with validation rules
- Failover Configuration Specification: redundancy model (active-passive, active-active), switchover triggers, test procedures
- Gateway Security Hardening Checklist: firmware version verification, unused service disablement, access control enforcement, audit log configuration
- Network Connectivity Diagram showing gateway placement within zone/conduit architecture

## Tools

- Vendor-specific gateway configuration software (e.g., Siemens TIA Portal for PROFINET, ABB Freelance/800xA for IEC 61850)
- Protocol analyzer / packet sniffer (Wireshark, Kismet) for validating protocol translation
- Excel / structured markdown for mapping tables
- Network documentation tools (Visio, draw.io) for connectivity diagrams
- SSH / serial console for firmware and configuration management

## Roles

- DES (Design Engineer) — primary: selects gateway hardware, designs protocol and data point mapping
- SYS (System Architect) — provides system architecture context, validates protocol translation feasibility
- STC (Security Technical Consultant) — specifies security hardening requirements, reviews access control and audit logging
- OPS (Operations Engineer) — supports field configuration, validates failover behavior

## Dependencies

### Hard Dependencies (Required Before Execution)

- SK-D01-001: Zone/Conduit Architecture Design — gateway configuration must respect zone/conduit boundaries and inter-zone communication rules
- SK-D02-004 ⏳: Data Flow Diagram Development — identifies which protocols must be translated and data flow requirements
- Vendor gateway datasheet and technical documentation — essential for understanding hardware capabilities and configuration options

### Soft Dependencies (Enhances Effectiveness)

- SK-D02-003 ⏳: Firewall Rule Development — gateway configuration integrates with firewall policies controlling inter-zone traffic
- SK-D02-006 ⏳: Redundancy and Failover Strategy — informs gateway failover configuration
- SK-D07-001 ⏳: Protocol Selection and Justification — provides context for protocol choice at each end of the gateway

## Acceptance Criteria

1. Gateway hardware is certified/approved for the industrial environment (temperature, vibration, IP rating per site requirements)
2. Protocol mapping table is complete: every source protocol data point has a defined mapping to destination protocol, with documented transformation rules
3. Data point mapping table is bidirectional (where applicable) and includes validation rules, data type conversions, and timing specifications
4. Failover configuration is tested: manual switchover triggers, recovery timing, and data loss verification
5. Security hardening checklist is 100% complete: firmware version documented and validated, all unused services disabled, access control enforced, audit logging enabled and tested
6. Gateway configuration has been reviewed and approved by SAC or STC; protocol translation has been validated via packet capture or simulation

## Estimated Effort

| Level | Effort | Notes |
|-------|--------|-------|
| Junior (< 2 yr) | 5–8 person-days | Assumes single gateway, 2–3 protocol translations, straightforward 1:1 mapping; includes configuration, testing, documentation |
| Senior (5+ yr) | 2–3 person-days | Same scope; senior can leverage prior gateway configurations and protocol knowledge |

Notes: Complex gateways with many-to-many protocol translations or custom transformation logic may require 1.5–2× effort. Failover and redundancy configuration adds 2–3 days. Vendor-specific learning curves vary; initial projects with a new gateway platform may add 1–2 days.
