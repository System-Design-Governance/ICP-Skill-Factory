## Description

Conduct a preliminary (High-Level Cyber Risk Assessment, HLCRA) threat and risk assessment for the SuC based on the asset inventory, preliminary network architecture, and threat landscape analysis. This skill produces the initial risk profile that drives zone/conduit architecture design and security requirements allocation. Under ICP governance (GOV-SD), this is the "Gate 1 Lite" assessment path — sufficient for Gate 1 design baseline establishment but must be upgraded to a complete detailed assessment (SK-D01-007) by Gate 3. The boundary of this skill is the preliminary risk identification and classification; the detailed risk assessment with full FMEA/HAZOP integration falls under SK-D01-007, and the integrated triple-method assessment methodology (IEC 62443-3-2 + FMEA + HAZOP) is governed by SK-D01-035 ⏳.

## Inputs

- Completed asset inventory with criticality ratings (from SK-D01-005: Asset Inventory Development)
- SuC description and system context documentation
- Preliminary network architecture diagram (from SK-D02-001 ⏳)
- Data flow diagram identifying storage points and dataflows (from SK-D02-004 ⏳)
- Threat source catalogue (industry-standard threat libraries: ICS-CERT, MITRE ATT&CK for ICS)
- Recovery Point Objective (RPO) and Recovery Time Objective (RTO) targets from customer requirements
- Data safeguarding requirements (legal/regulatory, PE/O confidential, configuration, operational, audit logs)

## Outputs

- Preliminary TRA Report (structured document per ID11 exemplar format):
  - Section 1: Introduction, purpose, objectives
  - Section 2: Security context (SuC description, asset summary, network architecture, data flow, threat sources)
  - Section 3: Risk assessment results (HLCRA proforma with threat-vulnerability matrix, impact analysis, risk classification)
- Threat Source Identification Table: threat source ID, threat type, likelihood, capability, motivation
- Threat-Vulnerability Matrix: mapping threats to assets/zones with preliminary impact scoring
- Preliminary Risk Register: risk ID (T-XXX format per GOV-SD), threat description, affected zone/asset, likelihood, impact, risk level (Low/Medium/High/Critical), proposed treatment direction
- Data Safeguarding Requirements Summary: protected data types with classification and safeguarding approach
- Security Profile Summary: consolidated risk posture for the SuC feeding downstream design decisions

## Standards

- IEC 62443-3-2: Security Risk Assessment for System Design — primary methodology for zone-based risk assessment
- IEC 62443-1-1: Terminology, concepts and models — threat, vulnerability, risk definitions
- ISO 27005: Information Security Risk Management — risk assessment process framework (supplementary)
- NIST SP 800-82 Rev. 3: Guide to OT Security — threat landscape context (supplementary)

## IEC 62443 Lifecycle Stages

## Roles

- Security Engineering Role (GOV-SDP) — primary: owns integrated risk assessment methodology, executes IEC 62443-3-2 threat analysis, produces risk register with T-XXX coded risk sources
- SAC (Security Architect) — provides zone/conduit context for threat scoping, validates risk-to-zone mapping
- SYS (System Architect) — provides system architecture context for threat scenario definition
- Design QA Role (GOV-SDP) — verifies TRA completeness against Gate 1 Lite requirements

## Automation Potential

**Level:** Partial

Threat library lookup and matrix population can be partially automated using threat intelligence feeds and templated risk scoring. However, threat scenario contextualization to the specific SuC, likelihood judgment for the operational environment, and risk treatment direction decisions require human expertise. Report generation from structured data is automatable.

## Acceptance Criteria

1. TRA report covers all zones and assets identified in the approved asset inventory (SK-D01-005 output)
2. Threat source identification table includes at least the ICS-CERT top threat categories relevant to the SuC type
3. Every entry in the preliminary risk register has a coded risk source ID (T-XXX format per GOV-SD traceability requirement)
4. Risk classification uses the approved Likelihood × Impact matrix with documented scoring rationale
5. Data safeguarding requirements are identified for all protected data types (legal/regulatory, PE/O confidential, configuration, operational, audit, backup, historical)
6. TRA report is sufficient for Gate 1 Lite approval — meets GOV-SD Gate 1 prerequisite for design baseline establishment

## Estimated Effort

| Level | Effort | Notes |
|-------|--------|-------|
| Junior (< 2 yr) | 10–15 person-days | Assumes single-site, ~50 assets; includes threat research, matrix building, and report writing |
| Senior (5+ yr) | 5–8 person-days | Same scope; senior leverages threat libraries and prior project risk patterns |

Notes: ID04 §9.0 baseline indicates ~10 working days for initial-level risk assessment. Complex multi-zone SuCs with diverse asset types may require 1.5× effort. Projects requiring RPO/RTO analysis add 2–3 days for business impact assessment.
