## Description

This skill encompasses the authoring of a comprehensive System Design Description (SDD) document that consolidates the outputs of security architecture (D01), system architecture (D02), control system design (D05), panel design (D06), and commissioning planning (D07) into a single, coherent master reference. The SDD serves as the design baseline for implementation, validation, and operational teams, and is a required deliverable per ID01 §7.4.1 (design documentation requirements). The skill focuses on technical clarity, internal consistency, traceability across design domains, and compliance with IEEE 1016 documentation standards adapted for OT/ICS environments. This skill does not include the execution of design activities themselves (those fall to D01–D07), but rather the synthesis and structured documentation of completed designs into an approved control document.

## Inputs

- Zone/Conduit architecture diagram and specification (from SK-D01-001: Zone/Conduit Architecture Design)
- Security requirements and security level allocations (from SK-D01-003: Security Requirements Specification)
- OT network topology and interconnection design (from SK-D02-001: OT Network Topology Design)
- Data flow diagrams and communication protocols (from SK-D02-004: Data Flow Diagram Development)
- Control system functional design and logic documents (from SK-D05-001: Control System Functional Specification)
- Panel layout, SCADA HMI, and graphical interface designs (from SK-D06-001: Panel Layout and Design Specification)
- Approved Bill of Materials (BOM) with hardware and software specifications
- All prior design review comments, acceptance records, and design approval sign-offs
- System architecture narrative, resilience strategy, and integration approach documentation

## Outputs

- System Design Description (SDD) document: consolidated master reference organized per IEEE 1016 sections (system overview, design rationale, decomposition, interface specifications, quality attributes)
- Design traceability matrix: mapping requirements → architecture → components → implementation
- Consolidated design review matrix: showing all design review findings, resolutions, and approval signatures
- Design baseline release notes: version control, change history, and design baseline approval record
- Integration checklist: verification that all design elements are complete, consistent, and ready for implementation phase

## IEC 62443 Lifecycle Stages

- R2 (Detailed Design) — SDD consolidates all detailed design outputs and provides the design baseline
- R3 (Implementation) — SDD serves as the authoritative reference for implementation teams and guides component procurement, configuration, and integration
- R4 (Commissioning/Startup) — SDD is used for commissioning verification and as-built documentation cross-reference

## Roles

- SYS (System Architect) — primary: leads SDD authoring, integrates cross-domain designs, ensures overall consistency and traceability
- DES (Design Engineer) — contributes design details from their discipline (security, control, HMI, etc.), reviews SDD for technical accuracy
- STC (Security Technical Consultant) — reviews security architecture sections, validates SL allocations and security requirement traceability
- SAC (Security Architect) — reviews security-specific SDD sections and approves security baseline content
- QAM (Quality Assurance Manager) — reviews SDD for completeness, consistency, and conformance to design standards

## Acceptance Criteria

1. SDD document is organized per IEEE 1016 structure and includes all mandatory sections: system overview, design entities, design views, design rationale, and interface specifications
2. Every requirement from the Security Requirements Specification (SK-D01-003) is traceable to a design element or subsystem in the SDD via a documented traceability matrix
3. All architecture diagrams (Zone/Conduit, network topology, functional decomposition) are embedded and cross-referenced with narrative explanation
4. SDD covers 100% of the Bill of Materials (BOM) components with specifications, configuration, and integration approach documented
5. Design review findings from all design gate reviews are documented, with resolution status and incorporation into SDD clearly marked
6. SDD has been formally reviewed and approved by SAC, SYS, and QAM, with signatures and approval dates recorded in the design baseline release record
