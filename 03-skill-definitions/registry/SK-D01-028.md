## Description

Configure secure remote access mechanisms for OT/ICS environments including VPN setup, jump server configuration, multi-factor authentication (MFA), session recording, and access logging. This skill implements the remote access architecture and controls defined in the Zone/Conduit design (SK-D01-001) and security monitoring plan, translating abstract security requirements into operational remote access systems that enable authorized personnel (engineering, support, vendors) to safely access OT/ICS systems across geographical boundaries. Per IEC 62443-3-3 (ID02 Annex A.9 §9), remote access must traverse defined conduits with appropriate authentication and encryption; this skill ensures that remote access endpoints are hardened, access is strictly controlled and auditable, and all remote sessions are recorded for forensic investigation if needed. It covers selection and configuration of VPN technology (site-to-site, clientless), deployment of jump servers (bastion hosts) that mediate access to critical systems, enforcement of multi-factor authentication (MFA) for all remote sessions, implementation of session recording and keystroke logging for high-risk remote activities, configuration of access control lists (ACLs) limiting remote access to authorized zones and conduits, and comprehensive logging of all remote access events. The boundary of this skill is configuration of remote access infrastructure; the network architecture and conduit design falls under SK-D01-001, and incident response procedures for compromise of remote access infrastructure fall under D07.4.

## Inputs

- Approved Zone/Conduit Architecture (from SK-D01-001: Zone/Conduit Architecture Design), particularly remote access conduits and allowed protocols
- Security Functional Description Specification (SFDS) with remote access requirements (from SK-D01-011 ⏳: SFDS Development)
- List of authorized remote users and use cases (maintenance, vendor support, engineering, monitoring) with required access levels
- Business continuity requirements and remote access SLAs (uptime targets, performance requirements)
- Approved remote access architecture and technology baseline (VPN type, jump server requirements, MFA method)
- Security monitoring requirements for remote access (from SK-D01-009 ⏳: Security Monitoring Plan Development)
- Network and data flow diagrams showing remote access entry points (from SK-D02-004 ⏳)
- VPN and jump server platform specifications (vendor documentation, hardware specs, capacity planning)

## Outputs

- VPN Configuration Baseline: VPN technology selection, tunnel encryption algorithm, authentication method (certificate, PSK), key exchange parameters, session timeout settings
- VPN Authentication Configuration: certificates and certificate authority (CA) hierarchy for site-to-site VPN, or client certificate distribution and enrollment procedure for client-to-site VPN
- Jump Server (Bastion Host) Configuration: hardened OS baseline, firewall rules, allowed protocols, user account provisioning procedure, privileged access management (PAM) integration
- Multi-Factor Authentication (MFA) Configuration: authentication factors (something you have, something you know, something you are), MFA provider integration (RADIUS, LDAP, SAML, Okta, etc.), enrollment procedure, recovery procedures for MFA loss
- Session Recording and Keystroke Logging Policy: which remote sessions require recording (all, or high-risk only), recording storage and retention, playback procedure for investigation
- Remote Access Control Policy: access matrix (user role → authorized zones/conduits → allowed protocols), geographic access restrictions (IP whitelisting if applicable), time-based access restrictions
- Remote Access Logging Configuration: log source types (VPN logs, jump server logs, authentication logs), log format standardization, log aggregation into SIEM (per SK-D01-014: SIEM Configuration and Tuning)
- Operational Runbook: VPN client provisioning procedure, jump server access procedure, session recording retrieval procedure, MFA troubleshooting, emergency access procedure (break-glass)
- Compliance verification checklist: encryption strength, authentication strength, audit trail completeness, regulatory requirements (HIPAA, PCI-DSS, NIST if applicable)

## Acceptance Criteria

1. VPN tunnels are established between remote access client and VPN gateway, or between remote sites; tunnel encryption is verified (TLS 1.2+, AES-256 or equivalent)
2. Jump servers are hardened per industry baseline (minimal exposed services, regular patching, host-based firewall enabled)
3. Multi-factor authentication is enforced for all remote access sessions; at least two authentication factors are configured (certificate + password, or password + TOTP)
4. Session recording is configured for all remote access to critical zones (SL 2 and above); session playback capability is tested and verified
5. Remote access control matrix is implemented in firewall/NAC rules; access is restricted to authorized users, zones, and protocols; out-of-policy access attempts are logged and alerted
6. All remote access events (VPN login, jump server access, MFA challenges, file transfer) are logged with user identity, timestamp, source IP, action, and result; logs are integrated into SIEM
7. Operational runbook covers normal access procedures, emergency access procedures (break-glass), MFA recovery, and incident response for suspected compromise of remote access credentials

## Estimated Effort

| Level | Effort | Notes |
|-------|--------|-------|
| Junior (< 2 yr) | 12–18 person-days | Assumes single VPN gateway, 1–2 jump servers, ~50 authorized users; includes configuration, testing, documentation, and MFA setup |
| Senior (5+ yr) | 5–8 person-days | Same scope; senior can leverage templates, automated provisioning, and PAM integration from prior projects |

Notes: Highly distributed remote access (multiple VPN gateways, multiple jump servers per zone) may require 1.5–2× effort. Integration with existing corporate identity management (Active Directory, Okta) may reduce MFA configuration effort by 30–40%. Emergency access and break-glass procedures add 2–3 person-days of design and testing.
