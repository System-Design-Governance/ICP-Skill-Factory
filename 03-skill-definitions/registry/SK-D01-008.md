## Description
Apply STRIDE (Spoofing, Tampering, Repudiation, Information Disclosure, Denial of Service, Elevation of Privilege) and DREAD (Damage, Reproducibility, Exploitability, Affected Users, Discoverability) frameworks to systematically identify and rate threats to OT/ICS components. This skill produces threat catalogs and risk rankings that feed into SK-D01-006 (preliminary threat risk assessment) and SK-D01-007 (detailed risk assessment), and supports the integrated risk assessment triple-method approach (SK-D01-035) by providing the cyber risk foundation for cross-method traceability.

## Description — Objectives
- Apply STRIDE methodology to enumerate threats across six threat categories for OT/ICS systems
- Perform DREAD scoring to assign quantified risk ratings to identified threats
- Create threat catalog with complete threat descriptions, attack vectors, and impact scenarios
- Establish traceability between STRIDE threats and IEC 62443 security requirements
- Integrate STRIDE/DREAD findings into broader risk assessment workflows (GA, FMEA, HAZOP)

## Description — Scope
**In Scope:** Systematic enumeration of threats using STRIDE categories; DREAD scoring methodology; threat catalog creation; OT/ICS-specific threat adaptation; integration with IEC 62443 requirements.

**Out of Scope:** Detailed vulnerability assessment (handled separately); exploitation tool development; threat hunting or active testing; implementation of controls; operational threat monitoring.

## Description — Key Concepts
- **STRIDE:** A threat enumeration framework organized by six threat categories: Spoofing (identity fraud), Tampering (data/process modification), Repudiation (denial of actions), Information Disclosure (unauthorized data access), Denial of Service (availability attacks), Elevation of Privilege (unauthorized capability acquisition)
- **DREAD:** A threat rating framework scoring threats on five dimensions: Damage Potential (impact severity), Reproducibility (ease of attack), Exploitability (technical complexity), Affected Users (scope), Discoverability (attacker visibility)
- **Threat Catalog:** Structured inventory of identified threats with STRIDE classification, DREAD scores, and OT/ICS-specific context
- **OT/ICS Adaptation:** Customization of threat modeling to account for process safety, availability criticality, and legacy system constraints

## Description — Methodology
STRIDE/DREAD threat modeling follows a systematic process:

**Phase 1: System Decomposition**
- Map OT/ICS architecture into threat-modeling units (assets, processes, data flows, actors)
- Align with Zone/Conduit design from SK-D01-001
- Identify trust boundaries and external interfaces

**Phase 2: STRIDE Threat Enumeration**
For each system component, systematically enumerate threats across six categories:
- Spoofing: Can an adversary impersonate a trusted component (e.g., fake SCADA server, spoofed sensor)?
- Tampering: Can an adversary modify data in transit or at rest (e.g., alter control parameters, modify historian logs)?
- Repudiation: Can an adversary deny having performed an action (e.g., disable audit logging)?
- Information Disclosure: Can an adversary access unauthorized information (e.g., eavesdrop on real-time data, extract credentials)?
- Denial of Service: Can an adversary disrupt system availability (e.g., flood network, crash host)?
- Elevation of Privilege: Can an adversary gain unauthorized capabilities (e.g., shift operator to administrator, exploit vulnerability)?

**Phase 3: DREAD Scoring**
Score each threat on five dimensions using ordinal scale (1=Low, 2=Medium, 3=High):
- **Damage:** Impact to process, safety, environment, reputation (1: minimal | 3: catastrophic)
- **Reproducibility:** Can attacker reliably repeat the attack? (1: very difficult | 3: trivial)
- **Exploitability:** What technical skill/tools required? (1: expert only | 3: script-kiddie level)
- **Affected Users:** Scope of impact—single unit, facility, enterprise? (1: few | 3: all)
- **Discoverability:** How easily can attacker find this threat? (1: very obscure | 3: obvious, documented)

**Phase 4: Risk Ranking**
Calculate overall DREAD Risk Score = (Damage + Reproducibility + Exploitability + Affected Users + Discoverability) / 5, range 1-3. High-risk threats (≥2.4) prioritized for control design.

**Phase 5: Catalog Creation**
Document each threat with:
- Threat ID (T-XXX format for traceability with SK-D01-035)
- STRIDE category and DREAD scores
- Attack vector and prerequisite conditions
- Impact scenario (process, safety, data, availability)
- IEC 62443-3-3 requirement linkage
- Applicable security controls (from SK-D01-002 or later implementation)

## Description — Activities
1. Decompose OT/ICS system into threat-modeling components aligned with Zone/Conduit design
2. Conduct STRIDE brainstorming workshops with security architects and OT domain experts
3. Document all identified threats with STRIDE category assignment
4. Perform DREAD scoring consensus with multidisciplinary team
5. Create threat catalog with T-XXX identifiers and linkage to IEC 62443 requirements
6. Validate threat catalog against known OT/ICS threat intelligence (CISA advisories, ICS-CERT)
7. Integrate threat findings into SK-D01-006 preliminary risk assessment
8. Prepare threat summary for presentation to governance (Risk Committee)

## Outputs
- STRIDE Threat Catalog: Structured document listing all identified threats with STRIDE category, DREAD scores, and OT/ICS context
- DREAD Risk Ranking Table: Threats ranked by overall DREAD score (descending)
- Threat-to-IEC 62443 Traceability Matrix: Cross-reference between threat identifiers (T-XXX) and IEC 62443-3-3 functional requirements
- Threat Validation Report: Evidence of threat catalog validation against external threat intelligence sources
- Risk Summary Briefing: Executive summary of high-risk threats (≥2.4 DREAD) with recommended controls

## Acceptance Criteria
- Threat catalog contains ≥20 threats (or 100% of identified threats if fewer) across all six STRIDE categories
- Each threat includes explicit DREAD scores (not ranges or qualitative estimates)
- DREAD scoring rationale is documented for high-risk threats (≥2.4)
- All high-risk threats have explicit linkage to specific IEC 62443-3-3 requirements
- Threat catalog is validated against ≥2 external threat intelligence sources with findings documented
- STRIDE/DREAD findings are explicitly integrated into SK-D01-006 preliminary risk assessment (cross-reference required)
- No threat descriptions use vague language ("may," "could," "potentially"); each specifies concrete attack mechanism

## Description — Practical Guidance
- **Pitfall:** Generic STRIDE threat lists not adapted to OT/ICS specifics. **Guidance:** Tailor STRIDE categories to OT hazards (e.g., tampering = modification of safety-critical setpoints, not just data fields).
- **Pitfall:** DREAD scores assigned without team consensus, leading to score variance. **Guidance:** Conduct consensus scoring sessions with security architects and OT engineers; document dissenting views for high-risk items.
- **Pitfall:** Threat catalog isolated from other risk assessment methods. **Guidance:** Ensure explicit cross-reference to FMEA failure modes and HAZOP deviations in SK-D01-035 integrated assessment.
- **Pitfall:** Over-scoring threats due to theoretical attack paths, ignoring practical feasibility. **Guidance:** Use "Exploitability" and "Discoverability" scores to reflect real-world attack difficulty; note legacy system constraints.
- **Pitfall:** STRIDE/DREAD scores not linked to security control design. **Guidance:** Each high-risk threat must map to specific controls in SK-D01-002 (Defense-in-Depth) or later design phases.
