## Description

Conduct a detailed risk assessment (DTRA) that deepens the preliminary TRA findings into specific security control requirements, countermeasure specifications, and risk treatment plans. Under ICP governance (GOV-SD), this is the mandatory "Complete" assessment that must be achieved by Gate 3 — upgrading the Gate 1 Lite preliminary assessment to a fully traceable risk treatment package. The detailed assessment derives specific functional requirements (FR) and system requirements (SR) per IEC 62443-3-3, maps countermeasures to each identified threat, and produces the residual risk register that feeds Gate 3 handover. The boundary of this skill is detailed risk analysis and control derivation; the integrated triple-method execution (IEC+FMEA+HAZOP) is a distinct methodological skill (SK-D01-035 ⏳), and risk source traceability management is covered by SK-D01-036 ⏳.

## Inputs

- Preliminary TRA report and risk register (from SK-D01-006: TRA Preliminary)
- Finalized zone/conduit architecture (from SK-D01-001: Zone/Conduit Architecture Design)
- Security Level Targets (SL-T) per zone (from SK-D01-010 ⏳)
- Detailed system architecture and design specifications
- Existing security controls inventory (current state)
- IEC 62443-3-3 FR/SR requirements catalogue
- FMEA worksheets (from SK-D01-035 ⏳, if integrated risk assessment executed)
- HAZOP deviation analysis (from SK-D01-035 ⏳, if integrated risk assessment executed)

## Outputs

- Detailed Risk Assessment Report (DTRA): comprehensive document covering all threat scenarios with detailed impact analysis, control derivation, and treatment decisions
- Security Controls Requirements Matrix: FR/SR mapping per zone with status (Implemented / Planned / N/A) and design evidence reference
- Countermeasure Specification by Functional Area:
  - Data protection (storage point and dataflow safeguarding)
  - Hardware protection (physical access, tamper detection)
  - System administration security (account management, audit logging)
  - Application security (input validation, session management)
  - Network security (segmentation enforcement, protocol filtering)
- Risk Treatment Plan: per-risk treatment decision (Mitigate / Accept / Transfer / Avoid) with responsible party and timeline
- Residual Risk Register: risk ID (T-XXX/FM-XXX/HAZ-XXX), original risk level, applied controls, residual risk level, risk acceptance authority per GOV-SD hierarchy (Low→PM; Medium→PM+Security; High→PM+Security+Business; Critical→Engineering Management)
- Updated threat-vulnerability matrix with control effectiveness ratings

## Automation Potential

**Level:** Partial

FR/SR mapping from SL-T levels can be partially automated (lookup table from IEC 62443-3-3). Risk scoring calculation and matrix visualization are automatable. However, countermeasure selection, risk treatment decisions, residual risk judgment, and the contextual assessment of control effectiveness in the specific operational environment require human expertise.

## Acceptance Criteria

1. DTRA covers 100% of risks identified in the preliminary TRA, with no downgrade to "N/A" without documented justification and approver sign-off
2. FR/SR mapping is complete for all zones: every applicable SR has status "Implemented" with design evidence reference (no "Planned" status allowed at Gate 3 per GOV-SD)
3. Every residual risk entry has a traceable risk source ID (T-XXX, FM-XXX, or HAZ-XXX) — untraceable risks trigger Gate 3 failure
4. Risk acceptance is signed by the correct authority per GOV-SD hierarchy (PM for Low; PM+Security for Medium; PM+Security+Business for High; Engineering Management for Critical)
5. Countermeasure specifications cover all functional areas: data protection, hardware, system admin, application, network
6. 20% sampling verification of risk source traceability passes without finding untraceable entries (Design QA Gate 3 verification)
