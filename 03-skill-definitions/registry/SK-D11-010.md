## Description

Maintain the library of internal design standards, templates, and reference architectures used across projects to ensure consistency, quality, and efficiency in engineering delivery. This governance skill encompasses the complete lifecycle of design standards: creation (authoring new standards based on project learning and best practices), review and approval, versioning and change management, applicability mapping (defining when and where each standard is used), retirement (deprecating obsolete standards), and training on standard updates. Design standards form the foundation for repeatable quality and accelerate project delivery by providing proven patterns and templates. This skill works closely with SK-D11-019 (Standards Ownership & Exception Arbitration) to manage standards exceptions and governance. The Head of System Design (GOV-SDP) is responsible for standards library stewardship.

## Inputs

- Project design artifacts and lessons learned (successful designs, common design errors, improvement areas)
- Industry best practices and standards (IEC 62443, NIST SP 800-82, utility/industrial automation best practices)
- Customer requirements and preferences (may drive customization of standard templates)
- Feedback from project teams (usability of standards, gaps in coverage, requests for new standards)
- Regulatory and compliance requirements (must be incorporated into standards)
- Technology and product evolution (new tools, libraries, or approaches that should be standardized)
- Change requests and exceptions (from SK-D11-019) indicating areas where standards need evolution

## Outputs

- Standards library (centralized, versioned repository):
  - Design templates (architecture patterns, security zone templates, network segmentation templates)
  - Technical standards (naming conventions, notation standards, technical decision documentation format)
  - Process standards (design review procedures, design change management process, approval workflow)
  - Reference architectures (proven solutions for common system types: greenfield SCADA, brownfield integration, distributed control)
- Standards documentation for each standard:
  - Applicability (project type, system size, security level, when this standard applies)
  - Detailed standard specification and rationale
  - Examples and case studies (previous projects using this standard)
  - Version history and change log
- Standards lifecycle procedures:
  - Creation/authoring process (who can propose, review criteria, approval authority)
  - Change management (how to update or retire standards)
  - Training and communication plan for standard adoption
- Metrics and dashboard (standards adoption rates, exception frequency, feedback from teams)
- Standards training materials (video walkthroughs, documentation, examples)

## Tools

- Version control system (Git) for standards library storage and change tracking
- Wiki or documentation platform (Confluence, MkDocs, SharePoint) for standards publication
- Design template tools (draw.io, Visio, PowerPoint for architecture templates)
- Change management (Jira, GitHub Issues for tracking standards improvements and exceptions)
- Feedback and survey tools (Slido, SurveyMonkey for collecting project team feedback)
- Metrics and analytics (SQL/BI tools for adoption tracking)

## Standards

- IEC 62443-1-1: Terminology, concepts and models — foundational concepts for all standards
- IEC 62443-3-2: Security Risk Assessment for System Design — standards for zone/conduit and risk assessment
- ISO 9001: Quality Management — standards lifecycle and documentation control
- Best practice: Configuration Management and Change Control from CMMI, RUP, and industry standards organizations

## IEC 62443 Lifecycle Stages

- Applicable across all R0–R5 stages (design standards are created, updated, and retired over project portfolio lifecycle)

## Roles

- Head of System Design (GOV-SDP) — primary: standards library stewardship, standards approval authority
- Design Architects and Lead Engineers — contribute: propose standards based on project learning
- QA (Quality Assurance) — supports: validates standards compliance in projects
- Training and Communications — supports: standards publication and training delivery
- Process Admin — supports: repository maintenance, change tracking

## Dependencies

### Hard Dependencies (Required Before Execution)

- Internal design processes and project execution ongoing — standards are derived from and applied to real project work
- Organizational structure and decision authority defined — standards governance requires clear approval paths

### Soft Dependencies (Enhances Effectiveness)

- SK-D11-006: Quality Plan Development — quality standards drive design standards and acceptance criteria
- SK-D11-019: Standards Ownership & Exception Arbitration — exception tracking reveals standards gaps and improvement opportunities
- SK-D13-003: AI-Assisted Design Review — design standards feed into automated rule sets for review

## Acceptance Criteria

1. Standards library is centralized, versioned, and accessible to all project teams
2. Each standard includes: clear applicability statement (project type, system size, SL, when to use), detailed specification with rationale, and at least one example
3. Standards creation procedure documented and approved: project teams can propose, standards committee reviews, GOV-SDP approves
4. Version control functional: all standard changes tracked with change log, prior versions accessible for reference
5. Standards adoption tracked: metrics show percentage of projects using approved standards (target ≥ 80%)
6. Exception tracking functional: deviations from standards documented and reviewed per SK-D11-019 procedures
7. Annual standards review conducted: feedback from projects collected, obsolete standards retired, new standards added
8. Training program operational: new staff and project teams trained on applicable standards; training materials kept current
9. Standards library audit completed: consistency check across standards, cross-reference validation, applicability mapping verified
10. Governance structure and approval authority documented and approved by organizational leadership

## Estimated Effort

| Level | Effort | Notes |
|-------|--------|-------|
| Senior (5+ yr) | 2–3 days/month ongoing | Monthly maintenance including feedback processing, minor updates, metrics reporting; plus quarterly deep reviews (1–2 days) |
| Standards Committee | 0.5 FTE minimum | Dedicated staff or rotating senior engineers; responsible for standards creation approval, exception review, training |

Notes: Initial standards library build (first 30–50 standards) requires 3–6 months with dedicated team. Mature organizations maintain 50–200+ standards. Annual deep review of entire library requires 1–2 weeks. Major standards overhaul (every 3–5 years) may require 4–6 weeks effort.
