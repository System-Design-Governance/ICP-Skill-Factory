## Description

This skill encompasses the authoring of a comprehensive Hardening Recommended Practices document that provides device-specific and system-specific hardening guidelines for all OT/ICS components in the project scope. The document translates the abstract hardening strategy (from SK-D01-019: Endpoint Hardening) into concrete, verifiable hardening procedures for operating systems (Windows, Linux variants), applications (SCADA, HMI, historian, gateways), and network devices (switches, routers, firewalls). The hardening document serves as both a design baseline and an implementation guide; it documents the "golden standard" configuration for each device type and provides step-by-step procedures for hardening devices at deployment. The scope includes OS kernel parameter tuning, service and feature disabling, application security settings, network device access controls, and security configuration baseline documentation per device type.

## Inputs

- Hardening Strategy (from SK-D01-019: Endpoint Hardening) — high-level hardening objectives and security controls to implement
- System Architecture and Device Inventory (from SK-D02-001 and SK-D01-005) — device types, quantities, and roles in system
- Security Requirements Specification (from SK-D01-003) — security level targets and baseline requirements per device role
- Vendor hardening guides and security baselines (from device manufacturers: Microsoft, Linux distributions, application vendors)
- Industry hardening benchmarks and baselines (CIS Benchmarks, DISA STIGs for OT/ICS systems)
- IEC 62443-3-3 security requirements mapped to hardening controls (from SK-D01-003)
- Access control and credential policies (from SK-D01-020: Account and Access Control)

## Outputs

- Hardening Recommended Practices Document: comprehensive guide organized by device/system type (OS hardening, application hardening, network device hardening) with procedures, configuration parameters, and verification steps
- Hardening Configuration Baselines: templates and parameter tables for each device type, showing "hardened" vs. "standard" settings with justification
- Device-Specific Hardening Checklists: per-device-type implementation checklists with step-by-step procedures, prerequisites, and verification steps
- Security Configuration Reference Tables: parameter names, recommended values, security rationale, and verification methods
- Hardening Verification Procedures: post-deployment tests and automated checks to confirm hardening controls are in place
- Exceptions and Deviation Log: documented deviations from hardening baseline with risk acceptance and compensating controls

## Standards

- IEC 62443-3-3: System security requirements and security levels — baseline security requirements that hardening must satisfy
- IEC 62443-4-2: Technical security measures for industrial automation and control systems
- CIS Benchmarks: Center for Internet Security hardening baselines for Windows, Linux, applications
- DISA STIGs: Defense Information Systems Agency Security Technical Implementation Guides for system hardening
- Microsoft Security Baselines (for Windows hardening)
- Linux Foundation security hardening guidelines
- ID13 format exemplar: documentation structure and style requirements

## IEC 62443 Lifecycle Stages

- R2 (Detailed Design) — hardening strategy and baselines are designed and documented
- R3 (Implementation) — hardening procedures are executed during device configuration and deployment
- R4 (Commissioning/Startup) — hardening is verified and tested during commissioning phase
- R5 (Operations) — hardening baseline is maintained and enforced through configuration management

## Roles

- STC (Security Technical Consultant) — primary: leads hardening document authoring, selects hardening controls aligned with security requirements
- DES (Design Engineer) — supports with device-specific technical details and implementation constraints
- SYS (System Architect) — reviews hardening strategy alignment with overall security architecture
- SAC (Security Architect) — reviews hardening baseline for completeness and appropriateness
- Vendor/OEM Representatives — may provide guidance on vendor-recommended hardening baselines

## Acceptance Criteria

1. Document covers all device types in the system scope: each OS variant, each application platform, and each network device type has hardening procedures documented
2. For each device type, hardening controls are mapped to IEC 62443-3-3 security requirements; coverage matrix shows all applicable requirements are addressed by hardening
3. Hardening procedures include specific parameter names, recommended values, and security rationale for each control; procedures are verifiable and measurable
4. Document includes device-specific hardening checklists for implementation teams with prerequisites, step-by-step procedures, and post-implementation verification steps
5. All hardening deviations from vendor baselines or industry standards (CIS, DISA STIG) are documented with risk acceptance and compensating controls
6. Hardening configuration baselines have been reviewed and approved by STC, SAC, and a system implementation engineer; document is baseline-controlled in configuration management
