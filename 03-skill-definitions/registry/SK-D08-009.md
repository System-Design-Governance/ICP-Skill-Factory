## Description

Execute penetration testing against OT/ICS systems following pre-approved rules of engagement that prioritize operational safety and system continuity. Penetration testing simulates real-world attack scenarios to identify exploitable security weaknesses that an attacker could leverage to compromise system confidentiality, integrity, or availability. OT/ICS penetration testing differs from IT penetration testing in that safety-critical systems cannot be taken offline without business-critical impact; testing must employ non-destructive techniques, use isolated test environments where possible, and coordinate with operations to minimize production disruption. Testing includes network penetration testing (network reconnaissance, vulnerability exploitation, lateral movement), application testing (application-level attacks, authentication bypass), social engineering (if in scope), and OT-specific testing (protocol fuzzing, device/PLC exploitation, HMI interface attacks). Penetration testing findings directly inform security risk acceptance decisions and prioritization of remediation efforts.

## Inputs

- Penetration testing scope definition: systems in scope, test objectives, rules of engagement, exclusions (from SK-D08-002 or similar scoping document)
- Security test plan and methodology (from SK-D08-002 or organizational penetration testing standard)
- Network topology and architecture diagrams (from SK-D01-001, SK-D02-001 ⏳)
- Asset inventory with criticality and safety classification (from SK-D01-005 ⏳)
- System design and security controls documentation
- Known vulnerabilities and advisories for identified technologies
- Authorized credentials for authenticated testing scenarios (if applicable)
- Operational constraints and maintenance windows when testing can occur
- Emergency shutdown and rollback procedures

## Outputs

- Penetration Testing Engagement Report: executive summary, scope, methodology, testing phases, findings summary by severity
- Network Penetration Testing Report: network reconnaissance findings, identified vulnerabilities, successful exploitations, lateral movement chains, persistence techniques attempted
- Application Penetration Testing Report: web application findings, API vulnerabilities, authentication bypass techniques, privilege escalation paths, data exposure risks
- Social Engineering Testing Report (if in scope): phishing campaign results, credential collection success rates, physical security weaknesses, human vulnerability metrics
- OT-Specific Testing Report: protocol fuzzing results, PLC/device exploitation attempts, unsafe configurations identified, HMI interface attacks, safety system interactions
- Exploitation Chains and Attack Scenarios: documented paths from initial access through to critical asset compromise, demonstrating chained vulnerabilities
- Root Cause Analysis: analysis of underlying security control gaps that enabled exploitation
- Remediation Recommendations: prioritized by severity and exploitability, with implementation guidance and effort estimation
- Evidence and Proof-of-Concept: technical artifacts (screenshots, logs, captured traffic) demonstrating each finding
- Risk Assessment Update: impact on overall system risk posture, residual risk after remediation

## Tools

- Network Reconnaissance: nmap, Shodan, Censys, DNS enumeration tools
- Vulnerability Assessment: OpenVAS, Tenable Nessus, Qualys for automated vulnerability discovery
- Exploitation Frameworks: Metasploit, Cobalt Strike for common vulnerability exploitation
- Network Access Testing: Nmap scripts, custom Python scripts for network-level attacks
- Protocol-Specific Tools: custom fuzzing frameworks for OT protocols (MODBUS, DNP3, IEC 60870-5-104), industrial protocol analyzers
- Web/API Testing: Burp Suite, OWASP ZAP, Postman for application-level testing
- Social Engineering Tools: phishing simulation platforms, custom social engineering scripts
- Wireless Testing: Aircrack-ng, Kismet if wireless systems are in scope
- Password Testing: Hashcat, John the Ripper for credential testing
- Post-Exploitation: reverse shells, lateral movement tools, persistence mechanisms appropriate for red team testing
- Documentation: note-taking tools, proof-of-concept video capture, reporting frameworks

## Roles

- STC (Security Technical Consultant) — primary: oversees penetration testing execution, interprets findings, ensures safety compliance, approves test activities
- Penetration Tester / Red Team Lead — executes network and application penetration testing, documents findings
- OT Security Specialist — advises on OT-specific risks, protocol-level testing, safety implications of testing activities
- OPS (Operations Engineer) — provides operational context, coordinates testing windows, monitors operational systems during testing, responds to safety concerns
- Network Administrator — provides network access, assists with test environment setup, validates network impacts
- Application Developer / System Owner — provides system context, assists with application testing, validates findings

## Dependencies

### Hard Dependencies (Required Before Execution)

- Penetration testing scope and rules of engagement approved — penetration testing cannot begin without clear authorized scope and safety boundaries
- SK-D08-002: Security FAT Test Case Design — test objectives guide penetration testing scope
- System design and architecture documentation — penetration testing targets must be well understood
- Operational constraints documented — testing must accommodate operational safety requirements

### Soft Dependencies (Enhances Effectiveness)

- SK-D08-001 ⏳: Factory Acceptance Test Development — penetration testing may be part of FAT security validation
- SK-D08-003: SAT Procedure Development — penetration testing may be executed during SAT in production-like environment
- Vulnerability scan results (SK-D08-010 or prior scanning) — informs penetration test targeting

## Acceptance Criteria

1. Penetration testing scope and rules of engagement are documented and approved by STC, OPS, and project management
2. Scope clearly defines: systems in scope, testing objectives, authorized testing windows, safety constraints, exclusions, emergency procedures
3. Pre-testing coordination meeting with operations: operational constraints, emergency contacts, safety procedures, monitoring during testing
4. Network penetration testing phase executed: network reconnaissance, vulnerability identification, exploitation attempts (with coordination), lateral movement chains documented
5. Application penetration testing phase executed: authentication testing, API testing, web interface testing, privilege escalation attempts
6. Social engineering testing (if in scope): phishing campaigns conducted, credential testing, results documented
7. OT-specific testing executed: protocol fuzzing, device-level exploitation, HMI interface attacks, safety system interactions tested
8. Each finding documented with: affected system/asset, vulnerability description, exploitation technique, impact assessment, evidence/proof-of-concept
9. Exploitation chains documented: paths from initial access to critical asset compromise, demonstrating business risk
10. Root cause analysis performed for each finding: underlying security control gap
11. Remediation recommendations provided with: severity rating, implementation guidance, effort estimate, risk acceptance if not remediated
12. Penetration testing report reviewed and approved by STC, OPS, and project management
13. Post-testing coordination: debrief with operations team, verification that no residual testing artifacts remain, operational system validation

## Estimated Effort

| Level | Effort | Notes |
|-------|--------|-------|
| Junior (< 2 yr) | 15–25 person-days | Assumes moderate system complexity, straightforward network and application testing, limited OT protocol testing; includes reconnaissance, exploitation, documentation, and initial analysis |
| Senior (5+ yr) | 10–18 person-days | Same scope; senior can leverage prior penetration testing experience, advanced exploitation techniques, and faster root cause analysis |

Notes: Complex systems with OT protocols, PLCs, or specialized hardware require advanced OT-specific testing expertise and may add 5–15 days. Distributed systems across multiple sites require 3–5 days for coordination and travel. First-time penetration testing projects may add 2–3 days for stakeholder education and trust building. Systems requiring coordination during active operations (no test windows) may require staged testing over extended timeline.
