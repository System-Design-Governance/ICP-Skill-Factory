## Description

Develop Programmable Logic Controller (PLC) ladder logic programs for industrial automation sequences including motor control, valve sequencing, interlocking logic, and safety-related functions. Ladder logic is programmed in accordance with IEC 61131-3 standard conventions and best practices for readability, maintainability, and safety. This skill translates detailed logic design specifications (from control design phase D05.3) into executable PLC rungs, managing input/output mappings, timer/counter utilization, and safety interlock logic. Programs enforce operational constraints (e.g., preventing simultaneous contradictory commands, enforcing startup/shutdown sequences), and integrate with supervisory control systems through defined communication interfaces per SK-D05-001 (SCADA/RTU Architecture). Security considerations include PLC access control (per SK-D01-020), configuration change management (per SK-D10-003), and firmware integrity verification (per SK-D10-002).

## Inputs

- Logic Design Specification: functional requirements for motor start/stop sequences, valve control, interlocking rules, and safety logic from control design phase
- PLC Hardware Configuration: I/O module types and counts, available memory, processor scan time, communication modules (Ethernet, Modbus, Profibus, etc.)
- I/O Mapping Table: tag definitions (motor names, valve names, sensor names), address allocation, signal type (discrete/analog), safety-related vs. non-safety logic segregation
- Supervisory Interface Requirements (from SK-D05-001 ⏳: SCADA/RTU Architecture Design): command/status signal list, setpoint ranges, alarm thresholds
- PLC Access Control Policy (from SK-D01-020 ⏳: Cybersecurity Access Control Design): program upload/download authorization, password requirements, firmware version control
- Change Management Procedure (from SK-D10-003 ⏳: System Change Management Procedure Design): configuration baseline documentation, change approval workflow
- Safety Standards Applicability: SIL rating (IEC 61508 / IEC 62061) if applicable, functional safety requirements

## Outputs

- PLC Ladder Logic Source Code: documented rung-by-rung implementation with inline comments, addressing IEC 61131-3 conventions
- Logic Documentation: rung descriptions, function block usage, variable/tag definitions, cross-reference lists (which rung affects which output)
- I/O Mapping Verification: source/sink addressing validation, unused I/O identification, conflict detection (e.g., contradictory command logic)
- Test Plan: unit test cases for each major logic block (motor sequence, valve control, interlocking), acceptance criteria
- Safety Logic Verification Report: proof that safety interlocks are correctly implemented, timing analysis for critical sequences
- Programmer's Guide: installation instructions, program load procedure, parameter adjustment procedures, troubleshooting checklist
- Audit Trail and Version Control: PLC program version number, change history, developer identification

## Tools

- PLC Programming IDEs: vendor-supplied (Siemens TIA Portal, Allen-Bradley Studio 5000, Beckhoff TwinCAT, Rockwell RSLogix)
- IEC 61131-3 Compliance Checkers: static code analysis tools (some integrated in vendor IDEs)
- Spreadsheet / Visio (I/O mapping, logic flow diagrams, timing analysis)
- Git / SVN (source code version control for ladder logic files)
- Simulation and Testing Tools: PLC emulation environments (some vendor IDEs include simulation)

## Standards

- IEC 61131-3: Programmable controllers — Part 3: Programming languages (ladder diagram syntax and semantics)
- IEC 61508: Functional Safety of Electrical/Electronic/Programmable Electronic Safety-Related Systems (if SIL-rated logic required)
- IEC 62061: Safety of machinery — Functional safety of safety-related control systems
- IEC 62443-3-3: System Security Requirements and Security Levels — control system security requirements for PLC configuration
- IEC 62443-4-1: Security of Product Development — secure PLC firmware and configuration management
- Vendor-Specific Programming Standards: manufacturer best practices for code readability and maintainability

## IEC 62443 Lifecycle Stages

- R2 (Detailed Design — logic specification and algorithm design)
- R3 (Implementation — PLC program development, unit testing, and field installation)

## Roles

- DES (Design Engineer) — primary: develops ladder logic, validates I/O mapping, documents logic flow
- TST (Test Engineer) — develops and executes unit test cases, validates safety interlocks, performs field commissioning tests
- OPS (Operations Engineer) — provides operational input on motor control sequences, maintenance windows, safety override procedures
- STC (Security Technical Consultant) — reviews PLC access control and change management implementation per SK-D01-020

## Dependencies

### Hard Dependencies (Required Before Execution)

- SK-D05-001 ⏳: SCADA/RTU Architecture Design — establishes PLC/SCADA signal interface and communication protocol

### Soft Dependencies (Enhances Effectiveness)

- SK-D01-020 ⏳: Cybersecurity Access Control Design — program upload/download authorization and password policy
- SK-D10-003 ⏳: System Change Management Procedure Design — PLC program version control and change approval workflow
- SK-D10-002 ⏳: Algorithm Integrity Verification Design — PLC firmware integrity monitoring and change detection

## Acceptance Criteria

1. All logic blocks specified in the design are implemented in ladder logic with clear rung descriptions and inline comments
2. I/O mapping is verified end-to-end: every tag is addressed, no conflicts between rungs, unused I/O is explicitly listed
3. Safety-critical interlocks (e.g., preventing simultaneous contradictory commands) are implemented correctly and verified through test cases
4. Timing analysis confirms that critical sequences (e.g., motor start ramp time) meet operational requirements
5. Static code analysis (IEC 61131-3 compliance checker) confirms syntax correctness and no undefined variables
6. Unit test cases cover 100% of major logic blocks with documented pass/fail results
7. PLC program is under version control (Git/SVN) with documented change history and developer attribution
8. Program change authorization follows SK-D10-003 procedures: approval log, baseline configuration, rollback procedure
9. Documentation has been reviewed and approved by DES, TST, and OPS

## Estimated Effort

| Level | Effort | Notes |
|-------|--------|-------|
| Junior (< 2 yr) | 5–10 person-days | Assumes straightforward logic (20–50 rungs), single PLC, <30 I/O points; includes documentation and unit testing |
| Senior (5+ yr) | 2–5 person-days | Same scope; senior leverages reusable ladder logic templates and rapid debugging methodology |

Notes: Complex interlocking (>100 rungs), multi-PLC coordination, or SIL-rated safety logic may require 1.5–2× effort. Field commissioning and hardware integration testing can add 2–3 days.
