## Description

Conduct systematic security verification of third-party software, firmware, and hardware components before integration into the System under Consideration (SuC) to ensure they meet project security requirements, are free of known critical vulnerabilities, and have documented patch/support status. This skill implements the supply-chain security controls required by IEC 62443-3-3 and IEC 62443-2-1 by establishing a gate before third-party components enter the development or deployment environment. The verification covers vulnerability scanning, security configuration review, compliance with project security policies (e.g., approved vendor lists, cryptography standards), and compatibility with the SuC's security architecture. The boundary of this skill is pre-integration assessment; post-integration security testing and compliance validation fall under device-level or system-level verification skills (D06.x, D09.x).

## Inputs

- Third-party component specifications (name, version, vendor, intended use case within SuC)
- Bill of Materials (BoM) with component inventory and versions (from SK-D01-005 ⏳: Asset Inventory Development)
- Project security requirements specification (from SK-D01-007 ⏳: Security Requirements Specification, or SOW)
- Approved vendor list (if customer-specified)
- Known vulnerability databases (CVE, CVSS, vendor security advisories)
- Vendor security documentation: security datasheet, patch/support lifecycle, cryptography algorithms employed, hardening guides
- IEC 62443 supply-chain security checklist per ID01 §6.4 and ID02 Annex B.3

## Outputs

- Third-Party Component Security Verification Report (per component or component family): component name, version, vendor, assessment date, vulnerability scan results (CVE count, severity breakdown, CVSS scores), patch/support status (end-of-life date, patch frequency history), security configuration compliance, project security requirements alignment assessment, risk rating (approved/conditional/rejected), and remediation recommendations
- Compliance Assessment Matrix: component security properties vs. project requirements (encryption algorithms, key lengths, authentication methods, logging capabilities) with gaps and mitigations
- Vulnerability Remediation Plan (if conditional approval): list of identified vulnerabilities, vendor patch availability, patch validation timeline, interim compensating controls, acceptance criteria for moving to approved status
- Supply-Chain Risk Register: tracked third-party components with patch status, end-of-life dates, and escalation triggers (e.g., zero-day announcement, vendor discontinuation)
- Approved Components List: curated inventory of pre-approved third-party components with version, assessment date, and re-verification schedule (per ID02 B.3)

## Tools

- Vulnerability scanning platforms (Qualys, Tenable Nessus, Rapid7 InsightVM, OWASP Dependency-Check)
- CVE/NVD databases (National Vulnerability Database, vendor-specific security advisories)
- Bill of Materials (BoM) analysis tools (Black Duck, WhiteSource, Synopsys)
- Security configuration analysis tools (vendor-specific configuration auditors)
- IEC 62443 supply-chain security checklists and assessment templates
- Risk scoring matrices (CVSS, custom project-specific risk models)
- Spreadsheet / database for component tracking and re-verification scheduling

## IEC 62443 Lifecycle Stages

- R1 (Security Requirements — component risk assessment against project security targets)
- R2 (Detailed Design — component pre-integration verification and approved components list)

## Roles

- STC (Security Technical Consultant) — primary: conducts vulnerability assessment, evaluates security compliance, determines risk ratings
- SAC (Security Architect) — approves conditional approvals and risk escalations; defines security requirements for component selection
- Procurement / Supply Chain — coordinates vendor documentation requests and approval lifecycle
- System Integrator / Development Team — provides component list and intended integration context

## Dependencies

### Hard Dependencies (Required Before Execution)

- SK-D01-005 ⏳: Asset Inventory Development — complete BoM with third-party components is required for assessment
- SK-D01-007 ⏳: Security Requirements Specification — project security requirements define verification criteria

### Soft Dependencies (Enhances Effectiveness)

- SK-D01-001: Zone/Conduit Architecture Design — component placement in zones informs risk assessment (e.g., components in critical zones face stricter scrutiny)
- SK-D01-010 ⏳: Security Level Target Assessment — SL requirements refine component security requirements
- Supply-chain risk management policies (customer or corporate) — inform component approval criteria and escalation procedures

## Acceptance Criteria

1. Every third-party component in the SuC BoM has a corresponding security verification report completed and signed by STC
2. Vulnerability scan reports document CVE count, CVSS scores, and vendor patch/remediation status (no "unknown" components)
3. Security requirements compliance matrix shows alignment of component cryptography, authentication, logging, and configuration hardening with project requirements
4. Approved Components List is curated, version-specific, and approved by SAC; conditional approvals include explicit remediation plans and timelines
5. Re-verification schedule is defined for all components (annually for stable components, after major vendor updates, or upon disclosure of critical vulnerabilities)
6. All rejected components have documented denial reasons and alternative component recommendations
7. Supply-Chain Risk Register is actively maintained with escalation triggers (vendor discontinuation, end-of-life, critical CVE) and monitoring schedule

## Estimated Effort

| Level | Effort | Notes |
|-------|--------|-------|
| Junior (< 2 yr) | 4–6 person-days | Assumes ~20 components, standard tools and checklists; includes vulnerability scanning and report compilation |
| Senior (5+ yr) | 2–3 person-days | Same scope; senior leverages vendor relationships and known component profiles from prior projects |

Notes: Complex custom or proprietary components may require additional vendor communication and security evaluation, adding 1–2 days per component. Large-scale multi-vendor environments (50+ components) may require specialized BoM management tooling, adding setup overhead of 2–3 days. Recurring re-verification effort is typically 50% of initial assessment (annual refreshes, targeted updates).
