## Description

This skill encompasses the development and maintenance of a Requirements Traceability Matrix (RTM) that links customer requirements to design elements, test cases, and verification evidence throughout the project lifecycle. The RTM establishes bidirectional traceability: every customer requirement has a corresponding design response, every design element traces back to a customer requirement, and every test case validates one or more requirements. Effective RTM management ensures complete requirements coverage, prevents scope creep, and provides evidence of systematic verification and validation aligned with IEC 62443 secure system engineering principles.

---

## Description — Business Context

Requirements traceability is a fundamental discipline in secure system engineering per IEC 62443-3-3. Without a formal RTM, organizations risk implementing designs that do not satisfy customer needs, missing verification of critical requirements, or failing to demonstrate that security controls are effective. The RTM serves as the authoritative record of project scope, design decisions, and validation evidence. In compliance audits, regulators examine the RTM as primary evidence that the design was systematic and security-focused.

## Description — Procedural Steps

**Phase 1: RTM Structure Definition**
1. Define RTM columns:
   - Requirement ID: unique identifier for each requirement (e.g., REQ-001, SYS-002)
   - Requirement Text: concise statement of customer/system need
   - Requirement Type: functional, non-functional, performance, security, environmental
   - Acceptance Criteria: testable conditions for requirement satisfaction
   - Design Element ID: design component or subsystem that responds to requirement
   - Design Document Reference: specification or design drawing that documents the response
   - Test Case ID(s): test(s) that validate the requirement
   - Verification Evidence: test report reference, inspection results, analysis outputs
   - Status: complete, in-progress, deferred, not-applicable
   - Risk/Priority: relative importance and risk of non-satisfaction
   - Notes: traceability gaps, assumptions, dependencies
2. Establish RTM naming conventions and ID schemes
3. Select RTM storage format: spreadsheet (Excel, Sheets), dedicated tool (DOORS, Helix), or custom database
4. Create RTM template with metadata header: project name, version, approval date, maintainer

**Phase 2: Requirements Identification & Analysis**
1. Extract requirements from CRS/FSD; categorize by type (functional, non-functional, security, environmental)
2. For each requirement, define:
   - Testable acceptance criteria: clear, measurable conditions for validation
   - Priority/Risk ranking: relative importance; risk of non-satisfaction
   - Assumptions and constraints
   - Related requirements (dependencies, conflicts)
3. Verify requirement characteristics:
   - Clear and unambiguous language
   - Testable and verifiable
   - Feasible within project constraints
   - Aligned with IEC 62443 security level target
4. Identify "derived requirements" emerging from design: internal requirements not in CRS but necessary for design completeness
5. Update RTM with complete requirement list; obtain stakeholder approval

**Phase 3: Design Response Mapping**
1. For each requirement, identify design element(s) responding to it:
   - Architecture component (e.g., "Authentication Module", "Firewall")
   - Design subsystem or functional block
   - Specification section or design drawing reference
2. Document design response in RTM:
   - Design element ID and name
   - Design document reference (specification section, drawing number, document version)
   - Design rationale or key implementation approach (brief summary)
3. Verify bi-directional traceability:
   - Each requirement has ≥1 design response
   - Each design element responds to ≥1 requirement (no orphan designs)
4. Analyze design-requirement alignment:
   - Do design implementations fully satisfy requirement acceptance criteria?
   - Are there design elements addressing security concerns identified in threat analysis?
5. Document any requirement-design gaps or mismatches; escalate for resolution

**Phase 4: Test Case Development & Verification Mapping**
1. For each requirement, develop test case(s):
   - Test Case ID: unique identifier (e.g., TC-001, TC-002)
   - Test Scenario: inputs, conditions, stimuli
   - Expected Results: observable outputs per acceptance criteria
   - Verification Method: test execution, simulation, inspection, analysis, demo
   - Acceptance Criteria: pass/fail conditions
2. Document test cases in RTM:
   - Map each test case to requirement(s) it validates
   - Verify that all acceptance criteria are covered by test cases
3. Establish verification evidence collection:
   - Test reports documenting test execution, results, anomalies
   - Inspection records (design review, code review)
   - Analysis results (FMEA, safety analysis, security assessment)
   - Simulation/modeling evidence
4. Update RTM with verification evidence references:
   - Test report ID and date
   - Evidence document reference and version
   - Pass/fail status
5. Analyze coverage:
   - Requirement-to-test coverage: verify all requirements have test cases
   - Test-to-requirement coverage: verify all tests validate a requirement (no orphan tests)

**Phase 5: Traceability Analysis & Gap Closure**
1. Conduct traceability impact analysis:
   - **Backward traceability:** Every design element traces to a requirement
   - **Forward traceability:** Every requirement has design response
   - **Verification traceability:** Every requirement has test case and verification evidence
2. Identify gaps:
   - Unmapped requirements (no design response) → escalate for design completion
   - Orphan design elements (no requirement) → assess necessity; remove if not justified
   - Uncovered requirements (no test case) → plan verification activity
   - Missing verification evidence → assign to V&V team for completion
3. Create traceability closure list:
   - Gap description, responsible party, target closure date
   - Track gap resolution through project status reviews
4. Periodic gap audit (weekly/bi-weekly during design phase):
   - Verify closure of previously identified gaps
   - Identify any new gaps introduced by requirements changes or design evolution

**Phase 6: Requirements Change Management**
1. Establish change control for RTM:
   - Requirements changes require change request approval (per Management of Change process)
   - Impact analysis: effect on design, test cases, schedule, budget, security posture
2. Update RTM upon change approval:
   - Modify requirement text or acceptance criteria
   - Identify affected design elements and test cases
   - Flag design/test updates required; assign to responsible parties
3. Maintain change history:
   - Version number, change date, author, approval
   - Change summary (what changed, why)
4. Track requirement dependencies:
   - Identify requirements that depend on the changed requirement
   - Assess cascade effects; update traceability as needed

**Phase 7: RTM Maintenance & Verification**
1. RTM review gates at project milestones:
   - End of requirements phase: 100% requirement collection and analysis complete
   - End of design phase: 100% design-requirement mapping and verification mapping complete
   - Pre-test phase: 100% test cases developed; verification evidence plan finalized
   - Post-test: 100% verification evidence collected; all requirements validated
2. Periodic audits (monthly or per project schedule):
   - Verify RTM data consistency and accuracy
   - Check for new gaps introduced
   - Validate requirement status reflects actual design/test progress
3. Maintenance of RTM database:
   - Archive superseded versions
   - Update status fields as design and testing progress
   - Ensure document references remain valid (update if documents relocate)

## Acceptance Criteria

- RTM structure defined with all required columns (requirement, design response, test case, verification evidence, status)
- 100% of customer requirements identified, prioritized, and entered into RTM with testable acceptance criteria
- Backward traceability verified: every design element mapped to ≥1 requirement
- Forward traceability verified: every requirement has ≥1 design response documented with design reference
- Verification traceability verified: every requirement has ≥1 test case and planned verification evidence
- Bi-directional traceability complete: no unmapped requirements, no orphan design elements, no uncovered requirements
- Change control process implemented: all RTM changes require approval and impact analysis
- RTM baseline approved by customer, system architect, and QA lead
- Gap closure list tracked and managed; all critical gaps resolved before design phase completion

## Description — Common Pitfalls

**Pitfall: Incomplete or ambiguous requirements leading to traceability gaps**
- *Mitigation:* Establish clear requirement specification guidelines; conduct requirements review before RTM entry; use structured templates for requirement writing

**Pitfall: Design elements implemented without corresponding requirements (scope creep)**
- *Mitigation:* Enforce RTM-based design: verify requirement exists before design begins; periodic audit of design artifacts against RTM

**Pitfall: Weak design-requirement mapping: mapping exists but design does not actually satisfy requirement**
- *Mitigation:* Design review process validates that design response fully addresses requirement; verify acceptance criteria; gap analysis bridges mismatches

**Pitfall: Test cases developed independently of requirements; poor test-requirement traceability**
- *Mitigation:* Develop test plan and test cases in parallel with RTM; establish requirement-test mapping during test development

**Pitfall: RTM becomes outdated as requirements change; traceability loses accuracy**
- *Mitigation:* Enforce change control: no requirement changes without RTM update and impact analysis; periodic audit of RTM consistency

**Pitfall: Manual RTM maintenance prone to errors; traceability breaks when documents updated**
- *Mitigation:* Use dedicated requirements management tool with change tracking and impact analysis; periodic automated verification of document references

## Estimated Effort

- RTM completeness: 100% of known requirements entered; 0 undocumented requirements
- Forward traceability: 100% of requirements have ≥1 design response; average design response time <3 business days
- Backward traceability: 100% of design elements mapped to requirements; 0 orphan designs identified
- Verification traceability: 100% of requirements have test case(s); average test case development time <2 business days
- Bi-directional traceability: 0 unmapped requirements, 0 orphan design/test elements
- Gap closure rate: all identified gaps tracked; 100% critical gaps closed; closure cycle time <5 business days
- Change management: 100% of RTM changes have approved change request; impact analysis completed before design/test update
- Audit findings: RTM audits identify <5% data inconsistencies; issues corrected within 3 days

## Description — Continuous Improvement

- Analyze post-test gap closure data: which requirement types consistently show late closure? Refine requirement specification approach
- Track design-requirement mismatches: do certain design teams struggle with requirement interpretation? Provide targeted training
- Monitor test-requirement mapping effectiveness: are test cases discovering requirement ambiguities? Strengthen requirements review process
- Periodic review of RTM tool effectiveness: does current tool meet traceability needs? Consider alternative tools if limitations identified
- Benchmark RTM completeness/closure metrics against industry standards and historical project data
