## Description

Configure Security Information and Event Management (SIEM) systems for OT/ICS environments, including log source integration, correlation rule development, alert threshold tuning, and dashboard creation. This skill implements the security monitoring architecture designed under SK-D01-015 (Security Alarm Rule Design), translating abstract alarm rules and correlation logic into operational SIEM configurations that detect and respond to cybersecurity threats in real time. It covers integration of diverse log sources (firewalls, access control systems, field devices, historian), development of practical correlation rules that balance alert fidelity with operational burden, configuration of alert severity and routing, and creation of security dashboards for operational awareness. The boundary of this skill is hands-on SIEM configuration; the design of alarm rules falls under SK-D01-015, and incident response procedures fall under D07.4.

## Inputs

- Security Alarm Rule Design specification (from SK-D01-015: Security Alarm Rule Design)
- Approved Zone/Conduit Architecture (from SK-D01-001: Zone/Conduit Architecture Design)
- Inventory of OT/ICS assets and systems that generate logs (asset inventory with system types, locations, protocols)
- Network and data flow diagrams showing log source locations (from SK-D02-004 ⏳)
- Security monitoring requirements (from SK-D01-009 ⏳: Security Monitoring Plan Development)
- SIEM platform specifications (vendor documentation, available integrations, storage capacity, retention policy)
- Business continuity and operational constraints (acceptable log volumes, alert fatigue thresholds)

## Outputs

- SIEM configuration baseline (XML, JSON, or native format exports per SIEM vendor)
- Log source integration checklist: source ID, source type, protocol/method, data format, sample validation
- Correlation rule library: rule ID, trigger condition, severity level, action (alert, escalation, block), testing results
- Alert threshold tuning documentation: metric, baseline, threshold value, justification, anomaly detection parameters
- SIEM dashboard specifications: dashboard name, key metrics, alert aggregation, SLA indicators
- Operational runbook: log data retention policy, archive procedure, search methodology, backup/recovery
- Integration test report validating end-to-end flow from log source through alert generation and response

## Tools

- SIEM platform (Splunk, ELK Stack, ArcSight, etc.) with OT-specific connectors
- Log collection agents (syslog, WinRM, SNMP, Syslog-ng, Fluentd)
- Protocol analyzers (Wireshark) for validating log format before integration
- Excel / structured markdown (correlation rule library, threshold documentation)
- Dashboard builder tools native to SIEM platform

- STC (Security Technical Consultant) — primary: configures SIEM, develops correlation rules, tunes thresholds
- SYS (System Architect) — provides operational context, validates feasibility of log collection without impacting OT

## Dependencies

### Hard Dependencies (Required Before Execution)

- SK-D01-015: Security Alarm Rule Design — cannot configure SIEM without knowing what rules and alarm categories to implement
- SK-D01-001: Zone/Conduit Architecture Design — log sources are mapped to zones and conduits; architecture drives log routing strategy

### Soft Dependencies (Enhances Effectiveness)

- SK-D01-009 ⏳: Security Monitoring Plan Development — monitoring strategy informs which data sources are critical to collect
- SK-D02-004 ⏳: Data Flow Diagram Development — data flows identify additional log sources and correlation opportunities
- SK-D01-002 ⏳: Defense-in-Depth Strategy Design — monitoring is a key pillar of defense-in-depth implementation

## Acceptance Criteria

1. All log sources specified in the alarm rule design are integrated into SIEM and producing data
2. At least 80% of log entries are successfully parsed and normalized into searchable fields
3. Every correlation rule has been tested with synthetic/historical data and produces expected alerts
4. Alert threshold tuning baseline is documented with rationale; false positive rate is < 5% over 2-week test period
5. SIEM dashboards display key metrics (active alerts, event throughput, zone-level anomalies) with < 5 min refresh latency
6. Operational runbook includes step-by-step procedures for dashboard review, alert investigation, and log archival
7. Integration testing confirms end-to-end alert flow: trigger condition → alert generation → dashboard display → escalation routing

## Estimated Effort

| Level | Effort | Notes |
|-------|--------|-------|
| Junior (< 2 yr) | 12–18 person-days | Assumes ~30 log sources, 15–20 correlation rules, basic dashboard; includes integration testing and threshold tuning |
| Senior (5+ yr) | 5–8 person-days | Same scope; senior can leverage SIEM templates, automated tuning, and pattern libraries from prior projects |

Notes: Complex OT environments with diverse protocols (Modbus, IEC 60870-5-104, DNP3) may require custom parsers and add 5–10 days. Multi-SIEM deployments (federated architecture) may require 1.5–2× effort.
