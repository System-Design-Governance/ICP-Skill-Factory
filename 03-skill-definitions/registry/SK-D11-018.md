## Description

Verify design quality and traceability per the Design QA role responsibilities defined in governance structure (GOV-SD: Design QA). This verification skill is executed at critical IEC 62443 lifecycle gates (R1 Design Baseline Verification and R3 Pre-Delivery Verification) to confirm that design artifacts meet quality standards and that all requirements are traceable to implementation and test. Verification encompasses requirements-to-design traceability matrix completeness, document register currency and consistency, design review finding closure and resolution verification, and acceptance criteria satisfaction. The skill ensures design artifacts are audit-ready and meet contractual and regulatory documentation requirements. Verification is a gate function that precedes design baseline approval (R1) and delivery (R3).

## Inputs

- Design artifacts (all deliverables from current phase: specifications, architecture diagrams, detailed designs, datasheets, interface control documents)
- Requirements specification (from SK-D01-008 ⏳, SK-D02-001 ⏳, SK-D03-001 ⏳, etc.)
- Quality plan and acceptance criteria (from SK-D11-006: Quality Plan Development)
- Traceability matrix (requirements-to-design mapping, design-to-test mapping if R3)
- Document register (list of all design deliverables, version control, approval status)
- Design review findings and closure records (from SK-D02-012: Design Review — Detailed and SK-D11-001: Design and Requirements Ownership)
- Previous design verification reports (if re-verification or phased design)
- GOV-SD Design QA role procedures and checklist

## Outputs

- Design Quality Verification Report including:
  - Traceability matrix completeness assessment (percentage of requirements traced to design; any gaps identified and disposition)
  - Document register audit results (all expected deliverables present; versions match requirements; approval signatures complete)
  - Design review finding closure verification (all findings resolved and closed with evidence; any open findings escalated)
  - Acceptance criteria satisfaction check (for each requirement, evidence that design satisfies acceptance criteria)
  - Quality standard compliance (naming conventions, design diagram standards, documentation completeness per SK-D11-006)
- Verification sign-off (GOV-SDP signature on design baseline approval or pre-delivery release)
- Disposition of findings (if any gaps or non-conformances found: corrective action assignments, revised closure target date)
- Design artifact quality metrics (consistency scoring, compliance scoring, readiness index)
- Audit trail (verification evidence, inspection records, reviewer sign-offs)

## IEC 62443 Lifecycle Stages

- R1 (Security Requirements Verification — design baseline readiness assessment)
- R3 (Implementation Verification — pre-delivery design and documentation completeness)

## Roles

- Head of System Design (GOV-SDP) — primary verification authority; signs off on design baseline and pre-delivery status
- Design QA (from GOV-SD) — execution: conducts verification, prepares report, tracks findings to closure
- Requirements Engineer — supports: validates traceability matrix completeness
- Document Control / Configuration Management — supports: maintains document register, version control

## Dependencies

### Hard Dependencies (Required Before Execution)

- All design artifacts for the current phase completed or marked as draft for review — cannot verify incomplete designs
- Requirements specification finalized and baselined — traceability verification requires stable requirements
- Quality plan and acceptance criteria defined (from SK-D11-006) — verification checklist based on quality plan
- Design review process completed (from SK-D02-012, SK-D11-001) — finding closure verification requires design review findings

### Soft Dependencies (Enhances Effectiveness)

- SK-D13-003: AI-Assisted Design Review — automated completeness and consistency checking can pre-screen artifacts before manual verification
- SK-D09-005: Document Management System — DMS provides artifact version history and approval chain visibility
- SK-D11-010: Internal Design Standards Maintenance — design standards compliance is part of quality verification

## Acceptance Criteria

1. Traceability matrix populated for 100% of requirements; any requirements without design trace documented with rationale and risk assessment
2. Document register audit complete: every expected deliverable present, versions aligned with approval records, no orphaned documents
3. Design review findings closure verified: all findings from SK-D02-012 and SK-D11-001 review marked closed with evidence of resolution
4. Acceptance criteria for each requirement verified against design artifacts: evidence captured (e.g., "requirement 4.2.1 satisfied by architecture diagram section 3.2, page 7")
5. Quality standard compliance confirmed: naming conventions, diagram standards, document structure per SK-D11-006 and SK-D11-010
6. Design artifacts assessed as audit-ready: ready for customer delivery and compliance review
7. Verification report issued with sign-off by GOV-SDP (Design QA authority); any non-conformances assigned corrective action
8. Verification audit trail retained: inspection records, reviewer evidence, approval signatures

## Estimated Effort

| Level | Effort | Notes |
|-------|--------|-------|
| Senior QA Engineer (5+ yr) | 2–5 days per gate | R1 or R3 verification depending on design complexity and size; includes artifact review, traceability audit, report generation |

Notes: Effort depends heavily on design complexity (single-zone system vs. multi-area network), artifact count (10 documents vs. 100+), and prior quality. Well-managed projects with AI-assisted review (SK-D13-003) may require lower effort. Major design changes requiring re-verification may add 1–3 days.
