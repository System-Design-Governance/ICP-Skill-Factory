## Description

Execute vulnerability scanning using OT-aware vulnerability scanning tools, analyze discovered vulnerabilities, classify each finding by severity using CVSS (Common Vulnerability Scoring System) scoring, and produce comprehensive vulnerability assessment reports with prioritized remediation recommendations. Vulnerability scanning is a key verification activity during commissioning (FAT/SAT) and ongoing security assurance throughout operations. OT/ICS vulnerability scanning differs from IT scanning in that many OT environments cannot tolerate active scanning due to process disruption risk; scanning must be scheduled during maintenance windows or executed using passive scanning techniques that do not generate disruptive network traffic. Scanning covers network-layer vulnerabilities (open ports, services, banner information), application-layer vulnerabilities (web services, APIs, custom applications), device firmware vulnerabilities, and configuration weaknesses. Scanning results are analyzed to understand risk context (is vulnerability exploitable, what impact would exploitation have), classified by severity, and presented with remediation guidance prioritized by risk and implementation effort.

## Inputs

- System inventory and asset list (from SK-D01-005 ⏳: Asset Inventory Development)
- Network topology and architecture diagrams (from SK-D01-001, SK-D02-001 ⏳)
- System design and security requirements (from SK-D01-005, SK-D01-006)
- Known vulnerabilities and security advisories for identified technologies
- Historical vulnerability scan data (if available, for trend analysis)
- Operational constraints: maintenance windows, sensitivity to network scanning, critical process schedules
- Scanning tool configuration and credentials (if authenticated scanning is planned)
- Remediation policies and severity classification standards (from organizational security standards)

## Outputs

- Vulnerability Assessment Report: executive summary, scan scope, scanning methodology, vulnerability statistics (by severity, by asset type, by vulnerability category)
- Detailed Vulnerability Findings: per-vulnerability listing with: vulnerability ID (CVE), affected asset/service, vulnerability description, CVSS score and vector, severity rating (Critical/High/Medium/Low), impact assessment
- Vulnerability Remediation Roadmap: prioritized list of remediation actions with severity, exploitability assessment, business impact, implementation effort, recommended timeline
- Remediation Recommendations: per-vulnerability guidance including: temporary mitigations (if full remediation delayed), permanent remediation steps, vendor patches or configuration changes, testing requirements
- Risk Assessment Update: updated risk register entries reflecting vulnerability findings and proposed remediation
- CVSS Scoring Worksheet: detailed CVSS scoring for each vulnerability with temporal and environmental modifiers
- Scan Evidence and Supporting Data: raw scan results, supporting network logs, screenshot evidence of findings
- Trends and Analysis Report (if historical data available): vulnerability trend analysis, improvement/degradation assessment over time

## Tools

- OT-Aware Vulnerability Scanners: Tenable Nessus with OT plugin, OpenVAS, Qualys VMDR with OT signatures
- Passive Network Scanning: Wireshark, network TAP with passive vulnerability detection, NetFlow analysis
- Service Enumeration: nmap, Shodan, Censys for identifying exposed services and versions
- Vulnerability Databases: NVD (National Vulnerability Database), vendor security advisories, CVE databases
- CVSS Scoring Tools: NVD CVSS calculator, CVSS v3.1 vector string tools
- Reporting and Analysis: custom scripts (Python, Excel) for vulnerability aggregation, Splunk for log-based vulnerability detection
- Configuration Review Tools: OpenSCAP for configuration vulnerability assessment
- Firmware Analysis: binwalk, firmwalker for embedded system vulnerability analysis (if applicable)

## Roles

- STC (Security Technical Consultant) — primary: designs scanning strategy, executes scans, analyzes results, prioritizes remediation
- QA (Quality Assurance Engineer) — coordinates scanning execution, manages scanning schedules, coordinates with operations
- Vulnerability Analyst — performs detailed analysis of findings, CVSS scoring, impact assessment
- OPS (Operations Engineer) — coordinates scanning windows, monitors operational impact, validates that scanning does not disrupt processes
- System Owner / Asset Manager — provides context on asset criticality, validates findings, confirms remediation feasibility
- Remediation Team — implements remediation actions, validates patches and configuration changes

## Acceptance Criteria

1. Scanning scope is clearly defined: assets to be scanned, scan types (active/passive), scanning windows, exclusions
2. OT-aware scanning tools are used: Nessus with OT plugin or equivalent; passive scanning employed where active scanning risks disruption
3. Scanning is executed during approved maintenance windows or using passive techniques that do not impact operations
4. All discovered vulnerabilities are documented with: asset/service affected, vulnerability description, CVE (if applicable), affected version
5. CVSS scoring is applied to each vulnerability: base score, temporal modifiers (exploit maturity, available patches), environmental modifiers (OT context)
6. Severity classification is consistent: Critical (CVSS 9.0–10.0), High (7.0–8.9), Medium (4.0–6.9), Low (0.1–3.9)
7. Vulnerability analysis includes: exploitability assessment, business impact, context within OT environment
8. Remediation recommendations are provided for each vulnerability: temporary mitigations (if applicable), permanent fixes, required testing
9. Remediation roadmap prioritizes by: severity, exploitability, remediation effort, operational risk
10. Risk register is updated reflecting vulnerability findings and proposed remediation plan
11. Trend analysis performed (if historical data available): improvement/degradation over time, effectiveness of prior remediation efforts
12. Vulnerability assessment report is reviewed and approved by STC and project management

## Estimated Effort

| Level | Effort | Notes |
|-------|--------|-------|
| Junior (< 2 yr) | 5–10 person-days | Assumes moderate system complexity, straightforward scan execution, standard vulnerability analysis; includes scan setup, analysis, CVSS scoring, and reporting |
| Senior (5+ yr) | 2–6 person-days | Same scope; senior can leverage automated analysis, prior vulnerability context, and faster prioritization |

Notes: Large systems with many devices (>200 assets) may require 1.5–2× effort for comprehensive analysis. Systems with many custom applications or proprietary protocols may require specialized scanning and analysis. Comprehensive remediation roadmap development may add 2–4 days for detailed impact assessment and resource planning.
