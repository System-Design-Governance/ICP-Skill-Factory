## Description

This skill encompasses the systematic evaluation of change requests by analyzing technical impact, schedule impact, cost impact, and security impact, producing comprehensive change evaluation reports that support change approval decisions. Per IEC 62443 ID01 §7.8.7, changes that affect security controls require explicit security impact assessment to ensure that modifications do not degrade security posture or violate the target security level. The change evaluation process integrates technical feasibility analysis, project schedule impact, budget implications, and risk assessment to inform Configuration Control Board (CCB) decision-making.

---

## Description — Business Context

During the project lifecycle, stakeholder needs evolve, technical constraints become apparent, and customer requirements may be refined. The change management process provides a disciplined mechanism to evaluate proposed changes before approval, preventing ad-hoc modifications that could compromise security, cause schedule overruns, or exceed budget. Per IEC 62443 ID01 §7.8.7, security-impacting changes must undergo security impact assessment and may trigger re-certification if the target security level is affected. Robust change evaluation ensures organizational visibility into project scope evolution and supports evidence of secure engineering practices.

## Roles

**Primary Stakeholders:**
- Project Manager (PM): initiates change evaluation process
- System Architect: assesses technical impact and feasibility
- Security Officer/Security Architect: evaluates security impact
- Quality Assurance (QA): assesses quality and verification impact
- Finance/Budget Owner: assesses cost impact and budget availability
- Configuration Control Board (CCB): reviews evaluation and approves/rejects change
- Requirement Originator: clarifies change rationale and acceptance criteria

**Downstream Skill Dependencies:**
- Feeds SK-D10-003 (Management of Change Execution)
- May trigger SK-D11-020 (Design Change Impact Analysis & SL Recertification) per GOV-SD Gate 2

## Description — Knowledge Domains

- Technical feasibility analysis: design complexity, component availability, integration risk
- Requirements impact: affected requirements, traceability implications, scope creep assessment
- Schedule impact: task duration changes, dependency effects, critical path analysis
- Cost impact: labor, materials, third-party services, contingency reserve
- Security impact assessment: effect on security controls, threat model changes, SL implications
- Risk analysis: risk identification, probability/impact assessment, mitigation strategies
- Verification and validation impact: test plan changes, schedule implications, resource requirements
- Change classification: emergency vs. standard, critical vs. non-critical, security-affecting vs. non-security

## Inputs

- Change request form and process defined: submission format, required information, routing procedures
- CCB (Configuration Control Board) established with clear authority matrix: approvers by change type/scope
- Baseline requirements, design, and test plan established and baselined
- Project schedule, budget, and resource plan baselined
- IEC 62443 security level (SL) target and threat model documented
- Security impact assessment criteria defined: what constitutes a security-affecting change
- Evaluation template and decision criteria documented
- Team member training completed on change evaluation procedures and IEC 62443 requirements

## Description — Procedural Steps

**Phase 1: Change Request Intake & Initial Review**
1. Receive change request from submitter; verify completeness:
   - Change request ID and date
   - Submitter name, organization, contact information
   - Description of proposed change: clear, concise statement of what is changing
   - Reason for change: rationale, business justification, problem being solved
   - Affected items: requirements, design elements, components, test cases, documents
   - Proposed implementation: brief technical approach, if known
2. Perform initial screening:
   - Is change within project scope? (out-of-scope changes may be rejected or deferred)
   - Is change clear and well-documented? (request missing information; do not evaluate incomplete changes)
   - Is change a duplicate of previous request? (consolidate if appropriate)
3. Assign change request ID and log in change request tracking system
4. Route to appropriate evaluation team based on change type and scope

**Phase 2: Technical Impact Analysis**
1. Identify affected design elements:
   - Which requirements are affected?
   - Which design components/subsystems are affected?
   - Which documents require update (specification, design drawing, wiring diagram, CBOM)?
2. Assess technical feasibility:
   - Can the change be implemented with existing technology and components?
   - Are required components available with adequate lead time?
   - What is the implementation risk? (complexity, dependencies, unknowns)
   - Does change create new dependencies or interfaces?
3. Evaluate design impact:
   - Does change improve design quality, maintainability, performance, or security?
   - Does change introduce new risks or failure modes?
   - What is the effort required to update design documentation?
4. Document technical findings:
   - Feasibility assessment: technically feasible, feasible with risks, not feasible
   - Implementation approach: high-level technical strategy
   - Affected design elements and documents
   - Technical risks and mitigation strategies
   - Design review/approval requirements

**Phase 3: Security Impact Assessment**
1. Determine if change is security-affecting per IEC 62443 criteria:
   - Does change affect security control implementation or effectiveness?
   - Does change involve modification to: authentication, cryptography, access controls, secure communication, security monitoring?
   - Does change affect system interfaces with untrusted networks or users?
   - Does change modify threat model assumptions?
2. If change is security-affecting:
   - Review threat model to understand security controls potentially impacted
   - Assess whether change maintains or reduces security posture
   - Evaluate effect on target security level (SL): can SL be maintained or must it be revised?
   - Identify required security verification activities (testing, assessment, re-certification)
3. Security impact findings:
   - Security-affecting: Yes/No
   - Impact on target SL: Maintains SL, degrades SL, SL revision required
   - Required security verification: security test cases, architectural assessment, re-certification activities
   - Security risk assessment: low/medium/high risk to security posture
4. Security officer sign-off on security impact assessment

**Phase 4: Schedule Impact Analysis**
1. Identify schedule-affected activities:
   - Design update effort and duration
   - Test plan/test case updates required
   - Component procurement lead time (if new components required)
   - Integration and testing rework effort
   - Approval and review cycle time
2. Estimate effort for each activity:
   - Resource hours by role (design engineer, QA, security, procurement)
   - Duration: effort hours ÷ team availability
   - Critical path analysis: which activities are on critical path?
3. Calculate schedule impact:
   - Best case: assume no rework, minimal risk
   - Most likely case: realistic assessment with risk buffer
   - Worst case: if risks materialize, additional rework required
4. Assess schedule recovery options:
   - Can schedule slip be absorbed within project contingency?
   - Can other activities be compressed to offset delay?
   - Can resources be reallocated to accelerate change implementation?
5. Document schedule impact findings:
   - Activities affected and duration changes
   - Overall schedule impact (days/weeks)
   - Critical path implications
   - Risk factors and recovery options
   - Recommendation: proceed as planned, proceed with contingency plan, or defer change

**Phase 5: Cost Impact Analysis**
1. Identify cost-affected categories:
   - Design labor: hours by role (engineer, architect, manager) × hourly rate
   - Component costs: material cost for new components, price deltas for substitutes
   - Testing and verification: additional test execution, test harness updates, security assessment
   - Procurement and supply chain: lead time, expedite fees, alternative supplier costs
   - Change documentation and training: procedure updates, user/operator training
2. Estimate costs for each category:
   - Gather quotes from suppliers for new components
   - Estimate engineering effort using historical data or expert judgment
   - Consider learning curve and productivity factors
3. Calculate total change cost:
   - Direct costs: labor, materials, third-party services
   - Indirect costs: management overhead, contingency reserve (10-20% of direct costs)
   - Opportunity cost: impact on other project activities or resource conflicts
4. Compare against budget:
   - Is cost within remaining budget contingency?
   - Can budget be reallocated from other areas to fund change?
   - Does change require budget increase request?
5. Document cost impact findings:
   - Cost breakdown by category
   - Total estimated cost (with contingency)
   - Budget availability assessment
   - Recommendation: approve cost, request budget increase, or defer change

**Phase 6: Quality & Verification Impact Analysis**
1. Assess impact on verification and validation:
   - Which test cases are affected by change?
   - Do existing test cases remain valid?
   - What new test cases are required to validate change?
   - Is regression testing required?
2. Identify impact on quality assurance:
   - Design review/inspection requirements
   - Code review (if change includes software)
   - Requirements traceability matrix (RTM) updates
   - Document update and control requirements
3. Estimate verification effort:
   - New test case development
   - Test execution and rework effort
   - Test result documentation and defect management
4. Quality impact summary:
   - Verification activities required
   - Effort and schedule impact
   - Resource requirements
   - Risk to quality if change approved vs. deferred

**Phase 7: Change Evaluation Report & Recommendation**
1. Compile change evaluation report:
   - Executive summary: change description, recommendation, key impacts
   - Technical feasibility assessment
   - Security impact assessment (if applicable)
   - Schedule impact analysis with critical path implications
   - Cost impact analysis and budget assessment
   - Quality/verification impact analysis
   - Risk assessment: top 3 risks and mitigation strategies
   - Alternative approaches: if change is not approved, what are consequences?
2. Develop recommendation matrix (if multiple change packages under evaluation):
   - Priority ranking: business value, urgency, risk
   - Impact assessment: schedule, cost, quality, security, scope
   - Recommendation: approve, approve with conditions, defer, reject
3. Prepare presentation for CCB:
   - Key findings and recommendation
   - Decision factors: business value, technical feasibility, schedule/budget impact, security implications
   - Required actions if approved (design update, test planning, etc.)
4. Obtain sign-off from evaluation team:
   - Technical lead, security officer, QA lead, finance representative
   - All signatory agreement required before CCB review (no split decisions)

## Acceptance Criteria

- Change request form complete with clear description, rationale, affected items, and proposed implementation
- Technical feasibility assessment documented: design impact, implementation approach, risks, mitigation
- Security impact assessment completed and documented (if change is security-affecting):
  - Impact on target SL identified
  - Required security verification activities defined
  - Security officer sign-off obtained
- Schedule impact analysis documented: affected activities, effort estimates, critical path implications, recovery options
- Cost impact analysis documented: cost breakdown, budget assessment, contingency included
- Quality/verification impact assessment documented: test case impacts, regression testing scope, verification effort
- Change evaluation report complete with all analyses, risks, and CCB recommendation
- If recommendation is security-impacting: security officer explicitly approves recommendation
- Evaluation team signatures obtained on final report before CCB review
- Change tracking system updated with evaluation report reference and status

## Description — Common Pitfalls

**Pitfall: Superficial impact analysis leading to underestimated schedule/cost or missed security implications**
- *Mitigation:* Establish detailed evaluation checklist; require multi-discipline evaluation team; peer review of impact assessments before CCB

**Pitfall: Evaluators lack technical depth or domain knowledge; impact assessments lack credibility**
- *Mitigation:* Assign evaluation tasks to engineers with design/testing experience; provide training on change evaluation methodology; require subject-matter expert review

**Pitfall: Security impact assessment overlooked for changes perceived as "non-security"**
- *Mitigation:* Establish clear criteria for security-affecting changes; require security review of all changes affecting SL-relevant components; training on IEC 62443 security controls

**Pitfall: Change evaluation viewed as bureaucratic delay; pressure to approve changes without proper analysis**
- *Mitigation:* Communicate business value of change management; demonstrate risks avoided through proper evaluation; provide fast-track approval for low-impact changes

**Pitfall: Change evaluation report too lengthy or technical; CCB cannot quickly understand recommendation**
- *Mitigation:* Use executive summary format; highlight top 3 impacts and recommendation; visual charts showing schedule/cost/risk trade-offs

**Pitfall: Interdependencies between multiple changes not captured; cumulative impact underestimated**
- *Mitigation:* Maintain change tracking dashboard showing all pending changes; evaluate change interactions; consider combined impact on schedule/cost

## Estimated Effort

- Change evaluation completeness: 100% of submitted changes receive documented evaluation before CCB review
- Evaluation cycle time: average time from change submission to CCB decision ≤5 business days (for standard changes)
- Evaluation accuracy: actual schedule/cost impact within ±20% of estimate; security impact assessments validated post-implementation
- Security impact identification: 100% of security-affecting changes identified; 0 security-impacting changes approved without security assessment
- CCB decision efficiency: ≥95% of CCB recommendations approved without significant additional evaluation; <5% changes require re-evaluation
- Change approval rate: balance between flexibility and stability; recommend deferring 10-20% of requested changes to protect baseline schedule/budget
- Post-change validation: 100% of approved changes verified to have actual impact within estimated ranges; lessons learned captured for future evaluations

---

## Description — Continuous Improvement

- Analyze post-implementation change data: which impact areas consistently show estimation errors? Refine estimation methods
- Track deferred vs. approved changes: which change characteristics lead to deferral decisions? Communicate patterns to stakeholders
- Review CCB decision rationale: are approval decisions consistent with evaluation findings? Identify decision biases or criteria misalignment
- Benchmark change management metrics against industry standards and historical project performance
- Periodically review and update security impact assessment criteria based on emerging IEC 62443 guidance
- Gather feedback from evaluation team on template clarity, tool usability, and process efficiency; implement improvements
