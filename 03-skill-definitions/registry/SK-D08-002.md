## Description

Design security-specific FAT (Factory Acceptance Test) test cases that comprehensively validate the security controls and hardening of the OT/ICS system before site deployment. This skill encompasses test case design for access control verification, network segmentation testing, encryption validation, log integrity checks, security alarm triggering, and compliance verification against hardening baselines. Each test case is traceable to security requirements defined in SK-D09-002 (Security Functional Specification) and verifies compliance with IEC 62443-3-3 Functional Security Requirements (FR-SR). Security FAT testing differs from functional FAT testing (covered separately) in its exclusive focus on security behavior: authentication failures, authorization boundary violations, audit trail integrity, and alarm responsiveness. This skill provides the security test case designs; the execution and documentation of FAT results falls under SK-D08-001 ⏳.

## Inputs

- Security Functional Specification (from SK-D09-002)
- Security Requirements specification including FR-SR mapping (from SK-D01-006, SK-D01-007)
- System architecture and zone/conduit diagram (from SK-D01-001)
- Access control matrix and RBAC policies (from SK-D01-003)
- Encryption and key management specification (from SK-D05-001 ⏳)
- System hardening checklist and baseline standards (from SK-D01-002)
- IEC 62443-3-3 technical security requirements (normative reference)
- Test environment specification and system configuration

## Outputs

- Security FAT Test Plan: overview of security testing scope, test environment setup requirements, safety considerations, and test execution schedule
- Security FAT Test Case Catalog: prioritized list of all security test cases with test ID, description, preconditions, test steps, expected results, and traceability to FR-SR/security requirements
- Access Control Test Cases: user authentication (valid/invalid credentials, multi-factor), authorization (role-based access, zone boundaries, conduit restrictions), privilege escalation attempts
- Network Segmentation Test Cases: inter-zone communication validation (allowed vs. blocked traffic), conduit rule enforcement, zone boundary crossing verification
- Encryption and Cryptography Test Cases: encryption validation (TLS version, cipher strength), key exchange verification, data-in-transit encryption, certificate validation
- Logging and Audit Trail Test Cases: log generation on security events, audit trail completeness, timestamp validation, log integrity verification (tamper detection if applicable)
- Security Alarm Test Cases: alarm triggering on unauthorized access attempts, network anomalies, policy violations; alarm delivery and responsiveness verification
- Hardening Compliance Test Cases: firmware version verification, disabled service validation, unused port closure, default password removal, security patch application
- Security FAT Test Data Package: credentials for testing, sample attack payloads (for vulnerability validation), encrypted test data for encryption testing

## Acceptance Criteria

1. Every Functional Security Requirement (FR) in SK-D09-002 has at least one corresponding security FAT test case; mapping is documented
2. Access control test cases cover: valid authentication, invalid credentials, role-based authorization, privilege escalation attempts, zone boundary enforcement
3. Network segmentation test cases cover: allowed inter-zone communication, blocked unauthorized traffic, conduit rule enforcement (protocols, direction, authentication)
4. Encryption test cases cover: TLS version and cipher strength validation, certificate validation, key exchange verification, data-in-transit encryption
5. Logging and audit trail test cases cover: event logging completeness, timestamp accuracy, log tamper detection (if applicable), audit trail retention
6. Security alarm test cases cover: alarm triggering on policy violations, alarm delivery, alarm responsiveness, false positive assessment
7. Hardening compliance test cases cover: firmware version, disabled services, removed default passwords, closed unused ports, applied security patches
8. Test cases are documented in a consistent format with clear preconditions, test steps, and expected results; test data is prepared (credentials, attack payloads)
9. Security FAT test case design has been reviewed and approved by SAC or senior STC

## Estimated Effort

| Level | Effort | Notes |
|-------|--------|-------|
| Junior (< 2 yr) | 8–15 person-days | Assumes moderate system complexity (~5 zones, ~3 user roles, basic access control); includes test case design, documentation, and test data preparation |
| Senior (5+ yr) | 4–8 person-days | Same scope; senior can leverage prior security testing experience and reusable test case patterns |

Notes: Complex systems with multi-zone architectures, many user roles, or stringent encryption requirements may require 1.5–2× effort. Compliance with advanced IEC 62443-3-3 requirements (e.g., anomaly detection, security function testing) may add 3–5 days. Initial security testing for unfamiliar systems or emerging OT technologies may add 2–3 days.
