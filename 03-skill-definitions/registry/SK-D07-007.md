## Description

Integrate third-party APIs (weather services, market data feeds, grid operator interfaces, cloud platforms) with OT/ICS systems, enabling data exchange while maintaining security and reliability. This skill covers API authentication and credential management, data format handling (REST/JSON and SOAP/XML), rate limiting and throttling, error handling and retry logic, and security considerations for API keys, credentials, and data-in-transit encryption. API integration frequently spans multiple security zones and conduits; all integrations must comply with zone/conduit boundaries per SK-D01-001 and network segmentation controls. This skill focuses on the integration architecture, protocol configuration, and security controls; the actual firewall rule implementation falls under SK-D02-003 ⏳.

## Inputs

- Third-party API technical documentation (REST/SOAP specification, authentication schemes, data models, rate limits)
- Network architecture and zone/conduit diagram (from SK-D01-001)
- Data requirements and information classification (from SK-D01-005, SK-D01-007)
- Security requirements and authentication policies (from SK-D01-002, SK-D01-003, SK-D01-006)
- Zone/conduit crossing approval documentation (per SK-D01-001 boundary)
- Redundancy and failover strategy (from SK-D02-006 ⏳)
- Encryption requirements and certificate management standards (from SK-D05-001 ⏳)

## Outputs

- Third-Party API Integration Specification: API endpoint, authentication method, data format, rate limits, timeout behavior
- API Authentication and Credential Management Plan: credential storage, rotation schedule, access control (who can access credentials), audit trail requirements
- API Data Format Mapping Document: external API data format → internal system data format, including validation rules and type conversions for both REST/JSON and SOAP/XML
- Error Handling and Retry Logic Specification: timeout values, retry attempts, exponential backoff, fallback behavior, alert triggers
- API Security Hardening Checklist: TLS/SSL version verification, certificate pinning (if applicable), API key encryption, data-in-transit encryption, secure logging (secrets redaction)
- Zone/Conduit Crossing Configuration: explicit approval of inter-zone API traffic, firewall rule specifications (to be implemented in SK-D02-003 ⏳)
- API Integration Test Plan: authentication test cases, data format validation, rate limiting validation, error condition testing

## Tools

- API testing and mocking tools (Postman, Swagger/OpenAPI, SoapUI)
- Network analyzer / packet sniffer (Wireshark) for validating encryption and data-in-transit security
- Credential management tools (HashiCorp Vault, AWS Secrets Manager, or equivalent) for secure API key storage
- REST client libraries or SOAP client libraries (language-dependent: Python requests, Java Apache HttpClient, C# RestSharp)
- Excel / structured markdown for integration specification and mapping tables
- SSL/TLS certificate validation tools (OpenSSL, Qualys SSL Labs)

## Roles

- DES (Design Engineer) — primary: specifies API integration requirements, designs data mappings, coordinates zone/conduit crossing
- SYS (System Architect) — provides system architecture context, validates API integration feasibility
- STC (Security Technical Consultant) — specifies authentication and encryption requirements, reviews security hardening checklist
- OPS (Operations Engineer) — supports credential rotation, monitors API integration health and security logs

## Dependencies

### Hard Dependencies (Required Before Execution)

- SK-D01-001: Zone/Conduit Architecture Design — API integration must respect zone/conduit boundaries; any inter-zone API traffic requires explicit conduit specification
- Third-party API technical documentation — essential for understanding authentication schemes, data formats, and rate limits
- Encryption and credential management standards (from SK-D01-006, SK-D05-001 ⏳) — defines API key storage and data encryption requirements

### Soft Dependencies (Enhances Effectiveness)

- SK-D02-003 ⏳: Firewall Rule Development — coordinates firewall rules controlling API traffic across zones
- SK-D02-006 ⏳: Redundancy and Failover Strategy — informs API integration failover behavior (e.g., multiple API endpoints for load balancing)
- SK-D01-005 ⏳: Information Classification — clarifies which data classifications are permitted to transit API boundaries

## Acceptance Criteria

1. API endpoint, authentication method, data format, and rate limits are explicitly documented and match third-party API specification
2. API authentication plan specifies credential storage mechanism, rotation schedule, and who is authorized to access credentials
3. Data format mapping is bidirectional (where applicable) and includes validation rules; JSON, XML, and binary data formats are handled explicitly
4. Error handling specification covers all documented error codes; timeout values and retry logic are defined and tested
5. Security hardening checklist is 100% complete: TLS version ≥ 1.2 is enforced, API keys are encrypted at rest, data-in-transit encryption is enabled, sensitive data (credentials, API keys) are redacted from logs
6. Zone/conduit crossing has been documented and approved by SAC/STC; firewall rule specification has been prepared (to be implemented in SK-D02-003 ⏳)
7. API integration test plan is complete; authentication, data validation, rate limiting, and error handling have been tested

## Estimated Effort

| Level | Effort | Notes |
|-------|--------|-------|
| Junior (< 2 yr) | 5–10 person-days | Assumes single API endpoint, straightforward REST/JSON data format, basic authentication; includes specification, implementation, testing, documentation |
| Senior (5+ yr) | 2–5 person-days | Same scope; senior can leverage prior API integration experience and patterns |

Notes: APIs with complex authentication schemes (OAuth 2.0, certificate-based), multiple data format types (REST/JSON + SOAP/XML), or many-to-many data mappings may require 1.5–2× effort. Zone/conduit crossing coordination adds 2–3 days. Initial integrations with new cloud platforms or grid operator interfaces may add 2–3 days due to vendor-specific learning curves.
