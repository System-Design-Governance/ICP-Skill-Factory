## Description

Configure Modbus register mapping between SCADA/RTU master stations and field devices (power meters, protection relays, switchgear controllers, energy storage inverters) in OT/ICS systems. This skill covers Modbus TCP and Modbus RTU (serial) configurations, register address assignment and management, data type mapping (coils, discrete inputs, holding registers, input registers), polling schedules, error handling and retry logic, and security considerations. Modbus mapping directly implements the data exchange requirements from the approved Data Flow Diagram (SK-D02-004) and respects Zone/Conduit security boundaries defined in SK-D01-001 (Zone/Conduit Architecture). The skill ensures register addresses do not conflict across devices, polling frequencies do not exceed device processing limits, and network bandwidth is respected. This is a foundational skill for SCADA/RTU integration and feeds into system commissioning and testing.

## Inputs

- Approved Data Flow Diagram and data exchange specifications (from SK-D02-004 ⏳: Data Flow Diagram Development)
- Device point lists: tag names, units, data types, ranges, update frequency requirements (from SK-D02-001 ⏳: OT Network Topology Design)
- Zone/Conduit architecture and security requirements (from SK-D01-001: Zone/Conduit Architecture Design)
- SCADA/RTU platform documentation (ModbusPlus configuration manual, Wonderware, IGS, etc.)
- Field device datasheets (meter registers, relay contact lists, inverter modbus register maps)
- Network topology and bandwidth constraints (from SK-D02-001)
- Security conduit specifications: authentication, encryption requirements (from SK-D01-001, SK-D01-002 ⏳)

## Outputs

- Modbus Register Mapping Table (Excel/CSV): device, register address, data type (coil/discrete/holding/input), units, scaling factor, polling frequency, security zone assignment
- Modbus Configuration Files (per SCADA platform export format): .csv, .xml, or vendor-specific format
- Modbus Address Conflict Report: verification that all addresses are unique within the system
- Polling Schedule Specification: polling intervals, timeout values, retry logic, bandwidth utilization analysis
- Security Implementation Checklist: confirmation that all Zone/Conduit conduit specifications are reflected in Modbus authentication/encryption configuration
- RTU Device Configuration Scripts (if applicable): Modbus device configuration files for field equipment
- Integration Test Cases: Modbus read/write test sequences for commissioning

## Acceptance Criteria

1. All data points in the approved Data Flow Diagram have corresponding Modbus registers with consistent naming and addressing
2. Register addresses are unique across all devices; no address collisions or conflicts
3. Data type mapping (coil vs. discrete vs. holding/input register) is appropriate for the data semantic (discrete input → coil, analog value → holding register, etc.)
4. Scaling factors (if applicable) are documented and verified to preserve measurement accuracy (e.g., 10-bit ADC mapped to 0–100 scaling)
5. Polling frequencies are feasible for the SCADA platform and do not exceed RTU/device processing capability
6. Network bandwidth utilization is within 70% of available bandwidth under peak polling load
7. Security requirements from Zone/Conduit conduit specifications are reflected: authentication credentials, encryption ciphers, or segregation measures
8. Error handling strategy (timeouts, retries, failover) is specified and consistent across all devices
9. Register mapping documentation is approved by DES and reviewed by STC for security compliance
