## Description

Design high-availability (HA) architecture for operational technology (OT) and industrial control systems (ICS) to ensure service continuity in the face of hardware failures, software faults, or environmental disruptions. This skill encompasses server and system redundancy strategies, database replication mechanisms, automatic failover procedures, and recovery methodologies. The design covers three standby configurations: hot standby (active/active or active/passive with real-time synchronization), warm standby (periodic synchronization with manual or semi-automatic switchover), and cold standby (offline backup activated only after failure detection and manual intervention). HA architecture design is driven by the target Recovery Time Objective (RTO) and Recovery Point Objective (RPO) defined in SK-D02-007 (RTO/RPO Planning), and integrates with SK-D01-001 (Zone/Conduit Architecture) to ensure HA mechanisms do not compromise security zone boundaries or introduce unintended communication paths. This skill outputs specifications that feed SK-D02-007 (RTO/RPO Planning) with architectural feasibility assessments and enable SK-D01-022 (Backup and Restore) implementation planning.

## Inputs

- System Architecture Design (from SK-D02-001: System Partitioning and Functional Architecture)
- Zone/Conduit Architecture Design (from SK-D01-001: Zone/Conduit Architecture Design)
- Operational Requirements and SLAs (Availability targets, MTBF/MTTR requirements, tolerable outage windows)
- Infrastructure Constraints (physical site layout, network topology, power distribution, environmental conditions)
- Business Continuity Requirements (from customer SOW or SLAs)
- Threat and Risk Assessment (from SK-D01-006 ⏳: Threat and Risk Assessment — Preliminary; common-cause failure modes)
- Technology and Vendor Constraints (approved hardware/software platforms, licensing limitations)

## Outputs

- High-Availability Architecture Diagram (D2 / Visio format)
  - Logical architecture showing redundancy relationships
  - Physical site layout showing standby resource locations
- HA Configuration Specification Document with sections for:
  - Server/System Redundancy Model (active/active, active/passive, N+1, N+M)
  - Database Replication Strategy (synchronous, asynchronous, failover trigger conditions)
  - Network Redundancy and Load Balancing (multiple network paths, virtual IPs, connection pooling)
  - Automatic Failover Procedures (health check intervals, failure detection logic, switchover decision trees)
  - Service Continuity Plan (prioritized service restoration sequence, dependencies between services)
- Standby Configuration Comparison Matrix (hot/warm/cold standby trade-offs per system tier)
- HA Readiness Checklist (pre-deployment validation requirements)
- Integration points with Security Zone boundaries (conduit updates for HA communication, authentication requirements for standby synchronization)

## IEC 62443 Lifecycle Stages

- R1 (Security Requirements) — initial HA strategy based on preliminary availability targets and threat landscape
- R2 (Detailed Design) — complete HA architecture design with specific redundancy mechanisms and failover procedures

## Roles

- SYS (System Architect) — primary: owns HA architecture, balances availability targets with security constraints
- INF (Infrastructure/Network Architect) — collaborates on network redundancy, physical site layout, power distribution
- DES (Design Engineer) — translates HA architecture into detailed system design and failover logic
- QA (Quality Assurance) — develops and executes HA testing and failover simulation procedures

## Dependencies

- SK-D01-001: Zone/Conduit Architecture Design — HA mechanisms must not violate security zone boundaries

### Soft Dependencies (Enhances Effectiveness)

- SK-D02-007 ⏳: RTO/RPO Planning — RTO/RPO targets drive HA configuration selection; can be done iteratively
- SK-D01-006 ⏳: Threat and Risk Assessment (Preliminary) — common-cause failure modes inform redundancy strategy
- SK-D01-022 ⏳: Backup and Restore Planning — HA works in conjunction with backup/restore for comprehensive continuity

## Acceptance Criteria

1. HA architecture design covers all system tiers and critical services identified in the system architecture
2. Each system tier has an assigned redundancy model (hot/warm/cold standby) with documented rationale
3. Failover procedures are fully specified: health check intervals, failure detection criteria, switchover decision logic, and switchback procedures
4. HA mechanisms (database replication, network synchronization) have explicit security classifications aligned with their Security Level Target
5. Architecture is traceable to the operational requirements specification: each availability target has a corresponding HA mechanism
6. HA design has been reviewed for consistency with the Zone/Conduit Architecture; any HA communication paths have been added as conduits or explicitly approved exceptions
7. Design has been reviewed and approved by SYS, INF, and PE/O
