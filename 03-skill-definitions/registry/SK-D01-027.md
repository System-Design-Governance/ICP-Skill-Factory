## Description

Design and implement cybersecurity controls for Safety Instrumented Systems (SIS) that enforce IEC 62443 security requirements while preserving the Safety Integrity Level (SIL) and functional safety assurance of IEC 61511 safety systems. This specialized skill addresses the unique conflict domain where security measures (e.g., encryption, authentication, network isolation) may impact safety system response time, availability, or functional testing procedures. The skill covers SIS network isolation (segmentation from non-safety OT and IT networks), SIS-specific access controls and authentication that do not compromise manual safety overrides, change management procedures that maintain SIL certification, and security-safety requirement reconciliation when conflicts arise. The boundary of this skill is SIS-specific security architecture and control implementation; general OT security controls (Zone/Conduit, firewall rules) are covered in D01.1–D01.5, and SIL validation/functional safety testing fall under functional safety engineering (IEC 61511, D09.x).

## Inputs

- Approved SIS System Architecture (safety system P&ID, functional safety block diagrams from IEC 61511 engineering phase)
- SIS Safety Requirements Specification (SIL targets per IEC 61511 Annex B)
- Zone/Conduit Architecture Design with SIS zone definition (from SK-D01-001, where SIS is designated as a dedicated high-SL zone)
- IEC 62443 Security Requirements for the SIS zone (SL-T assignment, from SK-D01-007 ⏳)
- Existing SIS network topology, communication protocols, and device list (HMI, PLC, RTU, I/O modules, operator interfaces)
- Safety-critical network communication requirements: latency bounds, reliability requirements, diagnostic message frequencies
- Functional safety certification baseline (if upgrading existing SIL-certified system)

## Outputs

- SIS Security Architecture Specification: dedicated network segmentation for SIS, isolation mechanisms (air gap, demilitarized zone, application-level gateway), access control policy for SIS administrative access, and SIS-specific authentication methods with documented compatibility with manual safety overrides
- SIS Network Isolation Design: network topology with firewall rules separating SIS from non-safety OT and IT networks, conduit specifications for SIS-to-DCS or SIS-to-HMI communication with allowed protocols and certification of latency/reliability preservation
- SIS Change Management Procedure: change process for SIS security controls, including change impact assessment for functional safety, SIL re-validation triggers, and approval by both safety engineer and security architect (dual-sign-off requirement)
- Security-Safety Conflict Resolution Log: documented cases where security requirements and safety requirements compete, the chosen resolution (security/safety trade-off), rationale, and risk acceptance by project leadership (PtW if required)
- SIS Cybersecurity Assessment Report: mapping of IEC 62443 security requirements to SIS control implementation, verification that SIL certification is not compromised, and compliance statement
- SIS Operator Training Plan: administrative access control, authentication procedures, and security awareness training with explicit coverage of safety-critical override procedures

## IEC 62443 Lifecycle Stages

- R1 (Security Requirements — SIS security targets based on SIL requirements and threat landscape)
- R2 (Detailed Design — SIS isolation architecture and control specification)
- R3 (Implementation Planning — SIS change management procedure and dual-approval process)

## Roles

- SAC (Security Architect) — primary: designs SIS security architecture, leads security-safety conflict resolution, ensures IEC 62443 compliance
- Functional Safety Engineer (IEC 61511 certified) — reviews SIS security controls for SIL impact, approves change procedures, signs off safety-security conflict resolutions
- STC (Security Technical Consultant) — supports with network isolation implementation details and firewall/gateway configuration
- SIS Vendor / System Integrator — provides SIS architecture and vendor guidance on compatible security controls
- Operations / Safety Manager — approves SIS administrative access control and training plan

## Dependencies

### Hard Dependencies (Required Before Execution)

- Approved SIS System Architecture from functional safety engineering — security design must align with safety system boundary and critical functions
- SK-D01-001: Zone/Conduit Architecture Design (with designated SIS security zone) — SIS isolation is the primary security control
- SK-D01-007 ⏳: Security Requirements Specification — SIS-specific SL targets and requirements

### Soft Dependencies (Enhances Effectiveness)

- SK-D01-003 ⏳: Firewall Rule Planning — SIS firewall rules and gateway policies
- SK-D01-010 ⏳: Security Level Target Assessment — detailed SL assignment for SIS zone
- IEC 61511 functional safety engineering documentation — concurrent development with security engineering for conflict detection

## Automation Potential

**Level:** Low

Safety-security conflict resolution requires expert judgment from both security and functional safety domains and cannot be fully automated. Network isolation topology can be templated, but SIS-vendor-specific control adaptation requires manual engineering. Change management procedures are partly automatable (impact assessment triggers), but final approval requires dual human review.

## Acceptance Criteria

1. SIS is designated as a dedicated high-SL security zone with explicit isolation from non-safety OT and IT networks (per Zone/Conduit Architecture)
2. Network isolation mechanism is specified and documented: air gap, dedicated firewall/gateway, or demilitarized zone with application-level filtering
3. SIS access control policy permits administrative access only through defined authenticated paths, without compromising manual safety override procedures (tested and verified)
4. SIS communication conduits specify protocols, latency bounds, and reliability guarantees with documented evidence that security controls preserve these performance characteristics
5. SIS Change Management Procedure defines dual-sign-off requirement (Security Architect + Functional Safety Engineer) for all SIS security control changes
6. Security-Safety Conflict Resolution Log documents at least one identified conflict (if any) with rationale, chosen resolution, and stakeholder approval signatures
7. Functional Safety Engineer has reviewed SIS security architecture and issued written confirmation that SIL certification is not compromised by applied security controls

| Level | Effort | Notes |

| Junior (< 2 yr) | 12–18 person-days | Requires collaboration with safety engineer; includes conflict analysis, dual-approval workflow setup, and training plan development |
| Senior (5+ yr) | 8–12 person-days | Same scope; senior leverages prior safety-security integration experience and SIS vendor relationships |

Notes: New SIS deployments (greenfield) typically require moderate effort. Upgrades to existing SIL-certified systems may require 1.5–2× effort due to SIL re-validation and change impact assessment overhead. SIS vendor expertise is critical; engagement with vendor technical support adds 2–5 days depending on vendor responsiveness.
