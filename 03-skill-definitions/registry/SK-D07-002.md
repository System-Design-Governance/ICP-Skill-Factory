## Description

System Integration Architecture Diagram design defines the complete visual and logical representation of heterogeneous OT/ICS system interconnections within a power utility or industrial facility. This skill encompasses the creation of architectural diagrams that illustrate SCADA, EMS, DERMS, historian, protection relay, and metering system relationships, along with their data flow patterns and security zone boundaries. The diagrams form the foundation for integration design by establishing clear system topology, protocol stacks, and communication pathways before implementation begins. This skill operates at the design phase (R2) and serves as a prerequisite for detailed protocol and data mapping work.

## Inputs

- Security architecture design from SK-D01-001 (security zones, trust boundaries, access rules)
- System requirements and functional specifications from project scope documentation
- Network topology and physical site layout diagrams
- Legacy system inventories and technical datasheets
- Regulatory compliance requirements (IEC 62443, NERC CIP, grid codes)
- Protocol and interface specifications for candidate integration technologies

## Outputs

- System integration architecture diagram (Visio, Draw.io, or AutoCAD format) showing all subsystems and interconnections
- Data flow diagram depicting information movement between systems with directionality and volume indicators
- Protocol stack matrices mapping which protocols operate on which communication links
- Security zone boundary overlay diagrams aligned with SK-D01-001 design
- Integration topology documentation including equipment lists and communication link specifications
- Diagram validation checklist and completeness assessment

## Tools

- Microsoft Visio or equivalent diagramming software
- Draw.io (free alternative with collaborative features)
- AutoCAD or technical drawing tools for physical layout integration
- Protocol analyzers (Wireshark) for reference data
- Network modeling and simulation tools for topology validation

## Standards

- IEC 62443-2-1: Establishing an industrial automation and control systems security program
- IEC 62443-3-3: System security requirements and security levels
- NIST SP 800-82: Guide to Industrial Control Systems (ICS) Security
- IEEE 1686: Standard for Intelligent Electronic Devices (IED) Cyber Security Capabilities
- IEEE 1815 (DNP3) and IEC 61850 standards for protocol reference structures

## IEC 62443 Lifecycle Stages

- **R0 (Preparation)**: Gather system requirements and security policies that constrain architecture choices.
- **R1 (Design Planning)**: Develop conceptual diagrams and topology alternatives; evaluate integration feasibility.
- **R2 (Design Development)**: Create final architecture diagrams with protocol stacks, security zones, and data flows clearly defined.
- **R3 (Implementation Planning)**: Validate diagram completeness and prepare specifications for integration engineering work in SK-D07-004 and SK-D07-005.

## Roles

- **SYS**: System Architect responsible for overall integration topology, alignment with business requirements, and coordination with security and operations teams.
- **SEC**: Security Officer validates that security zone boundaries and trust relationships are correctly represented.
- **OPS**: Operations stakeholder ensures diagram reflects actual system capabilities and operational reality.

## Acceptance Criteria

1. All major subsystems (SCADA, EMS, DERMS, historian, protection, metering) are identified and represented in the architecture diagram with clear labeling and roles.
2. Data flow directions between systems are explicitly shown with annotations indicating information type, frequency, and protocol.
3. Security zone boundaries from SK-D01-001 are overlaid on the architecture diagram, and all inter-zone communication links are documented.
4. Protocol stacks for each communication link are specified in a matrix format, including network layer (Ethernet, serial), data link layer (TCP/IP, DNP3), and application layer protocols.
5. The diagram has been reviewed and validated by at least the System Architect, Security Officer, and Operations representative, with sign-off documented.
6. Completeness assessment confirms that all known system interfaces, data requirements, and security constraints are represented in the final diagram set.
