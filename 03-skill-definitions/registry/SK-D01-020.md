## Description

Design and implement the account management policy and Role-Based Access Control (RBAC) framework for all systems within the SuC, covering centralized accounts (Microsoft Active Directory), local device accounts, and service accounts. This skill defines the authorization level hierarchy, rights matrix across device types, password policies, account lifecycle management (creation, review, expiry, deletion), and exception handling procedures. The boundary of this skill is account/access policy design and initial implementation; ongoing account maintenance during operations falls under R4 operational procedures, and the underlying network segmentation that supports access control enforcement falls under SK-D01-001.

## Inputs

- Asset inventory with device types and OS platforms (from SK-D01-005)
- Zone/Conduit architecture with SL-T assignments (from SK-D01-001) — SL drives access control rigor
- FR/SR mapping for access control requirements (from SK-D01-007: SR 1.x Human User Identification/Authentication, SR 2.x Use Control)
- Customer requirements for least privilege and mutual authentication
- Organizational Active Directory structure and Group Policy baseline
- Existing account inventory (for brownfield: current user accounts across all devices)

## Outputs

- Account Management Policy Document (per ID07 §4.0 exemplar):
  - Centralized account management approach (AD + GPO for Windows-based hosts)
  - Local account management rules (one local account per host, least privilege, encrypted transit)
  - Account naming conventions and prohibited practices (no hidden accounts, no unchangeable passwords)
  - User identity validity rules (configurable expiry, immediate deletion of invalid accounts)
  - Default account handling (rename default admin, change default passwords post-installation)
- Authorization Level Hierarchy Definition (4-tier per ID07 exemplar):
  - Viewer: read/browse system/HMI only
  - Control: limited control operations (e.g., device switching)
  - Engineering: all rights except user management
  - System Manager/Administrator: full rights including user/group management
- Rights Matrix: role × system access mapping across all device types (SCADA HMI, IED Control/Protection, Ethernet Switch, Wireless AP)
- Password Policy Specification: complexity requirements, rotation schedule, and enforcement method per account category (AD-managed, local, service)
- Account Exception Register: documented exceptions with justification, compensating controls, and review schedule
- FAT/SAT Verification Appendix: account verification procedures integrated into testing

## Automation Potential

**Level:** Partial

AD account creation, GPO deployment, and password policy enforcement can be automated through scripting and IAM tools. Rights matrix generation from templates is automatable. However, authorization level definition, role-to-function mapping, exception judgment, and customer coordination for account naming/approval require human decision-making.

## Acceptance Criteria

1. RBAC framework defines at least 4 authorization levels with clear separation of privileges (no privilege creep between levels)
2. Rights matrix covers all device types in the asset inventory with explicit role × system access assignments
3. No default passwords remain on any device post-implementation; all default admin accounts renamed
4. Password policies enforced: complexity meeting customer/employer standards, rotation schedule per account category, encrypted transit for all credential exchanges (HTTPS, SSH, SFTP, SCP)
5. All accounts reviewed and approved by employer/customer as documented in account tracking register
6. Exception register documents all deviations from standard policy with compensating controls and scheduled review dates
