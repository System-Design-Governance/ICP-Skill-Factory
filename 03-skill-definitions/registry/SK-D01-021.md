## Description

Design and implement the security patch management procedure for all system categories within the SuC — covering Windows-based host servers (WBHS), ICS/IACS devices, and network infrastructure devices. This skill establishes the end-to-end patch lifecycle: acquisition source identification, applicability assessment, risk-based evaluation, test-environment validation, approval workflow, controlled deployment, and post-deployment verification. Under ICP governance (GOV-SD), configuration changes including patches are Gate 2 change management triggers — significant patches that affect architecture, SL level, or contracted commitments require formal Gate 2 review. The boundary of this skill is the patch management procedure and process design; the actual endpoint hardening that patches support falls under SK-D01-019, and the broader change management governance falls under SK-D11-020 ⏳.

## Inputs

- Asset inventory with firmware/software versions (from SK-D01-005)
- Configuration baseline documentation per device category (established during hardening: SK-D01-019)
- Vendor patch releases, security bulletins, and advisories
- Risk assessment context (from SK-D01-007) — vulnerability severity informs patch prioritization
- Customer/employer patch approval requirements and maintenance windows
- IEC 62443-2-4 CCM (Configuration and Change Management) process requirements

## Outputs

- Patch Management Procedure Document (per ID07 §8.0 exemplar):
  - Patch acquisition channels identification (vendor portals, ICS-CERT, security feeds)
  - Applicability assessment methodology (Applies / N/A / Deferred with justification)
  - Risk-based evaluation criteria (criticality × exploitability × operational impact)
  - Testing and validation procedures (test environment requirements)
  - Approval workflow (standard vs. emergency patch paths)
  - Deployment procedures per device category (WBHS, ICS, network)
  - Post-deployment verification and rollback procedures
- Patch Management Map: visual identification of all patch sources and distribution paths per device category
- Configuration Revision Control & Management Table: version history per device (current version, patch applied, date, approver, rollback version)
- Patch Applicability Register: per-patch tracking (patch ID, vendor, affected systems, applicability decision, test result, deployment status, deployment date)
- Emergency Patch Procedure: expedited process for critical zero-day or actively exploited vulnerabilities

## Tools

- Vendor patch management portals (Microsoft WSUS/SCCM, Siemens ProductCERT, ABB Cybersecurity Portal)
- Configuration management tools (Acronis, or equivalent for baseline/snapshot management)
- Test environment (isolated network segment mirroring production for pre-deployment validation)
- Patch tracking spreadsheet or ITSM platform (ServiceNow, Jira)
- Version control for configuration files (Git or vendor-specific configuration management)

## Roles

- Security Engineering Role (GOV-SDP) — primary: designs patch management procedure, defines risk-based prioritization criteria
- DES (Design Engineer) — executes patch deployment, maintains configuration revision tables
- SAC (Security Architect) — evaluates patch impact on zone/conduit architecture and SL levels
- System Architect (GOV-SDP) — Gate 2 approver for patches classified as minor changes; escalates major changes to Head of System Design

## Automation Potential

**Level:** Partial

Patch acquisition monitoring, applicability assessment against asset inventory, and deployment to Windows-based systems (via WSUS/SCCM) can be largely automated. Configuration version tracking is automatable. However, risk-based evaluation of OT-specific patch impact, test environment validation in ICS contexts, and approval decisions for IACS devices require human judgment due to safety and availability implications.

## Acceptance Criteria

1. Patch management procedure covers all three device categories (WBHS, ICS, network) with category-specific processes
2. Patch applicability register maintained: every vendor patch release has a documented decision (Apply / N/A / Defer) with rationale
3. Configuration Revision Control table is current: every patched device has pre-patch version, post-patch version, date, and approver recorded
4. Emergency patch procedure defined: expedited path for critical vulnerabilities with maximum acceptable time-to-patch target
5. Rollback procedure documented and tested: every patch deployment has a verified rollback path before production application
6. Patch deployment coordination with backup procedures verified: backup taken before each patch application (cross-reference SK-D01-022 ⏳)
