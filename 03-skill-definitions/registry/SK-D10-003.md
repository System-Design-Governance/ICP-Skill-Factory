## Description

This skill encompasses the execution of the Management of Change (MOC) process per IEC 62443 ID01 §7.8.7.1 and ID03 §5.7.1, managing approved changes from implementation through verification and documentation update. MOC ensures that changes are implemented in a controlled manner with appropriate technical review, testing, and approval at each implementation step. The MOC process includes: change implementation planning, technical execution with peer review, verification testing to validate change effectiveness, documentation updates to maintain consistency, and formal closure with stakeholder communication. Per GOV-SD governance, security-impacting changes may trigger Gate 2 re-evaluation to ensure continued compliance with target security level.

---

## Description — Business Context

Once a change is approved by the Configuration Control Board (CCB), execution must follow disciplined procedures to ensure the change is implemented correctly, validated through appropriate testing, and integrated into project baselines and documentation. Uncontrolled change implementation risks introducing design defects, creating undocumented configurations, and potentially degrading security posture. The MOC execution skill establishes procedures for change implementation, technical review, verification, documentation, and closure to ensure organizational control and auditability of project modifications throughout the lifecycle.

## Roles

**Primary Stakeholders:**
- Project Manager (PM): owns MOC execution process and coordinates across disciplines
- Design Engineer: implements change in design documents and specifications
- Quality Assurance (QA): verifies change implementation against approved change request
- System Integrator/Manufacturing: implements change in hardware assembly or software build
- Security Officer/Security Architect: verifies security-impacting changes maintain security posture
- Requirement Originator: accepts change implementation and verifies correctness
- Configuration Management: updates baselines and maintains version control
- System Owner/End User: validates change meets operational expectations

**Downstream Skill Dependencies:**
- Depends on SK-D10-002 (Change Request Evaluation and Impact Analysis) for change approval
- May trigger SK-D11-020 (Design Change Impact Analysis & SL Recertification) if security-impacting per GOV-SD Gate 2

## Description — Procedural Steps

**Phase 1: Change Implementation Planning**
1. Develop detailed change implementation plan:
   - Change overview: summary of what is changing and why
   - Affected items: requirements, design elements, documents, components, tests
   - Implementation approach: step-by-step technical strategy
   - Task breakdown: specific implementation tasks, sequence, dependencies
   - Resource plan: team members assigned, effort estimates, schedule
   - Risk mitigation: identified risks, contingency plans, rollback procedures
2. Identify design/documentation impacts:
   - Requirements Traceability Matrix (RTM) updates: modified requirements, new requirements, requirement deletions
   - Design specification updates: architecture changes, design drawing modifications, component substitutions
   - Bill of Materials (BOM) updates: component additions/deletions/substitutions, quantity changes
   - Wiring diagrams, P&IDs, electrical schematics: impact assessment and update approach
   - Software code changes (if applicable): code review process, testing approach
3. Define verification approach:
   - Test cases required: new tests to validate change, regression tests to verify no unintended effects
   - Test environment: hardware test fixture, simulation environment, or production system
   - Test acceptance criteria: clear pass/fail conditions
   - Test schedule: integration with overall project schedule
4. Schedule change implementation:
   - Critical path analysis: which tasks are on critical path?
   - Dependency management: which tasks must complete before others can begin
   - Resource leveling: ensure resources adequately allocated without overload
   - Timeline: milestones for design completion, verification start, closure
5. Identify stakeholder approvals required:
   - Design review/approval authority for each affected document
   - Security review (if security-impacting change)
   - Quality assurance sign-off on implementation
   - Change originator acceptance

**Phase 2: Design Implementation & Technical Review**
1. Update design documentation:
   - Modify requirements specification: document requirement changes, additions, deletions
   - Update design drawings and schematics: incorporate design changes, maintain consistency with requirements
   - Revise component specifications: update component selections, part numbers, specifications
   - Update Bill of Materials: add/remove/modify components with effective dates and version tracking
   - Modify configuration items: source code, firmware, hardware assemblies, as applicable
2. Maintain traceability:
   - RTM updates: reflect new/modified requirements; verify backward/forward traceability
   - Design-requirement mapping: ensure modified design elements still trace to requirements
   - Change cross-references: document which design artifacts are affected by this change
3. Conduct design peer review:
   - Assign peer reviewer with domain expertise (independent of implementation team, if possible)
   - Review technical correctness: does design properly implement the approved change?
   - Check for completeness: are all affected documents updated consistently?
   - Verify traceability: are requirements, design, and verification elements aligned?
   - Identify secondary effects: could design change create unintended interactions?
   - Document review findings: issues, recommendations, approval sign-off
4. Address review findings:
   - Implement peer reviewer recommendations
   - If disagreement on design approach: escalate to technical authority for resolution
   - Document design decisions and rationale
   - Conduct secondary review if significant changes made post-peer-review
5. Design baseline update:
   - Update design configuration baseline with approved modified documents
   - Assign new version number per version control scheme
   - Maintain version control history: previous version remains archived
   - Update design documentation index/table of contents if document structure changed
   - Communicate baseline update to downstream teams (manufacturing, integration, QA)

**Phase 3: Change Verification Testing**
1. Develop verification test plan:
   - Identify test cases for change validation:
     - Positive tests: verifying change implements required functionality
     - Negative tests: verifying change does not create unintended side effects
     - Regression tests: verifying existing functionality remains unaffected
   - Define test environment: hardware test fixture, simulation, or production system
   - Establish success criteria: clear pass/fail conditions for each test
   - Plan test data: representative data sets for testing change
2. Prepare test environment:
   - Set up test hardware/software configuration
   - Load baseline configuration and apply change
   - Verify test environment readiness: all tools, instrumentation, documentation in place
   - Confirm test environment traceability: document version of change under test
3. Execute verification tests:
   - Run test cases per approved test plan
   - Document test execution: which tests ran, when, by whom
   - Record test results: pass/fail for each test case
   - Capture test data and evidence: logs, measurements, screen captures, if relevant
   - If test fails: investigate root cause, determine if change implementation issue or test issue
4. Handle test failures:
   - Determine if failure is critical (change does not work) or non-critical (minor functionality issue)
   - If critical failure: halt change implementation; return to design phase for redesign
   - If non-critical failure: document issue; determine fix scope; revise change implementation if needed; re-test
   - Update change implementation plan if rework required; re-schedule remaining tasks
5. Document test results:
   - Test report: executive summary of test results, critical findings, pass/fail status
   - Test evidence package: test cases executed, test data, test logs, test results
   - Traceability: map test results back to change requirements and test cases
   - Approval: QA sign-off confirming change meets acceptance criteria

**Phase 4: Security Verification (if Security-Impacting Change)**
1. Execute security verification activities (per change evaluation):
   - Security-related tests: cryptographic functionality, access control validation, secure communication verification
   - Security architectural review: does change maintain security control effectiveness?
   - Threat model re-assessment: does change introduce new threats or affect threat assumptions?
   - Security standards compliance: verify change maintains IEC 62443 compliance
2. Document security verification:
   - Security test results: pass/fail for each security test case
   - Security officer review and sign-off
   - If issues found: determine criticality; if SL could be affected, escalate to SK-D11-020
3. Determine security impact outcome:
   - Change maintains security posture and target SL: proceed with closure
   - Change affects security posture but target SL still achievable: document mitigation; may require additional controls
   - Change violates SL requirements: escalate to Design Change Impact Analysis & SL Recertification (SK-D11-020)

**Phase 5: Documentation Update & Baseline Integration**
1. Finalize all affected documentation:
   - Requirements specification: final version reflecting all change-related modifications
   - Design specifications and drawings: final versions with peer review sign-off
   - Test plan and test results: final versions with QA sign-off
   - User/operator documentation: update if change affects system operation or maintenance
   - Training materials: update if operator/maintainer training required
   - Configuration management documentation: baseline version number, component list, change cross-reference
2. Update Requirements Traceability Matrix (RTM):
   - Reflect new/modified requirements with test evidence
   - Verify bi-directional traceability: requirement → design → test → verification evidence
   - Document completeness: 100% of change-related requirements have design response and test evidence
3. Update configuration baselines:
   - Release new baseline version: requirements baseline, design baseline, test baseline
   - Version number reflects change: e.g., R1.B2 to R1.B3
   - Archive previous baseline version: maintain historical record
   - Update configuration management database: current baseline version, component list, document list
4. Archive implementation artifacts:
   - Store change implementation plan with completion status
   - Archive design peer review reports
   - Archive test results and test evidence package
   - Archive change evaluation report and CCB decision
   - Update change tracking system with implementation completion evidence

**Phase 6: Stakeholder Communication & Change Closure**
1. Communicate change completion:
   - Notify design team: new baselines available; update local copies with new versions
   - Notify manufacturing/assembly team: if hardware components changed, procurement/assembly procedures may be affected
   - Notify system integration team: if change affects integration, update integration approach
   - Notify operations/maintenance team: if change affects system operation or maintenance procedures
   - Notify customer (if applicable): provide summary of change, impact on system capabilities
2. Conduct change closure review:
   - Verify all implementation plan tasks completed
   - Confirm design peer review completed and approved
   - Confirm verification testing completed with passing results
   - Confirm security verification completed (if security-impacting)
   - Confirm all documentation updated and baselines released
   - Obtain formal sign-off from: Design Lead, QA Lead, Project Manager
3. Update change tracking system:
   - Mark change status as "Closed"
   - Record closure date, responsible party, evidence references
   - Document any open issues or follow-up activities (e.g., field testing, long-term monitoring)
4. Lessons learned (post-change closure):
   - Analyze change implementation: did schedule/cost estimates prove accurate?
   - Identify process improvements: were MOC procedures effective?
   - Document lessons learned: what would be done differently on future similar changes?
   - Update MOC process if systemic issues identified

**Phase 7: Change Rollback (if Required)**
1. If change implementation encounters critical issues that cannot be resolved:
   - Evaluate rollback feasibility: can baseline be reverted?
   - Determine rollback scope: which documents/components require reversion?
   - Execute rollback: restore previous baseline versions, notify affected teams
   - Document rollback decision and rationale: why was change rolled back?
   - Determine next steps: modify change request and re-plan implementation, or close change request?
2. Post-rollback communication:
   - Notify all stakeholders of rollback decision
   - Explain impact on project schedule/cost
   - Clarify path forward: revised change, deferred change, or abandonment

## Acceptance Criteria

- Change implementation plan complete with: tasks, schedule, resource plan, verification approach, risk mitigation, rollback procedures
- All design documentation updated consistently: requirements, design specifications, drawings, BOM, RTM
- Design peer review completed and approved: no critical findings outstanding
- Verification testing completed: all test cases executed, results documented, pass/fail status clear
- If security-impacting change: security verification completed and approved; SL impact assessed
- Test evidence package complete: test results, test logs, traceability to change requirements and acceptance criteria
- Documentation baseline updated and released: version number, effective date, previous version archived
- Change tracking system updated: closure record with evidence references, sign-offs, lessons learned
- Formal sign-offs obtained: Design Lead, QA Lead, Security Officer (if security-impacting), Project Manager
- Stakeholder notification completed: affected teams informed of change completion and baseline updates

## Description — Common Pitfalls

**Pitfall: Change implementation begins before plan is approved; rework required when issues discovered**
- *Mitigation:* Enforce gate approval: no implementation tasks begin until plan approved by PM and technical lead; use project management tool to track plan approval status

**Pitfall: Design documentation updated but peer review skipped or performed superficially**
- *Mitigation:* Make peer review mandatory gate: design baseline cannot be released without documented peer review; review must address completeness, consistency, and secondary effects

**Pitfall: Verification testing insufficient; change does not actually work post-implementation**
- *Mitigation:* Develop test cases during change evaluation phase; involve QA in change implementation planning; execute comprehensive test plan with traceability to change requirements

**Pitfall: Security verification skipped for changes thought to be "non-security"**
- *Mitigation:* Establish clear criteria for security-impacting changes; require security officer involvement in change evaluation; make security verification mandatory for all SL-relevant components

**Pitfall: Documentation updates incomplete; inconsistencies between design specification, drawings, and actual configuration**
- *Mitigation:* Establish checklist of documents requiring update; require peer reviewer to verify consistency across all affected documents; use document management system to flag related documents

**Pitfall: Baseline released with change, but version numbering inconsistent or previous baseline not properly archived**
- *Mitigation:* Enforce version numbering scheme; use configuration management tool to manage baseline versions; automate baseline release with version control system

**Pitfall: Stakeholders unaware of change completion; teams continue using old baselines**
- *Mitigation:* Develop communication plan during implementation phase; send change completion notification to all affected teams; require acknowledgment of baseline update

## Estimated Effort

- MOC execution compliance: 100% of approved changes implemented via formal MOC process; 0 changes implemented ad-hoc
- Implementation plan accuracy: actual schedule/effort within ±20% of estimate; critical path managed to minimize delays
- Design peer review completion: 100% of design changes reviewed by qualified peer; average review cycle time <3 business days
- Verification testing completeness: 100% of planned test cases executed; 0 test cases deferred or skipped
- Security verification (if applicable): 100% of security-impacting changes verified; 0 security changes approved without security assessment
- Test results: <5% of test cases fail on first execution; rework identified and corrected rapidly
- Documentation consistency: audit of released baselines shows 100% consistency between requirements, design, BOM, and RTM
- Baseline release timeliness: new baseline released within 2 business days of change closure
- Stakeholder communication: 100% of affected teams notified of baseline updates; acknowledgment documented
- Change closure timeliness: average time from change approval to closure <2 weeks (for straightforward changes)
- Rollback avoidance: <2% of changes rolled back post-implementation; lessons learned captured to prevent recurrence

## Description — Continuous Improvement

- Analyze MOC execution metrics: which types of changes consistently exceed schedule/cost estimates? Refine planning approach
- Review design peer review findings: are certain design areas prone to review issues? Provide targeted training or process improvements
- Analyze test results: are certain test categories discovering more issues than others? Strengthen test approach
- Track change rollback reasons: why are changes rolled back? Are issues preventable through better planning, review, or testing?
- Benchmark MOC cycle time and effort against historical project data and industry standards
- Gather feedback from implementation teams: are MOC procedures efficient? What process improvements would help?
- Periodically review and update change implementation templates and checklists to incorporate lessons learned

## Source Traceability

**Lifecycle Phases:** R1 (Detailed Design), R2 (System/Component Assembly), R3 (System Integration), R4 (Factory Acceptance Test)

**Integration Points:**
- Upstream: Change Request Evaluation and Impact Analysis (SK-D10-002) - must complete before MOC execution begins
- Downstream: Design Change Impact Analysis & SL Recertification (SK-D11-020) if security-impacting change requires SL re-evaluation per GOV-SD Gate 2

**MOC Triggers for Gate 2 Re-evaluation (GOV-SD):** If change:
- Affects security controls or their implementation
- Modifies threat model assumptions
- Could impact target security level achievement
- Introduces new security-relevant components or interfaces
Then escalate to SK-D11-020 for Design Change Impact Analysis & SL Recertification.

**Evidence of Competency:** Project manager demonstrates ability to plan change implementation systematically, coordinate design/verification activities, obtain necessary approvals, manage change closure, and communicate changes to stakeholders. Design engineer demonstrates ability to update design documentation consistently, participate in peer review, and incorporate feedback effectively. QA demonstrates ability to develop comprehensive verification test plans, execute tests, and document results clearly.
