## Description

Author defect reports that document issues, anomalies, and failures discovered during Factory Acceptance Test (FAT), Site Acceptance Test (SAT), and System Integration Testing (SIT) activities. Defect reports provide structured documentation of what was wrong (defect description), how to reproduce the issue (reproduction steps), why it matters (impact assessment and severity classification), and what needs to be done to fix it (resolution recommendations and tracking). Defect severity classification follows a standard scheme (Critical/Major/Minor/Cosmetic) that aligns business impact with remediation urgency. Well-written defect reports are essential for project communication, test result tracking, defect trend analysis, and configuration management. Defect reports feed into nonconformance management processes (SK-D11-007 ⏳) that track systemic issues and quality improvements across the project lifecycle. The skill encompasses both the mechanics of defect report writing (structure, completeness, clarity) and the judgment required to classify severity appropriately in the OT/ICS context where impact extends beyond typical IT concerns.

## Inputs

- Test execution logs and test case results (from SK-D08-001 ⏳, SK-D08-003, SK-D08-002, SK-D08-008, SK-D08-009, SK-D08-010)
- System design and specification documents (baseline against which defects are assessed)
- Security requirements and acceptance criteria (from SK-D01-005, SK-D01-006)
- Severity classification standard and business impact criteria (from organizational test and quality standards)
- Evidence of discovered issues: screenshots, logs, network traces, system state data
- Reproduction environment and test data (for verifying and documenting reproducibility)
- Historical defect data (for understanding typical severity distributions and project trends)
- Business context: operational impact, safety implications, schedule urgency

## Outputs

- Defect Report (per issue): defect ID, title, description, reproduction steps, test case reference, expected behavior vs. actual behavior, severity classification, affected component/system, discovery phase (FAT/SAT/SIT)
- Defect Severity Assessment: severity level justification (Critical/Major/Minor/Cosmetic), impact statement (operational, safety, schedule, cost)
- Defect Impact Narrative: business impact description (e.g., "Prevents system startup", "Causes intermittent data loss during peak load", "Cosmetic UI misalignment")
- Defect Reproduction Package: step-by-step reproduction procedure, test data, environment configuration, screenshots/logs as evidence
- Defect Resolution Recommendation: suggested fix, acceptance criteria for resolved defect, responsible party, target resolution date
- Defect Tracking Records: linkage to nonconformance management system (SK-D11-007 ⏳), resolution status tracking, reopened/rejected status
- Defect Summary Report: aggregated defect statistics by severity, defect trend analysis (discovery rate, resolution rate), project quality metrics
- Defect Root Cause Analysis (for systemic issues): underlying cause categories (design flaw, implementation error, incomplete testing, requirement misunderstanding)

## Roles

- DES (Design Engineer) — primary: writes defect reports, classifies severity, documents resolution recommendations
- QA (Quality Assurance Engineer) — reviews defect reports for completeness and accuracy, manages defect tracking system
- Testers / Test Engineers — discover defects and provide initial documentation for DES refinement
- STC (Security Technical Consultant) — classifies security-related defect severity, assesses security impact
- Development Team / Vendors — receives defect reports, assesses reproducibility, implements fixes
- OPS (Operations Engineer) — validates operational impact of defects, confirms test environment behavior matches production risk
- Project Manager — reviews defect trends, makes go/no-go decisions based on critical/major defects

## Acceptance Criteria

1. Defect report structure is complete: defect ID, title, description, reproduction steps, expected vs. actual behavior, severity, affected component, discovery phase
2. Defect description is clear and specific: not vague or subjective; describes observable behavior not interpretation
3. Reproduction steps are detailed enough for independent verification: test data, environment configuration, step-by-step procedure
4. Evidence (screenshots, logs) is included: supports reproduction and severity assessment
5. Severity classification is appropriate and consistently applied: Critical defects prevent system function or create safety risk, Major defects significantly impact functionality, Minor defects cause minor operational inconvenience, Cosmetic defects do not affect function
6. Business impact is articulated: operational impact, safety implications, schedule impact, cost/resources required for fix
7. All Critical defects have root cause assessment: identify whether issue is design flaw, implementation error, or testing gap
8. Defect reports are reviewed by QA and domain expert (DES, STC, OPS) before formal tracking
9. Defect report is linked to nonconformance management system (SK-D11-007 ⏳) for systemic tracking
10. Defect summary report shows trend: defect discovery rate, resolution rate, remaining critical/major defects at test phase conclusion

## Estimated Effort

| Level | Effort | Notes |
|-------|--------|-------|
| Junior (< 2 yr) | 0.5–1.5 person-days per 10 defects | Assumes standard severity mix, straightforward reproduction steps, typical system complexity; includes report writing, review, and severity justification |
| Senior (5+ yr) | 0.25–0.75 person-days per 10 defects | Same scope; senior can prioritize and triage faster, leverage domain knowledge for severity assessment |

Notes: Complex defects requiring root cause analysis or requiring developer/system expert involvement may require additional time for diagnosis and impact assessment. Security-related defects require STC review and may add 0.5–1 day per defect for security impact assessment. Systemic defects affecting multiple components may require additional analysis for pattern identification and process improvement recommendations.
