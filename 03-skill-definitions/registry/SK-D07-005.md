## Description

Cross-System Data Model Alignment establishes semantic consistency and unified naming conventions for data exchanged between heterogeneous OT/IT systems by analyzing, documenting, and harmonizing the information models of each subsystem. This skill applies Common Information Model (CIM) standards, OPC UA information model alignment, and point mapping techniques to ensure that the same physical or logical entity is represented consistently across all integrated systems. Alignment at the data model level (R2) prevents confusion, improves data quality, and reduces errors in downstream conversion and integration logic. This skill is essential for systems integrating SCADA, EMS, DERMS, metering, and IT enterprise systems that may use completely different naming and classification schemes.

## Inputs

- System architecture diagram from SK-D07-002 (identifies system boundaries and data exchange requirements)
- Data model specifications from each subsystem (register maps, tag databases, IEC 61850 data objects, OPC UA node sets, database schemas)
- Protocol specifications and technical documentation for each system
- CIM and OPC UA reference models and information model definitions
- Existing point naming conventions and legacy naming standards
- Business requirements for data semantics and shared understanding

## Outputs

- Data model alignment specification document with unified naming conventions and point mapping rules
- CIM/OPC UA information model mappings showing alignment between source systems and standardized models
- Point mapping matrix or spreadsheet documenting cross-system mappings for all exchange points
- Semantic data dictionary defining meaning, units, and acceptable value ranges for all unified points
- Data model validation report confirming semantic consistency across system boundaries
- Implementation guidelines for SK-D07-003 conversion logic to enforce aligned data models

## Tools

- CIM (Common Information Model) tools and CIM-based modeling software
- OPC UA Information Model Editor for OPC UA node set development
- Enterprise Architect or Visio for data model visualization and diagramming
- Spreadsheet tools (Excel, Google Sheets) for point mapping matrices and semantic dictionaries
- Database schema analysis and comparison tools
- Data profiling tools to validate existing data against aligned models

## Standards

- IEC 61970: Energy management system application program interface (EMS-API)
- IEC 61968: Application of IEC 61970 to distribution management
- IEC 61850: Communication networks and systems in substations (information model reference)
- OPC Foundation: OPC UA Information Model Specification
- IEC 62443-2-1: Establishing an industrial automation and control systems security program
- NERC CIP: Critical Infrastructure Protection standards for nomenclature and point identification

## IEC 62443 Lifecycle Stages

- **R0 (Preparation)**: Inventory all subsystems and their data models; identify potential inconsistencies and conflicts.
- **R1 (Design Planning)**: Analyze data model gaps and overlaps; evaluate CIM and OPC UA applicability for alignment.
- **R2 (Design Development)**: Create detailed data model mappings, unified naming conventions, and semantic dictionaries; validate alignment across system boundaries.
- **R3 (Implementation Planning)**: Document point mapping for SK-D07-003 integration logic and prepare data model enforcement mechanisms.

## Acceptance Criteria

1. Data model alignment specification document is complete and includes unified naming conventions for all data points exchanged between subsystems.
2. CIM/OPC UA information model mappings are defined, showing how source system data models map to or align with standard reference models.
3. Point mapping matrix is comprehensive, documenting all cross-system mappings with source system identifier, target system identifier, units, scaling, and semantic equivalence.
4. Semantic data dictionary defines meaning, acceptable value ranges, units of measure, and data quality requirements for all unified points.
5. Data model validation confirms semantic consistency across all subsystem boundaries and identifies any unresolved conflicts or ambiguities.
6. Documentation includes guidelines for future point additions and modifications to maintain alignment as systems evolve.
