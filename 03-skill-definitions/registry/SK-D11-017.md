## Description

Govern the 4-gate review process (Gate 0–Gate 3) per GOV-SD, verifying that all gate-specific blocking conditions are satisfied before approving stage transitions. This is the governance-authoritative gate review skill that superseded SK-D11-017 (Stage-Gate Review Facilitation) in R5. It establishes the gating authority and enforcement mechanism for the stage-gate process throughout the IEC 62443 project lifecycle. Gate reviews act as quality checkpoints and phase-transition gateways: each gate defines mandatory inputs, quality thresholds, blocking conditions, and decision criteria that must be satisfied before the project can proceed to the next lifecycle stage. The skill covers: Gate 0 (5 mandatory inputs + 4 quality thresholds), Gate 1 (Lite/Complete path selection, Threat and Risk Assessment verification), Gate 2 (change triggers, Security Level Decision Record update), Gate 3 (12-item delivery checklist, 20% sampling verification, residual risk acceptance hierarchy). The boundary of this skill is governance verification and gate transition authorization; the actual preparation of gate deliverables falls under their respective skills (SK-D14-015 for Gate 0 preparation, SK-D01-013 for design gate deliverable preparation).

## Inputs

- GOV-SD (Tier 1): Gate Review Governance Framework — defines gate structure, blocking conditions, decision criteria, and escalation paths for all 4 gates
- Gate 0 submission: 5 mandatory inputs (project scope, preliminary SL targets, preliminary threat landscape, resource plan, contract) + 4 quality thresholds (scope completeness, threat landscape sufficiency, SL rationale clarity, resource feasibility)
- Gate 1 submission: Threat and Risk Assessment (TRA) report, Lite/Complete pathway decision rationale, security requirements baseline, architectural concept (where applicable)
- Gate 2 submission: Detailed design outputs (functional specification, security architecture), Security Level Decision Record (SLDR), change log documenting scope/requirement changes since Gate 1
- Gate 3 submission: Complete deliverables per 12-item checklist (functional specification, security implementation plan, test plan, operations manual, security functional description, etc.), 20% sampling of test results, residual risk assessment, customer acceptance evidence
- Project metadata: project ID, project phase, resource assignments, timeline tracking
- Escalation inputs (if required): risk acceptance requests, exception requests, deviation proposals

## Outputs

- Gate Review Record (per-gate): gate decision (Pass/Conditional Pass/Fail), evidence of condition verification, approval authority signature, date/time of decision
- Gate 0 Decision: approval to proceed to R1 security requirements analysis, OR block with remediation requirements
- Gate 1 Decision: approval to proceed to R2 detailed security design, OR block with remediation requirements; includes Lite/Complete pathway authorization
- Gate 2 Decision: approval to proceed to R3/R4 security implementation and testing, OR block with remediation requirements; includes SLDR approval
- Gate 3 Decision: approval to release to operations and close R5 lifecycle stage, OR conditional approval with residual risk acceptance (L1-L2-L3 escalation if applicable)
- Blocking Condition Verification Matrix: checklist of all gate-specific blocking conditions, verification evidence per condition, condition status (Met/Not Met/Waived with exception), date verified
- Gate Transition Authorization Record: project stage transition approval, signed by gating authority (Head of System Design for all gates)
- Escalation Outcome (if triggered): exception decision, risk acceptance rationale, conditions for acceptance (if any), appeal/reconsideration path

## Standards

- GOV-SD (Tier 1): Gate Review Governance Framework — authoritative definition of 4-gate structure, blocking conditions, decision criteria, escalation
- IEC 62443-2-4 SP.04.01: Develop a security implementation plan — project planning and gate review integration
- IEC 62443 lifecycle stages (R0–R5): gate review occurs at each stage transition per IEC 62443-1-1
- ID03 (Tier 3): Project Management and Lifecycle Procedures — project control and gate review procedures (may reference or be aligned with GOV-SD)
- PRAC: Industry standard practice for stage-gate project governance in IEC 62443 projects

## IEC 62443 Lifecycle Stages

- R0 (Foundational) → Gate 0 (Project Initiation Gate)
- R1 (Requirements) → Gate 1 (Security Requirements Gate)
- R2 (Design) → Gate 2 (Design Completion Gate)
- R3/R4 (Implementation & Verification/Validation) → Gate 3 (Implementation & Verification Completion Gate, with residual risk acceptance decision)
- R5 (Operation): post-implementation gate review for operational security posture

## Roles

- Head of System Design (GOV-SDP) — primary gating authority: verifies blocking conditions, makes gate decisions, authorizes stage transitions for all 4 gates
- PM (Project Manager) — submits gate deliverables, responds to blocking condition gaps, coordinates remediation
- SAC (Security Architect) — contributes technical verification for security-specific blocking conditions (TRA completeness, SL assignment rationale, design security completeness)
- Design QA Role (GOV-SDP) — may serve as gate reviewer, verifies documentation quality and completeness against 12-item Gate 3 checklist
- Customer Project Representative (or PE/O) — reviews gate submission for compliance with contract requirements, provides customer acceptance evidence for Gate 3

## Dependencies

### Hard Dependencies (Required Before Execution)

- None formally; however, gate execution requires the gate-specific deliverables to exist (SK-D14-015 for Gate 0 preparation inputs, SK-D01-013 for design-phase gate preparation, etc.). If those deliverables are not ready, gates will appropriately block and require remediation.

### Soft Dependencies (Enhances Effectiveness)

- SK-D14-015: Gate 0 Deliverable Preparation — prepares the 5 mandatory inputs for Gate 0 review; enhances gate efficiency
- SK-D01-013: Security Design Deliverable Preparation — prepares design-phase outputs for Gate 2 review
- SK-D11-019: Standards Ownership & Exception Arbitration — exception arbitration process (L1-L2-L3 escalation) may be invoked during gate reviews if blocking conditions cannot be met and exceptions are requested

## Automation Potential

**Level:** Partial

Gate scheduling and reminder automation can be fully implemented (calendar integration, automated milestone reminders). Blocking condition verification checklist can be partially automated (system can validate document completeness, date ranges, signature presence). Gate decision recording and audit trail logging can be fully automated. However, technical verification of blocking conditions (TRA completeness, SL rationale soundness, design security adequacy) requires domain expertise and cannot be automated. Risk acceptance decision-making (L1-L2-L3 escalation) requires human judgment and cannot be automated.

## Acceptance Criteria

1. All 4 gates (Gate 0, Gate 1, Gate 2, Gate 3) have blocking condition checklists defined per GOV-SD: every blocking condition is documented with specific verification criteria (not vague or subjective)
2. Blocking condition verification is evidenced in every gate review record: for each blocking condition, evidence of verification status (Met/Not Met/Waived) is recorded with supporting documentation link
3. Gate decisions are authorized by Head of System Design: every gate approval signature is from the designated gating authority; no delegated authority without documented exception
4. Stage transitions are blocked until blocking conditions are verified as Met (or approved Waived exception): no project proceeds to next stage without gate approval
5. Gate 3 sampling verification is documented: minimum 20% sample of test results verified, sampling methodology documented (random, risk-based, or other), sample evidence attached to gate record
6. Exception escalation outcomes (L1-L2-L3) are recorded with rationale: if any blocking condition is waived, exception decision record includes business justification and risk acceptance statement

## Estimated Effort

| Level | Effort | Notes |
|-------|--------|-------|
| Junior (< 2 yr) | 2–4 person-days per gate | Assumes 4 gates per project lifecycle; includes checklist completion, evidence collection, and documentation; does not include remediation of failed conditions |
| Senior (5+ yr) | 1–2 person-days per gate | Same scope; senior can efficiently verify conditions and coordinate approvals |

Notes: Total gate review effort per project lifecycle: 8–16 person-days (4 gates × 2–4 days). If blocking conditions are not met, additional remediation effort falls on the project team (not included in gate review effort). Gate 3 sampling verification and residual risk assessment may require 3–5 additional person-days due to test result review complexity. Escalation time for exceptions (L1-L2-L3): 2–7 days depending on escalation level and approval availability. GOV-SD defines the gate blocking conditions; the effort estimate assumes conditions are already defined and do not require customization per project.
