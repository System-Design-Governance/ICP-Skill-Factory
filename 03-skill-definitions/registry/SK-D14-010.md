## Description

Define and document the security requirements for a tender/bid response, mapping customer security expectations to IEC 62443 framework requirements, proposing Security Level Targets (SL-T) per zone, and specifying the security scope that ICP commits to deliver. This skill translates customer security language into structured IEC 62443-aligned requirements that feed into the Gate 0 decision package. Under GOV-SD, tender security requirements must reference Gate 0 quality thresholds (comprehensibility, evaluability, accountable owner, scope stability). The boundary of this skill is security-specific requirements for tender response; the general requirements specification falls under SK-D14-001, and the preliminary security classification that assigns initial SL-T falls under SK-D14-014.

## Inputs

- Customer/employer tender documents with security requirements or compliance mandates
- General requirements specification (from SK-D14-001) — project scope and functional requirements
- IEC 62443-3-2: Zone/Conduit model and SL framework
- ID01 §6.5.1.1.1: Security requirements definition guidance for project initiation
- ID03 Table 1 SSA-PRO-001: Security requirements definition as a required process activity
- Applicable industry sector standards (e.g., NERC CIP for power utilities, IEC 61850 for substation automation)
- GOV-SD Gate 0 quality thresholds: comprehensibility, evaluability, accountable owner, scope stability

## Outputs

- Tender Security Requirements Specification:
  - Security scope definition: what security functions are included/excluded in the tender response
  - Compliance framework mapping: customer requirements mapped to IEC 62443 requirements (IEC 62443-2-4, -3-3, -4-2)
  - Proposed SL-T per zone: initial Security Level Target proposals based on customer requirements and industry practice
  - Security deliverable inventory: list of security documents, tests, and activities included in the bid (mapped to IEC 62443-2-4 SP categories)
  - Security assumptions and exclusions: what ICP assumes about the employer's environment and what is explicitly excluded
  - Compliance gap pre-assessment: areas where customer requirements exceed or diverge from IEC 62443 standard practices
- Security Cost Impact Summary: security-specific cost drivers for CBOM integration (from SK-D14-005 ⏳)
- Gate 0 Security Input: security requirements quality assessment against Gate 0 thresholds (comprehensible? evaluable? owner assigned? scope stable?)

## Automation Potential

**Level:** Partial

Customer requirement extraction and IEC 62443 mapping can be partially automated (keyword matching, compliance matrix pre-population). SL-T proposal generation from industry sector templates can be automated for standard configurations. However, security scope definition requires expert judgment (balancing customer expectations with achievable commitments), compliance gap assessment requires contextual analysis, and Gate 0 quality assessment requires strategic evaluation.

## Acceptance Criteria

1. Tender security requirements mapped to IEC 62443 framework: every customer security requirement linked to at least one IEC 62443 SP/SR category
2. Proposed SL-T per zone documented with justification based on customer requirements and industry practice
3. Security deliverable inventory produced: complete list of security documents, tests, and activities with IEC 62443-2-4 SP category mapping
4. Security assumptions and exclusions explicitly documented — no ambiguous security scope boundaries
5. Gate 0 quality assessment completed: security requirements evaluated against four thresholds (comprehensible, evaluable, owner assigned, scope stable)
6. Compliance gap pre-assessment identifies divergence points between customer requirements and IEC 62443 — gaps documented with proposed resolution approach

## Estimated Effort

| Level | Effort | Notes |

| Junior (< 2 yr) | 5–8 person-days | Assumes medium-complexity tender with IEC 62443 compliance requirement; includes mapping, SL-T proposal, and documentation |
| Senior (5+ yr) | 3–5 person-days | Same scope; senior leverages sector-specific templates and IEC 62443 mapping experience |

Notes: Effort varies with customer security requirement complexity. Tenders with prescriptive security specifications (detailed SR lists) require more mapping effort but less interpretation. Tenders with vague security language require more elicitation and assumption documentation. International tenders may require regulatory research for local compliance requirements.
