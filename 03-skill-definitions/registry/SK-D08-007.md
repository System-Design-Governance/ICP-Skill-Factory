## Description

Establish performance baselines for OT/ICS systems during commissioning that serve as the reference metrics for subsequent anomaly detection, operational monitoring, and capacity planning. Performance baselines capture normalized measurements of network throughput, response times, CPU/memory utilization across critical devices, protocol-level latency (MODBUS, DNP3, IEC 60870-5-104, Ethernet/IP), and security event rates under normal operating conditions. Baselines are collected systematically after system stabilization (typically 2–4 weeks post-startup when transient startup behaviors have settled) using standardized instrumentation and measurement protocols. Baselines directly feed downstream anomaly detection systems (SK-D01-014 SIEM configuration) and inform capacity planning decisions. Baseline data is stored in a configuration management database (CMDB) or monitoring system for long-term trend analysis and tuning of alerting thresholds.

## Inputs

- System architecture documentation and network topology (from SK-D01-001, SK-D02-001 ⏳)
- Device inventory with performance requirements (from SK-D01-005 ⏳: Asset Inventory Development)
- Operational procedures and normal operating hours (from SK-D10-001 ⏳)
- Security monitoring infrastructure (SIEM, network analyzer, device logs) deployed and operational
- Monitoring tool configuration and data collection agents installed on critical systems
- Historical baseline data from similar systems (if available)
- Performance requirements and SLA targets from system design specification

## Outputs

- Performance Baseline Report: summary of baseline metrics, collection methodology, measurement period, statistical summaries (mean, std deviation, min/max percentiles), validation results
- Baseline Metrics Table: per-device/per-protocol baseline data organized by metric type (throughput Mbps, latency ms, CPU %, memory %, security events/hour), tagged with measurement date, collection duration, sample size
- Network Throughput Baseline: aggregate and per-conduit network throughput measurements (bytes/sec, packets/sec), peak and off-peak characterization
- Response Time Baseline: protocol response times (MODBUS register read latency, DNP3 poll cycle time, HMI command response time), percentile distributions (50th, 95th, 99th percentile)
- Device Resource Utilization Baseline: per-device CPU, memory, disk I/O, network interface utilization under normal loads
- Protocol Latency Baseline: application-layer protocol latencies (IEC 60870-5-104 ASDU latency, Ethernet/IP message latency, serial protocol response times)
- Security Event Rate Baseline: security event frequency during normal operations (login attempts/hour, policy violations/day, authentication failures/hour), normal vs. suspicious thresholds
- Baseline Configuration Records: CMDB entries documenting baseline metadata (collection date, system state, software versions, network conditions, environmental factors)
- Anomaly Detection Threshold Recommendations: proposed alert thresholds for deviation from baseline (e.g., alert if throughput exceeds baseline by >30%, latency by >50%)
- Baseline Validation Report: assessment of baseline quality, data completeness, statistical confidence, fitness for use in monitoring

## Tools

- Network Analysis Tools: Wireshark, tcpdump, network TAP/SPAN ports for packet capture and throughput measurement
- System Monitoring Tools: Prometheus, Grafana, ELK Stack (Elasticsearch, Logstash, Kibana), Splunk for time-series data collection and visualization
- Device Monitoring Agents: collectd, Telegraf, Windows Performance Monitor, SNMP agents for CPU, memory, disk, network metrics
- Protocol Analyzers: OPC UA client tools, MODBUS simulators, DNP3 test tools for protocol-level latency measurement
- SIEM Platform: for centralized security event collection and baseline calculation (integrated with SK-D01-014)
- Data Analysis: Python (pandas, NumPy), Excel, or business intelligence tools (Tableau, Power BI) for baseline statistical analysis

## Acceptance Criteria

1. Baseline collection methodology is documented: measurement period duration, sampling frequency, data collection tools, measurement locations, validation procedures
2. Collection period is appropriate: minimum 2–4 weeks post-startup to ensure system stabilization and capture normal variation
3. Network throughput baseline covers all conduits: aggregate throughput, per-protocol breakdown, peak vs. off-peak characterization
4. Response time baseline includes all critical protocols: MODBUS, DNP3, HMI, OPC UA latencies with percentile distributions (50th, 95th, 99th)
5. Device resource utilization baseline is available for all critical devices: CPU, memory, disk utilization under normal loads
6. Protocol latency baseline is documented with sample size and measurement methodology
7. Security event rate baseline is established with clear normal vs. suspicious event rate thresholds
8. Baseline validation confirms data quality: completeness (>95% data points collected), outlier analysis, statistical confidence assessment
9. Anomaly detection thresholds are proposed with documented rationale (e.g., mean + 2 std deviation, percentage deviation from baseline)
10. Baseline data is integrated into CMDB and monitoring system configuration; SIEM alerting thresholds are tuned based on baseline

## Estimated Effort

| Level | Effort | Notes |
|-------|--------|-------|
| Junior (< 2 yr) | 5–8 person-days | Assumes moderate system complexity, straightforward measurement setup, standard protocols; includes methodology design, data collection execution, and reporting |
| Senior (5+ yr) | 2–4 person-days | Same scope; senior can leverage prior baseline methodologies and automation scripts; faster analysis and threshold tuning |

Notes: Complex systems with many unique protocols or distributed architectures may require 1.5–2× effort due to multiple measurement locations and protocol-specific analysis. Systems with seasonal operational variations may require extended baseline collection (6–12 weeks) to capture full operational variation.
