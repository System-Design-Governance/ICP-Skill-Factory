## Description

Estimate the labor hours required for all engineering activities in the proposed OT/ICS cybersecurity project, broken down by lifecycle phase, skill category, and seniority level. Labor hour estimation feeds directly into CBOM services costing (SK-D14-005) and provides the resource planning baseline for project scheduling. This skill covers estimation methodology, effort calculation, and estimation documentation; the CBOM that consumes these estimates falls under SK-D14-005, and the cost risk contingency analysis falls under SK-D14-016 ⏳.

## Inputs

- Requirements specification (from SK-D14-001) — scope defines what activities are needed
- Feasibility assessment (from SK-D14-003) — technical approach determines activity breakdown
- Preliminary security classification (from SK-D14-014) — SL-T levels affect effort per security activity
- ICP historical project data: actuals from similar completed projects (effort benchmarks)
- ICP Skill Factory effort estimates: per-skill effort ranges from skill definitions (where authored)
- Site survey findings (from SK-D14-011) — travel and on-site effort factors
- Team composition: available personnel, seniority mix, learning curve factors

## Outputs

- Labor Hour Estimation Document:
  - Activity breakdown structure: hierarchical list of all engineering activities organized by lifecycle phase (Pre-R0, R0, R1, R2, R3, R4)
  - Per activity: estimated person-days (range: optimistic / expected / pessimistic), required seniority level, skill domain reference (SK/SC ID where applicable)
  - Phase summary: total person-days per lifecycle phase, by seniority level (junior / senior)
  - Role allocation: estimated person-days per GOV-SDP role (SAC, SYS, DES, STC, etc.)
  - Risk factors: estimation uncertainty factors (brownfield premium, multi-site factor, complexity factor, learning curve factor)
  - Estimation methodology: basis of estimate (analogous, parametric, expert judgment) per activity
- Effort Summary Table: total person-days by phase and role, formatted for CBOM services section integration
- Resource Loading Profile: monthly/quarterly resource demand forecast for project scheduling

## Automation Potential

**Level:** Partial

Activity breakdown generation from templates can be automated for standard project types. Effort calculation from parametric models (device count × effort-per-device) can be automated. Historical project benchmarking lookup can be automated. Three-point estimation aggregation (PERT weighting) can be computed automatically. However, estimation for novel project elements requires expert judgment, risk factor calibration requires project context, and seniority mix optimization requires organizational knowledge.

## Acceptance Criteria

1. Activity breakdown covers all lifecycle phases where ICP has delivery responsibility — no phases empty without documented rationale
2. Every activity has three-point effort estimate (optimistic / expected / pessimistic) with estimation basis documented (analogous / parametric / expert judgment)
3. Seniority level specified per activity: junior and senior effort separately estimated, enabling flexible team composition
4. Role allocation table produced: estimated person-days per GOV-SDP role summed across all phases — validates against available team capacity
5. Risk factors documented: at minimum brownfield premium, multi-site factor, and complexity factor with quantified multipliers
6. Effort summary table formatted for direct CBOM integration — total person-days by phase and seniority level ready for services costing

## Estimated Effort

| Level | Effort | Notes |
|-------|--------|-------|
| Junior (< 2 yr) | 3–5 person-days | Assumes medium-complexity project with standard activity structure; includes breakdown development, effort estimation, and documentation |
| Senior (5+ yr) | 2–3 person-days | Same scope; senior leverages historical benchmarks and rapid parametric estimation |

Notes: Labor estimation is typically performed by senior engineers or project managers with historical project experience. Accuracy improves with project similarity to prior engagements. First-of-a-kind projects require more estimation effort and carry higher uncertainty. Estimation is often iterated in parallel with CBOM development (SK-D14-005).
