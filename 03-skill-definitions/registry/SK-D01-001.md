## Description

Design the logical and physical Zone/Conduit segmentation architecture for an OT/ICS automation solution (SuC), partitioning assets into security zones based on their security level requirements and defining conduits that govern inter-zone communication. This skill translates the outputs of preliminary risk assessment (threat landscape, asset criticality) into a concrete network segmentation topology that enforces defense-in-depth. It covers zone boundary definition, conduit specification (allowed protocols, direction, authentication requirements), and alignment with the approved network and data flow diagrams. The boundary of this skill is architecture-level design; the actual firewall rule implementation falls under SK-D01-003 ⏳, and device-level protocol configuration falls under D05.5.

## Inputs

- Asset inventory with criticality ratings (from SK-D01-005 ⏳: Asset Inventory Development)
- Preliminary Threat and Risk Assessment report (from SK-D01-006 ⏳: Threat and Risk Assessment — Preliminary)
- Customer requirements specification / SOW (security and operational requirements)
- Approved network and data flow diagrams (from SK-D02-004 ⏳: Data Flow Diagram Development; per ID01 §7.3.12)
- Security Level Targets (SL-T) per zone (from SK-D01-010 ⏳: Security Level Target Assessment)
- Existing site architecture documentation (brownfield projects)

## Outputs

- Zone/Conduit Architecture Diagram (D2 or Visio format, per arch-diagram skill standards)
- Zone Definition Table: zone ID, zone name, SL-T, contained assets, physical location
- Conduit Specification Table: conduit ID, source zone → destination zone, allowed protocols, direction (uni/bidirectional), authentication method, encryption requirement
- Network Segmentation Narrative Document (section within the Security Functional Description Specification, per ID01 §7.4.1 and Annex A.9)
- Updated risk register entries reflecting residual risk after segmentation design

## Tools

- D2 / Mermaid (architecture diagramming, per ICP arch-diagram skill)

- ETAP / PSS/E (for understanding power system topology that informs zone boundaries)
- Excel / structured markdown (zone/conduit tables)

## Standards

- IEC 62443-3-2: Security Risk Assessment for System Design — zone/conduit model definition
- IEC 62443-3-3: System Security Requirements and Security Levels — SL allocation per zone
- IEC 62443-1-1: Terminology, concepts and models — foundational zone/conduit concepts
- NIST SP 800-82 Rev. 3: Guide to OT Security — network segmentation guidance (supplementary)

## IEC 62443 Lifecycle Stages

- R1 (Security Requirements — initial zone/conduit concept based on preliminary TRA)
- R2 (Detailed Design — finalized zone/conduit architecture based on detailed risk assessment)

## Roles

- SAC (Security Architect) — primary: designs zone/conduit topology, assigns SL-T per zone
- STC (Security Technical Consultant) — supports with protocol-level conduit specifications
- SYS (System Architect) — provides system topology input, validates feasibility
- DES (Design Engineer) — integrates segmentation into detailed design drawings

## Dependencies

### Hard Dependencies (Required Before Execution)

- SK-D01-005 ⏳: Asset Inventory Development — cannot design zones without knowing what assets exist and their criticality
- SK-D01-006 ⏳: Threat and Risk Assessment (Preliminary) — zone boundaries are driven by threat landscape and risk appetite
- SK-D02-004 ⏳: Data Flow Diagram Development — conduit design requires understanding of data flows between subsystems

### Soft Dependencies (Enhances Effectiveness)

- SK-D01-010 ⏳: Security Level Target (SL-T) Assessment — SL-T assignments refine zone granularity; can be done iteratively with zone design
- SK-D02-001 ⏳: OT Network Topology Design — physical network topology informs feasible zone placement
- SK-D01-002 ⏳: Defense-in-Depth Strategy Design — provides the strategic framework that this skill instantiates at the architecture level

## Acceptance Criteria

1. Zone/Conduit diagram covers 100% of assets identified in the approved asset inventory
2. Every zone has an explicitly assigned Security Level Target (SL-T) with documented rationale
3. Every conduit specifies: allowed protocols, communication direction, and authentication method
4. No zone contains assets with SL-T difference > 1 level (unless documented exception with risk acceptance)
5. The architecture is traceable to the preliminary TRA: each high-risk finding has a corresponding segmentation control
6. Design has been reviewed and approved by both SAC and the PE/O (per ID01 §7.4.2 review process)
