## Description

Establish and operate continuous security monitoring for OT/ICS environments during the operational phase (R4) to detect, alert on, and investigate security events in real-time. This skill operationalizes deployed security controls by continuously observing system behavior, identifying anomalies and threats against established baselines and threat intelligence, and escalating findings through the organization's security operations center (SOC). Continuous monitoring feeds SK-D01-016 (Incident Response Procedure) with early detection of breaches, supports SK-D01-017 (Incident Investigation) with comprehensive event logging, and continuously refines anomaly detection through operational feedback. This skill implements the detection and response layer of IEC 62443-2-1 §6.5.2.6 continuous monitoring requirements and builds on SK-D01-014 (SIEM Configuration) for event collection and SK-D08-007 (Performance Baseline) for anomaly detection.

## Description — Objectives

- Establish continuous collection and monitoring of security events from OT/ICS systems (firewalls, intrusion detection systems, access controls, system logs)
- Develop and maintain security event detection signatures and anomaly detection rules based on threat intelligence and operational baselines
- Operate 24/7 security monitoring function (internally or via managed security service provider) to detect threats in real-time
- Correlate security events to identify sophisticated attack patterns, lateral movement, and data exfiltration attempts
- Generate alerts on suspicious activity with appropriate severity levels and escalation procedures
- Maintain alert tuning to minimize false positives while ensuring real threats are detected
- Produce security posture reports showing event trends, threat exposure, and control effectiveness
- Integrate OT monitoring findings with organizational SOC and IT security monitoring for holistic visibility

## Description — Scope

**In Scope:** Security event collection and retention; detection signature/rule development and tuning; real-time alert generation and escalation; event correlation and pattern analysis; anomaly detection based on operational baselines; security posture reporting; threat intelligence integration; SOC communication and alert handoff.

**Out of Scope:** Incident response actions (handled by SK-D01-016); forensic investigation details (handled by SK-D01-017); operational system performance monitoring (separate operational function); control implementation or remediation (assigned to engineering teams).

## Description — Key Concepts

- **Security Monitoring:** Continuous observation and analysis of system behavior and security events to detect threats and verify control effectiveness
- **Security Event:** Any observable action or change in system state that may indicate a security incident (login attempts, configuration changes, data access, network connections)
- **Anomaly Detection:** Identification of events or patterns that deviate significantly from established operational baselines
- **Detection Rule/Signature:** Logic that identifies specific security events or patterns indicative of known attacks or suspicious behavior
- **Alert:** Generated notification of a security event or anomaly that meets alert criteria and requires investigation or response
- **False Positive:** An alert triggered on benign activity that is not actually a security threat (increases alert fatigue and reduces monitoring effectiveness)
- **Event Correlation:** Linking multiple related security events to identify attack patterns or complex threats
- **Threat Intelligence:** Information about known threats, vulnerabilities, threat actors, indicators of compromise (IOCs), and attack techniques

## Description — Methodology

Continuous security monitoring for OT/ICS follows a structured six-phase operational model:

**Phase 1: Monitoring Infrastructure Deployment and Configuration**
- Deploy SIEM platform with OT/ICS data collection capabilities (from SK-D01-014: SIEM Configuration)
- Configure log collection from all security event sources:
  - Network devices (firewalls, routers, switches, IDS/IPS systems)
  - Access control systems (authentication servers, privilege management platforms)
  - System and application logs (Windows Event Viewer, syslog, control system event logs)
  - Network traffic analysis (NetFlow, full packet capture where operationally feasible)
  - Operational technology systems (DCS logs, SCADA events, IED (Intelligent Electronic Device) logs)
- Establish centralized log repository with:
  - Log retention policy (minimum 1 year per IEC 62443-2-1 §7.8.4.2)
  - Log integrity protection (encryption, tamper detection)
  - Secure access controls (least privilege, audit logging of access)
- Validate data collection through test events and baseline confirmation

**Phase 2: Baseline Establishment and Anomaly Detection Rule Development**
- Establish operational baselines from SK-D08-007 (Performance Baseline):
  - Normal user login patterns (times, locations, frequency)
  - Normal network traffic patterns (protocols, data flows, bandwidth)
  - Normal system performance parameters (CPU, memory, disk usage)
  - Normal data access patterns (users, files, timing)
- Develop anomaly detection rules for:
  - Off-hours login attempts, failed authentication attempts, privilege escalation
  - Unusual network connections to external systems or suspicious internal communication
  - Data access patterns inconsistent with job responsibilities
  - System configuration changes or policy modifications
  - Control system anomalies (unexpected commands, state changes, communication patterns)
- Integrate threat intelligence signatures for:
  - Known malware/ransomware indicators
  - Known vulnerability exploitation patterns
  - Threat actor tactics, techniques, and procedures (TTPs) relevant to OT/ICS
  - Zero-day indicators from ICS-CERT and sector-specific threat feeds
- Establish alert severity levels and escalation thresholds

**Phase 3: Alert Generation and Initial Triage**
- Configure SIEM to generate alerts when:
  - Anomalies exceed baseline thresholds
  - Detection rules match known attack patterns
  - Threat intelligence indicators appear in monitoring data
  - Security policy violations occur (e.g., unauthorized access, compliance exceptions)
- Establish alert triage process:
  - Severity classification (Critical, High, Medium, Low) based on business impact and compliance significance
  - Initial validation to confirm alert is not false positive based on operational context
  - Escalation to incident response team or SOC for confirmed alerts
- Route critical/high alerts to incident response team immediately; medium/low alerts to SOC for queuing and investigation
- Maintain alert queue dashboard showing open, investigating, and resolved alerts

**Phase 4: Event Correlation and Pattern Analysis**
- Implement SIEM correlation rules to identify:
  - Attack chains (reconnaissance → lateral movement → exfiltration)
  - Coordinated suspicious activity from single source or target
  - Data exfiltration patterns (large data transfers, unusual protocols, encrypted tunneling)
  - Privilege escalation chains (account takeover → credential harvesting → elevated access)
- Perform human-driven analysis:
  - Review correlated event patterns to identify sophisticated threats SIEM may miss
  - Investigate unexpected correlations to confirm or rule out attacks
  - Document findings and adjust detection rules based on analysis results
- Produce Security Event Reports showing:
  - Event trends (volume, types, severity over time)
  - Top sources of alerts and investigation outcomes
  - Confirmed incidents and root cause determination
  - Control effectiveness assessment based on detection data

**Phase 5: Threat Intelligence Integration and Rule Refinement**
- Subscribe to and integrate threat intelligence feeds:
  - ICS-CERT vulnerability and incident advisories
  - Vendor security bulletins and patch notifications
  - Sector-specific threat intelligence (utility, critical infrastructure threat feeds)
  - Generic cybersecurity threat intelligence (malware signatures, IOCs, threat actor profiles)
  - Organizational threat intelligence from SK-D01-032 (Threat Intelligence Collection and Analysis)
- Update SIEM signatures and detection rules quarterly (or more frequently for zero-days):
  - Add indicators of compromise (IOCs) from new vulnerabilities or threat campaigns
  - Update attack pattern signatures for new variants of known threats
  - Refine baseline thresholds based on seasonal operational changes (e.g., maintenance windows, production schedules)
- Tune alerting to reduce false positives:
  - Establish exclusions for known benign activity (scheduled maintenance, automated backups, system tests)
  - Correlate false positive alerts with operational events and adjust detection thresholds
  - Track false positive rate and document tuning decisions
- Maintain alert tuning log documenting rule changes, justification, and effectiveness

**Phase 6: Operational Reporting and Continuous Improvement**
- Produce regular security posture reports:
  - Monthly Security Event Report: event summary, alert distribution, incident count, trend analysis
  - Quarterly Threat Assessment: threat intelligence updates, new vulnerabilities affecting systems, recommended controls enhancements
  - Annual Security Monitoring Effectiveness Report: detection performance, false positive rate, incident detection latency, control effectiveness assessment
- Conduct after-action reviews for each detected incident:
  - Document how incident was detected (which alert triggered investigation)
  - Assess detection latency (time from attack occurrence to alert generation)
  - Identify detection gaps (attacks that should have been detected but were not)
  - Recommend monitoring enhancements or rule refinements
- Provide findings to:
  - SK-D01-017 (Incident Investigation) to support forensic analysis
  - SK-D01-032 (Threat Intelligence) to feed operational threat intelligence back to threat team
  - SK-D01-014 (SIEM Configuration) to recommend infrastructure upgrades or data source additions
  - SK-D01-016 (Incident Response Procedure) to recommend procedure updates based on monitoring insights

## Description — Activities

1. Establish monitoring operations capability:
   - Deploy SIEM platform with OT/ICS collectors (from SK-D01-014)
   - Configure log collection from all security event sources
   - Establish log retention and integrity protection
   - Train SOC/monitoring team on SIEM operations, alert investigation, and escalation procedures

2. Build detection capability:
   - Establish operational baselines from SK-D08-007 (Performance Baseline)
   - Develop anomaly detection rules for common OT/ICS attack patterns
   - Integrate threat intelligence signatures
   - Configure alert severity levels and escalation procedures

3. Operate continuous monitoring:
   - Monitor alerts 24/7 (internally or via managed security service provider)
   - Triage alerts, validate against operational context, and escalate confirmed incidents
   - Perform event correlation to identify sophisticated attack patterns
   - Investigate security events and document findings

4. Continuously improve monitoring:
   - Integrate new threat intelligence quarterly (or more frequently for zero-days)
   - Tune detection rules based on false positive analysis and operational feedback
   - Conduct after-action reviews for detected incidents
   - Update SOC procedures and training as threats evolve

5. Report and communicate:
   - Produce monthly/quarterly/annual security posture reports
   - Communicate monitoring findings to incident response, threat intelligence, and engineering teams
   - Escalate critical findings to governance/management
   - Document trend analysis and control effectiveness assessment

## Description — Practical Guidance

- **Pitfall:** Deploying SIEM without establishing baselines, leading to excessive false positive alerts. **Guidance:** Establish operational baselines from SK-D08-007 before configuring anomaly detection; tune detection rules based on operational context; maintain false positive tracking and tuning log.
- **Pitfall:** Alert fatigue from too many low-severity alerts, overwhelming SOC and reducing real threat detection. **Guidance:** Tune detection rules to reduce false positives; focus SOC investigation on high/critical severity alerts; correlate low-severity events to identify attack patterns; regularly review alert queue for stale or non-actionable alerts.
- **Pitfall:** Monitoring is deployed but not operationalized, with no 24/7 coverage. **Guidance:** Establish clear operational procedures and staffing plan for 24/7 monitoring; train SOC team on incident escalation; define alert response SLAs; monitor coverage rate and escalation effectiveness.
- **Pitfall:** Threat intelligence is not integrated into detection rules, missing known attacks. **Guidance:** Subscribe to relevant threat intelligence feeds; review threat intelligence quarterly (or more frequently for zero-days); update detection rules based on new threats; document threat intelligence subscription and update schedule.
- **Pitfall:** No after-action review process, missing opportunities to improve detection. **Guidance:** For each detected incident, document detection method and latency; identify detection gaps (threats that should have been detected); recommend monitoring improvements; track implementation of recommendations.
- **Pitfall:** Monitoring insights are not communicated to incident response or threat intelligence teams. **Guidance:** Produce monthly/quarterly security posture reports; share findings with SK-D01-017, SK-D01-032, and other relevant teams; establish feedback loop from monitoring findings to system improvements.
