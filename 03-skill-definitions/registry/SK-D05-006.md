## Description

Design the alarm management system including alarm priority hierarchy (Critical/High/Medium/Low/Information), alarm rationalization, alarm shelving and suppression rules, and alarm response procedures. This skill applies ISA-18.2 (Alarm Management for the Process Industries) and IEC 62682 (Procedures for Performance Evaluation of Alarm Systems for Nuclear Power Stations — applicable for industrial plants with demanding alarm performance requirements). Design covers alarm tag definition, priority assignment based on operational impact and response urgency, alarm rationalization to eliminate nuisance alarms, and integration with security-specific alarms per SK-D01-015 (Security Alarm Rule Design). The output is a formalized Alarm Management Strategy and detailed Alarm Database that guides alarm configuration in the SCADA/EMS/DERMS system.

## Inputs

- System Architecture Design and SCADA/EMS/DERMS functional specification (from SK-D05-001 ⏳ and SK-D05-002 ⏳)
- Process Flow Diagrams and process operating envelopes (normal, abnormal, emergency conditions) (from SK-D05-001 ⏳: Process Flow Mapping)
- Equipment specifications and failure modes (FMEA or HAZOP results, if available)
- Operational procedures and operator response expectations (from SK-D05-003 ⏳: Operational Procedures Development)
- Security Alarm Rule Design output (from SK-D01-015: Security Alarm Rule Design)
- Historical alarm data or benchmark alarm counts from similar systems
- Customer regulatory requirements and corporate alarm management policies

## Outputs

- Alarm Management Strategy Document: philosophy and governance framework for alarm management, including alarm performance targets (e.g., max 10 alarms per hour under normal operation per ISA-18.2)
- Alarm Rationalization Report: justification for each alarm (why this parameter is alarmed, trigger thresholds, operator response)
- Alarm Database/Register: structured list of all alarm tags with fields: alarm ID, description, priority level, trigger condition/threshold, alarm message text, recommended operator action, response time expectation, suppression/shelving rules, integration with automation (auto-remediation or escalation)
- Alarm Display Specification: alarm presentation rules for HMI screens, filtering, sorting, grouping, and acknowledgment logic
- Security Alarm Integration Matrix: mapping security alarms from SK-D01-015 to the general alarm hierarchy
- Alarm Testing and Validation Plan: procedure for commissioning and validating alarm behavior

- IEC 62682:2015 — Procedures for Performance Evaluation of Alarm Systems: quantitative alarm performance metrics and testing procedures

- IEC 60812 — Failure Mode and Effects Analysis (FMEA): systematic approach to identifying alarms for failure modes
- NIST SP 800-82 Rev. 3 — Guide to OT Security: security alarm design for ICS/SCADA systems

## Roles

- DES (Design Engineer) — primary: creates alarm hierarchy, prioritization, and configuration specifications
- Process Engineer / Domain Expert — supports with process-level alarm thresholds and operator response procedures
- SCADA/EMS/DERMS Specialist — validates alarm feasibility, data availability, and SCADA implementation approach
- STC (Security Technical Consultant) — integrates security alarms and validates security alarm messaging
- Operations Manager / Operator Representative — approves alarm strategy and validates operator response expectations

- SK-D01-015: Security Alarm Rule Design — security-specific alarms must be integrated into the general alarm hierarchy

### Soft Dependencies (Enhances Effectiveness)

## Acceptance Criteria

1. Alarm hierarchy (Critical/High/Medium/Low/Information) is fully defined with objective criteria for each level (e.g., "Critical = operator must respond within 5 minutes or facility safety is jeopardized")
2. Every safety-critical failure mode and security-critical event has a corresponding alarm rule in the database
3. Alarm rationalization report justifies each alarm and documents why non-alarmed parameters are excluded
4. All alarms from SK-D01-015 (Security Alarm Rules) are integrated into the alarm database with appropriate priority and response procedures
5. Alarm thresholds are technically sound and traceable to process engineering or equipment manufacturer specifications
6. Suppression/shelving rules are documented with clear conditions for when alarms should be masked (e.g., during startup, maintenance)
7. Alarm message text is clear, actionable, and includes recommended operator response (per ISA-18.2 Section 4)
8. Database completeness verified: 100% of process parameters monitored by SCADA are reviewed and classified (alarmed or justified as non-alarmed)
9. Design has been reviewed and approved by DES, operations representative, STC, and process/domain expert
