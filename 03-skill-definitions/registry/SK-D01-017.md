## Description

Investigate security incidents in OT/ICS environments using digital forensics methodology adapted for operational technology systems. This skill provides systematic evidence preservation, analysis, and documentation procedures that support incident response teams in determining the scope, impact, and root cause of security breaches while maintaining operational continuity and chain of custody. Incident investigation in OT/ICS contexts differs significantly from IT forensics due to time-critical production constraints, specialized control system protocols, and limited tolerance for system downtime. This skill covers evidence preservation (volatile and non-volatile data), timeline reconstruction, attack vector analysis, root cause determination, and formal investigation report generation per ID01 §6.5.2.5 incident investigation requirements. This skill builds on SK-D01-016 (Incident Response Procedure) and complements SK-D01-018 (Continuous Security Monitoring) which detects incidents that trigger investigation.

## Description — Objectives

- Preserve digital evidence from OT/ICS systems in a manner that maintains chain of custody and forensic integrity
- Reconstruct security incident timelines from system logs, network traffic, and operational records
- Conduct root cause analysis to identify attack vectors, compromised assets, and control failure points
- Document findings in formal investigation report with technical details and management summary
- Comply with operational technology constraints (production uptime, safety requirements) during evidence collection
- Provide investigation findings to inform SK-D01-025 (SBOM Analysis) and SK-D01-032 (Threat Intelligence)
- Support regulatory/legal proceedings if incident escalates to external authorities

## Description — Scope

**In Scope:** Evidence preservation planning and execution; volatile and non-volatile data collection; system log analysis and timeline reconstruction; network traffic analysis; attack vector determination; root cause analysis; investigation report generation; legal/regulatory documentation preparation.

**Out of Scope:** Immediate operational incident response (handled by SK-D01-016); remediation and corrective action implementation (assigned to engineering teams); preventive measure development (assigned to SK-D01-018 monitoring enhancement); legal prosecution decisions (assigned to governance/legal team).

- Organization's evidence handling and retention policy

## Description — Key Concepts

- **Digital Forensics:** Systematic preservation, acquisition, analysis, and documentation of digital evidence from computing systems
- **Chain of Custody:** Documented record of evidence handling, storage, and analysis to maintain forensic integrity and admissibility in legal proceedings
- **Volatile Data:** Time-sensitive data stored in memory that will be lost upon system shutdown or power loss (e.g., running processes, network connections, memory contents)
- **Non-Volatile Data:** Persistent data stored on disk or other durable media (e.g., system logs, configuration files, deleted file fragments)
- **Attack Vector:** The path or method used by an attacker to compromise the system (e.g., unpatched vulnerability, malware delivery, credential misuse)
- **Root Cause:** The fundamental reason why the security control failed and allowed the attack to succeed (e.g., missing patch, weak authentication, misconfiguration)
- **Timeline Reconstruction:** Chronological sequencing of events leading up to, during, and after the incident based on system logs and forensic artifacts

## Description — Methodology

Security incident investigation in OT/ICS follows a structured six-phase forensic process:

**Phase 1: Investigation Authorization and Planning**
- Receive incident alert from SK-D01-016 (Incident Response) or monitoring team
- Assess operational impact and determine if investigation can proceed without system disruption
- Obtain authorization from incident commander to initiate forensic investigation
- Document investigation scope (affected systems, time window, objectives)
- Identify required expertise (OT/ICS forensics, control system knowledge, network analysis)
- Brief investigation team on operational constraints (production schedules, safety interlocks, system availability requirements)
- Establish secure evidence storage and chain-of-custody procedures

**Phase 2: Volatile Data Preservation**
- Document current system state (live processes, network connections, memory contents, real-time logs)
- Execute memory acquisition if target system can tolerate brief activity capture without operational disruption
- Capture active network connections and session information from affected assets
- Document running processes and loaded modules that may indicate malware or attack tools
- Preserve system log buffers (Windows Event Log, syslog buffers) before they wrap around
- Record system time and timezone configuration for timeline reconciliation
- Minimize system modifications during volatile data collection to maintain forensic integrity

**Phase 3: System Stabilization and Non-Volatile Acquisition**
- Determine safe point to acquire forensic images of affected storage media (in consultation with operational team)
- Create forensically sound disk images of affected systems using write-blocking tools
- Maintain chain of custody documentation for every forensic acquisition
- Verify image integrity using hash values (MD5, SHA-256) on both source and copy
- Preserve non-volatile evidence: system logs (Windows Event Viewer, syslog files), audit logs, configuration files, database records, email archives
- Document acquisition timestamps, tools used, and hash verification results

**Phase 4: Evidence Analysis and Timeline Reconstruction**
- Establish master timeline by correlating events from multiple sources:
  - System audit logs (authentication, file access, privilege elevation)
  - Application logs (control system events, data modifications, communications)
  - Network logs (firewall logs, intrusion detection alerts, NetFlow data)
  - Operational logs (maintenance activities, configuration changes, uptime records)
- Identify suspicious artifacts:
  - Unusual login patterns (off-hours, from unexpected IP addresses, multiple failed attempts)
  - Unauthorized file creation or modification times
  - Unexpected network connections to external systems
  - Privilege escalation events or access to restricted resources
  - Malware signatures or suspicious process executions (if applicable to operational technology)
- Perform deleted file recovery if evidence suggests intentional data destruction
- Conduct memory forensics to identify remnants of malware, attack tools, or exfiltration activity

**Phase 5: Root Cause Determination**
- Identify attack vector: which vulnerability, misconfiguration, or procedural failure enabled initial compromise
- Trace attack progression: which assets were accessed, what actions were performed, what data was accessed
- Determine control failure points: which security controls were bypassed, disabled, or ineffective
- Assess impact scope: which systems were compromised, which data was accessed or modified
- Identify supporting evidence of attacker intent: lateral movement patterns, data collection activity, persistence mechanisms
- Correlate findings with SK-D01-032 (Threat Intelligence) and vulnerability databases to identify specific CVEs or threat actor patterns

**Phase 6: Investigation Reporting and Evidence Preservation**
- Prepare formal Investigation Report including:
  - Executive Summary (incident overview, timeline, root cause, impact assessment)
  - Investigation Scope and Methodology (what was examined, tools used, limitations)
  - Detailed Findings (chronological timeline with evidence citations, attack vector, root cause)
  - Evidence Inventory (all artifacts collected with hash values, acquisition dates, storage location)
  - Chain of Custody Documentation (custody records for all evidence from collection through analysis)
  - Technical Appendices (log excerpts, forensic tool reports, network analysis, timeline graphics)
- Store investigation report and evidence in secure archive per organizational retention policy
- Provide investigation findings to SK-D01-018 (Continuous Monitoring) for signatures/baselines enhancement
- Support legal/regulatory proceedings if incident requires external escalation

## Description — Activities

1. Establish forensic investigation readiness:
   - Acquire forensic tools (forensically sound disk imaging, memory analysis, log parsing, timeline reconstruction tools)
   - Train investigation team on OT/ICS forensics methodology and chain of custody procedures
   - Document operational constraints and approved investigation procedures in incident response plan
   - Establish secure evidence storage facility with access controls and audit logging

2. Upon incident detection:
   - Execute incident response plan (SK-D01-016) which triggers investigation authorization
   - Preserve volatile data per Phase 2 procedures before system state changes
   - Acquire non-volatile forensic images and evidence per Phase 3 procedures

3. Conduct evidence analysis:
   - Build comprehensive timeline from multiple log sources per Phase 4 procedures
   - Perform root cause analysis identifying attack vector and control failures per Phase 5 procedures
   - Document all findings with forensic tool outputs and evidence citations

4. Generate investigation report and archive evidence:
   - Prepare formal investigation report per Phase 6 deliverables template
   - Store evidence and investigation records in secure archive with access controls
   - Provide findings to relevant stakeholders (incident commander, management, legal if applicable)

5. Continuous improvement:
   - Review investigation findings for gaps in SK-D01-018 (Continuous Monitoring)
   - Update SK-D01-032 (Threat Intelligence) with new attack vectors and indicators
   - Incorporate forensic lessons learned into SK-D01-016 (Incident Response Procedure) updates

## Outputs

- **Investigation Authorization Document:** Formal approval to conduct forensic investigation with scope and constraints defined
- **Volatile Data Preservation Log:** Documentation of memory capture, running processes, network connections captured with timestamps
- **Forensic Image Inventory:** Listing of all systems imaged with timestamps, hash values (MD5/SHA-256), acquisition tool, and storage location
- **Chain of Custody Records:** Documented custody trail for all evidence from collection through analysis and storage
- **Master Timeline:** Chronological reconstruction of all events leading to, during, and after incident with evidence source citations
- **Formal Investigation Report:** Executive summary, detailed timeline, root cause determination, impact assessment, and technical appendices
- **Evidence Archive:** Secure storage of forensic images, evidence artifacts, and investigation documentation per retention policy
- **Investigation Findings Brief:** Summary presentation for management/governance highlighting key findings and recommendations

## Acceptance Criteria

- Investigation is authorized in writing by incident commander and documents scope, constraints, and objectives
- All volatile data is preserved before system state changes; memory acquisition (if applicable) includes documented hash verification
- Forensic images are created using write-blocking tools with chain of custody documentation and hash verification for integrity
- Master timeline is built from multiple independent log sources (system logs, application logs, network logs, operational logs) with event correlations documented
- Attack vector is explicitly identified and supported by forensic evidence (not speculation)
- Root cause points to specific control failure (missing patch, weak authentication, misconfiguration, procedural failure) with evidence citations
- Investigation report is formal document with executive summary, detailed timeline, root cause, and technical appendices suitable for stakeholder review
- All evidence is stored in secure, audited facility with access controls and retention documentation per organizational policy
- Investigation team maintains chain of custody documentation throughout analysis and evidence handling
- Investigation findings inform updates to SK-D01-018 (Continuous Monitoring) and SK-D01-032 (Threat Intelligence) recommendations

## Description — Practical Guidance

- **Pitfall:** Modifying evidence during analysis, compromising chain of custody. **Guidance:** Use write-blocking tools for disk imaging; analyze evidence on isolated forensic workstation; maintain documented custody trail for all evidence handling.
- **Pitfall:** Failing to preserve volatile data before system shutdown or reboot. **Guidance:** Prioritize volatile data collection immediately upon incident detection; establish pre-incident acquisition procedures in incident response plan.
- **Pitfall:** Relying on single log source for timeline reconstruction. **Guidance:** Correlate events from multiple independent sources (system logs, network logs, application logs, operational logs) to build comprehensive, verifiable timeline.
- **Pitfall:** Confusing attack vector with root cause. **Guidance:** Attack vector is the entry point (e.g., unpatched vulnerability); root cause is why the control failed (missing patch, misconfiguration, inadequate monitoring); clearly separate in investigation report.
- **Pitfall:** Conducting investigation without operational team coordination, risking system damage or safety hazards. **Guidance:** Brief investigation team on production constraints; obtain operations approval before evidence collection; document any operational disruptions encountered.
- **Pitfall:** Losing investigation findings after report completion. **Guidance:** Archive investigation records and evidence per organizational retention policy with access controls; establish process to periodically review investigations for lessons learned and monitoring improvements.
