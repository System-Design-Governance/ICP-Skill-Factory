## Description

Design the SCADA real-time database structure including point database schema, historical data tables, event logging schema, and calculated point definitions. This skill maps the SCADA point list (SK-D05-001) to database structures optimized for real-time monitoring, historical trend analysis, and audit logging. Covers both relational (SQL Server, PostgreSQL) and time-series database designs, incorporating performance tuning, data retention policies, and security controls for EMS/SCADA/DERMS platforms.

---

## Inputs

- SCADA point list (SK-D05-001) with all field definitions: point ID, data type, scaling, scan rate
- SCADA platform architecture and database backend selection (relational, time-series, hybrid)
- Real-time data flow requirements (polling intervals, latency targets, data rate projections)
- Historical data retention policy (archival duration, query performance targets)
- Calculated point specifications (derived values, aggregations, filtering logic)
- Security and audit requirements (access logging, data change tracking, encryption)
- Performance and scalability targets (storage capacity, query response time, concurrent connections)
- Operational reporting needs (KPI dashboards, alarm reporting, trend analysis)

---

## Outputs

- Real-time database schema design document
  - Point database table structure (point metadata, current values, timestamps)
  - Calculated point definition and refresh schedules
  - Data type mapping (SCADA abstraction → database native types)
- Historical data tables and partitioning scheme
  - Time-series table design with appropriate indexing
  - Data archival and purge policies
  - Aggregation tables (minute/hourly/daily summaries)
- Event logging schema
  - Alarm event table (alarm ID, timestamp, severity, point ID, value, threshold, operator action)
  - Change audit log (configuration changes, user access, data modifications)
  - Communications event log (RTU/FEP connection status, protocol errors)
- Calculated point specifications
  - Derivation logic (formulas, filtering, time-windowing)
  - Refresh rate and trigger conditions
- Database performance and sizing analysis
  - Storage capacity projection (raw + historical + archives)
  - Query performance estimates
  - Backup/recovery strategy
- Data security and encryption specifications
- Database creation and initialization scripts (SQL DDL)

## IEC 62443 Lifecycle Stages

- **Planning & Requirements**: Define data flow, retention policies, and security requirements
- **Design & Implementation**: Design database schema, create tables, configure archival
- **Verification & Validation**: Validate schema completeness, test data ingestion and query performance
- **Operations & Maintenance**: Monitor database health, manage archival/purge cycles, optimize queries
- **Decommissioning**: Archive final data, retire obsolete tables, decommission database instances

## Acceptance Criteria

1. Real-time point database schema includes: point ID (primary key), current value, timestamp, quality flag, last change time, scan interval
2. Calculated point table defined with: definition logic, input points, refresh rate, priority
3. Historical data tables partitioned by time (daily or monthly) for optimal query performance and archival management
4. Aggregation tables created for common query patterns (hourly/daily min/max/avg for trending)
5. Event logging schema captures: event ID, timestamp, point ID, old/new value, alarm threshold, severity, operator action
6. Change audit log records: change timestamp, user ID, object type, object ID, before/after values
7. Communications event log tracks: RTU/FEP connection status changes, protocol errors, data loss events
8. Data retention policy specified and enforced: raw data retention period, archive duration, purge schedule
9. Storage capacity projections validated: confirm database growth projections align with available disk space and backup strategy
10. Query performance targets met: real-time trending queries < 1 sec; historical queries < 5 sec
11. Backup and recovery strategy documented: RPO (recovery point objective), RTO (recovery time objective)
12. Encryption specified for sensitive data: at-rest encryption for database files, in-transit encryption for network communications
13. Sign-off by System Architect, DBA, and Security Engineer
