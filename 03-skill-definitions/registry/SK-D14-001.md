## Description

Develop the requirements specification document that captures, structures, and formalizes customer/employer security and functional requirements for an OT/ICS cybersecurity project, translating SOW language, tender documents, and stakeholder discussions into a structured requirements set suitable for feasibility assessment and Gate 0 evaluation. This skill covers requirements elicitation, structuring, traceability numbering, and specification document production. Under GOV-SD, requirements specification is a Pre-Gate 0 activity — outputs are advisory and non-binding until Gate 0 approval formalizes the project scope. The boundary of this skill is requirements documentation; the feasibility evaluation of those requirements falls under SK-D14-003 ⏳, and the security-specific requirements definition for tender response falls under SK-D14-010.

## Inputs

- Customer/employer SOW (Statement of Work) or tender documents
- Stakeholder meeting notes and Q&A records
- Applicable standards and regulatory requirements (IEC 62443, local regulations)
- Existing site documentation (brownfield — current system descriptions)
- ICP pre-sales engagement records (if available)
- ID03 §5.1.1: Requirements specification guidance
- ID03 Table 2 Doc 0.01: Requirements Specification as a required project deliverable

## Outputs

- Requirements Specification Document:
  - Project scope definition: SuC boundary, included/excluded systems, employer vs. integrator responsibilities
  - Functional requirements: system functionality, performance targets, availability requirements, capacity requirements
  - Security requirements: security objectives, compliance targets (IEC 62443 SL targets), specific security functions required
  - Operational requirements: maintenance windows, operational environment constraints, personnel availability
  - Interface requirements: integration points with employer systems, third-party systems, external networks
  - Requirements numbering: unique ID per requirement (REQ-nnn) for traceability
  - Priority classification: Must-Have / Should-Have / Nice-to-Have per requirement
  - Acceptance criteria: per-requirement verification method (test, inspection, review, demonstration)
- Requirements Traceability Matrix (initial): REQ ID → source document/stakeholder → priority → status
- Open Issues Register: ambiguous or unresolved requirements requiring stakeholder clarification

## Automation Potential

**Level:** Partial

Requirements extraction from structured tender documents can be partially automated (NLP-based clause identification). Requirements numbering and traceability matrix generation can be automated. Template-based document generation can streamline formatting. However, requirements elicitation from stakeholder discussions requires human communication skills, requirements interpretation requires domain expertise (translating business language to technical requirements), and priority classification requires stakeholder negotiation.

## Acceptance Criteria

1. Requirements specification covers all four requirement categories: functional, security, operational, and interface — no category empty without documented rationale
2. Every requirement has a unique ID (REQ-nnn), priority classification, and acceptance criterion with verification method specified
3. Security requirements include explicit IEC 62443 SL target references where applicable (which zones, which SL levels)
4. Requirements traceability matrix links every requirement to its source (SOW clause, stakeholder, standard reference) — no orphan requirements
5. Open issues register produced: all ambiguous or unresolved requirements flagged with owner and target resolution date
6. Document marked as Pre-Gate 0 advisory (non-binding) per GOV-SD boundary rule — no language implying formal commitment

## Estimated Effort

| Level | Effort | Notes |
|-------|--------|-------|
| Junior (< 2 yr) | 5–8 person-days | Assumes single-site, medium-complexity SOW; includes stakeholder meetings, document authoring, and one review cycle |
| Senior (5+ yr) | 3–5 person-days | Same scope; senior leverages requirements templates and rapid SOW analysis |

Notes: Effort depends heavily on SOW clarity and stakeholder availability. Complex multi-site projects or vague SOWs may require 2× effort. International projects with language barriers add communication overhead. Requirements specification often iterates 2–3 times before stabilization.
