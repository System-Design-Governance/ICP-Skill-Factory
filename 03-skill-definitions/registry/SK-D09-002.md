## Description

Author the Security Functional Description Specification (SFDS) document that maps IEC 62443-2-4 Service Provider (SP) requirements to the specific implementation design of the SuC, providing a comprehensive security design document that describes how each applicable Security Requirement (SR) is addressed through system countermeasures, configurations, and operational procedures. The SFDS serves as the bridge between the risk assessment outputs (FR/SR mapping) and the testing inputs (FAT/SAT test case derivation) — it is both a design specification and an SR checklist feeder. Under ICP governance (GOV-SD), the SFDS feeds the SR checklist that drives Gate 3 verification. This skill covers the full SP-requirement-to-implementation mapping document; the underlying risk assessment that determines SR applicability falls under SK-D01-007, and the actual testing that verifies the implementations falls under SK-D08-005.

## Inputs

- FR/SR mapping matrix (from SK-D01-007) — defines which SRs apply per zone and their target SL
- Zone/Conduit architecture (from SK-D01-001) — security topology being described
- Detailed risk assessment results (from SK-D01-007) — countermeasure specifications per risk
- Endpoint hardening design (from SK-D01-019) — hardening configurations to document
- Account and access control design (from SK-D01-020) — RBAC framework to document
- Security patch management design (from SK-D01-021) — patch procedures to document
- Backup and restore design (from SK-D01-022) — backup procedures to document
- Malware protection design (from SK-D01-023) — antivirus deployment to document
- Network segmentation design (from SK-D01-001) — firewall rules and conduit specifications
- IEC 62443-2-4: SP requirement categories (SP.01 through SP.12)
- IEC 62443-3-3: System security requirements (SR categories)

## Outputs

- Security Functional Description Specification Document (per ID12 exemplar format):
  - Document structure following IEC 62443-2-4 SP category organization:
    - SP.01: Solution staffing and qualifications
    - SP.02: Architecture design and engineering
    - SP.03: System hardening and configuration
    - SP.04: Wireless network security
    - SP.05: Security management system
    - SP.06: Configuration management
    - SP.07: Event management
    - SP.08: Backup and restore
    - SP.09: Verification and validation
    - SP.10: Malware protection
    - SP.11: Patch management
    - SP.12: Account management
  - Per SP requirement: SP ID, requirement description, applicability (Applicable/N/A with rationale), implementation description (how the SuC addresses this requirement), evidence reference (configuration file, document, test result), responsible role
  - Zone-specific implementation notes where SR implementation varies by zone SL
  - Cross-reference table: SP requirement → SR mapping → Zone applicability → Implementation status
- SR Checklist (Gate 3 feeder): per-SR status summary extracted from SFDS for Gate 3 verification use
- Implementation Gap Register: SP requirements marked as Planned or Partially Implemented, with remediation plan and target date

## Automation Potential

**Level:** Partial

SP requirement template population can be partially automated (pre-filling SP IDs, descriptions, and default applicability). Cross-reference matrix generation (SP → SR → Zone) can be automated from structured input data. Configuration evidence collection can be scripted (device config exports, screenshot automation). However, implementation description authoring requires expert judgment (explaining how specific countermeasures address SP requirements), applicability determination requires contextual analysis (why certain SP requirements are N/A for a given SuC), and evidence quality assessment requires human review.

## Acceptance Criteria

1. SFDS covers 100% of applicable IEC 62443-2-4 SP requirements: every SP category (SP.01–SP.12) addressed with per-requirement applicability determination
2. Each applicable SP requirement has: implementation description, evidence reference, responsible role, and SR mapping — no requirements left with blank implementation fields
3. N/A determinations include documented rationale: why the SP requirement does not apply to this SuC (not just marked N/A without explanation)
4. Zone-specific variations documented: where SR implementation differs by zone SL, the SFDS specifies the per-zone implementation
5. SR checklist extractable from SFDS: every applicable SR has a clear status (Implemented / Planned / N/A) that can be directly used for Gate 3 verification
6. SFDS reviewed and approved by Security Engineering Role and Design QA Role; cross-reference matrix verified for completeness (per GOV-SD Gate 3 evidence requirements)

| Senior (5+ yr) | 12–18 person-days | Same scope; senior leverages reusable SP implementation descriptions and efficient evidence collection workflows |

Notes: Effort is the highest among Batch 4 skills due to the breadth of SP coverage. ID12 exemplar is 97 pages for a medium-complexity SCADA project — this calibrates the expected document volume. Projects with higher SL targets (SL-3, SL-4) have more applicable SRs, increasing effort. The SFDS is typically authored iteratively: initial draft during R2 detailed design, then refined during R3 as implementations are completed. Reuse rate across similar projects can reach 40–50% for standard SP implementation descriptions.
