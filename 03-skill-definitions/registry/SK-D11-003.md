# Skill Definition — SK-D11-003

> **Critical Security Design Review**: A mandatory control gate that verifies security requirements are correctly translated into design, defense-in-depth principles are maintained, and no security gaps exist before proceeding to implementation. This skill executes the critical review procedures mandated by IEC 62443-3-2 §7.3.2, ensuring security-impacting design deliverables meet compliance requirements and are technically sound.

## Description

Execute critical security design reviews per IEC 62443-3-2 §7.3.2 for security-impacting design deliverables, verifying that security requirements (from the approved Security Functional Description Specification — SFDS) are correctly translated into design, defense-in-depth principles are maintained, and no critical security gaps exist. This is a verification gate that must be passed before proceeding to implementation (R3). The review covers all design domains where security decisions are made: Zone/Conduit Architecture (D01), Network Design (D02), Protection Device Design (D04), Device Configuration & Hardening (D05), and Data Security Design (D12). The skill uses checklists from SK-D11-001 (Design Review Checklist Development) and includes both formal structured review procedures and expert judgment on security design quality.

## Inputs

- Approved Security Functional Description Specification (SFDS) with security requirements traceability matrix (from SK-D01-001 ⏳: SFDS Development)
- Security-impacting design deliverables covering:
  - Zone/Conduit Architecture Diagram and specification (from SK-D01-001 ⏳: Zone/Conduit Architecture Design)
  - Network topology and data flow diagrams (from SK-D02-001 ⏳: OT Network Topology Design; SK-D02-004 ⏳: Data Flow Diagram Development)
  - Protection device selection and settings (from D04 protection engineering)
  - Device hardening and access control design (from SK-D05-001 ⏳: Device Hardening Design)
  - Cryptography, key management, and data encryption design (from SK-D12-004 ⏳: Cryptography and Key Management Design)
- Design Review Checklists for security verification (C.1–C.6 categories from SK-D11-001)
- Threat and Risk Assessment (Detailed) with identified threats and mitigations (from SK-D01-007 ⏳)
- Applicable security standards and compliance requirements (IEC 62443, NERC CIP, customer-specific)

## Outputs

- Critical Security Design Review Report documenting:
  - Scope of review (which design deliverables and domains were reviewed)
  - Verification results for each security requirement in the SFDS (requirement ID, design element that implements it, verification status: Pass, Fail, or Conditional Pass with mitigations)
  - Findings: defects, gaps, or deviations from security requirements or standards
  - Risk assessment of findings (Critical, High, Medium, Low) with recommended mitigations
  - Defense-in-depth assessment: verification that multiple independent security layers are implemented
- Updated requirements traceability matrix (RTM) showing design-to-requirements mapping
- Conditions or mitigations required before proceeding to implementation (if Conditional Pass)
- Sign-off by SAC and PE/O (Project Executive / Owner) per ID01 §7.4.2

## IEC 62443 Lifecycle Stages

- R2 (Detailed Design — critical security review as a gate before proceeding to R3)
- R3 (Implementation gate — verification that design is implementation-ready)

## Roles

- SAC (Security Architect) — primary: leads security design review, makes Pass/Fail/Conditional Pass decision
- PE/O (Project Executive / Owner) — approves review findings and any Conditional Pass mitigations
- STC (Security Technical Consultant) — supports technical depth of review, especially for cryptography and access control design
- DES (Design Engineer) — presents design artifacts, addresses technical clarifications during review
- Compliance Lead — provides interpretation of applicable standards and regulatory requirements

## Acceptance Criteria

1. All security requirements in the approved SFDS have been mapped to design elements (100% coverage of requirements traceability)
2. Each security requirement has a corresponding design element that implements it, with documented verification of correctness
3. No Critical findings remain unaddressed (Fail status converted to Pass or Conditional Pass with documented mitigations)
4. High-risk findings have documented mitigations or explicit risk acceptance by PE/O
5. Defense-in-depth assessment confirms that security controls are independent and not single-point-of-failure (no reliance on a single mechanism for all security)
6. Design review checklist (C.1–C.6) completion rate is 100%; any non-applicable items are explicitly documented
7. Cryptography design (if used) has been reviewed for key management, cryptographic standards compliance, and crypto-agility
8. Review has been signed off by SAC and PE/O per ID01 §7.4.2 review process

## Estimated Effort

| Level | Effort | Notes |

| SAC / Senior Security Architect | 10–15 person-days | Complete security design review for greenfield project, ~15–20 security requirements; includes RTM update and report |

Notes: Brownfield projects or projects with complex threat models may require 20+ days. Projects with Conditional Pass findings may require 2–3 additional days for mitigation design and re-verification.
