## Description

Data Format Conversion Design specifies the logical and technical approach for translating heterogeneous data formats and protocols across integrated OT/ICS systems, enabling semantic interoperability where native formats differ. This skill addresses the translation of Modbus to IEC 61850, DNP3 to OPC UA, and similar cross-protocol conversions that arise when legacy and modern systems must exchange information reliably. The design phase (R2) establishes conversion rules, data model mappings, unit transformations, and timestamp normalization requirements that guide implementation in SK-D07-003 and later test phases. Conversion design is critical for ensuring data fidelity, consistency, and security throughout the integrated system.

## Inputs

- System architecture diagram from SK-D07-002 (identifies system boundaries and communication links requiring conversion)
- Data model specifications from each subsystem (Modbus registers, IEC 61850 data objects, DNP3 points, OPC UA attributes)
- Protocol documentation (IEC 61850, Modbus, DNP3, OPC UA specification references)
- Unit conversion requirements and normalization rules from engineering specifications
- Timestamp and time synchronization requirements
- Security and logging requirements for conversion operations

## Outputs

- Data format conversion specification document with detailed mapping rules
- Protocol translation matrix showing source and target protocol elements, data types, and transformation logic
- Unit conversion and normalization rules handbook
- Data model mapping spreadsheets or UML diagrams
- Conversion test case definitions and expected result specifications
- Implementation guide for SK-D07-003 conversion logic development

## Tools

- UML modeling tools (Enterprise Architect, StarUML) for data model visualization
- Python or C# for prototyping conversion logic and validation
- Protocol analyzers (Wireshark) for reference and protocol signature analysis
- Data transformation tools (XSLT, graphical mapping tools)
- Spreadsheet tools for point mapping matrices and unit conversion tables

## IEC 62443 Lifecycle Stages

- **R0 (Preparation)**: Identify protocol heterogeneity gaps and conversion requirements from system architecture.
- **R1 (Design Planning)**: Assess conversion complexity and feasibility; select conversion approaches (bidirectional, unidirectional, rule-based).
- **R2 (Design Development)**: Create detailed conversion specifications, mapping tables, and transformation rules for all identified protocol pairs.
- **R3 (Implementation Planning)**: Validate conversion specifications with SK-D07-003 development team and prepare test case definitions.

## Roles

- **SYS**: System Architect ensures conversion design aligns with overall system topology and data flow requirements.
- **ENG**: Integration Engineer specifies technical conversion rules and validates feasibility with available tools and libraries.
- **SEC**: Security Officer reviews conversion logic for potential data disclosure, injection, or manipulation vulnerabilities.

## Acceptance Criteria

1. Data format conversion specification document is complete and includes conversion rules for all identified protocol pairs (e.g., Modbus ↔ IEC 61850, DNP3 ↔ OPC UA).
2. Protocol translation matrix is defined with explicit mappings between source and target protocol elements, including data types, units, and scaling factors.
3. Unit conversion rules are documented for all physical quantities crossing subsystem boundaries (e.g., MW to kW, degrees Celsius to Fahrenheit if required).
4. Timestamp normalization approach is defined, including time zone handling, synchronization tolerance, and timestamp format conversion.
5. Conversion test cases are defined with input/output examples that demonstrate correct transformation of representative data samples.
6. Security review confirms that conversion logic does not introduce data disclosure, injection, or integrity vulnerabilities.

## Source Traceability

- IEC 61850 standard defines semantic structure and information model for substation communication; conversion design must preserve semantic correctness.
- IEC 61970 EMS-API defines normalized data models for energy management; conversion to/from EMS must align with EMS-API structures.
- Modbus Specification defines register-based data representation; conversion rules must preserve register semantics and value ranges.
- IEEE 1815 (DNP3) defines point-based data model; conversion to/from DNP3 must respect point quality flags and object variations.
- OPC Foundation OPC UA Specification defines information model with attributes, events, and methods; conversion to/from OPC UA must map correctly to information model constructs.
