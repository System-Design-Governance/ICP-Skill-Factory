## Description

Implement systematic security hardening across all endpoint categories within the SuC — including Windows-based host servers (SCADA HMI), ICS embedded devices (IED, PLC/RTU), and network infrastructure devices (managed switches, wireless AP). This skill applies the defense-in-depth principle at the device level through physical hardening (disabling unused ports/interfaces), logical hardening (protocol restriction, access control enforcement), and software hardening (firmware updates, service minimization, default credential elimination). The boundary of this skill is device-level configuration hardening; the security architecture that determines which devices need hardening and to what level falls under SK-D01-001 (Zone/Conduit Architecture Design), and the ongoing patch management of these hardened devices falls under SK-D01-021 ⏳.

## Inputs

- Asset inventory with device specifications (from SK-D01-005: Asset Inventory Development)
- Zone/Conduit architecture with SL-T assignments per zone (from SK-D01-001)
- FR/SR requirements mapping per device type (from SK-D01-007: Detailed Risk Assessment — countermeasure specifications)
- Vendor hardening guides and CIS Security Configuration Benchmarks per device category
- Vulnerability scanning results (Nessus or equivalent) — baseline state before hardening
- Equipment baseline with firmware/software versions and serial numbers (from ID12 exemplar: SP06.02BR asset baselining)

## Outputs

- Hardening Implementation Report per device category:
  - **Server Hardening Checklist** (per ID13 exemplar): OS patched, unnecessary services removed, user authentication configured, access controls set, security controls installed, security tested, unnecessary accounts removed, strong remote access (MFA)
  - **Network Device Hardening Checklist**: firmware updated with hash verification, unused physical interfaces shut down, switch port access control, encrypted passwords, logging enabled, unneeded services disabled, secure remote management (SSH v2, no Telnet), restricted management access
  - **IACS Embedded Device Hardening Checklist**: remote program changes disabled, firmware updated with hash verification, unused network interfaces disabled, default passwords changed, logging enabled, unnecessary services/protocols disabled, firewall with rate-limiting and deep-packet inspection
- Physical Hardening Documentation: port/interface disable strategy per device (e.g., VGA/DVI/USB/RS232 disabled; RJ45/PS2 enabled for operations)
- Protocol Hardening Tables: enabled vs. disabled protocols per device with justification
- Post-Hardening Verification Report: vulnerability scan comparison (before vs. after), compliance score against CIS benchmark
- Exception Log: devices where full hardening cannot be applied with compensating controls and approval

## Tools

- Nessus / Tenable.io (vulnerability scanning — pre and post hardening comparison)
- CIS Security Configuration Benchmarks (per OS and device type)
- Device Manager / Windows Server Manager (configuration inspection)
- SSH / secure console access (device configuration)
- Palo Alto Cortex XDR (endpoint protection verification, per ID12 exemplar)
- Configuration management tools (Ansible, or vendor-specific: ABB, Siemens management consoles)

## Dependencies

### Hard Dependencies (Required Before Execution)

- SK-D01-005: Asset Inventory Development — must know which devices to harden
- SK-D01-001: Zone/Conduit Architecture Design — SL-T per zone determines hardening depth
- SK-D01-007: Detailed Risk Assessment — FR/SR mapping specifies which security controls apply per device

### Soft Dependencies (Enhances Effectiveness)

- SK-D01-021 ⏳: Security Patch Management — patches should be applied before or during hardening
- SK-D01-020 ⏳: Account and Access Control Management — account hardening is a sub-component often done simultaneously
- SK-D01-023 ⏳: Malware Protection Implementation — endpoint protection installation often occurs during hardening window

## Automation Potential

**Level:** Partial

Configuration scripting (Ansible playbooks, GPO templates) can automate repeatable hardening steps for Windows servers and network devices. Vulnerability scanning before/after is automated. However, IACS embedded device hardening often requires vendor-specific tools with manual interaction, physical port disabling requires on-site work, and exception judgment for devices that cannot be fully hardened requires human expertise.

## Acceptance Criteria

1. Hardening checklists completed for 100% of devices in the asset inventory, with each checklist item marked Pass/Fail/Exception
2. Post-hardening vulnerability scan shows no Critical or High findings that existed in pre-hardening baseline (or documented exception with compensating control)
3. Physical hardening documented: all unused ports/interfaces disabled with per-device justification for remaining enabled interfaces
4. No default passwords remain on any device; all credentials changed and documented in secure credential store
5. Secure remote management enforced: SSH v2 only (no Telnet), encrypted passwords, restricted management access IP ranges
6. Exception log documents all devices where full hardening is infeasible, with compensating controls approved by Security Engineering Role

## Estimated Effort

| Level | Effort | Notes |
|-------|--------|-------|
| Junior (< 2 yr) | 15–20 person-days | Assumes ~50 devices across 3 categories (server, network, IACS); includes scanning, hardening, documentation |
| Senior (5+ yr) | 8–12 person-days | Same scope; senior leverages scripted hardening templates and parallel execution |

Notes: Effort scales roughly linearly with device count. IACS embedded devices typically require 2–3× effort per device compared to Windows servers due to vendor-specific tooling. Physical hardening requires on-site presence. Multi-site deployments multiply effort by site count.
