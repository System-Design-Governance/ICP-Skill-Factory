## Description
Design the defense-in-depth security strategy for OT/ICS environments, layering security controls across physical, network, host, application, and data protection layers. This skill provides the strategic framework that SK-D01-001 (Zone/Conduit Architecture) instantiates at the network level, establishing the multi-layered control philosophy that subsequent D01 implementation skills operationalize.

## Description — Objectives
- Define a multi-layer security control framework aligned with IEC 62443 principles
- Establish clear architectural layering across physical, network, host, application, and data domains
- Document control dependencies and integration points across layers
- Create a prioritized roadmap for control implementation and validation
- Ensure alignment with operational technology (OT) and industrial control system (ICS) constraints

## Description — Scope
**In Scope:** Strategic design of defense-in-depth controls, layering philosophy, control architecture, cross-layer dependencies, and implementation sequencing.

**Out of Scope:** Individual control implementation (owned by SK-D01-019 endpoint protection, SK-D01-020 access controls, SK-D01-023 malware protection, etc.); detailed technical specifications for each layer; operational procedures and training.

## Inputs
- IEC 62443-3-3 architecture and control framework (ID01 §6.5.1.3)
- IEC 62443-2-1 compliance requirements (ID02 Annex A §9)
- Zone/Conduit design from SK-D01-001
- Preliminary threat assessment from SK-D01-006
- Current security posture baseline from SK-D01-011

## Description — Key Concepts
- **Defense-in-Depth:** Multi-layered security architecture where control failure at one layer is mitigated by overlapping controls in adjacent layers
- **Control Layer:** A logical or physical boundary incorporating related security mechanisms (e.g., network layer, host layer, application layer)
- **Cross-Layer Dependency:** A relationship where one layer's control effectiveness depends on or influences another layer's implementation
- **Control Redundancy:** Intentional duplication of control functions across layers to ensure resilience against single-point failures

## Description — Methodology
The defense-in-depth strategy design follows a systematic five-layer model:

1. **Physical Layer:** Physical access controls, facility security, equipment hardening, environmental controls
2. **Network Layer:** Perimeter defense, segmentation (zones/conduits), network monitoring, DLP controls
3. **Host Layer:** Operating system hardening, local firewalls, host-based intrusion detection, privileged access management
4. **Application Layer:** Secure coding practices, input validation, authentication/authorization, application monitoring
5. **Data Layer:** Encryption at rest, encryption in transit, data classification, access control to data objects

For each layer, the strategy identifies:
- Primary security objectives aligned with IEC 62443 functional requirements
- Key control functions and mechanisms
- Interdependencies with other layers
- Implementation priorities and sequencing
- Success metrics and validation criteria

## Description — Activities
1. Conduct layer-by-layer security requirements analysis aligned with IEC 62443-3-3
2. Map threat scenarios from SK-D01-006 to control layers and identify coverage gaps
3. Design control interaction matrix showing dependencies and reinforcements across layers
4. Develop layer-specific control architecture diagrams
5. Document residual risks and mitigation strategies for layer interactions
6. Create implementation sequence and resource allocation plan
7. Define validation and verification criteria for each layer

## Outputs
- Defense-in-Depth Strategy Document including five-layer architecture and control framework
- Control Layer Architecture Diagrams (physical, network, host, application, data)
- Cross-Layer Dependency Matrix showing control interactions and redundancies
- Implementation Roadmap with phased sequencing and resource requirements
- Validation Criteria and Success Metrics for each layer and cross-layer interactions
- Risk Mitigation Plan for residual risks at layer boundaries

## Acceptance Criteria

## Description — Practical Guidance
- **Pitfall:** Over-concentrating controls in a single layer (e.g., all access control at network perimeter). **Guidance:** Distribute overlapping controls to achieve true defense-in-depth redundancy.
- **Pitfall:** Ignoring layer interactions in implementation sequencing. **Guidance:** Use dependency matrix to identify mandatory sequencing (e.g., physical perimeter security before network segmentation).
- **Pitfall:** Designing an ideal strategy disconnected from OT operational constraints. **Guidance:** Validate strategy against operational technology downtime budgets, performance impact, and vendor support capabilities.
- **Pitfall:** Treating data layer security as optional or secondary. **Guidance:** Emphasize data encryption and access control as essential to defense-in-depth in critical OT environments.
