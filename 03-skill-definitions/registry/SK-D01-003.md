## Description

Translate the approved Zone/Conduit Architecture (SK-D01-001) into a detailed set of firewall rules and access control lists (ACLs) that enforce inter-zone communication policies. This skill operationalizes the logical zone/conduit design by specifying concrete source IP/zone, destination IP/zone, protocol, port, and action (allow/deny) for each allowed communication flow, along with logging and justification. The boundary of this skill is firewall rule specification and documentation; the actual firewall device configuration and testing falls under the device-level configuration skills in D05.5, and the broader network segmentation documentation strategy falls under SK-D01-004 ⏳.

## Inputs

- Approved Zone/Conduit Architecture Diagram (from SK-D01-001: Zone/Conduit Architecture Design)
- Conduit Specification Table: source zone → destination zone, allowed protocols, direction, authentication requirements (from SK-D01-001)
- Asset Inventory Register with IP/MAC addresses and zone assignments (from SK-D01-005 ⏳: Asset Inventory Development)
- Data Flow Diagram showing inter-asset communication flows (from SK-D02-004 ⏳: Data Flow Diagram Development)
- Firewall device specifications and available rule formats (e.g., Palo Alto Networks, Fortinet, Cisco ASA)
- Customer security policy and exception procedures

## Outputs

- Detailed Firewall Rule Specification Table: rule ID, source zone/IP (or subnet), destination zone/IP (or subnet), protocol, port(s), action (allow/deny/log), logging level, rule expiration date (if temporary), technical justification, and traceability to approved conduit
- Firewall Rule Priority/Order Documentation: rule precedence and conflicts resolution matrix
- Denial/Exception Log: any requested rules denied by security policy, with documented rationale and approval
- Firewall Configuration Narrative Document: section within Security Implementation Specification (per ID02 Annex A.9 §10.1)
- Rule Testing Plan: test cases for verification that rules enforce intended communication paths and block unintended paths (per ID02 C.3 design verification)

## Tools

- Firewall vendor rule specification templates (Palo Alto, Fortinet, Cisco, etc.)
- Excel / structured markdown (rule specification tables and traceability matrices)
- Network documentation tools (draw.io, Visio) for rule visualization
- Firewall simulator / test environment for rule validation
- Rule conflict detection tools (if available from vendor)

## Roles

- STC (Security Technical Consultant) — primary: translates zone/conduit into firewall rules, validates rule completeness and consistency
- SAC (Security Architect) — approves rule exceptions and escalations; resolves policy conflicts
- DES (Design Engineer) — integrates firewall rules into network design documentation
- Operations/Network Team — validates firewall rule feasibility and performance impact

## Dependencies

- SK-D01-001: Zone/Conduit Architecture Design — firewall rules operationalize the approved conduit specifications
- SK-D01-005 ⏳: Asset Inventory Development — rule specification requires knowledge of source/destination IPs and zone membership

### Soft Dependencies (Enhances Effectiveness)

- SK-D01-010 ⏳: Security Level Target Assessment — SL-specific access control policies shape rule strictness
- Device-level configuration skills (D05.5) — knowledge of firewall vendor syntax and capabilities helps rule specification

## Acceptance Criteria

1. Every allowed communication path defined in the approved Conduit Specification Table has at least one corresponding firewall rule
2. Every rule specifies: source (IP/zone), destination (IP/zone), protocol, port, action, logging, and justification with traceability ID
3. No overlapping or conflicting rules exist (conflicts identified and resolved with documented precedence)
4. Denial/exception log documents all requested rules that were denied by policy, with SAC-level approval
5. Rule specification covers both unicast and any broadcast/multicast communication requirements with documented security rationale
6. All rules include default-deny catch-all rules at the end of each direction to enforce "deny by default" principle
7. Rule testing plan is complete and traceability to zone/conduit architecture is bidirectional
