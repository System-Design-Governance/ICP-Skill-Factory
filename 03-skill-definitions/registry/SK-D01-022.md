## Description

Design the comprehensive backup and restore procedure covering all device categories within the SuC, including backup targets, methods, schedules, retention policies, security requirements, restoration procedures, and disaster scenario playbooks. This skill defines the backup infrastructure architecture, specifies per-device-category backup approaches, and establishes the recovery procedures that ensure system recoverability within defined RPO/RTO targets. The boundary of this skill is backup/restore procedure design and infrastructure specification; the actual disaster recovery planning at the business continuity level falls under organizational ISMS procedures, and the endpoint hardening that protects backup infrastructure falls under SK-D01-019.

## Inputs

- Asset inventory with device types and OS platforms (from SK-D01-005)
- RPO (Recovery Point Objective) and RTO (Recovery Time Objective) targets from customer requirements
- System architecture and network topology (from SK-D02-001 ⏳) — informs backup data flow paths
- Storage capacity planning requirements
- Employer's IT backup infrastructure and central data center policies
- Configuration baseline per device (from SK-D01-019: Endpoint Hardening — post-hardening configuration is what needs backing up)

## Outputs

- Backup and Restore Procedure Document (per ID07 §6.0 exemplar):
  - Backup target definitions for 3 device categories:
    - Windows-based host servers (SCADA HMI): Acronis Agent, centralized scheduling, system image + configuration
    - ICS devices (IED Control/Protection): configuration file export to on-site file server, event-driven backup on configuration change
    - Network devices (Ethernet Switch, Wireless AP): configuration file export, event-driven backup on change
  - Backup methods specification: Manual File Backup, Manual Image Backup, Automatic Image Backup (to NAS)
  - Backup supervision details: tool configuration, monitoring, alerting
  - Backup security: encryption requirements, access control to backup media, secure transfer to central repository
- Retention Policy Matrix:
  - Windows system image: every 6 months
  - Configuration file: every 6 months
  - Security logs (e.g., Palo Alto): per employer policy
  - On-site retention in Park DMZ → central data center for long-term archival
- Restoration Procedures per device category: step-by-step recovery from backup
- Disaster Scenario Playbooks:
  - Windows-based host server crash recovery
  - Malware-contaminated backup detection and clean recovery
  - Corrupt configuration file recovery
- Logical Backup Infrastructure Diagram: backup agents, on-site file servers, NAS, DMZ staging, central repository data flow

## Tools

- Acronis Cyber Protect (primary backup tool per ID07 exemplar: agent deployment, centralized scheduling, image backup)
- Windows Backup and Restore (native OS backup capabilities)
- NAS Server (automatic image backup destination)
- On-site Windows-based file server (ICS/network device configuration storage)
- Park DMZ storage (intermediate staging before central data center transfer)
- Vendor-specific configuration export tools (IED configuration managers, switch CLI)

## Automation Potential

**Level:** Partial

Automated backup scheduling, agent deployment, and retention enforcement are fully automatable (Acronis, Windows Backup). Backup integrity verification (hash checks) can be automated. However, disaster recovery playbook design, restoration procedure validation through testing exercises, and backup security architecture decisions require human judgment. Event-driven backup triggering on configuration changes requires integration design.

## Acceptance Criteria

1. Backup targets cover 100% of devices in the asset inventory across all 3 categories (WBHS, ICS, network)
2. Backup methods specified per device category with defined schedule and retention period
3. Restoration procedures documented and tested: at least one test restoration per device category verified during FAT/SAT
4. Disaster scenario playbooks cover minimum 3 scenarios: server crash, malware contamination, corrupt configuration
5. Backup security requirements defined: encryption, access control, secure transfer, and retention compliance
6. Logical backup infrastructure diagram documents complete data flow from device → on-site storage → DMZ staging → central repository
