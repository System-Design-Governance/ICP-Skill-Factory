# Skill Definition — SK-D12-001

> **Data Acquisition Architecture Design**: The foundational architectural skill for operational data collection in OT/ICS environments. This skill designs the complete data acquisition infrastructure that enables condition monitoring, SCADA data collection, and analytics while maintaining security and data quality requirements. It bridges system architecture (D02) and operational data systems (D12).

## Description

Design the data acquisition architecture for collecting operational data from OT field devices, SCADA systems, and smart meters. This skill covers the complete pipeline: data collection protocols (Modbus, DNP3, IEC 60870-5-104, etc.), polling and event-driven data collection strategies, edge processing and local intelligence, data quality validation and filtering, security requirements for data-in-transit, and integration with the Data Platform (D12 subsequent skills). The boundary of this skill is the architecture and design of data collection at the source and in-transit; the detailed implementation of protocol parsers falls under SK-D12-002 ⏳ (Protocol Parser Development), time-series database design falls under SK-D12-003 ⏳ (Time-Series Database Architecture), and encryption implementation falls under SK-D12-004 ⏳ (Cryptography and Key Management Design).

## Inputs

- System architecture and network topology (from SK-D02-001 ⏳: OT Network Topology Design)
- Single-Line Diagram with metering and SCADA monitoring points identified (from SK-D09-003 ⏳: SLD Development, or D09 equipment design)
- SCADA/DCS system architecture and available data points (from customer specifications or legacy system documentation)
- Operational data requirements specification:
  - What data must be collected (real-time power flows, voltages, temperatures, etc.)
  - Collection frequency / latency requirements (real-time vs. interval-based)
  - Data retention and historical analysis requirements
  - Operational dashboards and reporting requirements
- Smart meter communication specifications (if applicable): meter types, communication protocols, reading frequencies
- Security requirements for data-in-transit (from SFDS, per SK-D01-001 ⏳)
- Data quality requirements: accuracy, resolution, completeness targets
- Customer operational and compliance requirements (e.g., NERC CIP for North American utilities)

## Outputs

- Data Acquisition Architecture Diagram showing:
  - Data sources (field devices, SCADA systems, meters) and their locations
  - Data collection protocols per source/zone
  - Data collection points (RTU, gateway, edge processor)
  - Data flows (source → collector → processing → storage)
  - Polling frequencies / event triggers per data source
  - Quality validation points and data filtering logic
  - Security controls (encryption, authentication) for data-in-transit
- Data Collection Protocol Specification:
  - Protocol selection rationale per data source (why Modbus vs. DNP3, etc.)
  - Polling/subscription strategy: fixed-interval polling, change-of-value event-driven, hybrid
  - Communication parameters: baud rates, timeouts, redundancy/failover strategy
  - Bandwidth requirements and impact on network design
- Edge Processing and Local Intelligence Design (if applicable):
  - Real-time alerting on field devices (e.g., threshold violations)
  - Local data buffering and store-and-forward strategy for intermittent connectivity
  - Timestamp synchronization requirements
- Data Quality Architecture:
  - Validation rules for incoming data (range checks, plausibility checks, discontinuity detection)
  - Handling of bad/missing data (interpolation, smoothing, or flagging)
  - Quality metrics dashboard specification
- Security Requirements for Data-in-Transit:
  - Encryption requirements (which data streams require TLS/encryption)
  - Authentication requirements (how data source identity is verified)
  - Access control to sensitive data (e.g., metering data privacy per regulations)
- Integration specification for SK-D12-002, SK-D12-003, SK-D12-004 (implementation sequencing)

## Standards

- IEC 60870-5-104: Telecontrol equipment and systems — DNP3 alternative and industry standard protocol
- IEC 61850: Communication networks and systems for power utility automation — modern data model for power system data
- Modbus Organization Modbus/TCP specification — widely used legacy protocol in industrial automation
- DNP3 Technical Committee DNP3 standard — North American utility standard
- IEC 62443-4-1 & 4-2: Security for Industrial Automation and Control Systems — data protection and secure communication requirements
- NERC CIP (North America): Critical Infrastructure Protection standards for operational data handling
- IEC 61508: Functional Safety of Electrical/Electronic/Programmable Electronic Safety-related Systems — data quality and reliability

## IEC 62443 Lifecycle Stages

- R2 (Detailed Design — data acquisition architecture finalized)
- R3 (Implementation — data collection protocols implemented per architecture)

## Roles

- SYS (System Architect) — primary: designs data acquisition architecture, integrates with network and SCADA design
- DES (Design Engineer) — implements protocol specifications and detailed data collection configurations
- SAC (Security Architect) — advises on security requirements for data-in-transit, encryption strategy
- OPS (Operations Engineer) — provides operational data requirements and validation rules
- SCADA Integrator (specialist role, may be customer or external consultant) — provides SCADA system interface expertise

## Acceptance Criteria

1. Data Acquisition Architecture Diagram covers 100% of identified data sources: field devices, SCADA systems, smart meters
2. For each data source, the selected collection protocol is justified with reference to equipment capabilities and customer requirements
3. Polling frequencies and event-driven triggers are specified for each data stream, with rationale for real-time vs. interval-based strategy
4. Data quality validation rules are defined for all data streams; accuracy and completeness targets are documented
5. Security requirements for data-in-transit (encryption, authentication, access control) are aligned with SFDS and zone-based security levels
6. Edge processing strategy (if used) is documented with clear roles: local alerting vs. cloud-based analytics
7. Bandwidth and latency impact assessment is complete; impacts on network design (from SK-D02-001) are identified and mitigation strategies documented
8. Architecture has been reviewed and approved by SYS, SAC, and SCADA Integrator (or SCADA system owner)

## Estimated Effort

| Level | Effort | Notes |
|-------|--------|-------|
| Junior System Architect (< 2 yr) | 8–12 person-days | Moderate complexity project, ~20–30 data points, 2–3 SCADA systems; includes architecture, protocol specification, and documentation |
| Senior System Architect (5+ yr) | 4–6 person-days | Same scope; senior leverages protocol libraries and prior project templates |

Notes: Complex multi-site deployments or projects with aggressive data collection requirements (sub-second latency, high-frequency sampling) may require 15–20 days. SCADA system customization or non-standard protocols add 2–5 days. Edge processing design (real-time analytics at gateway) adds 3–5 days.
