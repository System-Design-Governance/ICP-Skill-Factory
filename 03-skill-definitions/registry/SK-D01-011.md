## Description
Perform systematic gap analysis against IEC 62443 standard series requirements, comparing current security posture with target compliance levels (SL1-SL4). This skill uses IEC 62443-2-1 Annex C verification checklists (C.1 through C.8) as the assessment framework. Gap analysis is conducted at R1 (initial baseline assessment) and R4 (post-deployment verification) to track remediation progress and validate control implementation effectiveness. The gap analysis feeds strategic roadmap development for compliance remediation (SK-D01-030 policies/procedures, SK-D01-035 integrated risk assessment, and D01/D02 control implementation skills).

## Description — Objectives
- Conduct comprehensive baseline assessment of current OT/ICS security posture against IEC 62443 requirements
- Apply IEC 62443-2-1 Annex C verification checklists systematically across all security management and process domains
- Quantify and prioritize compliance gaps with explicit remediation roadmap
- Establish traceability between identified gaps and specific IEC 62443-3-3 functional/assurance requirements
- Track gap remediation progress through R1 baseline, R2/R3 implementation, and R4 post-deployment verification phases
- Document gap closure evidence and residual risks

## Description — Scope
**In Scope:** Assessment of current security controls against IEC 62443-1-1 through 62443-3-3 requirements; application of ID02 Annex C checklists; gap identification and remediation roadmap; R1 baseline and R4 verification assessments; residual risk documentation.

**Out of Scope:** Implementation of controls (handled by respective D01/D02 skills); control effectiveness testing (handled by verification activities); operational security monitoring (post-deployment activity); vendor/product assessment.

## Inputs
- IEC 62443 standard series (ID01: 62443-3-3; ID02: 62443-2-1 with Annex C checklists)
- Current asset inventory, network diagrams, and system architecture documentation
- Existing security policy documentation (if any) and operational procedures
- Target compliance level decision (SL1, SL2, SL3, or SL4) from governance
- Preliminary threat assessment from SK-D01-006 or SK-D01-008
- Historical compliance/audit reports if available

## Description — Key Concepts
- **IEC 62443 Compliance:** Adherence to the IEC 62443 standard series (security terminology, functional requirements, system security requirements, security program requirements)
- **Security Level (SL):** Quantified target for security requirements, ranging from SL1 (basic awareness) to SL4 (critical facility with sophisticated threats); SL increases both functional and assurance requirements
- **Compliance Gap:** A discrepancy between current security posture and target IEC 62443 requirements for the assigned Security Level
- **Verification Checklist:** Structured set of assessment criteria from IEC 62443-2-1 Annex C (C.1-C.8) covering security management, asset management, process safety, supply chain, and organizational domains
- **Gap Remediation Roadmap:** Prioritized plan to close identified gaps, sequenced by criticality, resource availability, and operational constraints

## Description — Methodology
IEC 62443 compliance gap analysis follows a systematic five-phase process:

**Phase 1: Target Compliance Definition**
- Document target Security Level (SL1-SL4) and applicable IEC 62443 requirements
- Establish baseline functional and assurance requirements from IEC 62443-3-3 for target SL
- Identify management and process requirements from IEC 62443-2-1

**Phase 2: Current Posture Assessment**
- Inventory current security controls and practices across OT/ICS environment
- Map existing controls to IEC 62443-3-3 functional requirements (SR 1.1-1.13, SR 2.1-2.13, SR 3.1-3.14, SR 4.1-4.4)
- Assess control effectiveness and maturity level (Ad-Hoc, Repeatable, Defined, Quantitatively Managed, Optimized)
- Document gaps where controls are absent, incomplete, or immature

**Phase 3: Checklist-Based Assessment**
Apply IEC 62443-2-1 Annex C verification checklists systematically:
- **C.1:** Security management practices and governance (organizational, policy, responsibility)
- **C.2:** Asset management and change control (asset inventory, configuration management, patch management)
- **C.3:** Access control and privileged user management (account lifecycle, authentication, authorization)
- **C.4:** System hardening and secure configuration (OS hardening, application hardening, secure defaults)
- **C.5:** Data protection and cryptography (encryption, key management, data classification)
- **C.6:** Threat monitoring and detection (logging, alerting, incident response)
- **C.7:** Supply chain and third-party management (vendor assessment, secure procurement, incident information sharing)
- **C.8:** Process safety integration (hazard analysis, safety interlock verification, safety-security interaction)

For each checklist item, assess compliance status: Compliant, Partially Compliant, Non-Compliant, or Not Applicable.

**Phase 4: Gap Identification and Quantification**
- Consolidate assessment results into gap matrix: Requirement → Current State → Target State → Gap Description
- Categorize gaps by severity (Critical, High, Medium, Low) based on:
  - Impact on functional/assurance requirements
  - Risk exposure from non-compliance
  - Operational consequence of gap
- Estimate remediation effort (hours, cost, schedule impact)

**Phase 5: Roadmap Development**
- Sequence gap remediation by priority and dependencies
- Assign ownership and resource requirements for each gap closure activity
- Link gaps to specific D01/D02 implementation skills (SK-D01-019 through SK-D01-023, SK-D02-xxx)
- Establish milestone dates and success criteria for gap closure verification

## Description — Activities
1. Facilitate gap analysis kickoff meeting with security and OT stakeholders to define target SL and scope
2. Conduct current-state security control inventory and assessment across all OT/ICS components
3. Apply IEC 62443-2-1 Annex C checklists item-by-item with documented evidence for each checklist item
4. Perform gap consolidation and severity categorization with multidisciplinary review
5. Develop detailed remediation roadmap with resource estimates, scheduling, and ownership
6. Present gap analysis findings to governance with executive summary and risk implications
7. At R4 (post-deployment), repeat assessment to verify gap closure and document remediation evidence
8. Maintain gap tracking dashboard through implementation phases (R2, R3) with periodic review

## Outputs
- **IEC 62443 Compliance Gap Analysis Report:** Executive summary of target SL, current posture assessment, gap inventory, and remediation roadmap
- **Detailed Checklist Assessment:** Item-by-item assessment of all IEC 62443-2-1 Annex C checklist items with compliance status and evidence links
- **Gap Matrix:** Structured table showing requirement, current state, target state, gap, severity, effort estimate, and ownership
- **Gap Remediation Roadmap:** Phased plan with milestones, resource allocation, ownership, and success criteria for each gap closure
- **Requirement-to-Skill Traceability:** Cross-reference showing which D01/D02 implementation skills address each gap
- **Post-Deployment Verification Report (R4):** Assessment showing gap closure evidence and residual non-compliance items

## Acceptance Criteria
- Assessment is based on IEC 62443-2-1 Annex C checklists (C.1-C.8) applied systematically across all items
- Each checklist item has explicit compliance status (Compliant, Partially Compliant, Non-Compliant, N/A) with documented evidence
- All gaps are assigned severity (Critical, High, Medium, Low) with documented justification
- Remediation roadmap includes specific sequencing, resource estimates in hours/cost, and ownership assignment for each gap
- All gaps have explicit linkage to specific IEC 62443-3-3 functional/assurance requirements
- At R4, post-deployment assessment verifies gap closure with supporting evidence (test reports, configuration audits, policy documentation)
- Gap tracking dashboard shows progress across R1 baseline through R4 verification with dated updates
- No generic language in gap descriptions; each gap is specific and measurable (not "improve," "enhance," "strengthen")

## Description — Practical Guidance
- **Pitfall:** Treating gap analysis as one-time activity. **Guidance:** Conduct gap analysis at R1 (baseline), then track remediation progress and repeat at R4 (verification) to close the feedback loop.
- **Pitfall:** Using vague assessment criteria (e.g., "adequate controls exist"). **Guidance:** Each checklist item must have explicit compliance status (Compliant/Partially/Non-compliant) with documented evidence; maturity scoring quantifies control quality.
- **Pitfall:** Identifying gaps without remediation roadmap or ownership. **Guidance:** Every gap must have assigned remediation owner, resource estimate, target date, and linkage to specific D01/D02 skill execution.
- **Pitfall:** Confusing IEC 62443 requirements with specific implementation choices. **Guidance:** Gap analysis identifies what requirements are not met, not how to meet them; implementation details fall to SK-D01-019 through SK-D01-023.
- **Pitfall:** Underestimating effort and organizational change required. **Guidance:** Include change management, training, and cultural factors in remediation roadmap; align with operational constraints (maintenance windows, production schedules).
- **Pitfall:** Focusing only on technical gaps, ignoring management/organizational gaps (C.1, C.7, C.8). **Guidance:** Annex C checklists address governance, asset management, supply chain, and process safety; ensure comprehensive assessment across all domains.
