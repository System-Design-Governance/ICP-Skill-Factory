# 操作與訓練教材 (Operational Training Materials)

整合 2 個 SK，涵蓋操作維護手冊撰寫與培訓教材開發。

---

## 0. 初始化

1. 系統設計已完成 (SDD 已產出)
2. 告警清單與響應程序已定義
3. 目標受眾已識別 (操作員 / 管理員 / 維護人員)
4. 訓練時程已規劃

---

## 1. 工作流程

### Step 1: 操作維護手冊撰寫 (SK-D09-002)

**O&M 手冊章節結構**：

| 章節 | 內容 | 受眾 |
|------|------|------|
| 1. 系統概述 | 架構、功能、範圍 | 全員 |
| 2. 啟動/關機 | 步驟式操作程序 | 操作員 |
| 3. 正常操作 | 日常操作流程、監控重點 | 操作員 |
| 4. 告警響應 | 每告警：原因、影響、處置 | 操作員 |
| 5. 故障排除 | 症狀→診斷→修復 決策樹 | 維護 |
| 6. 安全操作 | 帳號管理、備份、事件回報 | 管理員 |
| 7. 維護排程 | 定期檢查項目與週期 | 維護 |
| 8. 備件清單 | 關鍵備件與供應商 | 維護 |

**寫作原則**：
- 為操作員寫，不是為工程師寫
- 步驟式 (Step 1, Step 2...) 而非敘述式
- 每步驟搭配截圖或照片
- 告警響應：先做什麼 (action) 再說為什麼 (explanation)

**步驟**：
1. 從系統設計提取操作流程
2. 將告警清單轉為告警響應程序
3. 建立故障排除決策樹 (症狀 → 可能原因 → 診斷步驟 → 修復)
4. 撰寫安全操作程序 (帳號、備份、patch、事件回報)
5. 定義維護排程 (daily/weekly/monthly/yearly)
6. Peer review by 實際操作員 (語言是否可理解)

**⚠️ 避坑**：
- 手冊太技術化 → 操作員看不懂，放著沒用
- 告警響應只寫「聯繫工程師」→ 操作員無法自行處置任何問題
- 未含安全操作 → 操作員不知道 security incident 如何回報

### Step 2: 培訓教材開發 (SK-D09-009)

**培訓教材架構 (Kirkpatrick L1-L3)**：

| 層級 | 目標 | 評估方式 |
|------|------|----------|
| L1 Reaction | 學員滿意度 | 問卷 |
| L2 Learning | 知識/技能掌握 | 測驗、實作考核 |
| L3 Behavior | 工作行為改變 | 現場觀察、主管回饋 |

**教材類型對應**：

| 受眾 | 內容 | 格式 |
|------|------|------|
| 操作員 | 系統操作、告警處理 | 投影片 + 實作練習 |
| 管理員 | 帳號管理、備份、安全 | 投影片 + Lab 環境 |
| 維護人員 | 故障排除、硬體更換 | 影片 + SOP 卡 |

**步驟**：
1. 分析受眾：角色、現有知識、需學習的差距
2. 定義學習目標 (可測量的行為描述)
3. 設計課程大綱與時數
4. 開發教材：投影片、Lab 練習、評量題
5. 設計 L2 測驗 (知識) + L2 實作考核 (技能)
6. 試教 → 收集回饋 → 修訂

**⚠️ 避坑**：
- 教材 = O&M 手冊複製貼上 → 不是教學設計
- 只有理論沒有實作 → 學員無法實際操作
- 未設計評量 → 無法確認學員是否真正學會

## 2. 驗收標準

| # | 條件 |
|---|------|
| 1 | O&M 手冊涵蓋啟動/關機/正常操作/告警/故障排除/安全 |
| 2 | 告警響應程序覆蓋所有定義的告警 |
| 3 | 手冊語言適合操作員 (非工程師) |
| 4 | 培訓教材涵蓋所有目標受眾 |
| 5 | 每模組有明確學習目標 + L2 評量 |
| 6 | 實際操作員已 review 手冊可讀性 |
